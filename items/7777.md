## Summary

Current main already provides the right extension seam for this behavior through plugin `before_tool_call` hooks plus plugin approvals, and the repo vision explicitly pushes optional workflow automation to plugins/ClawHub instead of OpenClaw core.

## Close Comment

Codex review on `cec34821754ad935c780805f3116f99449d46ac2` suggests closing this as a ClawHub/plugin request rather than a core OpenClaw issue. Current main already has the extension surface needed for this: plugin `before_tool_call` hooks can inspect a tool call, rewrite params, block it, or require a user approval (`src/plugins/hook-types.ts:324`, `src/agents/pi-tools.before-tool-call.ts:187`). The repo vision also sets the scope boundary that optional capability should usually ship as plugins and new skills should go to ClawHub first (`VISION.md:54`, `VISION.md:69`, `VISION.md:101`; `docs/plugins/building-plugins.md:16`, `docs/plugins/building-plugins.md:166`). For a workflow like “search Haivemind/Linear/GitHub before create/send and then ask for confirmation,” the right home is a ClawHub plugin or hook pack, not additional OpenClaw core plumbing.
