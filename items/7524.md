## Summary

Keep open. Current main still hard-routes non-direct chats to channel/group session keys, exposes only `session.dmScope` for DM collapsing, and documents groups as isolated/non-main. The requested generic `groupScope: "main"` or equivalent per-group config is not implemented.

## Evidence

- **Group routing still has no main-session branch:** `buildAgentPeerSessionKey()` only applies `dmScope` when `peerKind === "direct"`; otherwise it always returns `agent:<agentId>:<channel>:<peerKind>:<peerId>`. There is no configurable path for `group` to collapse into main.
  - file: [src/routing/session-key.ts:141](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/routing/session-key.ts#L141)
  - command: `nl -ba src/routing/session-key.ts | sed -n '130,177p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Session schema has no groupScope setting:** The session schema defines `scope` and `dmScope`, but no `groupScope` or equivalent per-group-to-main routing option.
  - file: [src/config/zod-schema.session.ts:27](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/zod-schema.session.ts#L27)

## Risks / Open Questions

- BlueBubbles and iMessage now expose current-conversation binding support (`extensions/bluebubbles/src/channel.ts:101`, `extensions/imessage/src/channel.ts:140`), so maintainers may want to decide whether that manual/channel-specific workaround is sufficient for some cases; it is not the generic config feature requested here and does not cover Signal.
- There may be undocumented workflow-level workarounds (for example binding a conversation to an existing session target), but the core routing/config/docs still do not provide the requested per-group main-session option.
