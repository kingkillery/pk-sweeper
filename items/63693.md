## Summary

Keep open. Current `main` still drops `cfg` during memory-search provider resolution, so the reported false-positive `plugins.allow` warning path remains. The issue discussion is consistent with the code, and the referenced fix patches exist but are not present on `main`.

## Evidence

- **Current main still calls provider lookup without cfg in memory-search merge/resolve paths:** `mergeConfig()` calls `getMemoryEmbeddingProvider(provider)` and `getMemoryEmbeddingProvider(fallback)` without forwarding `cfg`, and `resolveMemorySearchConfig()` also calls `getMemoryEmbeddingProvider(resolved.provider)` without `cfg`. This matches the issue's root-cause description.
  - file: [src/agents/memory-search.ts:154](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/memory-search.ts#L154)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '130,430p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Provider runtime only uses config-aware plugin capability resolution when cfg is passed:** `getMemoryEmbeddingProvider(id, cfg?)` forwards `cfg` into `resolvePluginCapabilityProvider(...)`; if callers omit it, resolution falls back to a config-less path.
  - file: [src/plugins/memory-embedding-provider-runtime.ts:33](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/plugins/memory-embedding-provider-runtime.ts#L33)
  - command: `nl -ba src/plugins/memory-embedding-provider-runtime.ts | sed -n '17,48p'`

- **Status path still routes through memory config resolution:** The human-readable status memory snapshot path passes `resolveMemorySearchConfig` into shared status scanning, so `openclaw status` still exercises the affected code path.
  - file: [src/commands/status.scan-memory.ts:32](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/commands/status.scan-memory.ts#L32)
