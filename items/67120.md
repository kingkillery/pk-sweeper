## Summary

Keep open. Current `main` still routes inbound Feishu audio payloads through the generic `parseMessageContent()` path, which returns raw JSON for non-text/post/share_chat/merge_forward messages, and the Feishu event model/tests on `main` show no `speech_to_text` extraction path for audio messages.

## Evidence

- **Inbound Feishu events feed raw message content into the agent context:** `parseFeishuMessageEvent()` reads `event.message.content` via `parseMessageContent(...)`, and the finalized inbound context forwards `ctx.content` as both `RawBody` and `CommandBody`.
  - file: [extensions/feishu/src/bot.ts:105](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/feishu/src/bot.ts#L105)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '90,150p'; nl -ba extensions/feishu/src/bot.ts | sed -n '978,992p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Current parser returns raw JSON for audio messages:** `parseMessageContent()` special-cases `post`, `text`, `share_chat`, and `merge_forward`; all other message types fall through to `return content`, so an audio payload like `{"file_key":...,"duration":...}` remains raw JSON.
  - file: [extensions/feishu/src/bot-content.ts:125](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/feishu/src/bot-content.ts#L125)
