## Summary

Keep #32868 open. Current main still matches the reported root cause: same-channel block replies are enqueued fire-and-forget, while `text_end` only awaits the block pipeline flush, not the dispatcher delivery chain. The newer #71664 appears to duplicate and reinforce this issue rather than supersede it.

## Evidence

- **Same-channel block reply is not awaited:** On the generic same-channel path, `dispatch-from-config.ts` calls `dispatcher.sendBlockReply(normalizedPayload)` without awaiting delivery or dispatcher idle; only cross-channel routing awaits `sendPayloadAsync`.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:1102](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/auto-reply/reply/dispatch-from-config.ts#L1102)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.ts | sed -n '1060,1135p'`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Dispatcher API still returns boolean:** `ReplyDispatcher.sendBlockReply` is typed as returning `boolean`, not a delivery promise, so callers cannot await per-message delivery through this API.
  - file: [src/auto-reply/reply/reply-dispatcher.types.ts:7](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/auto-reply/reply/reply-dispatcher.types.ts#L7)
  - command: `nl -ba src/auto-reply/reply/reply-dispatcher.types.ts | sed -n '1,80p'`

- **Actual delivery happens later on sendChain:** `createReplyDispatcher` appends async delivery to `sendChain` and awaits `options.deliver(...)` inside that chain, but `enqueue` returns `true` immediately after queuing.
  - file: [src/auto-reply/reply/reply-dispatcher.ts:247](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/auto-reply/reply/reply-dispatcher.ts#L247)
