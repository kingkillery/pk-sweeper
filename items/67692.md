## Summary

Keep PR #67692 open. Current main still returns `null` when ACP session resolution is `kind: "none"`, so an agent configured with `runtime.type: "acp"` can still fall through to embedded reply dispatch unless ACP metadata was created by another path. The PR targets a real, not-yet-implemented product gap, though its current branch needs maintainer review for the bot-raised policy and side-effect concerns before merge.

## Best Possible Solution

Keep the PR open and request a smaller, policy-safe implementation: lazily initialize ACP in `tryDispatchAcpReply` for `runtime.type: "acp"` agents only after the existing policy and no-op turn gates, normalize `runtime.acp` fields consistently, re-resolve the session unconditionally after init, and add focused tests for ACP-configured versus non-ACP agents. Treat the CLI `agent-command.ts` lazy-init changes as separate review scope unless maintainers explicitly want CLI parity in this PR.
