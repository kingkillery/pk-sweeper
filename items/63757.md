## Summary

Keep open. Current `main` still handles Telegram reactions by enqueueing plaintext system events, and the documented/plugin hook surfaces still only expose generic message hooks rather than a stable structured reaction event with emoji + sender metadata.

## Evidence

- **Telegram reaction handler still emits plaintext system events:** The Telegram runtime converts added reactions into text like `Telegram reaction added: ...` and calls `enqueueSystemEvent(...)`; it does not emit a typed reaction payload or hook event.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:977](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/telegram/src/bot-handlers.runtime.ts#L977)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '900,1035p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **User-facing docs describe reaction notifications as system-event text:** The Telegram docs say reaction notifications arrive as `message_reaction` updates and that OpenClaw enqueues system events like `Telegram reaction added: 👍 by Alice (@alice) on msg 42`. That matches the issue's complaint rather than disproving it.
  - file: [docs/channels/telegram.md:621](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/channels/telegram.md#L621)

- **Hook catalogs still expose only generic message hooks:** Plugin hooks document `message_received` and `message_sent` under message/delivery, with no reaction-specific hook. Internal hooks likewise list only `message:received`, `message:transcribed`, `message:preprocessed`, and `message:sent`.
  - file: [docs/plugins/hooks.md:81](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/plugins/hooks.md#L81)
