## Close Comment

Closing from a Codex review because the Telegram replay-storm path appears implemented on current `main`. The current Telegram polling code now scopes persisted offsets to bot identity and rejects stale/invalid offset files (`extensions/telegram/src/update-offset-store.ts:7`), confirms the stored offset with Telegram before polling resumes (`extensions/telegram/src/monitor.ts:175`), and only persists a safe watermark below any still-pending/failed updates (`extensions/telegram/src/bot-core.ts:287`). There is matching regression coverage for the startup confirmation and offset-state cases in `extensions/telegram/src/monitor.test.ts:766`, `extensions/telegram/src/update-offset-store.test.ts:40`, and `extensions/telegram/src/bot.create-telegram-bot.test.ts:1095`. The changelog shows this shipped as `Telegram/polling offset safety` in `2026.3.7` (`CHANGELOG.md:3191`). If you can still reproduce Telegram message replays on a current release, please open a fresh issue with the current version, exact Telegram account layout, and fresh logs; the Feishu-topic and update-authorization notes here should be tracked separately if they are still present.
