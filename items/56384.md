## Summary

Keep open. Current `main` still lacks the Feishu group-ID prefix normalization described in PR #56384, and the existing docs/runtime path make the reported mismatch plausible rather than obsolete.

## Evidence

- **Current resolver is still raw-only:** `resolveFeishuGroupConfig` on `main` only does exact lookup, then raw case-insensitive lookup, then wildcard fallback. It does not normalize `chat:`/`group:`/provider-prefixed IDs before matching group config keys.
  - file: [extensions/feishu/src/policy.ts:128](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/feishu/src/policy.ts#L128)
  - command: `nl -ba extensions/feishu/src/policy.ts | sed -n '128,150p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Runtime passes Feishu API chat_id straight into lookup:** The bot context stores `event.message.chat_id` directly as `ctx.chatId`, and both `resolveFeishuGroupConfig` and `resolveFeishuReplyPolicy` receive that raw value. There is no normalization on this path before lookup.
  - file: [extensions/feishu/src/bot.ts:119](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/feishu/src/bot.ts#L119)
