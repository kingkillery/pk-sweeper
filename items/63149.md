## Summary

Keep open. Current `main` has lane-concurrency limits and heartbeat backoff for `requests-in-flight`, but the code/docs still describe subagents as sharing the same gateway process resources and there is no strong evidence that the reported single-core CPU saturation under concurrent heartbeat/subagent load has been resolved on `main`.

## Evidence

- **Gateway already applies configurable lane concurrency, but only as queue limits:** `applyGatewayLaneConcurrency` sets separate limits for cron, main, and subagent lanes; the defaults are `4` for main agent work and `8` for subagents.
  - file: [src/gateway/server-lanes.ts:6](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-lanes.ts#L6)
  - command: `nl -ba src/gateway/server-lanes.ts | sed -n '1,40p'; nl -ba src/config/agent-limits.ts | sed -n '1,40p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Heartbeat runner skips when the main or resolved session lane is already busy:** `runHeartbeatOnce` returns `requests-in-flight` when either the main lane or the target session lane has queued work, which avoids piling heartbeat work onto an already-busy lane but does not prove CPU saturation is fixed.
  - file: [src/infra/heartbeat-runner.ts:759](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/infra/heartbeat-runner.ts#L759)

  - file: [src/infra/heartbeat-runner.ts:1533](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/infra/heartbeat-runner.ts#L1533)
