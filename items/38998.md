## Summary

Keep open. Current main still has no DM pairing-request hook/event: the shared pairing flow only stores the request and sends the pairing reply, docs still direct owners to `openclaw pairing list`, and representative channel handlers log then return before agent/message hook dispatch.

## Evidence

- **Hook registry has no pairing-request hook:** `PluginHookName` and `PLUGIN_HOOK_NAMES` enumerate the supported hook surface. They include `inbound_claim` and `message_received`, but no `pairing_request` or `pairing:request` hook.
  - file: [src/plugins/hook-types.ts:56](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/plugins/hook-types.ts#L56)
  - command: `nl -ba src/plugins/hook-types.ts | sed -n '50,120p'`
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Shared DM pairing flow does not emit a proactive event:** `issuePairingChallenge` only upserts the pairing request, optionally calls `onCreated`, builds the reply, and sends it. There is no plugin hook, internal hook, or system-event emission in the shared pairing path.
  - file: [src/pairing/pairing-challenge.ts:24](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/pairing/pairing-challenge.ts#L24)

- **Docs still describe manual DM pairing approval flow:** The pairing docs say unknown senders are not processed until approval and direct the owner to `openclaw pairing list` / `openclaw pairing approve`, matching the issue's stated workaround rather than a proactive notification flow.
  - file: [docs/channels/pairing.md:20](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/docs/channels/pairing.md#L20)
