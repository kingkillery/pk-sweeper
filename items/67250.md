## Summary

Keep open. Current `main` still has the empty-bubble suppression path that drops assistant bubbles whenever there is no markdown and no visible tool cards, and that guard does not exempt active streaming turns. The linked discussion matches the current code path, and there is no regression coverage for `isStreaming: true` with hidden tool calls.

## Evidence

- **Current main still suppresses the bubble without checking streaming state:** `renderGroupedMessage` returns `nothing` when there is no markdown, no visible tool cards, no images, no attachments, no canvas blocks, and no reply target. The guard does not include `!opts.isStreaming`, so an in-flight assistant turn can still render as blank.
  - file: [ui/src/ui/chat/grouped-render.ts:1417](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/ui/src/ui/chat/grouped-render.ts#L1417)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '1340,1435p'`
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **Streaming state is passed into the last message renderer:** Grouped chat rendering sets `isStreaming: group.isStreaming && index === group.messages.length - 1` before calling `renderGroupedMessage`, so the empty-bubble guard above is on the active streaming path for the final assistant message in a live group.
  - file: [ui/src/ui/chat/grouped-render.ts:343](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/ui/src/ui/chat/grouped-render.ts#L343)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '330,360p'`
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **No regression coverage for streaming-hidden-tool-call case:** The grouped-render test helper hardcodes `isStreaming: false`, and a repo search in this test file found no `isStreaming: true` case. Existing tests cover many `showToolCalls: false` paths, but not the reported streaming blank-bubble scenario.
  - file: [ui/src/ui/chat/grouped-render.test.ts:70](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/ui/src/ui/chat/grouped-render.test.ts#L70)
  - command: `nl -ba ui/src/ui/chat/grouped-render.test.ts | sed -n '68,86p' && rg -n "isStreaming: true|isStreaming\\s*\\?" ui/src/ui/chat/grouped-render.test.ts`
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **Issue discussion matches the current guard failure mode:** The provided comments add a second reproduction: the bubble blanks during the tool-call-to-text transition even with tool calls visible, because `visibleToolCards` becomes false before markdown tokens arrive while streaming is still active. That is consistent with the current `!markdown && !visibleToolCards ... return nothing` branch.

## Risks / Open Questions

- The report may affect more than the hidden-tool-calls setting; the same guard can blank the assistant bubble during any streaming phase gap before text arrives.
- Without an `isStreaming` regression test, future UI refactors can continue to reintroduce this behavior silently.
