## Summary

Keep open. Current `main` has gateway restart deferral and durable restart-continuation delivery, but the Telegram auto-reply path still sends directly through `deliverReplies` without the generic outbound write-ahead queue, and `sendChatAction` start failures are only logged. That does not high-confidently close the reported hot-reload ghost-turn loss case.

## Evidence

- **Gateway restart defers while work is active:** Hot reload restart now waits for active operations, pending replies, embedded runs, and task runs before restarting, with a timeout fallback. This is related hardening, but it only proves restart deferral, not durable Telegram reply delivery.
  - file: [src/gateway/server-reload-handlers.ts:247](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/gateway/server-reload-handlers.ts#L247)
  - command: `nl -ba src/gateway/server-reload-handlers.ts | sed -n '220,320p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Durable queue is wired for restart continuations:** Restart sentinel code enqueues only continuation payloads into the session-delivery queue before clearing the sentinel, then drains that continuation queue after the restart notice. This is narrower than normal Telegram mid-turn reply delivery.
  - file: [src/gateway/server-restart-sentinel.ts:485](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/gateway/server-restart-sentinel.ts#L485)
  - command: `nl -ba src/gateway/server-restart-sentinel.ts | sed -n '460,570p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Generic outbound path has write-ahead queue:** `deliverOutboundPayloads` persists payloads to the delivery queue before sending and only acks or fails the queue entry after send completion. That durable path exists in core, but it is not what Telegram auto-replies are using here.
  - file: [src/infra/outbound/deliver.ts:757](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/infra/outbound/deliver.ts#L757)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '740,845p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Telegram auto-replies still use direct bot delivery:** Telegram message dispatch calls `deliverReplies` for final payload delivery, and `deliverReplies` iterates replies and sends via `sendTelegramText` / media send helpers directly. There is no call from this path into `deliverOutboundPayloads` or the delivery queue.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:651](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/extensions/telegram/src/bot-message-dispatch.ts#L651)

I did not find code here that converts `sendChatAction` failure into an explicit user-visible turn failure or retryable abort.
