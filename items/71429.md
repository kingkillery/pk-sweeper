## Summary

Keep open. Current main has generic restart/reload hardening and durable queues for other outbound/restart-continuation paths, but I do not see the specific Telegram normal auto-reply path journaled before Bot API delivery or converted into a visible hard turn failure when typing/send delivery fails during reload. The existing maintainer comment on #71429 reached the same boundary and explicitly left the item open.

## Best Possible Solution

Keep #71429 open and add a focused Telegram auto-reply/reload regression that simulates a live turn, channel reload, sendChatAction failure, and final Telegram delivery failure. Then route normal Telegram final replies through an existing durable outbound/session-delivery queue before Bot API delivery, or deliberately fail the turn with visible Telegram/UI status and retry guidance. The shipped restart-continuation queue and generic outbound queue are likely reusable primitives, but the Telegram same-surface auto-reply path needs explicit wiring and coverage.
