## Summary

Keep open. Current `main` still builds the `session-memory` filename date from `Date.toISOString()`, so the filename uses the UTC calendar day instead of the host-local day. The reported mismatch reproduces for a Los Angeles timezone boundary case.

## Evidence

- **UTC date drives filename:** `src/hooks/bundled/session-memory/handler.ts:84-85` sets `const now = new Date(event.timestamp);` and then `const dateStr = now.toISOString().split("T")[0]`, which is always the UTC date.
  - file: [src/hooks/bundled/session-memory/handler.ts:84](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/hooks/bundled/session-memory/handler.ts#L84)
  - command: `nl -ba src/hooks/bundled/session-memory/handler.ts | sed -n '70,210p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Fallback slug time is also UTC:** `src/hooks/bundled/session-memory/handler.ts:161-177` derives the fallback `HHMM` slug and displayed time from `now.toISOString()`, and the markdown header is explicitly labeled `UTC`. That matches the reported UTC-based naming behavior.
  - file: [src/hooks/bundled/session-memory/handler.ts:161](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/hooks/bundled/session-memory/handler.ts#L161)
  - command: `nl -ba src/hooks/bundled/session-memory/handler.ts | sed -n '70,210p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Boundary-case repro matches the report:** With `TZ=America/Los_Angeles`, evaluating `new Date('2026-03-15T01:17:00.000Z')` produced `isoDate=2026-03-15` while the local date formatted in that timezone was `2026-03-14`. That is the exact off-by-one-day failure mode described in the issue.
  - command: `TZ=America/Los_Angeles node -e "const d=new Date('2026-03-15T01:17:00.000Z'); console.log('isoDate='+d.toISOString().split('T')[0]); console.log('localDate='+new Intl.DateTimeFormat('en-CA',{timeZone:Intl.DateTimeFormat().resolvedOptions().timeZone,year:'numeric',month:'2-digit',day:'2-digit'}).format(d)); console.log('resolvedTz='+Intl.DateTimeFormat().resolvedOptions().timeZone);"`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **No timezone regression test exists here:** `src/hooks/bundled/session-memory/handler.test.ts` has coverage for command gating, transcript extraction, reset fallback, and workspace routing, but no assertion for local-vs-UTC filename dates. A direct search for timezone/date-specific coverage in this test file returned no matches.
  - file: [src/hooks/bundled/session-memory/handler.test.ts:188](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/hooks/bundled/session-memory/handler.test.ts#L188)
  - command: `rg -n "timezone|local date|UTC date|filename" src/hooks/bundled/session-memory/handler.test.ts -S`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)

## Risks / Open Questions

- Closing this would leave a real user-visible bug on current `main` for users outside UTC, especially around local midnight.
- Any fix needs an explicit timezone source decision for this hook surface and boundary tests so future refactors do not reintroduce UTC-based filenames.
