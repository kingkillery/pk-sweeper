## Summary

Current `main` does not implement the requested admin-session unification. `identityLinks` still only canonicalizes the peer id within the selected DM scope, so `dmScope: "per-peer"` routes linked Matrix users to `agent:<agentId>:direct:<canonical>` rather than the main session key `agent:<agentId>:main`. TUI/webchat still use the main-session key, so the reported split-session behavior remains on `main`.

## Evidence

- **DM routing still stays inside per-peer/per-channel scopes:** `buildAgentPeerSessionKey()` applies `identityLinks`, then returns `agent:<agentId>:direct:<peer>` for `per-peer`, `agent:<agentId>:<channel>:direct:<peer>` for `per-channel-peer`, and only falls back to `buildAgentMainSessionKey()` outside those branches. There is no special case for canonical `main`.
  - file: [src/routing/session-key.ts:143](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/routing/session-key.ts#L143)

- **Test coverage still encodes the old behavior:** `resolve-route.test.ts` still expects `identityLinks` under `dmScope: "per-peer"` to resolve to `agent:main:direct:alice`, not `agent:main:main`. That matches the issue's reported actual behavior rather than the requested behavior.
