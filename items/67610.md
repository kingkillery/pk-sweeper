## Summary

Current main has not implemented the reported cross-module sandbox backend registry fix. `registerSandboxBackend` still writes to a module-scoped `Map`, `requireSandboxBackendFactory` reads that same module-local registry at sandbox resolution time, and the plugin SDK exposes this path to plugins. Existing tests cover direct same-module registration, but not the reported plugin/Jiti duplicate-module path. The related PRs in the provided GitHub context are not merged on main, so #67610 remains actionable.

## Best Possible Solution

Keep #67610 open and fix it in core. The likely direction is to make the sandbox backend registry process-level/shared, for example with a `globalThis` + `Symbol.for("openclaw.sandboxBackendFactories")` state holder, or to route plugin backend registration through an explicit host-owned runtime API. Add a regression test that loads a plugin through the real plugin loader/Jiti path, calls `registerSandboxBackend("abc", ...)`, then verifies core `resolveSandboxContext`/`requireSandboxBackendFactory("abc")` sees it. After that lands, close the issue and supersede or update PR #67623 as appropriate.
