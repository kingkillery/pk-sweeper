## Summary

Discord degraded/exhausted presence handling is already implemented on current `main` and was shipped in the changelog as a released feature, so this issue is obsolete.

## Evidence

- **Auto-presence maps degraded and exhausted runtime state:** Discord auto-presence classifies unavailable auth/profile reasons such as `rate_limit`, `overloaded`, `billing`, `auth`, and `auth_permanent`, maps exhausted availability to Discord `dnd`, and maps degraded availability to `idle`.
  - file: [extensions/discord/src/monitor/auto-presence.ts:104](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/extensions/discord/src/monitor/auto-presence.ts#L104)
- **Discord startup applies and refreshes presence:** The Discord ready listener refreshes auto-presence when enabled, and monitor startup creates the auto-presence controller and starts it against the gateway `updatePresence` API.
  - file: [extensions/discord/src/monitor/provider.startup.ts:83](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/extensions/discord/src/monitor/provider.startup.ts#L83)
- **Config surface exists for the requested feature:** Discord config exposes `channels.discord.autoPresence` with `enabled`, `healthyText`, `degradedText`, and `exhaustedText` fields for runtime/quota-based presence signaling.
  - file: [src/config/types.discord.ts:211](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/src/config/types.discord.ts#L211)
- **Regression tests cover exhausted and recovery behavior:** Tests verify exhausted runtime signals become Discord `dnd`, and that presence returns to `online` once a profile becomes usable again.
  - file: [extensions/discord/src/monitor/auto-presence.test.ts:54](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/extensions/discord/src/monitor/auto-presence.test.ts#L54)
- **Docs describe the exact healthy/degraded/exhausted mapping:** Discord docs show `autoPresence` config and document the mapping `healthy => online`, `degraded or unknown => idle`, `exhausted or unavailable => dnd`.
  - file: [docs/channels/discord.md:856](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/docs/channels/discord.md#L856)
- **Changelog shows the feature shipped:** The `2026.3.7` changelog includes `Discord/auto presence health signal: add runtime availability-driven presence updates plus connected-state reporting`.
  - file: [CHANGELOG.md:3067](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/CHANGELOG.md#L3067)
- **Implementation is already in the latest tagged release snapshot:** Comparing latest release `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` to current main shows no code diff in the relevant Discord auto-presence implementation files, so the latest release already contains this behavior.

  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
