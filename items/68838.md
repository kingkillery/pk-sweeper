## Summary

Keep open. Current `main` still has the reported late-`finally` delete pattern in the followup drain, and the surrounding clear/re-enqueue code still permits replacing the map entry for the same key without any identity check. I did not find current test coverage proving that interleaving is fixed.

## Evidence

- **Drain finally still deletes by key only:** `scheduleFollowupDrain` ends by clearing `queue.draining` and, when the captured queue is empty, unconditionally calling `FOLLOWUP_QUEUES.delete(key)` and `clearFollowupDrainCallback(key)`. There is no `FOLLOWUP_QUEUES.get(key) === queue` guard in current `main`.
  - file: [src/auto-reply/reply/queue/drain.ts:275](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/queue/drain.ts#L275)
  - command: `nl -ba src/auto-reply/reply/queue/drain.ts | sed -n '220,320p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Clear path removes the shared map entry before a new queue can be created:** `clearSessionQueues` still calls `clearFollowupQueue(cleaned)` and then `clearFollowupDrainCallback(cleaned)`. `clearFollowupQueue` mutates the captured queue in place and deletes the map entry with `FOLLOWUP_QUEUES.delete(cleaned)`, which matches the issue's prerequisite for replacing the queue object under the same key.
  - file: [src/auto-reply/reply/queue/cleanup.ts:67](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/queue/cleanup.ts#L67)
  - command: `nl -ba src/auto-reply/reply/queue/cleanup.ts | sed -n '52,72p' && nl -ba src/auto-reply/reply/queue/state.ts | sed -n '74,87p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Enqueue path still recreates and restarts an idle queue:** `enqueueFollowupRun` still calls `getFollowupQueue(key, settings)`, which creates and stores a fresh queue object when the map entry was deleted, then remembers the callback and restarts draining when `!queue.draining`. That means a post-clear enqueue can install a new queue object while an older drain still holds the previous queue reference.
  - file: [src/auto-reply/reply/queue/enqueue.ts:60](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/queue/enqueue.ts#L60)

- **Current tests do not cover the reported interleaving:** The nearby restart tests cover callback clearing after `clearSessionQueues`, but the asserted behavior stops at 'after-clear' not being processed. I did not find a regression test that clears a queue while an earlier drain is awaiting and then re-enqueues a new item on the same key before the old drain reaches `finally`.
