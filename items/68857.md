## Summary

Keep open. Current `main` does not implement `gateway.nodes.invokeTimeoutMs`, and the PR as described is incomplete: the runtime config validator and config metadata still exclude the new key, while `node.invoke` still hardcodes the 30s fallback and only forwards per-call `timeoutMs` to the node payload.

## Evidence

- **Current main still hardcodes the default timeout in NodeRegistry:** `NodeRegistry.invoke()` sends `timeoutMs: params.timeoutMs` to the node and falls back locally to a hardcoded `30_000` when no per-call timeout is provided, so there is no configurable gateway default on `main`.
  - file: [src/gateway/node-registry.ts:126](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/node-registry.ts#L126)
  - command: `nl -ba src/gateway/node-registry.ts | sed -n '30,180p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Gateway config type on main has no invokeTimeoutMs field:** `GatewayNodesConfig` currently exposes only `browser`, `allowCommands`, and `denyCommands`; there is no `invokeTimeoutMs` field in the checked-out `main` tree.
  - file: [src/config/types.gateway.ts:366](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/config/types.gateway.ts#L366)

## Risks / Open Questions

- Closing this PR would discard a still-relevant feature request that is not present on `main` or in the latest release.
- Merging the PR as-is would leave the advertised config unusable at runtime because the strict config schema does not yet accept `gateway.nodes.invokeTimeoutMs`.
- Even after schema wiring, the implementation should resolve one effective timeout and pass it both to the gateway timer and the `node.invoke.request` payload to avoid gateway/node timeout drift.
