## Summary

Current `main` already contains the existing-session Chrome MCP attach/readiness changes that address this timeout pattern: OpenClaw now waits for tabs to become usable after attach instead of treating the initial handshake as ready, and it resets poisoned Chrome MCP sessions on transport timeout/error so later calls can reconnect cleanly.

## Evidence

- **Shipped changelog entry:** `2026.3.23` changelog explicitly says existing-session Chrome MCP now waits for tabs to become usable after attach, reducing user-profile timeouts and repeated consent churn.
  - file: [CHANGELOG.md:2159](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/CHANGELOG.md#L2159)
- **Attach readiness gate in current code:** `ensureBrowserAvailable()` for Chrome MCP profiles calls `ensureChromeMcpAvailable()` and then polls `listChromeMcpTabs()` until tabs are usable before returning success.
  - file: [extensions/browser/src/browser/server-context.availability.ts:189](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/browser/src/browser/server-context.availability.ts#L189)
- **Session reset on poisoned/hung Chrome MCP calls:** Chrome MCP tool calls tear down the cached session on transport error or timeout so the next call reconnects instead of inheriting a bad session.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:355](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/browser/src/browser/chrome-mcp.ts#L355)
- **Regression coverage:** Tests cover both improved attach failure surfacing and recovery after failed attach / timed-out Chrome MCP calls.
  - file: [extensions/browser/src/browser/server-context.existing-session.test.ts:139](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/browser/src/browser/server-context.existing-session.test.ts#L139)
- **Also present in latest release:** `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:extensions/browser/src/browser/server-context.availability.ts` shows the same attach-readiness logic in `v2026.4.22`, so this is not main-only.
