## Summary

Keep open. Current `main` has partial CLI fast paths for `config get`/`config unset`, but `config file` and `openclaw config --help` still go through eager `config-cli` registration/loading, so the reported slowdown is not clearly fixed on `main` or in `v2026.4.23`.

## Evidence

- **Root-help fast path is root-only:** `shouldUseRootHelpFastPath()` returns true only for root help, and the test explicitly asserts subcommand help is not on that fast path. That means `openclaw config --help` still falls through to normal program construction.
  - file: [src/cli/run-main.ts:70](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/cli/run-main.ts#L70)
  - command: `nl -ba src/cli/run-main.ts | sed -n '55,85p' && nl -ba src/cli/run-main.test.ts | sed -n '85,92p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Command help still eagerly registers `config`:** After the route-first check, `run-main` registers the primary command for help/parsing, and the core registry resolves `config` by importing `../config-cli.js`. That keeps `openclaw config --help` on the heavy path.
  - file: [src/cli/run-main.ts:244](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/cli/run-main.ts#L244)
