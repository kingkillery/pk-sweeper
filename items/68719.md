## Summary

Keep open. Current main handles BlueBubbles voice notes through the generic media-understanding path, but it does not implement the BlueBubbles-specific `/api/v1/message/audio-transcript/:guid` enricher, the proposed `channels.bluebubbles.inboundAudioEnricher.*` config surface, or the scoped transcribed-hook payload requested in #68719.

## Risks / Open Questions

- Closing as implemented would overstate current behavior: main can transcribe inbound audio generically, but not through the BlueBubbles no-ASR-cost endpoint requested here.
- Closing as clawhub would also be premature: the current public hook surface documents inbound observation, not pre-dispatch inbound body rewriting for channel messages.
