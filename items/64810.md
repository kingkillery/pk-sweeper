## Summary

Keep open. On current `main` (`fd65caf4`), `src/infra/heartbeat-runner.ts` still skips heartbeats only when the main lane or resolved session lane has queued/active command-queue work; it does not check the reply-run registry for the race window described in the issue. The issue discussion already points to a narrow fix in PRs `#64823`/`#64963`, and that guard plus its regression test are still absent from current `main`.

## Evidence

- **Heartbeat runner still only gates on queue depth:** `runHeartbeatOnce` returns `requests-in-flight` for `CommandLane.Main` and the resolved session lane, but there is no `resolveActiveReplyRunSessionId(sessionKey)` check after preflight. That leaves the exact window described in the issue: queue depth can be 0 while a reply run is still active.
  - file: [src/infra/heartbeat-runner.ts:759](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/infra/heartbeat-runner.ts#L759)

- **Reply-run primitive exists on main but is not wired into heartbeat:** Current `main` exports `resolveActiveReplyRunSessionId(sessionKey)` from the reply-run registry, so the missing behavior is not blocked on missing plumbing; it is specifically absent from heartbeat-runner integration.
