## Summary

Keep open. Current main has important pieces: cache-TTL pruning for old tool results, documented config knobs, a Control UI context-usage notice, compaction notices, and the optional tokenjuice plugin for noisy exec/bash output. But the central request is broader visibility and control over pruning itself: preview/audit of what was removed, user-visible pruning notifications, and richer pruning strategy controls. Those are not implemented or superseded by the related items.

## Best Possible Solution

Keep this issue open and narrow it into a maintainer-owned product/API task: add a structured pruning result from the context-pruning path, surface brief Control UI/channel notifications when pruning actually changes context, expose an inspectable pruning log or session event, and decide whether richer retention policies belong in core config or a plugin/context-engine extension. Preserve the existing cache-TTL/tool-result defaults and tokenjuice as complementary mitigations.

## Evidence
