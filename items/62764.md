## Summary

Keep open. Current `main` still treats Linux gateway supervision as a user-systemd concern only: doctor/configure install paths write and enable `~/.config/systemd/user/openclaw-gateway.service`, load detection only checks that user unit, and the extra-service scan ignores canonical `openclaw-gateway*.service` units even when scanning system directories. The docs also explicitly describe system units as a supported deployment pattern, so the reported dual-unit conflict remains plausible and in scope.

## Evidence

- **Systemd unit path is hard-coded to the user unit directory:** `resolveSystemdUnitPathForName()` builds Linux unit paths under `~/.config/systemd/user/<name>.service`; there is no parallel `/etc/systemd/system` ownership probe in this code path.
  - file: [src/daemon/systemd.ts:47](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/daemon/systemd.ts#L47)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '40,80p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Install flow writes and activates only the user systemd unit:** `writeSystemdUnit()` writes the resolved user unit path, and `activateSystemdService()` runs `systemctl --user daemon-reload`, `enable`, and `restart` for `openclaw-gateway.service`. That matches the issue's claim that doctor/configure can regenerate and enable a user unit.
  - file: [src/daemon/systemd.ts:441](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/daemon/systemd.ts#L441)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '440,585p'`

- **Doctor decides 'not installed' from the user-unit probe, then offers installation:** `maybeRepairGatewayDaemon()` calls `service.isLoaded()`, and when that returns false it notes 'Gateway service not installed.' and offers 'Install gateway service now?' before calling `service.install()`. There is no check for an existing system-owned unit first.
  - file: [src/commands/doctor-gateway-daemon-flow.ts:101](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/commands/doctor-gateway-daemon-flow.ts#L101)

- **Load detection only recognizes the user unit:** `isSystemdServiceEnabled()` first checks for the user-unit file on disk, then runs `systemctl --user is-enabled <unit>`. A system unit at `/etc/systemd/system/openclaw-gateway.service` would not satisfy this probe.
  - file: [src/daemon/systemd.ts:647](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/daemon/systemd.ts#L647)

That means doctor would not flag a canonical system unit as an extra conflicting service.
