## Summary

Keep open. Current `main` still routes mirror-mode file writes through `syncLocalPathToRemote`, and that method still calls `openshell sandbox upload` with the original file path instead of staging a flat temp payload first. That matches the reported tar path-collision root cause. The only linked fix evidence is a fork commit referenced on the issue timeline, and it is not present on `origin/main` or the latest release ref inspected here.

## Evidence

- **Current main still uploads the raw file path:** `syncLocalPathToRemote` creates the destination parent, then invokes `openshell sandbox upload` with `localPath` as the upload source and `dirname(remotePath)` as the destination. That is the exact shape the report says causes tar to preserve the subdirectory component and collide with an existing remote directory.

- **Mirror-mode write tool still hits that path:** The OpenShell filesystem bridge writes the file locally, then immediately calls `backend.syncLocalPathToRemote(hostPath, target.containerPath)`. So a normal mirror-mode `write` still flows through the code above.
