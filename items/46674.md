## Summary

Keep open. Current `main` has safer Telegram update-watermark handling, but I did not find strong evidence that #46674's specific cross-restart dedupe gap is fully fixed. The polling runner still creates a fresh bot on each restart, the duplicate-update cache is still process-local, and I did not find a restart-boundary regression test or a merged fix PR that proves the replay-after-restart case is closed.

## Evidence

- **Dedupe cache is still in-memory and created per bot instance:** `createTelegramUpdateDedupe()` is instantiated inside the bot factory, alongside per-process pending/handled maps. That means dedupe state is recreated with each new bot instance rather than persisted across polling restarts.

## Risks / Open Questions

- The current watermark fixes may have reduced or changed the failure mode without eliminating the specific cross-restart replay window from #46674.
- Closing as implemented would overstate the evidence: there is still no persisted cross-restart dedupe state or explicit restart-boundary regression coverage in current main.
- Issue #58443 may be a broader duplicate-injection family rather than the same Telegram-only root cause, so it is supporting but not conclusive evidence.
