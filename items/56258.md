## Summary

Keep open. Current `main` still has no per-subsystem timeout around the early gateway shutdown steps, so the underlying reliability gap is not implemented on `main`. At the same time, the PR should not be conservatively closed as 'implemented' or 'cannot reproduce': `main` still awaits several subsystem stops serially before HTTP/WebSocket teardown, while the PR also carries unresolved correctness/regression concerns around dangling rejections and premature HTTP close completion.

## Evidence

- **Current main still blocks on early subsystem shutdown calls without per-subsystem caps:** `createGatewayCloseHandler` still awaits `bonjourStop`, `tailscaleCleanup`, `canvasHost.close`, `canvasHostServer.close`, each channel `stopChannel`, `disposeRegisteredAgentHarnesses`, `pluginServices.stop`, and `stopGmailWatcher` before moving on. There is no `raceTimeout` helper or per-subsystem timeout in this path on current `main`. That means one hung stop can still consume the shutdown budget before later teardown runs.
  - file: [src/gateway/server-close.ts:113](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-close.ts#L113)

This matches the PR motivation: without per-subsystem limits, one blocking stop can burn the whole budget.
