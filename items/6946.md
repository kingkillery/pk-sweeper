## Summary

Keep open. Current `main` does not implement the requested Telegram processing-indicator message flow. The codebase supports reaction-based acknowledgement/status feedback on the inbound message, but not sending a separate temporary ⌛ message, deleting it on reply, or editing it on failure.

## Evidence

- **Global message config only exposes reaction-based processing feedback:** `MessagesConfig` defines `ackReaction`, `ackReactionScope`, `removeAckAfterReply`, and `statusReactions`; there is no `processingIndicator` or equivalent message-sending config.
  - file: [src/config/types.messages.ts:113](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/types.messages.ts#L113)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Telegram channel config has no per-channel processing-indicator setting:** Telegram channel config includes `ackReaction` but no per-channel `processingIndicator`, `deleteOnReply`, or failure-edit setting.
  - file: [src/config/types.telegram.ts:211](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/types.telegram.ts#L211)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Inbound Telegram handling sets reactions on the trigger message, not a separate placeholder message:** The Telegram message-context path resolves `ackReaction` / `statusReactions` and calls `reactionApi(chatId, msg.message_id, ...)` against the inbound message. There is no corresponding send of a temporary processing message here.
  - file: [extensions/telegram/src/bot-message-context.ts:452](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/telegram/src/bot-message-context.ts#L452)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Reply cleanup removes reactions, not a sent indicator message:** After reply delivery, cleanup calls `removeAckReactionAfterReply` with a remover that clears `reactionApi(chatId, msg.message_id, [])`. This is reaction cleanup, not deletion of a separately sent processing message.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:1254](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/telegram/src/bot-message-dispatch.ts#L1254)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **User-facing docs describe Telegram ack/status reactions as the current processing signal:** Telegram docs say `ackReaction` sends an acknowledgement emoji while processing, and the config reference documents `messages.statusReactions.enabled` for lifecycle reactions. No processing-indicator message feature is documented.
  - file: [docs/channels/telegram.md:644](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/channels/telegram.md#L644)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Repo-wide symbol search found no processing-indicator config or implementation names:** A targeted search for `processingIndicator`, `deleteOnReply`, and `showErrorOnFailure` under `extensions/telegram`, `src/config`, and `docs` returned no matches.
