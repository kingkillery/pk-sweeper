## Summary

Keep open. Current `main` still contains the exact reported TUI pairing-required guidance without the concrete `openclaw devices approve <requestId>` command, and the TUI test only asserts that `devices list` appears. The approval command itself is supported and documented, so this remains a narrow core UX copy/test fix rather than a duplicate, ClawHub item, or already-shipped implementation.

## Best Possible Solution

Keep the issue open and make a small core TUI UX fix: update `src/tui/tui.ts` so the pairing-required hint explicitly tells users to run `openclaw devices approve <requestId>` after `openclaw devices list`, preferably using an exact request ID when the disconnect reason carries one and a placeholder otherwise. Add or tighten `src/tui/tui.test.ts` to assert the approval command appears.
