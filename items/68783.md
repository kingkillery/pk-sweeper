## Summary

Keep open. The underlying scheduled-run bug may be real, but this PR is not safe to close as-is: on current `main`, manual cron runs intentionally reload persisted delivery changes after execution, and this patch would overwrite that contract. The same overwrite risk exists on the timer/startup paths after their force-reload step.

## Evidence

- **Manual runs already isolate the execution copy:** `prepareManualRun` deep-clones the job before execution, so the PR's stated manual-run root cause ('passed by reference through the isolated agent execution pipeline') does not match current `main` for `cron.run`/force runs.

  - command: `nl -ba src/cron/service/ops.ts | sed -n '560,720p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Current main explicitly preserves delivery writebacks after manual runs:** `finishPreparedManualRun` force-reloads the store before the final persist, with an inline comment explaining this is required so isolated Telegram target writeback survives manual `cron run`. The PR's snapshot-restore logic would replace that reloaded `delivery` object and discard the persisted writeback.
