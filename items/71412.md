## Summary

Keep open. Current `main` still leaves timed-out channel stops marked `running: true` and returns before clearing `store.tasks`/`store.aborts`, while `startChannelInternal` still silently no-ops when a task entry exists. A current test in `src/gateway/server-channels.test.ts` explicitly asserts that restart remains blocked after a stop timeout, so the reported zombie-task behavior is still present rather than obsolete or already fixed.

## Evidence

- **Timeout path still returns before clearing task state:** `stopChannel` aborts, waits up to `CHANNEL_STOP_ABORT_TIMEOUT_MS`, and on timeout logs a warning, sets `running: true` plus a timeout `lastError`, then `return`s before `store.aborts.delete(id)` / `store.tasks.delete(id)` run.
