## Summary

Current main still matches the reported failure mode. The stop-timeout path marks the runtime as running and returns before clearing the task/abort maps, `startChannelInternal` still no-ops when a task entry exists, and the existing unit test currently asserts that a second start does not occur after a stop timeout. The related PR #71456 is a plausible fix but is open/unmerged in the provided context, so this issue should stay open until that or an equivalent patch lands.

## Best Possible Solution

Keep #71412 open and review or land PR #71456, or an equivalent patch, that releases the stale `store.aborts` and `store.tasks` entries on stop timeout, marks the runtime stopped instead of running, and updates the regression test to assert that a subsequent start registers a fresh task. Treat the `openclaw doctor` liveness improvement as a separate follow-up unless maintainers want it bundled.
