## Summary

Keep open. Current `main` does not implement npm `dev` publishing: the public release workflow only accepts `beta` or `latest`, the publish-plan code only resolves stable/beta publishes, and the current updater treats `dev` as a git-checkout flow rather than an npm dist-tag flow.

## Evidence

- **Release workflow has no dev dist-tag input:** The manual npm release workflow exposes `npm_dist_tag` only for stable releases, with choices limited to `beta` and `latest`.
  - file: [.github/workflows/openclaw-npm-release.yml:19](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/.github/workflows/openclaw-npm-release.yml#L19)
  - command: `nl -ba .github/workflows/openclaw-npm-release.yml | sed -n '1,80p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Publish plan only supports stable/beta:** `resolveNpmPublishPlan` accepts only `latest` or `beta` as publish tags and returns only `stable` or `beta` channels; there is no `dev` publish path.
  - file: [scripts/openclaw-npm-release-check.ts:162](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/scripts/openclaw-npm-release-check.ts#L162)

## Risks / Open Questions

- A private release workflow could publish a `dev` dist-tag outside this checkout, but there is no evidence of that path in public `main`.
- The docs still mention `dev` dist-tag publication as optional, so the issue's underlying request remains plausible rather than obsolete.
