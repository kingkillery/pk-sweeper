## Summary

Keep open. Current `main` at `51dd4f288f67ef4df4c7a121f078f508535df1fd` still lacks the MIME-type fallback and send-path preservation this PR adds, so the reported web UI bug is not implemented on main and the PR remains materially useful.

## Evidence

- **Current drop/file-select logic still rejects empty MIME types:** `handleFileSelect` and `handleDrop` still gate attachments with `isSupportedChatAttachmentMimeType(file.type)`. If the browser supplies an empty `File.type`, the attachment is skipped instead of falling back by filename extension.
  - file: [ui/src/ui/views/chat.ts:251](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/ui/src/ui/views/chat.ts#L251)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '200,320p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Main has no fallback resolver helper:** `attachment-support.ts` on current `main` only exports `CHAT_ATTACHMENT_ACCEPT` and `isSupportedChatAttachmentMimeType`; there is no `resolveSupportedChatAttachmentMimeType` or equivalent extension-based fallback implementation.
  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/ui/src/ui/chat/attachment-support.ts#L1)
