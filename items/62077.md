## Summary

Current `main` still intentionally injects full conversation metadata for non-webchat direct chats, including Telegram DMs; tests lock that behavior in, and there is no existing agent config or prompt-hook seam that provides the requested minimal-DM metadata mode.

## Evidence

- **Direct non-webchat chats still include conversation info:** `buildInboundUserContextPrefix()` treats direct chats as metadata-bearing whenever the resolved inbound channel is present and not `webchat`, so Telegram DMs still flow through the full conversation-info block with `chat_id`, `message_id`, `sender_id`, and related fields.
  - file: [src/auto-reply/reply/inbound-meta.ts:185](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/auto-reply/reply/inbound-meta.ts#L185)
  - command: `nl -ba src/auto-reply/reply/inbound-meta.ts | sed -n '175,245p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Tests explicitly preserve this direct-channel behavior:** Current tests expect direct `webchat` chats to hide identifiers, but expect direct external-channel chats to include message identifiers and sender metadata. That means the behavior reported in the issue is still the intended, asserted behavior on `main`.
