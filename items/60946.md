## Summary

Keep open. Current `main` still reports most tool failures as plain `{status:"error", error:"..."}` results; only a narrow edit-tool recovery now adds current file contents for exact-text mismatches, so the broader cross-tool rich-error feature in #60946 is not implemented yet.

## Evidence

- **Generic tool failures still collapse to a raw error string:** `toToolDefinitions` catches tool exceptions and returns `jsonResult({ status: "error", tool, error: described.message })` without structured causes, suggestions, or remediation metadata.
  - file: [src/agents/pi-tool-definition-adapter.ts:137](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/pi-tool-definition-adapter.ts#L137)
  - command: `nl -ba src/agents/pi-tool-definition-adapter.ts | sed -n '120,280p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Only the edit tool has a richer recovery path:** `wrapEditToolWithRecovery` is specialized for edit failures: it can append `Current file contents:` for exact-text mismatches and convert some post-write throws back to success, but it does not add the broader multi-match, similar-text, or exec-command suggestions requested in the issue.
  - file: [src/agents/pi-tools.host-edit.ts:135](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/pi-tools.host-edit.ts#L135)

- **Test coverage confirms the richer message is limited to one edit mismatch case:** The new test only asserts that an exact-match edit failure gains `Current file contents:`; there is no corresponding coverage for multiple-match edit errors or `exec` command-not-found suggestions from the issue body.
