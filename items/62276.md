## Summary

Keep open. Current main has better diagnostics for port/supervisor collisions, but the actual install/onboard flow still only checks the canonical managed service and does not scan for or block extra gateway-like services before they contend for port 18789, which matches both the original report and the follow-up Linux reproduction.

## Evidence

- **Daemon install only checks the canonical service:** `runDaemonInstall` resolves the platform gateway service, calls `service.isLoaded(...)`, and returns `already-installed` only for that one service. In this path there is no extra-service scan and no preflight that checks for a different gateway-like service already bound to the configured port.
  - file: [src/cli/daemon-cli/install.ts:89](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/daemon-cli/install.ts#L89)
  - command: `nl -ba src/cli/daemon-cli/install.ts | sed -n '70,135p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Collision handling is still reactive, not preventative:** `openclaw gateway run` still handles the conflict after startup fails: it prints `Gateway failed to start`, inspects port usage, and then tells the user to stop the supervised service. That is improved UX, but it does not prevent the duplicate-service/port-collision condition described in the issue.
  - file: [src/cli/gateway-cli/run.ts:689](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/cli/gateway-cli/run.ts#L689)
  - command: `nl -ba src/cli/gateway-cli/run.ts | sed -n '669,706p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Extra-service detection exists only in diagnostics flows:** The codebase already has a dedicated scan for `Other gateway-like services detected` plus cleanup guidance and a recommendation to run a single gateway per machine, but that logic lives in doctor/status-style diagnostics rather than the install path.
  - file: [src/commands/doctor-gateway-services.ts:412](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/commands/doctor-gateway-services.ts#L412)
  - command: `nl -ba src/commands/doctor-gateway-services.ts | sed -n '412,470p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **User-facing docs still frame this as a status/deep-scan troubleshooting case:** The CLI docs say `gateway status --deep` performs a best-effort scan for extra launchd/systemd/schtasks installs and warns when multiple gateway-like services are present. That documents the symptom, but it is not evidence of an install-time fix.
  - file: [docs/cli/gateway.md:195](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/docs/cli/gateway.md#L195)

- **Windows docs still describe managed startup plus manual-run split:** The Windows guide still says `openclaw onboard --non-interactive --install-daemon` / `openclaw gateway install` create managed startup (Scheduled Task or Startup-folder fallback), while the CLI-only path is `openclaw gateway run`. That means the reported 'service installed, then manual run hits port 18789' shape is still a plausible current-user trap.
  - file: [docs/platforms/windows.md:40](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/docs/platforms/windows.md#L40)

- **Independent corroborating reproduction in issue discussion:** The April 14, 2026 maintainer comment reports the same 18789 collision on Linux when a system-level service and the user-level `openclaw-gateway.service` both start, and the workaround is disabling the user service. That strengthens the case that this is not an obsolete one-off report.
