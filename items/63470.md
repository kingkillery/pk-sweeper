## Summary

Keep open. Current `main` still appears to reject `openclaw plugins enable ...` for `memory-lancedb` without embedding config: the enable command only flips `enabled`, config validation still runs for enabled plugins, and the bundled `memory-lancedb` manifest still requires `config.embedding` with `apiKey`. The issue comments document a manual config workaround, not a fix shipped on `main`.

## Evidence

- **Enable command does not seed required plugin config:** `plugins enable` reads config, calls `enablePluginInConfig`, applies slot selection, and immediately persists the result. There is no bootstrap/default config for plugin-required fields before write.
  - file: [src/cli/plugins-cli.ts:487](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cli/plugins-cli.ts#L487)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Config validation still blocks enabled plugins with invalid config:** Validation runs for plugin entries when the plugin is enabled or has config. If schema validation fails, it emits `invalid config` issues for `plugins.entries.<id>.config...`, which matches the reported failure mode during config write.
  - file: [src/config/validation.ts:1152](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/config/validation.ts#L1152)

- **Bundled memory-lancedb schema still requires embedding:** The shipped manifest still marks top-level `embedding` as required, and inside it requires `apiKey`. Enabling the plugin with only `{ enabled: true }` remains schema-invalid.
  - file: [extensions/memory-lancedb/openclaw.plugin.json:57](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-lancedb/openclaw.plugin.json#L57)

  - file: [docs/tools/slash-commands.md:349](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/tools/slash-commands.md#L349)

That resolves the local config, but it does not show a code change on `main` that fixes the CLI enable path.
