## Summary

Keep open. Current `main` still tears down the WhatsApp socket without draining pending debounced inbound messages, so this is not implemented on `main`, and the PR discussion also identified unresolved shutdown-race gaps in the proposed `flushAll()` approach.

## Evidence

- **Current WhatsApp close path still closes immediately:** `attachWebInboxToSocket().close()` detaches the listeners and then calls `closeInboundMonitorSocket(sock)` right away. There is no debounce-drain step between detach and socket close on current `main`.

  - file: [src/auto-reply/inbound-debounce.ts:235](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/auto-reply/inbound-debounce.ts#L235)
