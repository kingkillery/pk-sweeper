## Summary

Keep open. Current `main` still rejects `media://inbound/...` image references in the image tool, while adjacent agent runtime code already supports the same gateway claim-check URI form. The PR remains a plausible, scoped bug fix rather than something already implemented on `main`.

## Evidence

- **Current image tool still rejects custom schemes:** `createImageTool` treats any scheme other than `file:`, `http(s):`, or `data:` as unsupported and returns `unsupported_image_reference`, so `media://inbound/<id>` would still be rejected on current `main`.
  - file: [src/agents/tools/image-tool.ts:417](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tools/image-tool.ts#L417)

- **Native prompt image path already supports gateway media URIs:** The embedded runner detects `[media attached: media://inbound/<id>]` refs and resolves them through `resolveMediaBufferPath` plus `loadWebMedia`, which matches the behavior this PR is trying to bring to the image tool.
  - file: [src/agents/pi-embedded-runner/run/images.ts:272](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/pi-embedded-runner/run/images.ts#L272)
