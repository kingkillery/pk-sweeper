## Summary

Keep open. Current `main` still does not expose a configurable Undici TCP connect timeout for model providers, and the assistant timeout path still has multiple code branches (`fallback_model` vs `surface_error`/timeout synthesis) rather than a single documented timeout-fallback rule.

## Evidence

- **No model-provider config field for connect timeout:** `ModelProviderConfig` includes `baseUrl`, `apiKey`, `auth`, `api`, `injectNumCtxForOpenAICompat`, `headers`, `authHeader`, `request`, and `models`; there is no `connectTimeoutMs`-style field to carry an Undici TCP connect timeout from `openclaw.json`.
  - file: [src/config/types.models.ts:115](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/config/types.models.ts#L115)
  - command: `nl -ba src/config/types.models.ts | sed -n '60,130p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Runtime schema also lacks a connect-timeout knob:** `ModelProviderSchema` validates the same provider fields and has no connect-timeout property, so current `main` would reject or ignore the requested setting rather than plumb it through configuration.
  - file: [src/config/zod-schema.core.ts:347](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/config/zod-schema.core.ts#L347)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '344,364p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Global Undici dispatcher still does not set `connect.timeout`:** `ensureGlobalUndiciStreamTimeouts` applies `bodyTimeout` and `headersTimeout`; the `connect` block only carries `autoSelectFamily` and `autoSelectFamilyAttemptTimeout`. There is still no `connect: { timeout: ... }` assignment on the global `Agent`/`EnvHttpProxyAgent`.
  - file: [src/infra/net/undici-global-dispatcher.ts:136](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/infra/net/undici-global-dispatcher.ts#L136)
  - command: `nl -ba src/infra/net/undici-global-dispatcher.ts | sed -n '118,158p'`

- **Timeout handling still has distinct fallback vs surface-error branches:** Assistant failover still branches by runner state: after rotation exhaustion it returns `fallback_model` when `fallbackConfigured` is true, but otherwise returns `surface_error`; separate assistant-failover logic intentionally leaves plain timeout `surface_error` cases on the runner's timeout-payload path. That means timeout behavior is still conditional rather than a single, documented always-fallback rule.
  - file: [src/agents/pi-embedded-runner/run/failover-policy.ts:155](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/agents/pi-embedded-runner/run/failover-policy.ts#L155)

- **Current tests explicitly preserve non-fallback timeout surface behavior:** The assistant-failover tests assert that plain timeouts on the `surface_error` branch remain `continue_normal` so `run.ts` can synthesize a timeout payload, which matches the conditional timeout behavior the issue calls out rather than proving a unified fallback policy.
