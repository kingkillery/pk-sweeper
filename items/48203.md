## Summary

Current main already routes macOS Talk playback through the resolved gateway Talk config, preserves the configured ElevenLabs `voiceId`, and only falls back to system voice on explicit missing-credential or TTS/playback failure paths. The same relevant code is already present in release `v2026.4.22`.

## Close Comment

Closing from a Codex review because current main already implements the requested behavior well enough, and the same relevant code is already in `v2026.4.22`. On macOS, `TalkModeRuntime.reloadConfig()` loads Talk settings from the gateway `talk.config` payload and stores `cfg.voiceId` into runtime state (`apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:904`). Playback then prefers the configured voice (`resolvedVoice ?? currentVoiceId ?? defaultVoiceId`) and accepts raw ElevenLabs voice IDs directly in `resolveVoiceId()` / `isLikelyVoiceId()` instead of silently replacing them (`apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:583`, `apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:778`). The only remaining system-voice path is the explicit fallback when the key/voice is missing or ElevenLabs synthesis/playback actually fails (`apps/macos/Sources/OpenClaw/TalkModeRuntime.swift:470`). The config parser also now reads the canonical resolved Talk provider payload and rejects malformed normalized payloads (`apps/macos/Sources/OpenClaw/TalkModeGatewayConfig.swift:28`, `apps/shared/OpenClawKit/Sources/OpenClawKit/TalkConfigParsing.swift:20`), with regression coverage in the shared and macOS Talk config tests. I also checked the release boundary: these files are unchanged between current main and release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this implementation is already shipped.
