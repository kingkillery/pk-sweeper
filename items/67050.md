## Summary

Current `main` does not implement this change, and the PR as proposed would create a trivial install-scan bypass by letting runtime-capable files evade all scanner rules based only on test-like names. The existing install flow still blocks `critical` findings, so the reported false-positive path remains real, but this specific PR is not a conservative cleanup close candidate.

## Evidence

## Risks / Open Questions

- Leaving the PR open preserves an unresolved false-positive report for plugins that ship tests, but closing it would discard a still-plausible problem without a shipped safe fix.
- Merging the PR as written would weaken the install-time security boundary by making scanner coverage depend on attacker-controlled filenames.
