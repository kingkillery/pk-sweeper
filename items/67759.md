## Summary

Keep #67759 open. Current main still has the reported gateway decision point: WebChat asks `resolveGatewayModelSupportsImages` before parsing attachments, and that helper returns `false` when the selected model is absent from the gateway catalog unless it matches the existing Foundry or Claude CLI shims. I did not find the explicit `openclaw.json` provider-model fallback proposed in #67766, nor the Qwen vision-id fallback proposed in #67768. Current attachment handling is better than the original report because text-only runs offload images as media refs instead of dropping them, but native image delivery still depends on model metadata advertising `input: ["text", "image"]`, so the catalog-miss bug remains actionable.

## Best Possible Solution

Keep #67759 open and converge the fix in core gateway/session model capability resolution. The stronger direction is the explicit provider-config fallback from #67766, because it respects user-authored `models.providers.<id>.models[].input` without hardcoding Qwen model-name heuristics; add regression coverage for a catalog miss where the runtime config explicitly marks `bailian/qwen3.6-plus` or an equivalent DashScope-compatible provider as image-capable. If maintainers want Qwen Coding Plan to remain unsupported for `qwen/qwen3.6-plus`, WebChat should still surface a clear model/capability warning instead of silently treating an explicitly configured vision model as text-only.
