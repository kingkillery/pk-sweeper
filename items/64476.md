## Summary

Keep open. Current `main` contains targeted fixes for adjacent symptoms (`openclaw status` teardown hangs, long-running `agent.wait`, Windows Scheduled Task fallback behavior), but I did not find source-backed evidence that the reported intermittent Windows loopback RPC failures for `sessions_history`/ACP spawn are fixed end-to-end on `main`.

## Evidence

- **`gateway status` is still only a connectivity/auth probe by default:** The docs explicitly say default `openclaw gateway status` proves service state, WebSocket connect, and handshake-visible auth capability, but does not prove read/write/admin RPC operations; `--require-rpc` is the stronger read probe. That explains why `gateway status` can look healthy while `sessions_history` still fails, but it is diagnostic clarification, not proof the underlying partial-RPC bug is fixed.
  - file: [docs/cli/gateway.md:189](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/docs/cli/gateway.md#L189)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Current main fixes one reported symptom: CLI processes hanging after gateway work finishes:** `GatewayClient.stopAndWait()` now waits for socket teardown, and gateway call sites use it before exit. This matches the changelog entry for `openclaw status`/`openclaw version` hanging until an external timeout kills them, so part of the report is addressed.
  - file: [src/gateway/client.ts:352](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/gateway/client.ts#L352)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Gateway client no longer tick-times out while a request is still pending:** The client tick watchdog returns early when `pending.size > 0`, and `expectFinal` requests default to no request timeout. That aligns with the changelog fix for long-running RPCs such as ACP `agent.wait`, but it does not demonstrate a fix for intermittent `sessions_history` or spawn timeouts.
  - file: [src/gateway/client.ts:877](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/gateway/client.ts#L877)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Windows service handling improved, but only around install/restart/fallback paths:** Current Windows docs and service code show Scheduled Task creation now falls back to a Startup-folder launcher when denied or when `schtasks` wedges, and `schtasks` timeout/no-output is explicitly treated as a fallback condition. That helps service-management failure modes, but it is narrower than the reported partial-RPC behavior with a live listener on `127.0.0.1:18789`.
  - file: [src/daemon/schtasks.ts:38](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/daemon/schtasks.ts#L38)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Recent related report is still open:** The provided issue timeline shows a later cross-reference to issue `#67251` (`Windows: CLI subcommands hang with SIGKILL in exec environment (2026.4.14)`), and that referenced issue is still open. That weakens any claim that the Windows gateway/CLI timeout cluster is fully resolved on current `main`.
- **Shipped release notes only cover adjacent symptoms, not the full report:** The current changelog includes fixes for one-shot gateway RPC teardown hangs and for long-running `agent.wait` websocket handling, but I did not find a release-note entry or current-code change that clearly ties to intermittent Windows `sessions_history`/ACP spawn failures on a live local gateway.
  - file: [CHANGELOG.md:298](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/CHANGELOG.md#L298)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
