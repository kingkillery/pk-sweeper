## Summary

Keep open. Current `main` still drops CLI session reuse when `auth-profile` or same-version `auth-epoch` changes, so the reported context-loss bug is not implemented on `main` and is not obsolete. The supplied PR discussion also surfaced and addressed the config-priority edge case, which further supports the PR as still-useful work rather than cleanup-close material.

## Evidence

- **Current main still resets on auth-profile/auth-epoch changes:** `resolveCliSessionReuse()` returns only `invalidatedReason` for `auth-profile` and same-version `auth-epoch` mismatches, with no `sessionId`, so reuse is discarded on those paths.

- **Current tests still expect auth changes to invalidate instead of resume:** `cli-session.test.ts` asserts `{ invalidatedReason: "auth-profile" }` and `{ invalidatedReason: "auth-epoch" }` rather than preserving the existing session id, matching the buggy current-main behavior.
  - file: [src/agents/cli-session.test.ts:67](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/agents/cli-session.test.ts#L67)
