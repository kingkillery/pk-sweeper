## Summary

Keep open. Current `main` fixes the `help` half of the report by wiring `help` as a core Commander command, but the `tool` half is still misclassified: unknown primary commands still fall through to a generic `plugins.allow` diagnostic, and there is no registered root `tool` CLI on `main` to justify that message.

## Evidence

- **`help` is built into the root CLI:** The root program unconditionally registers `help [command]` via Commander, so `openclaw help` is no longer plugin-backed on current `main`.
  - file: [src/cli/program/help.ts:72](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/cli/program/help.ts#L72)
  - command: `nl -ba src/cli/program/help.ts | sed -n '40,80p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Missing primary commands still route into plugin-miss diagnostics:** After lazy plugin CLI registration, `run-main` checks whether the primary command exists; if not, it calls `resolveMissingPluginCommandMessage(primary, config)` and throws that error. This is the path that still handles `tool`.
  - file: [src/cli/run-main.ts:284](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/cli/run-main.ts#L284)
  - command: `nl -ba src/cli/run-main.ts | sed -n '272,292p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Fallback message still blames `plugins.allow` for arbitrary unknown ids:** When no parent plugin alias is found, `resolveMissingPluginCommandMessage` returns a `plugins.allow excludes "<id>"` message for any missing primary command if an allowlist is present. That logic would still misclassify `tool` as a bundled plugin surface.
  - file: [src/cli/run-main.ts:123](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/cli/run-main.ts#L123)
  - command: `nl -ba src/cli/run-main.ts | sed -n '70,135p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **There is no root `tool` CLI registered on current `main`:** The current core root-command catalog lists `setup`, `onboard`, `configure`, `config`, `backup`, `doctor`, `dashboard`, `reset`, `uninstall`, `message`, `mcp`, `agent`, `agents`, `status`, `health`, `sessions`, and `tasks`; `tool` is not present.
  - file: [src/cli/program/core-command-descriptors.ts:6](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/cli/program/core-command-descriptors.ts#L6)
  - command: `nl -ba src/cli/program/core-command-descriptors.ts | sed -n '1,110p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Current docs present tool usage as chat `/tool`, not `openclaw tool`:** The image-generation docs show `/tool image_generate ...` examples, which supports the narrower bug framing here: if `openclaw tool` is unsupported, the CLI should report that clearly rather than claiming `plugins.allow` excluded a synthetic plugin id.
  - file: [docs/tools/image-generation.md:78](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/docs/tools/image-generation.md#L78)
