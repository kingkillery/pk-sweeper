## Summary

Keep open. Current `main` does not implement this Feishu duplication fix, and the proposed one-line patch is not strong enough to close conservatively because it can also suppress legitimate repeated block deltas.

## Evidence

- **Current main still appends Feishu block payloads as raw deltas:** `queueStreamingUpdate()` only dedupes when `dedupeWithLastPartial` is set, and the active block-streaming path still calls `queueStreamingUpdate(text, { mode: "delta" })` with no dedupe. `onPartialReply` separately uses snapshot mode with `dedupeWithLastPartial: true`, so exact partial+block duplicates are still possible on current `main` and in `v2026.4.22`.

  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)

That shows `main` still treats block payloads as append-only deltas rather than reconciling them with the last snapshot.

- **The proposed patch has a plausible regression path:** `dedupeWithLastPartial` compares `nextText === lastPartial` and then overwrites `lastPartial = nextText`. Reusing that flag on the block delta path would not only skip block payloads that match the last snapshot; it would also skip identical adjacent block deltas after the first one, which are valid for a delta stream.

## Risks / Open Questions

- Current `main` likely still duplicates exact snapshot-equal Feishu block payloads when runtimes emit both `onPartialReply` and `block` events.
- Closing this PR would discard a still-relevant bug report even though the current patch needs revision rather than cleanup closure.
- A correct fix likely needs block-vs-snapshot-specific reconciliation plus a regression test, not just toggling `dedupeWithLastPartial` on the delta path.
