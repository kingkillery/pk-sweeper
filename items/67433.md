## Summary

Keep open. Current main still implements POST /hooks/agent as a fire-and-forget isolated agent trigger returning only { ok, runId }; it does not parse a blocking flag, await run completion, return text, or expose blocking error semantics. The PR is a focused implementation attempt with active review/follow-up, and related issue #49019 remains open for the same result-return capability.

## Best Possible Solution

Keep the PR open as the concrete implementation path. The best maintainer path is to review the latest PR head against current main, require green gateway-hook tests and changed-lane checks, and land a core gateway implementation only if the blocking response contract, failure semantics, idempotency behavior, docs, and tests are all correct. If maintainers choose a different API shape, keep #49019 or a narrower follow-up as the canonical product/design tracker.
