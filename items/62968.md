## Summary

Keep open. Current `main` still matches the reported bug: the direct-send failure is captured in `phases`, but when queue fallback succeeds the top-level result becomes delivered via queue/steered, and the caller only logs direct failures when the overall delivery failed. The issue timeline's RemoteClaw cross-references reinforce that this is a real downstream-observed path, not an incoherent report.

## Evidence

- **Dispatch returns successful fallback result:** In completion mode, `runSubagentAnnounceDispatch` appends the failed `direct-primary` phase, then returns the successful `queue-fallback` result when fallback delivery works. That drops the direct error from the top-level result except inside `phases`. See `src/agents/subagent-announce-dispatch.ts:88` through `:102`.
  - file: [src/agents/subagent-announce-dispatch.ts:88](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/agents/subagent-announce-dispatch.ts#L88)

  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Direct send catches error without logging:** `sendSubagentAnnounceDirectly` converts thrown errors into `{ delivered: false, path: "direct", error }` and returns them; it does not emit a log at the catch site. See `src/agents/subagent-announce-delivery.ts:615`.
  - file: [src/agents/subagent-announce-delivery.ts:615](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/agents/subagent-announce-delivery.ts#L615)

If fallback queue delivery succeeds, that condition is false because the returned result is delivered via `queued`/`steered`. See `src/agents/subagent-announce.ts:565` through `:569`.
