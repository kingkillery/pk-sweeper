## Summary

Current main already has Telegram polling stall recovery and network-error retry logic that addresses the reported symptom, and the browser proxy path is asynchronous with abort-based timeouts rather than a synchronous event-loop block.

## Close Comment

Closing after Codex review because current `main` already has the recovery logic for this class of Telegram stall. `extensions/telegram/src/polling-session.ts:313` adds a polling watchdog that marks the Telegram transport dirty, stops the runner/bot, and restarts the polling cycle after stall detection; `extensions/telegram/src/polling-session.ts:370` also retries recoverable polling network errors and rebuilds transport state. The behavior is covered in `extensions/telegram/src/polling-session.test.ts:324` and `extensions/telegram/src/polling-session.test.ts:545`. The reported root cause also does not match current code: browser proxy actions are forwarded as async `node.invoke` calls in `extensions/browser/src/browser-tool.ts:276`, and the timeout path is abort-based in `src/node-host/with-timeout.ts:1`, not a synchronous event-loop block. This is also reflected in the shipped `v2026.4.22` release notes at `CHANGELOG.md:400` and `CHANGELOG.md:401`.
