## Summary

Current `main` still keeps a session marked active until the full reply run unwinds, and block-reply delivery does not release that active state. New inbound messages on the default `collect` queue mode are therefore still queued behind the in-flight run rather than processed immediately.

## Evidence

- **Session lane holds the whole embedded run:** `runEmbeddedPiAgent` wraps the full agent execution in `enqueueSession(... enqueueGlobal(...))`, so the lane is occupied until that async task returns.
  - file: [src/agents/pi-embedded-runner/run.ts:249](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/agents/pi-embedded-runner/run.ts#L249)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '235,430p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Lane occupancy ends only when queued task resolves:** The command queue removes an active task from the lane only after `await entry.task()` finishes or throws, so partial/block output does not free the lane by itself.
  - file: [src/process/command-queue.ts:167](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/process/command-queue.ts#L167)
  - command: `nl -ba src/process/command-queue.ts | sed -n '1,260p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Block reply delivery does not clear reply operation:** `createBlockReplyDeliveryHandler` sends or enqueues block payloads, but it only tracks `directlySentBlockKeys`; it does not complete the reply operation or change run activity.
  - file: [src/auto-reply/reply/reply-delivery.ts:150](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/reply-delivery.ts#L150)
  - command: `nl -ba src/auto-reply/reply/reply-delivery.ts | sed -n '1,260p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Active session state persists until final cleanup:** `runReplyAgent` calls `replyOperation.complete()` only in its `finally` block after `runAgentTurnWithFallback(...)`, and `createReplyOperation.complete()/fail()` is what clears active run state.
  - file: [src/auto-reply/reply/agent-runner.ts:1777](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/agent-runner.ts#L1777)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '1180,1805p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **New inbound messages are queued while run is active by default:** When a run is active, `resolveActiveRunQueueAction` returns `enqueue-followup` for followup/collect modes, and the default queue mode is still `collect` for all channels, including Telegram.
  - file: [src/auto-reply/reply/queue-policy.ts:11](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/queue-policy.ts#L11)
