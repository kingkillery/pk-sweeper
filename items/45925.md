## Summary

Keep open. The underlying cron-session heartbeat bug still looks plausible on current `main`, but this PR drops cron `sessionKey` wake targeting without substituting `agentId`, which changes the wake from targeted to unscoped and can hit the default agent or all enabled heartbeat agents instead of the originating cron agent. The author's note that the red extensions CI job was unrelated appears credible, but the routing regression in the patch itself is still unresolved.

## Evidence

- **Current wake helper carries only `sessionKey` scope:** On current `main`, `scopedHeartbeatWakeOptions(...)` only preserves `sessionKey`, and exec-completion callers rely on that helper while passing no separate `agentId`.
  - file: [src/routing/session-key.ts:35](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/routing/session-key.ts#L35)
  - command: `nl -ba src/routing/session-key.ts | sed -n '35,40p'; nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '343,345p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Unscoped wakes are not single-agent:** The heartbeat runner only takes the targeted path when `requestedSessionKey || requestedAgentId`; otherwise it iterates enabled heartbeat agents. When no explicit per-agent heartbeat list exists, `resolveHeartbeatAgents` falls back to the default agent.
  - file: [src/infra/heartbeat-runner.ts:1482](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/infra/heartbeat-runner.ts#L1482)

  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Wake routing semantics depend on target fields:** Current tests explicitly verify that `agentId` and `sessionKey` are forwarded and preserved across retries/coalescing, so dropping the cron `sessionKey` is a real routing change, not just a cosmetic normalization.
  - file: [src/infra/heartbeat-wake.test.ts:262](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/infra/heartbeat-wake.test.ts#L262)

- **PR patch removes cron wake scope without replacement:** The PR diff in the provided GitHub context changes `scopedHeartbeatWakeOptions(...)` to return bare `wakeOptions` for cron session keys. Because the known exec wake callers still only pass `{ reason, coalesceMs }`, that drops the only targeting data before the runner sees the wake. The PR review thread already called this out as a P1 regression.

## Risks / Open Questions

- Current `main` still appears vulnerable to cron-session heartbeat misrouting because cron session keys are preserved end-to-end today.
- Merging this PR as-is risks a different regression: cron exec completions can trigger heartbeat for the default agent or every enabled heartbeat agent instead of only the originating cron agent.
