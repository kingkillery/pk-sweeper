## Summary

Keep open. Current main still builds Telegram model buttons from the provider bucket plus the bare model id, so a configured OpenRouter ref like `openrouter/openai/gpt-5.4-mini` resolves into provider `openrouter` and model `openai/gpt-5.4-mini`, then displays `openai/gpt-5.4-mini` unless a catalog display name overrides it. The cross-referenced #67797 is a separate reasoning-default issue and does not supersede this Telegram label bug.

## Best Possible Solution

Keep #67792 open. The likely fix belongs in the Telegram plugin's model keyboard path: display the fully qualified `provider/model` label for nested provider model ids such as OpenRouter, while preserving existing `modelNames` behavior where a real catalog display name is intentionally provided. Add a focused regression test in `extensions/telegram/src/model-buttons.test.ts` for `provider: "openrouter"` and `model: "openai/gpt-5.4-mini"`.
