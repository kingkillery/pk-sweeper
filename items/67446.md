## Summary

Keep open. Current main still does not implement MCP content-level error detection: `isToolResultError` only trusts `details.status`, MCP materialization only sets that status for `isError: true`, and `handleToolExecutionEnd` uses that classifier to decide whether `after_tool_call.error` is populated. The PR is a focused, recent contributor bugfix with active discussion, but the latest branch also changes `isError: false` to `status: "ok"`, which conflicts with the PR title/body claim and needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to decide the intended contract first: either trust explicit MCP `isError: false` and reword/narrow the PR to omitted-`isError` servers, or support known non-conformant `isError: false + Error:` servers with a stricter signal. Then land a focused implementation with unit coverage for `isToolResultError` and an integration-style assertion that `handleToolExecutionEnd` populates `after_tool_call.error` for the accepted failure shape without suppressing successful MCP outputs.
