## Summary

Keep open. Current `main` has TTS backend/config support, but the Control UI still does not ship the requested first-class Settings UX for TTS enable/disable, default provider selection, and default voice selection.

## Evidence

- **TTS is still documented as config/command driven:** The user docs say auto-TTS is off by default and should be enabled with `messages.tts.auto` or `/tts on`, which matches the issue’s complaint that configuration is still config/command centered rather than surfaced in Control UI Settings.
  - file: [docs/tools/tts.md:70](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/docs/tools/tts.md#L70)
  - command: `nl -ba docs/tools/tts.md | sed -n '1,120p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Default Settings UI has no TTS card:** `renderQuickSettings` is the default Settings experience and renders cards for Model, Security, Channels, Automations, API Keys, Appearance, Personal, and Profile only; there is no TTS settings card or toggle in that primary UI.
  - file: [ui/src/ui/views/config-quick.ts:579](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/ui/src/ui/views/config-quick.ts#L579)
  - command: `nl -ba ui/src/ui/views/config-quick.ts | sed -n '520,620p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Advanced mode falls back to the generic schema form:** When leaving Quick Settings, the Control UI switches to the full accordion config form rather than a dedicated TTS settings surface, so discoverability still depends on the generic config editor.
  - file: [ui/src/ui/app-render.ts:1055](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/ui/src/ui/app-render.ts#L1055)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '1038,1075p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Schema exposes raw TTS fields, not a voice-picker UX:** The config schema defines `messages.tts.provider` as a plain string and `messages.tts.providers` as freeform provider-specific objects. That supports raw config editing, but it does not implement the issue’s requested visible provider dropdown plus filtered default voice picker.
  - file: [src/config/schema.base.generated.ts:18787](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/config/schema.base.generated.ts#L18787)
  - command: `nl -ba src/config/schema.base.generated.ts | sed -n '18785,18990p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Backend primitives exist, but the UI request is still unsatisfied:** The gateway already exposes `tts.status`, `tts.enable`, `tts.disable`, `tts.convert`, `tts.setProvider`, and `tts.providers`; `tts.providers` even returns provider `models` and `voices`. This means the underlying capability exists in core, but current `main` still lacks the requested first-class web Settings controls that would use it.
  - file: [src/gateway/server-methods/tts.ts:160](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/gateway/server-methods/tts.ts#L160)

  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
