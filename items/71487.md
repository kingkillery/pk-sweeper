## Summary

Keep open. Current main has TTS backend, CLI, gateway RPCs, docs, and a per-message browser-native "Read aloud" button in webchat, but it does not have a first-class Control UI Settings surface that reflects/persists the TTS enabled state, default provider, or default voice. The provided related items do not supersede this request.

## Best Possible Solution

Keep the issue open and implement a first-class Text-to-Speech panel in Control UI Settings. It should use existing tts.status/providers/enable/disable/setProvider and tts.convert where possible, show configured vs unavailable providers, filter voices by selected provider, add or reuse a safe persistence path for default voice, and cover the UI/RPC behavior with focused tests.

## Risks / Open Questions

- The generic config editor may already expose some messages.tts fields, so a dedicated UI should coordinate clearly with openclaw.json defaults and local prefs instead of creating conflicting sources of truth.
- Current gateway APIs can support enable/disable, provider selection, provider/voice listing, and sample playback, but persistent default voice selection likely needs either a narrow new prefs/API surface or a deliberate provider-specific config write path.
