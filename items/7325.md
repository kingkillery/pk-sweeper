## Summary

Current main already treats media-understanding vendors as plugin-owned capability providers, and project scope explicitly says optional capability should usually ship as plugins rather than core. Soniox is absent from main and this request's proposed core-file edits do not match the current architecture.

## Evidence

- **Vision scope:** Project vision says optional capability should usually ship as plugins, the preferred path is npm/local extension loading, and the bar for adding optional plugins to core is intentionally high.

## Close Comment

Closing after Codex review. Current main routes media-understanding vendors through the plugin system rather than new core wiring in `src/media-understanding/*`: `src/media-understanding/provider-registry.ts:30` loads `mediaUnderstandingProviders` from plugin capability providers, `src/media-understanding/manifest-metadata.ts:6` reads provider metadata from plugin manifests, and the public SDK explicitly supports `api.registerMediaUnderstandingProvider(...)` in `docs/plugins/sdk-provider-plugins.md:577` and `src/plugin-sdk/media-understanding.ts:1`. `VISION.md:54` also says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high. Existing audio transcription support like Deepgram already follows that plugin shape in `extensions/deepgram/index.ts:5` and `extensions/deepgram/openclaw.plugin.json:7`. Based on that current architecture and scope, Soniox is a better fit as a ClawHub/community plugin than a core OpenClaw issue. If a Soniox plugin later needs a missing core seam, please open a focused follow-up issue for that seam.
