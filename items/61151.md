## Summary

Keep open. Current `main` still retains OpenAI `partialJson` on stored tool-call blocks, and `repairToolCallInputs()` does not strip it before replay/sanitization, so this PR addresses a real unfixed bug rather than stale work already implemented on `main`.

## Evidence

- **OpenAI stream assembly still stores `partialJson` on tool calls:** `openai-transport-stream` initializes streamed `toolCall` blocks with `partialJson`, and on `response.output_item.done` it parses arguments from `currentBlock.partialJson` but does not remove the field from the block already stored in `output.content`. That matches the PR's premise that completed OpenAI tool calls can retain `partialJson` in session history.
  - file: [src/agents/openai-transport-stream.ts:438](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/openai-transport-stream.ts#L438)

  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Replay repair on `main` has no `partialJson` strip path:** `repairToolCallInputs()` currently drops only tool-call blocks missing input/id/name and otherwise passes valid raw tool-call blocks through to sanitization/normalization. There is no `workBlock`-style branch that strips `partialJson` from otherwise-complete blocks before replay.
  - file: [src/agents/session-transcript-repair.ts:310](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/session-transcript-repair.ts#L310)
