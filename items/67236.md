## Summary

Keep open. The current `main` QQ Bot gateway still lacks inbound message-id deduplication, so the report remains plausible and there is no strong evidence that the duplicate-reply bug is already fixed or obsolete.

## Evidence

- **Gateway dispatch enqueues every QQBot message event:** `GatewayConnection` forwards every `DISPATCH` message result into `trySlashCommandOrEnqueue(result.msg)` with no cache or duplicate check before processing.
  - file: [extensions/qqbot/src/engine/gateway/gateway-connection.ts:243](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/qqbot/src/engine/gateway/gateway-connection.ts#L243)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/gateway-connection.ts | sed -n '210,265p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Dispatcher preserves messageId but does not dedupe on it:** `dispatchEvent()` maps QQ inbound events into `QueuedMessage` objects carrying `messageId: ev.id`, but immediately returns them as actionable messages; there is no idempotency filter at this layer.
  - file: [extensions/qqbot/src/engine/gateway/event-dispatcher.ts:53](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/qqbot/src/engine/gateway/event-dispatcher.ts#L53)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/event-dispatcher.ts | sed -n '35,150p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Per-user queue has no message-id suppression:** `createMessageQueue()` keys only by peer (`guild`, `group`, or `dm`) and unconditionally `push()`es each queued message. The queue limits drop oldest items on overflow, but they do not suppress repeated deliveries of the same `messageId`.
  - file: [extensions/qqbot/src/engine/gateway/message-queue.ts:104](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/qqbot/src/engine/gateway/message-queue.ts#L104)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/message-queue.ts | sed -n '20,185p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Outbound path can send multiple passive replies for one inbound message:** QQ Bot replies use `target.messageId` as the passive reply `msgId`, and the shared outbound layer allows up to four replies per inbound message via `checkMessageReplyLimit()` / `recordMessageReply()`. That limits counts, but it does not prevent 2-3 duplicate replies if the same inbound event is processed repeatedly.
  - file: [extensions/qqbot/src/engine/messaging/outbound.ts:809](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/qqbot/src/engine/messaging/outbound.ts#L809)

  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)

That matches the code inspection: there is no visible documented safeguard already shipped on `main`.
