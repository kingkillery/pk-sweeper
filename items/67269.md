## Summary

Keep #67269 open. Current main does not implement Feishu custom event handlers: the Feishu monitor registers only fixed built-in event types, the Feishu config schema is strict and has no `customEventHandlers` field, and the plugin manifest/docs do not advertise such a setting. The PR addresses a concrete integration need called out in the discussion, sharing the existing Feishu WebSocket/EventDispatcher for events such as `drive.file.bitable_record_changed_v1`; that is not currently expressible through the public plugin API alone, so this is not a conservative ClawHub close. The patch still needs maintainer review and likely schema/manifest/docs plus remaining review concerns before merge, but it is useful product work rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open for Feishu maintainer review. The best path is either to land a tightened Feishu-owned custom event seam, including config schema, manifest/docs, path normalization, fire-and-forget behavior, and regression tests, or to design a public Feishu/channel runtime hook API that lets third-party plugins subscribe to raw vendor events through the existing Feishu dispatcher without opening a second WebSocket.

## Risks / Open Questions

- Closing as implemented would be incorrect because main still lacks the requested Feishu custom event handler behavior and config field.
- Closing as ClawHub would likely strand the use case: current public plugin hooks can observe OpenClaw message/agent/tool lifecycle, but they do not let a third-party plugin subscribe to arbitrary raw Feishu events on the bundled Feishu WebSocket.
- The PR patch should not be merged as-is without maintainer review because the current patch does not update Feishu schema/manifest/docs and prior review comments flagged event-path and fire-and-forget concerns.
