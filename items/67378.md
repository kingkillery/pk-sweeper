## Summary

Keep #67378 open. Current main still normalizes proactive WhatsApp phone-number sends to `@s.whatsapp.net` and does not use the existing Baileys PN->LID mapping store or auth-dir `lid-mapping-{pn}.json` files before outbound delivery. The code does support inbound LID->PN resolution, but that is the reverse direction and is not wired into `createWebSendApi` outbound target selection.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin: add a focused outbound target resolver for direct-phone sends that can use the active socket's Baileys `lidMapping.getLIDForPN` and/or auth-dir `lid-mapping-{digits}.json` before falling back to `toWhatsappJid`, wire it through `createWebSendApi` and proactive send/poll/presence paths, and add regression tests for mapped PN->LID delivery plus fallback behavior. Preserve existing explicit JID/group handling and the current inbound LID->PN logic.
