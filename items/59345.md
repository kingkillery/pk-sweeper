## Summary

Keep open. The report is recent, author is not protected, and no protected labels are present, but I do not have high-confidence evidence that current main fixed the specific freshness concern. Current source still exposes totalTokensFresh as optional in the Control UI session row shape, and the shared session helper behavior visible on main treats only explicit false as stale rather than requiring explicit true, which is close to the issue's suspected failure mode.

## Evidence

- **Issue discussion provides concrete affected version and repro signal:** Issue #59345 reports OpenClaw app 2026.4.1 stable showing `100% context used 468.2k / 48k`, with the numerator matching historical `message.usage.totalTokens` and denominator matching the context window. The reporter also notes a local bundled-asset mitigation requiring `totalTokensFresh === true`.
- **Control UI session type still allows unknown freshness:** GatewaySessionRow includes `totalTokens?: number`, `totalTokensFresh?: boolean`, and `contextTokens?: number`, so the UI can receive a token total without an explicit freshness confirmation.
  - file: [ui/src/ui/types.ts:2524](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/ui/src/ui/types.ts#L2524)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **Shared freshness helper only rejects explicit false:** The session helper visible on main resolves fresh total tokens by returning undefined only when `entry?.totalTokensFresh === false`; undefined freshness is not rejected. That does not prove the Control UI badge is still wrong, but it prevents a conservative implemented-on-main close for this exact complaint.
  - file: [src/config/sessions/types.ts:463](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/config/sessions/types.ts#L463)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
