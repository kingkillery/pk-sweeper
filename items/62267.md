## Summary

Keep open. Current main has generic mitigations for slow model startup and long prompts, but it does not clearly fix this Ollama-specific report: native Ollama requests still use the model context window as `num_ctx`, timeout recovery is generic, and same-model idle-timeout retry is disabled once fallbacks are configured, so 'slow prompt ingestion causes cloud fallback' remains plausible.

## Evidence

- **Native Ollama requests still send the full model context window:** `createOllamaStreamFn()` builds `options.num_ctx` from `model.contextWindow ?? 65536`, so long-context Ollama runs still ask Ollama to ingest up to the configured/discovered window on every chat request.
  - file: [extensions/ollama/src/stream.ts:626](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/extensions/ollama/src/stream.ts#L626)
  - command: `nl -ba extensions/ollama/src/stream.ts | sed -n '600,700p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Ollama discovery uses `/api/show` context length and defaults to that in model metadata:** The provider reads `*.context_length` from Ollama `/api/show` responses and writes it into the discovered model definition as `contextWindow`; there is no Ollama-specific runtime cap here beyond whatever the config manually overrides.
  - file: [extensions/ollama/src/provider-models.ts:124](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/extensions/ollama/src/provider-models.ts#L124)

- **Main has generic timeout-compaction mitigation, not an Ollama-specific ingestion fix:** `run.ts` now attempts compaction when a model times out with high prompt-token usage (>65% of context), which is a mitigation for large prompts, but it is still generic retry/compaction logic rather than a fix that distinguishes healthy-but-slow Ollama prompt ingestion from actual model failure.

- **Idle-timeout retry is suppressed when fallbacks are configured:** Assistant-side same-model idle-timeout retry is only allowed when `!fallbackConfigured`; otherwise assistant timeout handling can escalate to `fallback_model`, which matches the issue's report that a slow local model falls back to cloud instead of being treated as healthy-but-slow.
