## Summary

Keep open. Current main has inbound `media://inbound/*` claim-check support in gateway parsing, prompt-image hydration, and `loadWebMedia`, but the `image` tool itself still rejects non-file/http/data schemes. That leaves the issue’s reported agent-session path unresolved on the specific tool entrypoint the linked PR describes.

## Evidence

- **Gateway emits claim-check URIs for text-only attachment flows:** `parseMessageWithAttachments` offloads images to opaque `media://inbound/<id>` refs when `supportsImages: false`, and its own docstring says this is so configured image tools can inspect the original file.
  - file: [src/gateway/chat-attachments.ts:263](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/chat-attachments.ts#L263)
