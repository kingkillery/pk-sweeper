## Summary

Keep open. Current `main` still omits `currentMessageId` in the followup delivery path, so `replyToMode: "all"` cannot implicitly set `replyToId` for queued followup replies; the PR remains a small, actionable fix rather than something already implemented or obsolete.

## Evidence

- **Followup delivery signature still lacks currentMessageId:** `resolveFollowupDeliveryPayloads` accepts config, payloads, provider, and routing fields, but no `currentMessageId`; its `applyReplyThreading(...)` call forwards only `payloads`, `replyToMode`, and `replyToChannel`. That means the implicit reply target is unavailable in this path on current `main`.
  - file: [src/auto-reply/reply/followup-delivery.ts:26](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/followup-delivery.ts#L26)

- **Followup runner does not pass queued.messageId:** The followup runner builds `finalPayloads` via `resolveFollowupDeliveryPayloads(...)`, but the current call site does not pass `queued.messageId` or any equivalent message identifier.
  - file: [src/auto-reply/reply/followup-runner.ts:427](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/followup-runner.ts#L427)
