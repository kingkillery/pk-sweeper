## Summary

Keep open. Current `main` does not implement the descriptive-avatar parsing this PR targets, and the PR discussion identifies plausible path-truncation regressions in the proposed regex-based extraction, so there is not strong evidence for a conservative close.

## Evidence

- **Current main still assigns raw avatar text verbatim:** `parseIdentityMarkdown` trims the value after the colon and, for `label === "avatar"`, stores that raw string directly. There is no tail-path extraction for descriptive `Avatar:` lines on `main`.
  - file: [src/agents/identity-file.ts:59](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/identity-file.ts#L59)
  - command: `nl -ba src/agents/identity-file.ts | sed -n '47,86p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Main test coverage does not include descriptive avatar lines:** Current tests cover placeholders and direct avatar values like `avatars/openclaw.png`, but there is no regression test proving support for descriptive `Avatar:` text with a trailing path.
  - file: [src/agents/identity-file.test.ts:4](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/identity-file.test.ts#L4)
  - command: `nl -ba src/agents/identity-file.test.ts | sed -n '1,36p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Current avatar normalization would mis-handle descriptive strings that contain a path:** `looksLikeAvatarPath` returns true for any value containing `/`, and `normalizeAvatarValue` accepts such strings as avatar paths. Without parser-side extraction, a descriptive string ending in `avatars/foo.png` is still treated as a local path candidate rather than normalized to the actual path.
  - file: [src/shared/avatar-policy.ts:74](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/shared/avatar-policy.ts#L74)
  - command: `nl -ba src/shared/avatar-policy.ts | sed -n '74,79p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Assistant identity resolution relies on that permissive path check:** `normalizeAvatarValue` returns any string that `looksLikeAvatarPath(...)`, so the descriptive-line bug remains user-visible until parsing is fixed upstream.
  - file: [src/gateway/assistant-identity.ts:42](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/gateway/assistant-identity.ts#L42)
  - command: `nl -ba src/gateway/assistant-identity.ts | sed -n '34,52p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **PR review discussion found unresolved regressions in the proposed fix:** The provided review comments on PR #62727 flag that splitting on dash separators can truncate legitimate filenames such as `avatars/hawk - final.png`, and that the en/em dash regex also matches without surrounding spaces, truncating values like `avatars/hawk—final.png`. That is evidence the PR should be reviewed/fixed, not conservatively closed.
