## Summary

Keep open. Current main still routes Mattermost `message` tool `send` through the plugin `handleAction`, and that handler still only forwards `replyToId`/`replyTo` plus `params.media`; it does not read `threadId` and does not pass the runner-provided `mediaLocalRoots`/`mediaReadFile` into `sendMessageMattermost`. Core now normalizes `path`/`filePath` into `params.media` before plugin dispatch, but the local-file upload path still needs those media roots, and explicit `threadId` still will not become Mattermost `rootId`. The related PR #67796 is an open partial fix rather than a shipped/superseding resolution.

## Best Possible Solution

Keep the issue open and fix the Mattermost plugin `handleAction` send path directly: read `threadId` as a fallback reply/root id, preserve the existing `replyToId`/`replyTo` precedence, use `media` -> `filePath` -> `path` if direct plugin calls should work without the generic runner, and forward `ctx.mediaLocalRoots`/`ctx.mediaReadFile` (or `ctx.mediaAccess`) into `sendMessageMattermost`. Add focused Mattermost `channel.test.ts` coverage for `threadId`, `filePath`/`path`, and local media roots on `action: send`; PR #67796 can be refreshed or superseded by that complete patch.
