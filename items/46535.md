## Summary

Current `main` already handles the built-in TTS voice-delivery case through the newer structured `details.media` path, so this PR’s legacy `[[audio_as_voice]]` propagation patch is superseded.

## Evidence

## Risks / Open Questions

- Legacy third-party tools that still emit `[[audio_as_voice]]` in text-only `MEDIA:` output instead of structured `details.media` are not covered by this superseding implementation path.

## Close Comment

Closing this PR after Codex review because current `main` already covers the built-in TTS delivery case through the newer structured media path. `src/agents/tools/tts-tool.ts:80` now returns audio in `details.media` with `audioAsVoice`, `src/agents/pi-embedded-subscribe.tools.ts:282` preserves that flag during extraction, and `src/agents/pi-embedded-subscribe.handlers.tools.ts:536` carries it into the outbound tool-media payload. The same behavior is covered by `src/agents/tools/tts-tool.test.ts:20` and `src/agents/pi-embedded-subscribe.handlers.tools.media.test.ts:476`, and I verified that implementation is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Given that, this PR’s older legacy-tag propagation change is superseded on `main`.
