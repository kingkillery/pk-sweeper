## Summary

Current main already routes the live gateway restart entry points through `scheduleGatewaySigusr1Restart(...)` instead of relying on `triggerOpenClawRestart()` for the in-process case, so the PR's reported exec-tool failure path no longer matches shipped main behavior. The Windows schtasks re-entry concern discussed on the PR is also avoided on current main because `triggerOpenClawRestart()` still dispatches win32 restarts to `relaunchGatewayScheduledTask(...)` without a helper-level SIGUSR1 fast path.

## Risks / Open Questions

- If a future caller invokes `triggerOpenClawRestart()` directly from inside the gateway process, the helper still does not have the generalized SIGUSR1 fast path proposed here.
- The current main behavior appears newer than the latest tagged release in the prompt, so users testing an older release may still observe behavior that no longer matches main.
