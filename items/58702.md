## Summary

Keep open. The report is concrete, non-stale, non-maintainer-authored, and has no protected labels, but current main does not provide high-confidence evidence that the WebChat overlap is fully fixed. The current renderer still constructs `bubbleClasses` without the `has-copy` class, while the CSS spacing rule only applies to `.chat-bubble.has-copy`; that matches the reporter and commenter’s CSS/JS mismatch theory. A related later issue, #61514, was closed via #61872 for similar icon-overlap behavior, but the current source still leaves enough doubt that this item should not be conservatively closed.

## Evidence

- **Reporter provided actionable reproduction:** Issue #58702 describes WebChat at `http://127.0.0.1:18789`, sending any message, and seeing copy/expand action icons permanently visible over the end of sent message text after upgrading to 2026.3.31 on Windows; the reporter also attached a screenshot.
- **Discussion identifies likely CSS/JS mismatch:** A commenter traced the issue to `.chat-bubble.has-copy` padding existing in CSS while the rendered bubble class did not include `has-copy`, leaving action buttons over the text. This directly matches the reported truncation/overlap.
- **Current renderer still omits `has-copy` from bubble classes:** Current `renderGroupedMessage` computes `canCopyMarkdown` and `canExpand`, but `bubbleClasses` is built as `['chat-bubble', streaming, 'fade-in']` without `has-copy`; actions are then rendered when `hasActions` is true.

  - command: `web inspection of GitHub main/raw because local read-only commands failed before execution`
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **CSS only reserves icon space when `has-copy` is present:** Current CSS defines `.chat-bubble.has-copy { padding-right: 70px; }` and positions `.chat-bubble-actions` absolutely at the top right, hidden until hover on pointer devices and always visible under `@media (hover: none)`. Without the class, the reserve-space rule does not apply.
