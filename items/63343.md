## Summary

Keep open. Current `main` still contains the exact `ensureTabAvailable` pattern the issue reports, and the linked fix PR #63355 was closed unmerged after review comments identified additional correctness problems.

## Evidence

- **Current code still discards `openTab()` result and filters local-managed tabs by `wsUrl`:** `ensureTabAvailable()` still does `const tabs1 = await listTabs(); if (tabs1.length === 0) { await openTab("about:blank"); } const tabs = await listTabs(); const candidates = capabilities.supportsPerTabWs ? tabs.filter((t) => Boolean(t.wsUrl)) : tabs;`. That matches the reported race: the newly opened tab is not retained, and `wsUrl`-less tabs are dropped before selection.
  - file: [extensions/browser/src/browser/server-context.selection.ts:43](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/browser/src/browser/server-context.selection.ts#L43)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Affected profile mode is still enabled on `main`:** `local-managed` profiles still set `supportsPerTabWs: true`, so the `wsUrl` filter is active for the localhost mode called out in the issue.
  - file: [extensions/browser/src/browser/profile-capabilities.ts:46](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/browser/src/browser/profile-capabilities.ts#L46)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)

I did not find a landed regression test on `main` for `openTab()` returning before `/json/list` exposes `wsUrl`, nor for the unfiltered fallback discussed in the issue/PR.
