## Summary

Keep open. Current `main` has some generic plugin-channel groundwork, but the Feishu-specific behavior this PR targets is still not implemented end to end. `/allowlist ... channel=feishu` still resolves only built-in channel ids, the Feishu plugin still does not expose an allowlist adapter or markdown-capable metadata, envelope stripping still omits `Feishu`, and Feishu security fixes do not provide `applyConfigFixes` hardening.

## Evidence

- **`/allowlist` still normalizes explicit channels with the built-in-only helper:** `handleAllowlistCommand` resolves `parsed.channel` with `normalizeChannelId(...)`, and `src/channels/registry.ts` defines that helper as `normalizeChatChannelId` only. Registered plugin ids instead go through `normalizeAnyChannelId`. So `channel=feishu` is still not accepted from non-Feishu surfaces.
  - file: [src/auto-reply/reply/commands-allowlist.ts:289](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/auto-reply/reply/commands-allowlist.ts#L289)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Built-in vs plugin channel normalization split is explicit on main:** `normalizeChannelId` returns only `ChatChannelId`, while `normalizeAnyChannelId` is the helper that resolves registered plugin channels. `commands-allowlist.ts` is not using the plugin-aware path.
  - file: [src/channels/registry.ts:64](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/channels/registry.ts#L64)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Allowlist command still requires a plugin allowlist adapter that Feishu does not provide:** Listing requires `plugin.allowlist.readConfig` and config edits require `plugin.allowlist.applyConfigEdit`; otherwise the command returns `does not expose allowlist configuration` / `does not support ... allowlist edits`. Searching `extensions/feishu/src/channel.ts` shows no `allowlist:` adapter field on the Feishu plugin.
  - file: [src/auto-reply/reply/commands-allowlist.ts:317](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/auto-reply/reply/commands-allowlist.ts#L317)
  - command: `rg -n "allowlist:" extensions/feishu/src/channel.ts`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Markdown-capable formatting still depends on plugin metadata, and Feishu metadata does not opt in:** `isMarkdownCapableMessageChannel` checks registered plugin metadata for `markdownCapable === true`. Feishu's `ChannelMeta` currently defines id/label/docs/aliases/order only; it does not set `markdownCapable`, so Feishu still falls back to plain formatting in this path.
  - file: [src/utils/message-channel.ts:73](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/utils/message-channel.ts#L73)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Envelope stripping still omits `Feishu`:** `stripEnvelope` only strips prefixes whose header starts with one of the hardcoded `ENVELOPE_CHANNELS`, and that list still excludes `Feishu`. A header like `[Feishu 2026-01-24 13:36] ...` would therefore not be stripped on current `main`.
  - file: [src/shared/chat-envelope.ts:2](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/shared/chat-envelope.ts#L2)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Security hardening is now plugin-owned, but Feishu still does not implement config fixes:** `src/security/fix.ts` now delegates channel-specific mutations through `plugin.security.applyConfigFixes`. Feishu's security section only exposes warning/audit collectors, not `applyConfigFixes`, so `security audit --fix` still has no Feishu group-policy hardening path.
  - file: [src/security/fix.ts:297](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/security/fix.ts#L297)

That means the PR would need rebase/scope trimming, not closure as implemented.
  - file: [src/config/zod-schema.hooks.ts:52](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/config/zod-schema.hooks.ts#L52)
