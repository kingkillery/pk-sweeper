## Summary

Keep open. Current `main` does not implement the requested behavior, but this PR's proposed channel-id suppression is not safe: OpenClaw documents `id === channels[0]` as the normal shape for a single channel plugin manifest, while the loader/registry treat distinct roots with the same plugin id as separate plugin candidates. Suppressing the duplicate warning for that case would hide real collisions and can silently drop the later plugin record.

## Evidence

- **Current duplicate logic only suppresses same-plugin duplicates:** `loadPluginManifestRegistry` only skips duplicate warnings when two candidates resolve to the same root path or same realpath. Distinct plugin roots with the same `manifest.id` still produce a warning and winner/loser override flow.
  - file: [src/plugins/manifest-registry.ts:703](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/plugins/manifest-registry.ts#L703)
  - command: `nl -ba src/plugins/manifest-registry.ts | sed -n '703,754p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Loader depends on every surviving manifest record being present:** `loadOpenClawPlugins` builds `manifestByRoot` from `manifestRegistry.plugins` and skips any discovery candidate whose root is absent from that map. A duplicate-suppression branch that `continue`s before `records.push(...)` would therefore silently drop that candidate from later loading/override handling.
  - file: [src/plugins/loader.ts:2164](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/plugins/loader.ts#L2164)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2164,2188p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Current tests only suppress duplicates for the same physical plugin:** Existing registry tests cover the intended no-warning cases: same physical directory via symlink and identical `rootDir` paths. There is no current-main test treating two distinct plugin roots as non-duplicates just because one manifest declares a channel with the same id.
  - file: [src/plugins/manifest-registry.test.ts:1276](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/plugins/manifest-registry.test.ts#L1276)
  - command: `nl -ba src/plugins/manifest-registry.test.ts | sed -n '1276,1329p'`

- **Public docs describe same-id channel registration inside one manifest:** The plugin docs show the expected channel-plugin manifest shape as a single plugin with `id: "acme-chat"`, `kind: "channel"`, and `channels: ["acme-chat"]`. That supports the issue report's pattern, but it does not imply that two separate plugin candidates with the same id should bypass duplicate-id detection.
  - file: [docs/plugins/sdk-channel-plugins.md:323](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/docs/plugins/sdk-channel-plugins.md#L323)
