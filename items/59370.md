## Summary

Keep the issue open. The issue is recent, authored by a non-maintainer, has no protected labels, and the discussion contains multiple concrete confirmations through 2026-04-15 that `logging.file` is ignored in packaged installs. Because local read-only inspection commands are blocked by the sandbox before execution, I do not have the source/test proof needed to close as implemented or unreproducible.

## Evidence

- **Issue reproduction:** The issue body reports a concrete npm-global packaged install reproduction on OpenClaw 2026.4.1: setting `logging.file` in `~/.openclaw/openclaw.json`, restarting the gateway, and observing logs still written under `/tmp/openclaw/openclaw-YYYY-MM-DD.log` while the configured file remains empty.
- **Discussion confirmation:** Comments include independent confirmations that packaged logging modules still resolve `../config/config.js`, swallow the failure, and fall back to `/tmp/openclaw`; one commenter specifically confirmed the same behavior on 2026.4.14, and the final comment summarizes three confirmations across shared logging modules.
- **Public behavior contract:** The public logging docs state that the default `/tmp/openclaw/openclaw-YYYY-MM-DD.log` path can be overridden via `logging.file` in `~/.openclaw/openclaw.json`, so the reported behavior remains a plausible product bug rather than an unsupported configuration request.
