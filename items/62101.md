## Summary

Keep open. Current `main` still leaves bare-integer BlueBubbles `allowFrom` entries unmatched, so the underlying bug is not implemented on `main`. But this PR’s current schema change is too broad because it rewrites a shared `AllowFromEntrySchema` used by other channels that legitimately accept numeric IDs, so closing it would be premature and merging as-is would risk regressions.

## Evidence

- **Shared schema still accepts bare numbers without BlueBubbles-specific coercion:** `AllowFromEntrySchema` on current `main` is still `z.union([z.string(), z.number()])`, so config parse does not add a `+` prefix or otherwise canonicalize numeric allowlist entries.

## Risks / Open Questions

- Closing as implemented would be wrong: current `main` still lacks the BlueBubbles numeric-handle normalization described in the PR.
- Closing as cannot_reproduce would be weak: the current code path still permits numeric `allowFrom` values and still compares them without `+E.164` coercion.
- Merging the PR unchanged would risk cross-channel regressions because `AllowFromEntrySchema` is shared beyond BlueBubbles.
