## Summary

Keep open. Current `main` still lacks requester-authorization checks for Discord guild-admin actions, so the security bug is real and not already fixed, but this PR's current fail-closed runtime approach appears to regress legitimate owner CLI/manual guild-admin commands that do not carry `requesterSenderId`.

## Evidence

- **Current main still forwards guild-admin actions without trusted sender identity:** `tryHandleDiscordMessageActionGuildAdmin` forwards guild-admin actions like `emoji-upload`, `role-add`, `channel-delete`, and `event-create` into `handleDiscordAction(...)` without adding `senderUserId`, unlike the moderation path later in the same file.
  - file: [extensions/discord/src/actions/handle-action.guild-admin.ts:64](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/discord/src/actions/handle-action.guild-admin.ts#L64)
  - command: `nl -ba extensions/discord/src/actions/handle-action.guild-admin.ts | sed -n '20,210p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Current main guild runtime has no requester permission check for admin mutations:** `handleDiscordGuildAction` executes privileged guild mutations (`roleAdd`, `roleRemove`, `channelCreate/Edit/Delete/Move`, `category*`, `eventCreate`, `channelPermissionSet/Remove`) directly and never reads `senderUserId` or calls `hasAnyGuildPermissionDiscord`.
  - file: [extensions/discord/src/actions/runtime.guild.ts:157](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/discord/src/actions/runtime.guild.ts#L157)

## Risks / Open Questions

- The underlying Discord guild-admin privilege-escalation path still appears open on current `main` for tool-driven/message-action flows.
- Merging the PR as-is risks breaking legitimate owner CLI/manual Discord admin commands because those paths currently do not carry `requesterSenderId`.
- Even after the main authz fix, the outer trusted-requester list needs to stay in sync with privileged guild actions such as `channelPermissionSet` and `channelPermissionRemove`.
