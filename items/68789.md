## Summary

Keep open. Current main still matches the report more than it refutes it: Slack `EAI_AGAIN` is intentionally treated as a non-fatal gateway error, outbound Slack sends still go through a no-retry WebClient path, and Socket Mode only has generic reconnect handling. I did not find a landed fix, a regression test for this DNS failure mode, or a safe reproduction path that disproves the report.

## Evidence

- **Issue report:** The issue body is recent (opened 2026-04-19) and concrete: host DNS and host Node HTTPS succeed repeatedly, while OpenClaw Slack sends intermittently fail with `getaddrinfo EAI_AGAIN slack.com`; the gateway stays up and Socket Mode shows pong timeouts/stale-socket restarts.
- **Reporter follow-up:** The only follow-up comment says quarantining the delivery queue stopped the replay storm but fresh direct Slack sends still failed with `EAI_AGAIN`, which argues the queue backlog was secondary rather than the root cause.
- **Current main still treats Slack DNS failures as non-fatal gateway errors:** `src/infra/unhandled-rejections.ts` includes `EAI_AGAIN` in transient network codes, and the tests explicitly cover Slack Web API request errors wrapping `getaddrinfo EAI_AGAIN slack.com`. That matches the report's "gateway remains running" symptom, but it does not fix the Slack delivery failure itself.
  - file: [src/infra/unhandled-rejections.ts:26](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/infra/unhandled-rejections.ts#L26)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Outbound Slack sends still use a no-retry client path:** `extensions/slack/src/send.ts` resolves outbound sends through `getSlackWriteClient(token)`, and `extensions/slack/src/client-options.ts` sets `SLACK_WRITE_RETRY_OPTIONS` to `{ retries: 0 }` with no DNS-specific handling. I found no current-main code that retries or re-diagnoses transient `EAI_AGAIN` failures on the Slack send path.
  - file: [extensions/slack/src/send.ts:404](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/slack/src/send.ts#L404)

I found no current-main DNS-specific recovery or extra transport diagnostics for the reported Slack failure mode.
