## Close Comment

Closing this as `cannot_reproduce` after a Codex review of current `main` (`4d7c4b3`). On current OpenClaw, `plugins.entries.<id>` does not create a second `config`-origin plugin candidate; discovery only creates `config` candidates from `plugins.load.paths` (`src/config/types.plugins.ts:1`, `src/plugins/discovery.ts:1139`). For the real false-positive case where a bundled plugin and a config-selected path resolve to the same on-disk plugin, `loadPluginManifestRegistry()` already suppresses the duplicate warning and keeps the higher-precedence origin (`src/plugins/manifest-registry.ts:627`), with coverage in `src/plugins/manifest-registry.test.ts:1112`. For distinct roots that share an id, current `main` still intentionally warns because that is a real precedence conflict, and the docs describe that behavior as expected (`docs/plugins/manifest.md:642`). Given that, this PR no longer matches current behavior well enough to land. If there is still a live repro on current `main`, the useful next step would be a fresh report with the exact config, whether `plugins.load.paths` is involved, and the emitted warning text.
