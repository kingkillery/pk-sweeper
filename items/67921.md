## Summary

Keep open. Current `main` does not implement the PR's core behavior change, and the existing startup flow now has explicit tests and contracts that conflict with the detached `startChannels()` approach proposed here.

## Evidence

- **Current main still waits for channel startup inside sidecars:** `startGatewaySidecars()` still awaits `params.startChannels()` before continuing, so channel startup remains part of sidecar readiness on `main`. That means the PR's core non-blocking change is not already implemented.

- **Current main documents waiting as the default contract:** `GatewayServerOptions.deferStartupSidecars` is documented as optional and defaults to `false`, with the explicit meaning that gateway startup waits until sidecars are ready. The PR would change that default behavior for channels without updating the broader contract.
  - file: [src/gateway/server.impl.ts:233](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/gateway/server.impl.ts#L233)

That is a stronger ordering guarantee than the PR's detached fire-and-forget change.

## Risks / Open Questions

- Closing as implemented would be incorrect: current `main` still defaults to waiting for sidecars, and this PR's key behavior change is not present.
- Closing as stale or insufficient would discard a still-coherent behavior proposal, but the proposal now conflicts with newer startup-ordering tests and contracts on `main`.
- Merging the PR as-is would likely regress startup hook ordering and readiness semantics unless it is redesigned around the current deferred-sidecars contract.
