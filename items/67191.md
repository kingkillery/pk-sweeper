## Summary

Current main does not provide enough evidence to close this bug. The BOOT.md runner still calls the agent with `deliver: false`, tests explicitly assert that behavior, and the existing boot-md integration test only proves startup dispatch, not that a BOOT.md `message` tool send reaches Telegram or another channel. The issue is recent and has an independent Telegram confirmation, so it should remain open until the intended boot delivery contract is either fixed or regression-tested and documented.

## Best Possible Solution

Keep the issue open. Add a focused regression that runs BOOT.md through the real boot runner with a mocked message action/channel and proves an explicit `message` tool send is dispatched while the final `NO_REPLY` remains silent. If the intended contract is explicit `channel` + `target` only, clarify the public BOOT.md docs and boot logs; if the intended contract is the configured main session, seed BOOT.md runs with that delivery context instead of relying on final `deliver: true`.
