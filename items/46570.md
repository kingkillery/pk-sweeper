## Summary

Keep open. Current main does not provide strong evidence that the reported mixed memory-file vs session-search problem is fixed for the default builtin backend, so closing as implemented or cannot-reproduce would be unsafe.

## Evidence

- **Builtin remains the default backend:** `resolveMemoryBackendConfig` still returns the builtin backend unless `memory.backend` is explicitly set to `qmd`, and the docs still describe the builtin engine as the default memory backend.
  - file: [packages/memory-host-sdk/src/host/backend-config.ts:348](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/packages/memory-host-sdk/src/host/backend-config.ts#L348)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Docs still anchor users on builtin memory search:** The memory docs say the builtin SQLite engine is the default backend with hybrid search, which matches the issue's reported local embedding + hybrid setup more closely than a QMD-only path.
  - file: [docs/concepts/memory-builtin.md:9](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/concepts/memory-builtin.md#L9)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Builtin search still slices purely by merged score:** Builtin `MemoryIndexManager.search` merges vector and keyword results, filters by score, and returns `strict.slice(0, maxResults)` with no per-source diversification step, so session hits can still crowd out memory-file hits.
  - file: [extensions/memory-core/src/memory/manager.ts:407](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/manager.ts#L407)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Only QMD has an explicit mixed-source retention safeguard:** `QmdMemoryManager.search` runs `diversifyResultsBySource(...)`, and there is a dedicated test asserting mixed session + memory results retain a memory hit. I did not find an equivalent safeguard or passing coverage for the builtin backend.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:1232](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/qmd-manager.ts#L1232)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Relevant builtin coverage is still skipped:** The builtin test file still has skipped tests for hybrid keyword/memory recall (`finds keyword matches via hybrid search when query embedding is zero` and `preserves keyword-only hybrid hits when minScore exceeds text weight`), which is not strong evidence that the reported behavior has been resolved on main.
  - file: [extensions/memory-core/src/memory/index.test.ts:358](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/memory-core/src/memory/index.test.ts#L358)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
