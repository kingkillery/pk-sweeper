## Summary

Current `main` has a real global subagent concurrency cap and queueing, so the issue's "no concurrency cap" claim is no longer true. But the subagent spawn path still has no memory-pressure-aware admission check or circuit breaker, and the default subagent cap is still 8. That is not strong enough evidence for a conservative close on an OOM report from an 8 GB / no-swap host.

## Evidence

- **Subagent concurrency is now configurable:** `agents.defaults.subagents.maxConcurrent` is part of the current config surface.
  - file: [src/config/types.agent-defaults.ts:392](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/config/types.agent-defaults.ts#L392)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Gateway applies the subagent cap to a dedicated lane:** Gateway startup wires `resolveSubagentMaxConcurrent(cfg)` into `CommandLane.Subagent`, and the command queue only starts work while `activeTaskIds.size < state.maxConcurrent`, which means excess subagent runs are queued instead of all starting at once.
  - file: [src/gateway/server-lanes.ts:6](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/gateway/server-lanes.ts#L6)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Queue implementation enforces lane concurrency:** The queue pump drains each lane only up to `state.maxConcurrent`, and `setCommandLaneConcurrency()` clamps and applies that value.
  - file: [src/process/command-queue.ts:155](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/process/command-queue.ts#L155)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Docs now describe queued subagent runs with a global cap:** The docs say `sessions_spawn` runs on global lane `subagent`, and document `maxConcurrent` as the global subagent lane cap.
  - file: [docs/tools/subagents.md:95](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/docs/tools/subagents.md#L95)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Spawn path still only checks depth, child count, and timeout:** Before starting a child run, the spawn flow resolves timeout, checks `maxSpawnDepth`, checks `maxChildrenPerAgent`, then dispatches the child on `lane: AGENT_LANE_SUBAGENT`; there is no memory-based refusal or backpressure check in this path.
  - file: [src/agents/subagent-spawn.ts:597](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/subagent-spawn.ts#L597)
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **No memory-pressure gate found in subagent scheduling path:** A targeted search across subagent spawn/registry, queue, and lane code found no `/proc/meminfo`, PSI, RSS-threshold, or memory-pressure circuit-breaker logic on the subagent admission path.
  - command: `rg -n "meminfo|PSI|memory pressure|rss|heap|circuit breaker" src/agents/subagent-spawn.ts src/agents/subagent-registry*.ts src/gateway/server-lanes.ts src/process/command-queue.ts src/config/agent-limits.ts`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
