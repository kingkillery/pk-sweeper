## Summary

Keep open. Current `main` still shows the reported cleanup gap: session reset cleans the target session and subagent registry children, but does not enumerate ACP child sessions spawned from that parent, and oneshot ACP close failures are still only logged with no PID-level fallback cleanup.

## Evidence

- **Parent reset only closes the target ACP session:** `cleanupSessionBeforeMutation()` runs `ensureSessionRuntimeCleanup()` and then calls `closeAcpRuntimeForSession()` once for the target key. There is no follow-up scan over ACP children by `spawnedBy` or `parentSessionKey`.
  - file: [src/gateway/session-reset-service.ts:397](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/session-reset-service.ts#L397)
  - command: `nl -ba src/gateway/session-reset-service.ts | sed -n '397,420p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Reset cleanup only cascades through subagent runs:** `ensureSessionRuntimeCleanup()` calls `stopSubagentsForRequester()` for the parent session key. That helper walks subagent run records and recursively stops subagents, but it does not inspect ACP session store entries.
  - file: [src/gateway/session-reset-service.ts:194](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/session-reset-service.ts#L194)
  - command: `nl -ba src/gateway/session-reset-service.ts | sed -n '194,245p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Subagent stopper is subagent-specific:** `stopSubagentsForRequester()` iterates `listSubagentRunsForController()` / `getLatestSubagentRunByChildSessionKey()` and aborts embedded/subagent runs. No ACP-session enumeration path exists there.
  - file: [src/auto-reply/reply/abort.ts:136](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/abort.ts#L136)
  - command: `nl -ba src/auto-reply/reply/abort.ts | sed -n '136,223p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **ACP oneshot close failure is still swallowed:** In `AcpSessionManager.runTurn()`, oneshot sessions call `runtime.close({ reason: "oneshot-complete" })` in `finally`, but any failure is only logged before cache cleanup. There is no PID/process-level fallback here.
  - file: [src/acp/control-plane/manager.core.ts:945](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/acp/control-plane/manager.core.ts#L945)
  - command: `nl -ba src/acp/control-plane/manager.core.ts | sed -n '945,965p'`

- **ACP children are tracked as parent-owned spawned sessions:** ACP spawn patches the child session with `spawnedBy: requesterInternalKey`, so child ownership metadata exists in the store and matches the issue's premise that parent-linked ACP child sessions can be identified.
