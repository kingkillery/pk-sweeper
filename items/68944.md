## Summary

Keep open. Current `main` still has the same plausible client-side handshake failure path the discussion identified, no relevant fix landed after the latest release, and the issue has fresh corroborating reports from April 24, 2026 rather than stale or obsolete evidence.

## Evidence

- **Current client still can swallow sendConnect failures during challenge handling:** `GatewayClient.sendConnect()` still builds and signs the device payload inline during `connect.challenge` handling, and `handleMessage()` still wraps the whole handler in a broad `try/catch` that logs `gateway client parse error` at debug level. A throw while signing or constructing the connect request would still prevent `connect` from being sent and get mislabeled as a parse error.
  - file: [src/gateway/client.ts:435](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/client.ts#L435)
  - command: `nl -ba src/gateway/client.ts | sed -n '430,560p' && nl -ba src/gateway/client.ts | sed -n '732,797p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Signing helper is still synchronous and unguarded:** `signDevicePayload()` still directly calls `crypto.createPrivateKey()` and `crypto.sign()` with no local error handling. That matches the failure mode discussed in the issue thread: if this path throws inside message handling, the client has no dedicated error surface before the broad debug-only catch in `handleMessage()`.
  - file: [src/infra/device-identity.ts:125](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/infra/device-identity.ts#L125)
