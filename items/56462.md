## Summary

Keep open. Current `main` does not show a full fix for Windows exec output decoding. The repo now has Windows codepage-aware decoding in the node-host `system.run` path, but the default exec path on Windows still routes non-PTY gateway exec through the child-process adapter, which decodes stdout/stderr with plain `chunk.toString()`. That leaves the reported garbled Chinese output plausible on current `main` and on the latest release.

## Evidence

- **Exec still defaults to gateway + non-PTY:** The exec docs say `pty` defaults to `false`, and `host=auto` resolves to `gateway` when no sandbox runtime is active. That matches the common Windows npm install path for local exec runs.
  - file: [docs/tools/exec.md:39](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/docs/tools/exec.md#L39)

  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Non-PTY exec uses the child adapter:** `bash-tools.exec.ts` computes `usePty = params.pty === true && !sandbox`, and `runExecProcess` falls back to `mode: "child"` when PTY is not requested, then spawns the supervisor in child mode.
  - file: [src/agents/bash-tools.exec.ts:1684](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/agents/bash-tools.exec.ts#L1684)
