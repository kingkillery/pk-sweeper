## Summary

Keep open. Current `main` still fails ACP task cancellation when the child ACP session has lost its ACP metadata, and the maintenance sweep does not convert that metadata-missing ACP task into a terminal/lost state if the session row itself still exists.

## Evidence

- **`openclaw tasks cancel` surfaces any failed cancellation directly:** The CLI resolves the task, calls `cancelDetachedTaskRunById`, and exits with an error whenever `result.cancelled` is false.
  - file: [src/commands/tasks.ts:384](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/commands/tasks.ts#L384)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **ACP task cancellation only marks the task `cancelled` after `cancelSession` succeeds:** For ACP tasks, `cancelTaskById` awaits `getAcpSessionManager().cancelSession(...)`. The task status is updated to `cancelled` only afterward; any thrown error is caught and returned as `cancelled: false`.
  - file: [src/tasks/task-registry.ts:1760](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/tasks/task-registry.ts#L1760)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **ACP sessions without ACP metadata are still treated as stale/missing-metadata by the control plane:** `resolveSession` returns `kind: "stale"` for ACP-shaped session keys that have no ACP metadata, and the error text is the same missing-metadata message reported in the issue.
  - file: [src/acp/control-plane/manager.core.ts:191](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/acp/control-plane/manager.core.ts#L191)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **`cancelSession` requires ready ACP metadata before it can cancel an inactive ACP session:** Outside the active-turn fast path, `cancelSession` resolves the session and immediately calls `requireReadySessionMeta(resolution)`, so metadata-missing ACP sessions fail before any cancel operation can run.
  - file: [src/acp/control-plane/manager.core.ts:1189](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/acp/control-plane/manager.core.ts#L1189)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Task maintenance does not treat metadata-missing ACP rows as missing backing sessions:** For ACP tasks, maintenance checks `Boolean(acpEntry.entry)`, not ACP metadata presence. If the session store row still exists but its `acp` metadata is gone, the task is considered backed and is not marked `lost`.
  - file: [src/tasks/task-registry.maintenance.ts:138](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/tasks/task-registry.maintenance.ts#L138)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Docs still describe ACP task cancellation and maintenance as working through the child session:** The public task docs say ACP tasks are cancelled by killing the child session and that maintenance reconciles ACP/subagent tasks by checking their backing child session, which does not match the metadata-missing ACP path above.
  - file: [docs/automation/tasks.md:181](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/docs/automation/tasks.md#L181)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
