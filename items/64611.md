## Summary

Keep open. Current `main` still lacks the PR's `lastRunMs > nowMs` guard, so a persisted future heartbeat timestamp would still fail the due check. The titled fix is not implemented on `main`, even though the unrelated heartbeat task-parsing test work from this PR's branch appears to have landed separately.

## Evidence

- **Current due check still fails future timestamps:** `isTaskDue` only special-cases `undefined` and otherwise returns `nowMs - lastRunMs >= intervalMs`. If `lastRunMs` is in the future after clock skew, that expression is negative and the task is not due.
