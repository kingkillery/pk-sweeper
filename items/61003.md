## Summary

Keep open. Current `main` still exposes session switching in Control UI, but not session creation from that switcher. The visible chat-level `New session` button maps to `/new`, which resets/rotates the current session state under the same session key rather than creating a separate selectable session. The backend already has `sessions.create`, so the issue remains a valid UI gap. The issue timeline also points to open PR #61423, which suggests the work is recognized but not yet landed on `main`.

## Evidence

- **Chat header only renders the existing session select plus model/thinking selectors:** `renderChatSessionSelect()` builds a `<select>` for existing sessions and then appends model/thinking controls; there is no create button, prompt, or modal in the header/session-switcher area.
  - file: [ui/src/ui/chat/session-controls.ts:23](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/ui/src/ui/chat/session-controls.ts#L23)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **The visible chat `New session` control is wired to `/new`, not to session creation:** The chat view passes `onNewSession: () => state.handleSendChat("/new", { restoreDraft: true })`, so the UI action is a slash-command reset path, not a call to create a new keyed session for the switcher.
  - file: [ui/src/ui/app-render.ts:2271](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/ui/src/ui/app-render.ts#L2271)

- **`/new` reuses the current session key and overwrites that store slot:** Session reset logic rotates the session ID for a fresh conversation, then writes the result back to `sessionStore[sessionKey]` / `store[sessionKey]`, which preserves the key rather than creating an additional separately switchable session entry.
  - file: [src/auto-reply/reply/session.ts:489](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/session.ts#L489)
