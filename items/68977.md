## Summary

Keep open. Current `main` still does not provide the requested scoped, durable idempotency contract on `enqueueSystemEvent`; it only offers an in-memory queue with consecutive-text suppression. `main` did add a narrower `exec.finished` dedupe on 2026-04-24, but that mitigation is path-specific and does not satisfy this plugin-facing API enhancement.

## Evidence

- **Core system-event queue is explicitly ephemeral and not durable:** `src/infra/system-events.ts` describes the queue as a lightweight in-memory queue that intentionally avoids persistence. That conflicts directly with the issue's requested restart-safe idempotency semantics.
  - file: [src/infra/system-events.ts:1](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/infra/system-events.ts#L1)
  - command: `nl -ba src/infra/system-events.ts | sed -n '1,30p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **`enqueueSystemEvent` still has no idempotency-key option:** `SystemEventOptions` only includes `sessionKey`, `contextKey`, `deliveryContext`, and `trusted`. `enqueueSystemEvent(...)` returns `false` only for empty text or consecutive duplicate text (`entry.lastText === cleaned`), not for a scoped `(sessionKey, idempotencyKey)` contract.
  - file: [src/infra/system-events.ts:36](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/infra/system-events.ts#L36)

There is no test coverage for durable cross-restart idempotency or a public `idempotencyKey` API.

## Risks / Open Questions

- Closing this as implemented would hide a still-missing public SDK contract for plugin resume/retry flows.
- The recent `exec.finished` mitigation may reduce one duplicate path, but it does not provide the requested durable semantics across plugin restarts or arbitrary system-event producers.
