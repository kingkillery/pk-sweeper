## Summary

Keep open. Current main still has two macOS LaunchAgent tracks by design: `ai.openclaw.mac` for the app login item and `ai.openclaw.gateway` for the local Gateway service. The app now tries to attach to an existing gateway before enabling launchd, which helps, but the CLI doctor/install repair paths still do not detect `ai.openclaw.mac` and skip or warn, and the extra-service scanner can still treat the app agent as an extra OpenClaw service. The issue remains repo-actionable rather than closeable.

## Best Possible Solution

Keep this open and fix the macOS ownership boundary explicitly. A conservative path is to make CLI doctor/gateway repair detect the Mac app LaunchAgent (`ai.openclaw.mac`) and either skip reinstalling `ai.openclaw.gateway` with a clear app-specific note or coordinate through one canonical gateway service owner. Also add the missing launchd scan ignore/regression test for `ai.openclaw.mac`, keep the existing attach-existing behavior, and document when terminal `openclaw doctor --fix` is safe for Mac app users.
