## Summary

Current main already covers the reported failure mode: interrupted one-shot cron jobs are cleared without startup replay, manual cron runs are detached/enqueued, and orphaned cron task records are reconciled to `lost` instead of staying stuck.

## Evidence

- **Startup skips replay for interrupted one-shot jobs:** `src/cron/service/ops.ts:93` clears stale `runningAtMs` markers on startup and adds interrupted `at` jobs to `skipJobIds` before `runMissedJobs`, which prevents self-restarting one-shot jobs from replaying on restart.
  - file: [src/cron/service/ops.ts:93](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/cron/service/ops.ts#L93)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Regression test for one-shot restart behavior:** `src/cron/service.restart-catchup.test.ts:197` asserts that an interrupted one-shot cron job is not replayed on startup and its stale running marker is cleared.
  - file: [src/cron/service.restart-catchup.test.ts:197](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/cron/service.restart-catchup.test.ts#L197)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Manual cron.run is detached:** `src/gateway/server-methods/cron.ts:379` routes `cron.run` through `enqueueRun`, and `src/gateway/server.cron.test.ts:896` verifies the RPC returns immediately with `{ ok: true, enqueued: true, runId }` while isolated work continues in the background.
  - file: [src/gateway/server-methods/cron.ts:379](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/gateway/server-methods/cron.ts#L379)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Stale cron tasks are reconciled instead of hanging forever:** `src/tasks/task-registry.maintenance.ts:124` treats cron tasks as backed by the active cron runtime only, and after the 5-minute grace window `shouldMarkLost` converts orphaned active cron tasks to `lost`.
  - file: [src/tasks/task-registry.maintenance.ts:124](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/tasks/task-registry.maintenance.ts#L124)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)

  - file: [src/tasks/task-registry.maintenance.issue-60299.test.ts:133](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/tasks/task-registry.maintenance.issue-60299.test.ts#L133)

## Close Comment

Closing after Codex review. Current main already has the safeguards this report asks for: startup clears stale `runningAtMs` markers and explicitly skips replaying interrupted one-shot `at` jobs (`src/cron/service/ops.ts:93`, covered by `src/cron/service.restart-catchup.test.ts:197`), which prevents the reported self-restart loop. Detached cron runs also no longer stay stuck indefinitely: cron tasks are reconciled to `lost` once the cron runtime no longer owns the job (`src/tasks/task-registry.maintenance.ts:124`, covered by `src/tasks/task-registry.maintenance.issue-60299.test.ts:133`, and documented in `docs/automation/tasks.md:23`). I also verified the same guards are present in the `v2026.4.22` release tag. If this is still reproducible on `v2026.4.22` or newer, please open a fresh issue with the exact `cron list --json`, `cron runs --id <job>`, and task status output from the same run.
