## Summary

Keep open. Current `main` still routes multiple image-processing paths through a bare `import("sharp")` on non-`sips` platforms, with no repo evidence of a Linux fallback or a disabled backend mode. The reported native-load crash class is therefore not clearly fixed on `main`.

## Evidence

- **Bare sharp import remains:** `prefersSips()` only recognizes `OPENCLAW_IMAGE_BACKEND === "sips"` or Bun-on-macOS, and `loadSharp()` memoizes a direct `import("sharp")` without a wrapper around native-module load failure.
  - file: [src/media/image-ops.ts:30](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/media/image-ops.ts#L30)

## Risks / Open Questions

- I did not reproduce an actual `SIGILL` locally because this review environment does not provide the older x64 CPU / VM configuration described in the report.
- The issue title says 'non-AVX', but sharp's current install docs describe Linux x64 prebuilts as requiring SSE4.2. That wording mismatch does not change the core keep-open call: unsupported CPUs can still fall into an unhandled native-load path.
