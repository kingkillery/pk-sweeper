## Summary

Keep open. Current `main` has a narrow `gateway status` startup optimization, but the broader request is not implemented: plugin discovery/manifest caches are still short-lived in-process caches only, and other short-lived commands like `status --json` and `sessions` still flow through config reads/validation paths that invoke plugin manifest loading.

## Evidence

- **No persisted plugin cache on main:** The plugin architecture docs still describe discovery, manifest, and registry caches as short in-process caches only, explicitly 'not persistence'.
  - file: [docs/plugins/architecture-internals.md:85](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/docs/plugins/architecture-internals.md#L85)
  - command: `nl -ba docs/plugins/architecture-internals.md | sed -n '82,105p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Discovery cache is process-local and 1s by default:** `src/plugins/discovery.ts` stores discovery results in a module-local `Map` and uses `DEFAULT_DISCOVERY_CACHE_MS = 1000`; there is no disk-backed cache in this path.
  - file: [src/plugins/discovery.ts:62](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/plugins/discovery.ts#L62)

- **Manifest cache is process-local and 1s by default:** `src/plugins/manifest-registry.ts` keeps manifest registries in a module-local cache with `DEFAULT_MANIFEST_CACHE_MS = 1000`, so repeated short-lived CLI invocations still miss cross-process reuse.
  - file: [src/plugins/manifest-registry.ts:160](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/plugins/manifest-registry.ts#L160)

  - file: [src/config/validation.ts:740](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/config/validation.ts#L740)
