## Summary

Keep open. Current `main` still appears to propagate Google Chat DM thread IDs even when `replyToMode` is `"off"`, so the underlying bug is not obsolete. The proposed PR also looks incomplete: its delivery-time guards would drop explicit `[[reply_to_*]]` overrides that core threading intentionally preserves when implicit threading is off.

## Evidence

- **Current main still seeds implicit threading from inbound Google Chat webhooks:** `processMessageWithPipeline` copies `message.thread?.name` into both `ReplyToId` and `ReplyToIdFull` without checking `account.config.replyToMode`, so a DM webhook still carries a thread reference into the shared reply pipeline.
  - file: [extensions/googlechat/src/monitor.ts:227](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/googlechat/src/monitor.ts#L227)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '210,236p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Current main delivers Google Chat replies using the resolved reply target directly:** The Google Chat monitor send path passes `payload.replyToId` straight through as the outbound `thread` for text and media sends, so any implicit reply target seeded from the inbound context will still thread later deliveries.
  - file: [extensions/googlechat/src/monitor.ts:406](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/extensions/googlechat/src/monitor.ts#L406)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '340,446p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Shared core contract preserves explicit replies when implicit threading is off:** Core reply handling distinguishes explicit versus implicit reply targets. `deliver.test.ts` asserts that `replyToMode: "off"` suppresses only the fallback reply target and still preserves explicit payload replies, and `reply-threading.ts` documents that explicit reply tags/directives remain allowed by default when mode is `"off"`.
  - file: [src/infra/outbound/deliver.test.ts:493](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/infra/outbound/deliver.test.ts#L493)

  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **PR discussion already identified the likely regression in this patch:** The Codex review comment on PR #67055 noted that replacing `payload.replyToId` with `account.config.replyToMode === "off" ? undefined : ...` would strip intentional `[[reply_to:*]]` / `[[reply_to_current]]` overrides, forcing flat replies even when the model explicitly requested threading. That aligns with the current shared-core reply contract.
