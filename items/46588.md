## Summary

Keep open. Current `main` still routes Gemini CLI audio auto-detect through a text/file-reading flow (`read_many_files`) instead of a real multimodal audio attachment path or a deterministic explicit failure, so the reported headless unreliability is not clearly fixed on `main`.

## Evidence

- **Gemini CLI audio auto-entry still uses file-reading fallback:** `resolveGeminiCliEntry()` returns a `gemini` CLI config with `--allowed-tools read_many_files` and the prompt `Use read_many_files to read {{MediaPath}} and respond with only the text output.` That matches the issue's reported non-binary `@path`/file-reading path rather than a dedicated audio attachment flow.
