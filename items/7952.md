## Summary

Issue #7952 is already implemented on current `main` and was shipped in `v2026.4.22`: `/hooks/agent` accepts `agentId`, routes to that agent, falls back to the default for unknown IDs, and rewrites agent-scoped session keys to the selected agent instead of nesting prefixes.

## Close Comment

Closing as implemented. Codex review of current `main` found that `/hooks/agent` now accepts optional `agentId` and routes the webhook run to that agent (`docs/gateway/configuration-reference.md:3259`, `src/gateway/hooks.ts:281`, `src/gateway/server-http.ts:664`). The current handler also rebases agent-scoped session keys onto the selected agent instead of nesting prefixes (`src/gateway/hooks.ts:383`, `src/gateway/server-http.ts:707`), which addresses the old `agent:default:agent:other:...` behavior described here. Coverage in `src/gateway/server.hooks.test.ts:178` and `src/gateway/server.hooks.test.ts:506` asserts both explicit `agentId` routing and the session-key rewrite. This implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
