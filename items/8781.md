## Summary

Atlas/ChatGPT-UI routing is not implemented on current `main`, and the current OpenClaw architecture and vision docs place this kind of optional browser/web integration on the external plugin or ClawHub path rather than OpenClaw core.

## Close Comment

Thanks for the report. I reviewed this with Codex against current `main` (`c0a7b6a510758645c5267734b984f4885a605b67`) and I’m closing it as `clawhub` rather than a core OpenClaw issue.

The specific Atlas/ChatGPT-UI routing described here is not implemented on `main`, but more importantly the current architecture already treats both sides of this request as plugin-owned surfaces: the bundled browser tool is explicitly replaceable by another plugin (`docs/tools/browser.md:46`), and `web_search` / `web_fetch` are first-class plugin capabilities (`docs/plugins/architecture.md:33`, `src/plugins/web-provider-public-artifacts.ts:23`). That matches the project scope guidance in `VISION.md`: optional capability should usually ship as plugins (`VISION.md:54`), new skills should go to ClawHub first (`VISION.md:68`), and commercial-service integrations that do not clearly fit the model-provider category are in the “will not merge for now” bucket (`VISION.md:103`).

If someone wants Atlas support, the supported direction is an external plugin / ClawHub package that replaces the browser surface and/or adds a web provider, rather than landing Atlas-specific browser actions in OpenClaw core.
