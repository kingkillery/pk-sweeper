## Summary

Keep open. Current `main` still treats stuck-session detection as warning-only diagnostics with no automatic abort/recovery hook, so there is not enough evidence to close this as fixed or obsolete. The report overstates one point because OpenClaw already has built-in manual abort surfaces, but that only narrows the issue to missing automatic recovery and/or the underlying hang trigger.

## Evidence

- **Heartbeat only warns on stuck sessions:** `startDiagnosticHeartbeat` computes `ageMs` and, when a processing session exceeds the threshold, only calls `logSessionStuck(...)`; there is no abort, cancel, queue-drain, or recovery call in that path.
  - file: [src/logging/diagnostic.ts:415](https://github.com/openclaw/openclaw/blob/734748d4f4fef7436def57f7a7ae3350bba9a091/src/logging/diagnostic.ts#L415)
  - command: `nl -ba src/logging/diagnostic.ts | sed -n '280,490p'`
  - sha: [734748d4f4fe](https://github.com/openclaw/openclaw/commit/734748d4f4fef7436def57f7a7ae3350bba9a091)
- **Stuck-session handler is diagnostics-only:** `logSessionStuck` logs a warning and emits a `session.stuck` diagnostic event, then returns. It does not invoke any runtime recovery behavior.
  - file: [src/logging/diagnostic.ts:296](https://github.com/openclaw/openclaw/blob/734748d4f4fef7436def57f7a7ae3350bba9a091/src/logging/diagnostic.ts#L296)

## Risks / Open Questions

- The specific hang trigger described in the issue may overlap with a provider-specific failure that was addressed elsewhere, so the bug report likely needs reframing around the generic lack of automatic recovery rather than the exact original trigger.
- Because built-in manual abort paths already exist (`chat.abort` / `sessions.abort`), the report's severity and workaround claims should be revalidated on a current build before any eventual fix is scoped.
