## Summary

Keep open. Current `main` still hardcodes a 30s timeout inside `runImagePrompt()`, while the config surface and adjacent media-understanding pipeline both support `tools.media.image.timeoutSeconds`. The PR addresses a real, current mismatch rather than an obsolete report.

## Evidence

- **Current main still ignores image timeout config in image tool:** At `runImagePrompt()`, all three image-description call sites still pass `timeoutMs: 30_000` directly, with no read of `tools.media.image.timeoutSeconds`.
  - file: [src/agents/tools/image-tool.ts:221](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/agents/tools/image-tool.ts#L221)
  - command: `nl -ba src/agents/tools/image-tool.ts | sed -n '180,270p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Adjacent media-understanding path already resolves per-capability timeout from config:** `resolveRunEntryOptions()` uses `resolveTimeoutMs(entry.timeoutSeconds ?? params.config?.timeoutSeconds ?? cfg.tools?.media?.[capability]?.timeoutSeconds, DEFAULT_TIMEOUT_SECONDS[capability])`, so the rest of the media-understanding pipeline already honors the same config contract the PR is trying to wire into the image tool.
  - file: [src/media-understanding/runner.entries.ts:363](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/media-understanding/runner.entries.ts#L363)

  - file: [src/config/schema.help.ts:632](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/config/schema.help.ts#L632)
