## Summary

Keep open. Current main does not expose a readiness-aware `until_ready` queue mode or option. The public config/schema/docs still define queue behavior around the existing modes and `debounceMs`, and the general followup drain still waits on the timer-based debounce before flushing. There is a narrower subagent announce deferral while a parent session remains busy, but that is not the requested operator-selectable `messages.queue` behavior for ordinary queued status/completion messages across chat surfaces.

## Best Possible Solution

Keep the issue open and handle it as a core queue design task. The conservative path is to add a documented `messages.queue` mode or option for readiness-aware aggregation, wire it through config types/schema/help/docs, tie flushing to session/run readiness with a clear starvation cap, and add focused tests for ordinary inbound followups plus async completion/status notices so channel-local serialization cannot bypass the behavior.
