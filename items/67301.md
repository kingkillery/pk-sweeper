## Summary

Current `main` has multiple code paths that explicitly preserve `SessionEntry.label`, including patch, turn/session rewrites, reset, checkpoint branch/restore, and compaction bookkeeping. But I did not find a regression test or other strong evidence that specifically covers the reported user flow of setting a custom label in Control UI, then losing it after compaction or a gateway restart. That is not enough to conservatively auto-close.

## Evidence

- **Custom labels persist when set via `sessions.patch`:** `applySessionsPatchToStore()` only changes `next.label` when `patch.label` is present, and only removes it when the caller explicitly sends `null`. There is no implicit overwrite from derived/session metadata in this path.
  - file: [src/gateway/sessions-patch.ts:219](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/gateway/sessions-patch.ts#L219)
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **Normal session rewrites carry forward the existing label:** The main session assembly path snapshots `entry.label` into `persistedLabel` and writes it back into the next `sessionEntry`, including reset-triggered rollover handling.
  - file: [src/auto-reply/reply/session.ts:487](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/auto-reply/reply/session.ts#L487)
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **Gateway reset preserves label on the replacement session entry:** `performGatewaySessionReset()` copies `currentEntry?.label` and `currentEntry?.displayName` into the new session entry before replacing the old one.
  - file: [src/gateway/session-reset-service.ts:594](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/gateway/session-reset-service.ts#L594)
  - sha: [c7f18a6b9d76](https://github.com/openclaw/openclaw/commit/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac)
- **Compaction restore/branch preserve label; manual compaction only updates counters:** Checkpoint clone logic carries `params.currentEntry.label`, and the manual compaction post-update only mutates timestamps/token counters. I do not see a label clear/overwrite in the compaction handler itself.
  - file: [src/gateway/server-methods/sessions.ts:252](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/gateway/server-methods/sessions.ts#L252)

- **Existing tests cover patch/list label persistence but not compaction/restart label retention:** There is a sessions test proving a patched label remains visible through `sessions.list`/`sessions.resolve`, but a targeted grep for `label` combined with `compaction` or `restart` in session-related tests returned no matches.
  - file: [src/gateway/server.sessions.gateway-server-sessions-a.test.ts:1192](https://github.com/openclaw/openclaw/blob/c7f18a6b9d76a31f4daed5096c0a25b3794c76ac/src/gateway/server.sessions.gateway-server-sessions-a.test.ts#L1192)
