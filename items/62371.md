## Summary

Keep open. Current `main` still silently ignores dropped image files when the browser provides an empty or non-image MIME type, which matches the report, and the linked PR #62392 indicates the same gap was already identified.

## Evidence

- **Drop handler still gates only on MIME type:** `handleDrop` iterates dropped files and immediately skips any file whose `file.type` does not pass `isSupportedChatAttachmentMimeType(...)`, so a dropped image with a missing MIME type is never converted into an attachment.
  - file: [ui/src/ui/views/chat.ts:272](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/ui/src/ui/views/chat.ts#L272)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '272,340p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Supported-type check has no filename fallback:** The helper accepts only strings starting with `image/`; there is no extension-based fallback for blank MIME types.
  - file: [ui/src/ui/chat/attachment-support.ts:3](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/ui/src/ui/chat/attachment-support.ts#L3)
  - command: `nl -ba ui/src/ui/chat/attachment-support.ts | sed -n '1,220p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **UI still advertises drop acceptance:** The chat surface registers `@drop` and `@dragover` on the main section, so the UI can show a valid drop target even when the dropped file will later be discarded by MIME filtering.
  - file: [ui/src/ui/views/chat.ts:1068](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/ui/src/ui/views/chat.ts#L1068)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '1040,1085p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Send pipeline already handles attachments once present:** `handleSendChatMessage` forwards `host.chatAttachments` into `sendChatMessageNow(...)`, so the likely failure point is attachment collection in the UI drop path, not downstream message sending.
  - file: [ui/src/ui/app-chat.ts:328](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/ui/src/ui/app-chat.ts#L328)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '320,400p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Issue timeline already points to the same diagnosis:** The provided GitHub timeline cross-references PR #62392, titled `fix(ui): accept dropped images with missing mime types`, which aligns directly with the current code path and the reported behavior.

## Risks / Open Questions

- No targeted regression test was found for dropped-image attachment handling in the Control UI, so this behavior can regress silently.
- The latest inspected `main` code at `51dd4f288f67ef4df4c7a121f078f508535df1fd` still shows the problematic MIME-only filter, so there is no strong basis to close as already fixed.
