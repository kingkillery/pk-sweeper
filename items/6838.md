## Summary

The request is a real idea, but current `main` treats this as community-plugin territory rather than an OpenClaw core cleanup item: new channel integrations are documented as plugins, community/ClawHub is the canonical distribution path, and the docs already point to an external Vapi bridge.

## Close Comment

Codex review: this looks like a better fit for the community plugin/ClawHub path than for OpenClaw core. Current main explicitly says optional capability should usually ship as plugins and be maintained outside the repo (`VISION.md:54`), and the roadmap guardrails call out commercial integrations and wrapper channels as out of scope for core by default (`VISION.md:101`). The repo already documents how to build channel plugins (`docs/plugins/sdk-channel-plugins.md:11`), treats community plugins/ClawHub as the canonical discovery path (`docs/plugins/community.md:9`), and even showcases an external Vapi bridge today (`docs/start/showcase.md:282`). Closing this as a scope-fit item rather than a missing core feature; if someone wants to pursue Vapi support, the right path is an external OpenClaw plugin/bridge published through the community plugin flow.
