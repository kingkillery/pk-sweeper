## Summary

Keep open. Current main still does not emit `stream: "compaction"` events from the embedded runner’s overflow/timeout recovery path; it only runs compaction hooks there. The PR targets a real remaining gap, but the existing review comments identify contract issues that should be fixed before merge, especially failed end events being treated as completed by at least one consumer.

## Best Possible Solution

Keep #67706 open for maintainer review. The right product/code direction is to add compaction start/end event emission to the embedded runner’s overflow and timeout recovery paths, preserve the canonical compaction event semantics, update or avoid consumers that treat any end event as success, and add targeted runner tests for success and failure cases.

## Evidence

## Risks / Open Questions

- Merging the PR as-is could emit failed recovery `phase: "end"` events that some consumers currently interpret as successful compaction completion.
- The PR’s proposed `completed` calculation should be aligned with the canonical subscribe handler before landing.
- The missing behavior should get focused regression tests in the overflow and timeout recovery tests, not just rely on existing hook coverage.
