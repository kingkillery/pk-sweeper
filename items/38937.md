## Summary

Keep open. Current `main` appears to have fixed the first half of the report by removing bare `[[tts]]` from the tagged-mode system prompt, but the streamed-block handling described in the issue still exists: TTS directives are still parsed on each block payload before delivery, and the accumulated-text retry only happens later as a partial fallback. That is not strong enough evidence to close #38937 as implemented or obsolete.

## Evidence

- **Prompt/parser mismatch was partially addressed:** The tagged-mode prompt no longer instructs the model to emit bare `[[tts]]`; it now says to use `[[tts:key=value]]` directives or a `[[tts:text]]...[[/tts:text]]` block.
  - file: [extensions/speech-core/src/tts.ts:399](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/extensions/speech-core/src/tts.ts#L399)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '387,407p'`
  - sha: [d3dd61f2c5bd](https://github.com/openclaw/openclaw/commit/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2)
- **Directive parser still only recognizes `[[tts:...]]` forms:** `parseTtsDirectives()` fast-exits unless the text contains `[[tts:` and only strips `[[tts:text]]...[[/tts:text]]` plus `[[tts:...]]` directives. It still does not do any cross-block assembly itself.
  - file: [src/tts/directives.ts:65](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/tts/directives.ts#L65)

- **Generic streaming path still parses each block before sending it:** `dispatch-from-config` accumulates block text, but it still runs each block through `maybeApplyTtsToReplyPayload()` and immediately sends that block onward. A tag split across blocks can therefore still leak raw text during streaming.

That mitigates some audio-generation cases but does not prove streamed tagged-mode cleanup is fixed.

- The original issue bundled two bugs; only the prompt mismatch is clearly fixed on `main`, so closing now would likely hide the remaining streamed-block defect.
