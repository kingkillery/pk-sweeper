## Summary

Keep open. Current `main` still uses a generic 8000-byte memory-embedding cap and the Ollama embedding adapter does not supply a model-specific `maxInputTokens`, so the reported `mxbai-embed-large` context-window failure remains plausible on current code. I found no change in this path between release `v2026.4.23` and `main` (`ec8dbc4595583816f71af2568df61dbe7c090e1f`).

## Evidence

- **Memory indexing still batches with a fixed 8000 cap:** `EMBEDDING_BATCH_MAX_TOKENS` is hardcoded to `8000`, and missing chunks are grouped with `buildMemoryEmbeddingBatches(..., EMBEDDING_BATCH_MAX_TOKENS)` before calling the embedding provider.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:45](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/extensions/memory-core/src/memory/manager-embedding-ops.ts#L45)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-ops.ts | sed -n '1,180p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Per-file chunk enforcement still uses the same generic cap:** During indexing, markdown chunks are passed through `enforceEmbeddingMaxInputTokens(this.provider, baseChunks, EMBEDDING_BATCH_MAX_TOKENS)`, so Ollama files are still constrained only by the provider's declared max or the shared 8000 fallback.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:624](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/extensions/memory-core/src/memory/manager-embedding-ops.ts#L624)

- **Remote embedding providers still default to 8192 when they do not declare a limit:** `resolveEmbeddingMaxInputTokens()` returns `8192` for non-local providers unless the provider sets `maxInputTokens` explicitly.
