## Summary

Keep open: current `main` still preserves the `sessions_send` peer-session A2A ping-pong-plus-announce flow that the issue reports as producing duplicate persistent-session artifacts. The linked timeline item appears to have landed only a narrower fix for parent-owned one-shot ACP children, not for ordinary persistent peer sessions.

## Evidence

- **`sessions_send` still has no per-call ping-pong control:** The current tool schema only accepts `sessionKey`, `label`, `agentId`, `message`, and `timeoutSeconds`. There is still no `maxPingPongTurns` or equivalent parameter on the `sessions_send` API itself.
