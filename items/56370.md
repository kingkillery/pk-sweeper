## Summary

Keep open. Current `main` still has the same split routing path the issue describes: implicit `--to` resolution hardcodes the `main` agent for direct-chat session keys, while explicit `--agent <id>` resolves that agent's main session key. Gateway-side canonicalization only remaps `agent:main:*` aliases when there is no configured `main` agent, so a setup with both `main` and `steward` can still route the default path to `main` and the explicit path to `steward`.
