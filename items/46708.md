## Summary

Keep open. Current `main` has substantial Discord approval work, but the cross-surface fallback problem described here is not strongly resolved: Control UI still appears live-event-only for approval prompts, and there is no clear gateway-backed TUI approval client path in the checked-in code/docs.

## Evidence

- **Control UI approval queue is event-driven only:** `ui/src/ui/app-gateway.ts` adds/removes approvals only from `exec.approval.requested` / `exec.approval.resolved` and `plugin.approval.requested` / `plugin.approval.resolved`. On reconnect it just prunes the existing local queue; it does not fetch pending approvals from the gateway.
  - file: [ui/src/ui/app-gateway.ts:124](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/ui/src/ui/app-gateway.ts#L124)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Gateway exposes replay APIs, but Web UI is not using them:** The gateway has durable pending-approval list methods (`exec.approval.list`, `plugin.approval.list`), and the shared native approval runtime explicitly replays them after connect. That makes the missing Web UI replay path a real gap, not an unavailable backend feature.
  - file: [src/infra/exec-approval-channel-runtime.ts:216](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/infra/exec-approval-channel-runtime.ts#L216)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Pending approval replay exists in server methods:** `src/gateway/server-methods/exec-approval.ts` and `src/gateway/server-methods/plugin-approval.ts` both expose `*.approval.list`, and `docs/gateway/protocol.md` documents them as pending-approval lookup/replay APIs.
  - file: [src/gateway/server-methods/exec-approval.ts:97](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/server-methods/exec-approval.ts#L97)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **TUI fallback remains unproven in current main:** `src/infra/exec-approval-surface.ts` treats `tui` as an enabled approval surface, but `rg -n "operator\.approvals|exec\.approval|plugin\.approval" src/tui src/cli` did not find a gateway-backed TUI approval client. `docs/cli/tui.md` only documents local-mode terminal approval prompts, not gateway exec-approval handling.
  - file: [src/infra/exec-approval-surface.ts:48](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/infra/exec-approval-surface.ts#L48)
  - command: `rg -n "operator\.approvals|exec\.approval|plugin\.approval" src/tui src/cli`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Discord-native approvals are improved, but that is only a partial match:** Current Discord code builds persistent approval messages with buttons and only updates/deletes them on resolve/expiry, and native approval handlers replay pending approvals after reconnect. That helps the Discord half of the report, but it does not prove the broader Web UI / TUI fallback issue is fixed.
  - file: [extensions/discord/src/approval-handler.runtime.ts:183](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/discord/src/approval-handler.runtime.ts#L183)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)

## Risks / Open Questions

- Closing this would overstate the state of the fallback surfaces: Discord-native handling has moved forward, but the report also covers Web UI / TUI recovery when approvals are missed or the session reattaches.
- The issue scope mixes at least two behaviors: native Discord approval delivery and cross-surface fallback/replay. Current main only gives strong evidence for the first one.
