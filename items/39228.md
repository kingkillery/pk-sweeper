## Summary

Current main already avoids the reported Docker crash-loop. Docker Compose still passes `--bind lan` on the CLI, but startup now seeds `gateway.controlUi.allowedOrigins` even when `gateway.bind` is unset in config by treating container startup as effective non-loopback bind before runtime config resolution.

## Evidence

- **Docker uses CLI-only bind override:** The compose service launches `node dist/index.js gateway --bind ${OPENCLAW_GATEWAY_BIND:-lan} --port 18789`, so the reported CLI-only bind path is the current Docker path.
  - file: [docker-compose.yml:29](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/docker-compose.yml#L29)
  - command: `nl -ba docker-compose.yml | sed -n '1,220p'`

- **Startup seeder now covers unset bind in containers:** `ensureControlUiAllowedOriginsForNonLoopbackBind` falls back from unset `config.gateway.bind` to `"auto"` when `isContainerEnvironment()` is true, then seeds localhost/127.0.0.1 Control UI origins before startup continues.
  - file: [src/config/gateway-control-ui-origins.ts:46](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/src/config/gateway-control-ui-origins.ts#L46)

## Close Comment

Closing from a Codex review: current `main` already covers the Docker crash-loop described here. `docker-compose.yml:29` still starts the gateway with a CLI-only `--bind lan`, and startup now seeds `gateway.controlUi.allowedOrigins` before runtime config resolution when `gateway.bind` is unset but the process is running in a container (`src/config/gateway-control-ui-origins.ts:46`, `src/gateway/startup-control-ui-origins.ts:8`, `src/gateway/server.impl.ts:321`). The runtime origin guard is still in place (`src/gateway/server-runtime-config.ts:57` and `src/gateway/server-runtime-config.ts:146`), so this startup seeding is the mechanism that prevents the crash-loop. This implementation is already present in the released `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so there is no remaining maintainer action on this issue.
