## Summary

Keep open. Current `main` still intentionally refuses a Docker gateway start when no config stamps `gateway.mode=local`, and the shipped Docker flow still requires a separate onboarding/setup step before `docker compose up`. That means the requested Dockge-style paste-and-run UX is not implemented on `main`.

## Evidence

- **Gateway still hard-blocks missing config:** `getGatewayStartGuardErrors()` returns `Missing config. Run \`openclaw setup\` or set gateway.mode=local (or pass --allow-unconfigured).` when no config exists and `--allow-unconfigured` is absent, matching the issue's reported container exit.
  - file: [src/cli/gateway-cli/run.ts:223](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cli/gateway-cli/run.ts#L223)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Bundled compose file does not bypass the guard:** `docker-compose.yml` starts `openclaw-gateway` with `node dist/index.js gateway --bind ... --port 18789` only; it does not add `--allow-unconfigured` or otherwise preseed `gateway.mode=local`. A plain compose start still depends on existing config.
  - file: [docker-compose.yml:29](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docker-compose.yml#L29)

It explicitly notes that pre-start onboarding and config writes must run through `openclaw-gateway` before the gateway container can start cleanly.
