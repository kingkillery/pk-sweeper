## Summary

Keep open. Current `main` still returns the raw `skills.update` config object, so the underlying secret-leak bug is not implemented on `main`. This PR also has unresolved correctness/test gaps: its review feedback identifies a `SecretRef` handling bug for `apiKey`, and the added test coverage is narrower than the new behavior.

## Evidence

- **Current main still echoes raw skill config from `skills.update`:** `skills.update` builds `current`, writes it, and responds with `config: current` directly. There is no redaction step in the current `main` handler.
  - file: [src/gateway/server-methods/skills.ts:300](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/gateway/server-methods/skills.ts#L300)

  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **`apiKey` is not always a string:** Skill config declares `apiKey?: SecretInput`, and `SecretInput` is `string | SecretRef`. That matches the PR review concern that masking logic must handle object-shaped SecretRefs, not only strings.
  - file: [src/config/types.skills.ts:3](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/config/types.skills.ts#L3)

- **PR review flags a real correctness bug in the proposed patch:** The review on PR commit `34f367ed14972eda4d3cf35cebce0ff91f2e5cb5` points out that the patch calls string methods on `apiKey` even though stored values may be `SecretRef` objects, which would both fail type-checking and corrupt the response shape when no new `apiKey` is supplied.
  - file: [src/gateway/server-methods/skills.ts:347](https://github.com/openclaw/openclaw/blob/34f367ed14972eda4d3cf35cebce0ff91f2e5cb5/src/gateway/server-methods/skills.ts#L347)
