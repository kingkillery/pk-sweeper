## Summary

Keep open. Current `main` still uses UTF-8 decoding in the exact execution paths this PR targets, so the reported Windows codepage bug is not implemented on `main`, and the PR also still has unresolved review concerns about multibyte chunk-boundary decoding.

## Evidence

- **`runExec()` still forces UTF-8 on current main:** `runExec()` still passes `encoding: "utf8"` to `execFileAsync` and returns the decoded strings directly, so raw Windows console bytes are still discarded before any codepage-aware decode could happen.
  - file: [src/process/exec.ts:131](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/process/exec.ts#L131)
  - command: `nl -ba src/process/exec.ts | sed -n '131,160p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Spawn-based callers still decode chunks with `toString("utf8")`:** Both local-shell and QMD process capture still convert each chunk with `toString("utf8")`, matching the bug description in the PR rather than an already-landed fix.
  - file: [src/tui/tui-local-shell.ts:117](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/tui/tui-local-shell.ts#L117)
  - command: `nl -ba src/tui/tui-local-shell.ts | sed -n '117,124p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **QMD path remains unchanged on current main:** `runCliCommand()` in the memory host still appends `data.toString("utf8")` for both stdout and stderr, so this affected surface is also not fixed on `main`.
  - file: [src/memory-host-sdk/host/qmd-process.ts:127](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/memory-host-sdk/host/qmd-process.ts#L127)
  - command: `nl -ba src/memory-host-sdk/host/qmd-process.ts | sed -n '127,137p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **The only codepage-aware decoder on main is still private to `invoke.ts`:** Current `main` defines `parseWindowsCodePage`, `resolveWindowsConsoleEncoding`, and `decodeCapturedOutputBuffer` only inside `src/node-host/invoke.ts`; there is no shared `src/infra/windows-encoding.ts` on `main`, so the PR's proposed extraction is not already present.
  - file: [src/node-host/invoke.ts:108](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/node-host/invoke.ts#L108)
  - command: `rg -n "decodeCapturedOutputBuffer|resolveWindowsConsoleEncoding|parseWindowsCodePage" src/node-host/invoke.ts src/infra/windows-encoding.ts`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Review discussion shows unresolved correctness concerns on the PR itself:** The provided PR review comments include an unresolved point that decoding each `data` chunk independently can still corrupt GBK/Shift_JIS output when multibyte characters straddle chunk boundaries, so this is not a safe conservative close candidate.
  - sha: [48818dce42b2](https://github.com/openclaw/openclaw/commit/48818dce42b23bf65c6fa88157e3dae7f6ab18d6)
