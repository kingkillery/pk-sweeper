## Summary

Keep open. Current `main` still lacks the loader-registration seam this PR adds, so the bundled/flattened gateway `logging.file` problem is not clearly implemented or obsolete on `main`.

## Evidence

- **Current logging config path still relies on fallback require:** `readLoggingConfig()` on current `main` still resolves `../config/config.js` via `createRequire(import.meta.url)` and returns `undefined` on failure; there is no registered runtime loader path for bundled/flattened runtimes.
  - file: [src/logging/config.ts:14](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/logging/config.ts#L14)
  - command: `nl -ba src/logging/config.ts | sed -n '14,32p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Runtime config loader is not wired into logging:** `loadConfig()` and `getRuntimeConfig()` are exported normally, but current `main` does not register any logging-config loader near that runtime config entrypoint.
  - file: [src/config/io.ts:1989](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/config/io.ts#L1989)

## Risks / Open Questions
