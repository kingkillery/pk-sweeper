## Summary

Do not close. Current `main` still does not contain the animation-path changes from the linked fix, so there is not strong evidence that Telegram GIF/looping-video uploads are fully handled on current OpenClaw.

## Evidence

- **Inbound media detection still omits `animation`:** `hasInboundMedia` and `resolveInboundMediaFileId` only check `photo`, `video`, `video_note`, `document`, `audio`, `voice`, and `sticker`. `animation` is absent, so that path is still incomplete for Telegram animation payloads.
  - file: [extensions/telegram/src/bot-handlers.media.ts:16](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/telegram/src/bot-handlers.media.ts#L16)
  - command: `sed -n '1,220p' extensions/telegram/src/bot-handlers.media.ts`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Primary media placeholder still omits `animation`:** `TelegramMediaMessage` excludes `animation`, and `resolveTelegramPrimaryMedia` falls through from `video`/`video_note` to `document`. That means animation messages still cannot produce the intended `<media:video>` classification in current code.
  - file: [extensions/telegram/src/bot/body-helpers.ts:8](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/telegram/src/bot/body-helpers.ts#L8)
  - command: `sed -n '1,140p' extensions/telegram/src/bot/body-helpers.ts`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Download resolver still does not select `msg.animation` as the file ref:** `resolveMediaMetadata` includes `animation.file_name` and `animation.mime_type`, but its `fileRef` chain stops at `photo`, `video`, `video_note`, `document`, `audio`, and `voice`. So an animation payload still will not enter the intended download path via its own file ref.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:89](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/telegram/src/bot/delivery.resolve-media.ts#L89)
  - command: `sed -n '1,260p' extensions/telegram/src/bot/delivery.resolve-media.ts`

- **Current regression coverage does not include the referenced video/animation inbound cases:** The e2e inbound media file only covers photo/file_path scenarios in current `main`; there is no current regression test for animation download or `<media:video>` placeholder handling there.
  - file: [extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts:1](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts#L1)
