## Summary

Keep open. The original Slack system-prompt mismatch still exists on current `main`: generic prompt assembly still treats Slack inline buttons as `inlineButtons`, while object-form Slack `capabilities: { interactiveReplies: true }` never reaches that capability check. Some follow-up Slack fixes are present on `main` (streaming final edits now forward blocks, and reply button action IDs are unique), but they do not resolve the issue this item was opened for.

## Evidence

- **Generic system-prompt gate still keys off `inlinebuttons` only:** `src/agents/system-prompt.ts:625` computes `inlineButtonsEnabled` from `runtimeCapabilitiesLower.has("inlinebuttons")`, and `src/agents/system-prompt.ts:370-373` still tells Slack users to set `${runtimeChannel}.capabilities.inlineButtons` when that flag is false.
  - file: [src/agents/system-prompt.ts:625](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/agents/system-prompt.ts#L625)
