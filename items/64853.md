## Summary

Keep open. Current `main` still sends `openclaw status` and other gateway-backed CLI commands through the same RPC/WebSocket call path, and I found no code or shipped-release evidence that this Windows SIGKILL/hang class was fixed. The report is also corroborated by two later open Windows issues, including one updated on 2026-04-19 saying the problem still reproduced in 2026.4.14.

## Evidence

- **`openclaw status` still reaches gateway RPC on main:** `statusCommand()` calls `resolveStatusRuntimeSnapshot(...)`, and its health path still invokes `resolveStatusGatewayHealth(...)` for gateway-backed status checks. That helper directly calls `callGateway({ method: "health", params: { probe: true } })`. This matches the reported failing surface (`openclaw status`).
  - file: [src/commands/status.command.ts:168](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/commands/status.command.ts#L168)
