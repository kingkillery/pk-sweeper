## Summary

Keep open. Current `main` still appears to reuse the same session identity across gateway execution and CLI embedded fallback, and the newer gateway abort plumbing does not cancel the timed-out gateway run before the CLI fallback window.

## Evidence

- **CLI fallback still calls embedded agent with the same resolved options object:** `agentCliCommand()` builds `localOpts` once, tries the gateway path, and on any error immediately calls `agentCommand(localOpts, ...)` without clearing or rotating session identity first.
  - file: [src/commands/agent-via-gateway.ts:180](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/agent-via-gateway.ts#L180)
  - command: `nl -ba src/commands/agent-via-gateway.ts | sed -n '176,198p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Embedded agent resolution still reuses a fresh stored sessionId:** `resolveSession()` keeps the existing `sessionEntry.sessionId` whenever the session is still fresh, so a fallback embedded run on the same session key will normally target the same transcript/lock owner rather than forcing a fresh session.
  - file: [src/agents/command/session.ts:241](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/agents/command/session.ts#L241)
  - command: `nl -ba src/agents/command/session.ts | sed -n '236,256p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Gateway agent path also reuses the existing sessionId for the requested session key:** When the gateway resolves a session key, it loads the session entry, uses `entry?.sessionId ?? randomUUID()`, stores that back into the session entry, sets `resolvedSessionId = sessionId`, and passes that same `sessionId` into the gateway-run `agentCommandFromIngress` call.
  - file: [src/gateway/server-methods/agent.ts:634](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-methods/agent.ts#L634)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '634,744p'; nl -ba src/gateway/server-methods/agent.ts | sed -n '963,1003p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Client fallback window is still earlier than gateway timeout cleanup:** CLI gateway calls wait `timeoutSeconds + 30s` before erroring, but gateway-side abort expiry is computed as `timeoutMs + 60s` grace and maintenance only aborts entries after `expiresAtMs` is passed. For a 60s agent timeout, the CLI can fall back around 90s while the gateway-run lock is still eligible to live until about 120s plus maintenance sweep cadence.
  - file: [src/commands/agent-via-gateway.ts:107](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/agent-via-gateway.ts#L107)

  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Current regression tests do not cover the reported lock/fresh-session behavior:** `agent-via-gateway.test.ts` still only checks that fallback invokes the embedded agent and preserves the cleanup flag behavior; there is no assertion that fallback rotates to a new sessionId or waits for the gateway lock to clear.
  - file: [src/commands/agent-via-gateway.test.ts:108](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/agent-via-gateway.test.ts#L108)
