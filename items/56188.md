## Summary

Keep open. Current `main` still contains the reported `/skill` regression: `resolveReplyDirectives` only loads workspace skill commands when configured model aliases exist, and `handleInlineActions` treats an empty `skillCommands` array as truthy so it skips the fresh-load fallback. That leaves `/skill <name> <input>` unrewritten when aliases are absent.

## Evidence

- **Directive stage still gates skill command loading on model aliases:** `resolveReplyDirectives` computes `hasConfiguredModelAliases`, sets `rawAliases` to `[]` when none exist, then only loads workspace skill commands when `rawAliases.length > 0`. With no aliases configured, `skillCommands` is returned as `[]`.
  - file: [src/auto-reply/reply/get-reply-directives.ts:224](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/get-reply-directives.ts#L224)
  - command: `nl -ba src/auto-reply/reply/get-reply-directives.ts | sed -n '220,280p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Inline action stage still skips fallback when passed []:** `handleInlineActions` uses `shouldLoadSkillCommands && params.skillCommands ? params.skillCommands : ...`. Because `[]` is truthy, an empty array from the directive stage blocks the fresh workspace load, so `/skill ...` cannot resolve a skill invocation.
  - file: [src/auto-reply/reply/get-reply-inline-actions.ts:227](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/get-reply-inline-actions.ts#L227)

If `skillCommands` stays empty, `resolveSkillCommandInvocation` returns `null` and the raw `/skill ...` text is left untouched.
