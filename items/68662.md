## Summary

Current `main` contains post-report fixes in the exact failure area: `chat.send` runs are now registered so lifecycle failures no longer disappear into a stuck waiting state, and session write-lock timeouts are now typed and explicitly prevented from cascading into model failover across providers. Given the issue discussion also showed the hang was not Ollama-specific and could poison later non-Ollama runs, this is best treated as implemented on `main` rather than still-open core work.

## Evidence

- **Cross-provider symptom from discussion:** The follow-up discussion showed this was not just one Ollama request shape: a commenter reported the same hang across multiple local models, and the reporter later said switching back to Anthropic still left the gateway unresponsive until restart. That points to gateway/session state rather than an Ollama-only transport bug.
- **chat.send now registers active runs:** `chat.send` now calls `registerChatAbortController(...)` and immediately `context.addChatRun(...)` before dispatch, which is the run-link needed for later lifecycle error delivery and abort tracking.
  - file: [src/gateway/server-methods/chat.ts:2241](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/server-methods/chat.ts#L2241)

- **Regression test for stuck waiting / dropped lifecycle error:** The gateway test now asserts that an active `chat.send` run with a chat-run link emits a visible `state: "error"` payload and clears the registry entry, covering the previously stuck waiting/no-active-run failure mode.
  - file: [src/gateway/server-chat.agent-events.test.ts:1329](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/server-chat.agent-events.test.ts#L1329)
