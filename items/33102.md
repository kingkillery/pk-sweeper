## Summary

Keep open. Current `main` still has no config-backed default for TUI reply delivery; `openclaw tui` only reads the explicit `--deliver` flag and defaults it to `false`. A related stuck-spinner failure mode has watchdog mitigation on `main`, but that does not implement the requested `tui.deliver`-style config support.

## Evidence

- **CLI only exposes a flag, not a config-backed default:** `openclaw tui` defines `--deliver` as a boolean option and passes `Boolean(opts.deliver)` into `runTui(...)`; there is no config lookup in the command path.
  - file: [src/cli/tui-cli.ts:19](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/cli/tui-cli.ts#L19)
  - command: `nl -ba src/cli/tui-cli.ts | sed -n '10,60p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **TUI runtime hardcodes default-off from options:** The TUI runtime derives `deliverDefault` from `opts.deliver ?? false`, so the effective default remains off unless the CLI flag is supplied.
  - file: [src/tui/tui.ts:315](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/tui/tui.ts#L315)
