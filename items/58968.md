## Summary

Keep open. Current main still routes persistent Playwright tab listing directly through listPagesViaPlaywright without an operation-level timeout or graceful degradation, and the linked timeout-fix PR is closed rather than clear evidence of a merged fix. Local read commands were blocked by the sandbox before execution, so this review relies on the provided GitHub context plus current-main raw source inspection.

## Evidence

- **Persistent Playwright list path has no timeout wrapper:** createProfileTabOps calls listPagesViaPlaywright({ cdpUrl: profile.cdpUrl }) directly for profiles using persistent Playwright, then maps the returned pages. There is no local timeout, fallback, or catch around that call in the current-main source inspected.
  - file: [src/browser/server-context.tab-ops.ts:1](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/browser/server-context.tab-ops.ts#L1)
  - command: `web raw current-main source inspection`
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **listPagesViaPlaywright still lacks an overall timeout:** listPagesViaPlaywright awaits connectBrowser, getAllPages, pageTargetId, and page.title without an operation-level timeout or Promise.race guard. connectBrowser has connection retry timeouts, but this does not protect the full listPagesViaPlaywright operation from later CDP/Page hangs described in the issue.
  - file: [src/browser/pw-session.ts:14](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/browser/pw-session.ts#L14)
  - command: `web raw current-main source inspection`
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **CDP HTTP helpers do have timeout defaults, making the Playwright path difference material:** fetchJson/fetchCdpChecked use a CDP HTTP request timeout via AbortController, while the persistent Playwright list path bypasses that HTTP helper and calls listPagesViaPlaywright directly.
  - file: [src/browser/cdp.helpers.ts:3](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/browser/cdp.helpers.ts#L3)
  - command: `web raw current-main source inspection`
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **Issue discussion points to an attempted fix, not a landed resolution:** The provided timeline cross-references PR #59043, titled 'fix(browser): add timeout protection to listPagesViaPlaywright', but that PR is closed. With current-main source still showing the timeout gap, the issue should not be closed as implemented.
