## Summary

Keep open. `main` still rejects skill symlinks that resolve into a different configured skills root, so the requested behavior is not already implemented. The proposed PR also appears to create an allowlist regression: a bundled skill loaded through a workspace/managed/extra symlink would retain the non-bundled caller source and bypass `skills.allowBundled`, which matches the unresolved review concern on the PR. ([github.com](https://github.com/openclaw/openclaw/pull/45912))

## Evidence

- **Current main still enforces same-root containment only:** `resolveContainedSkillPath` accepts a candidate only when its real path stays inside the current root's real path, then warns and drops it otherwise. There is no secondary allowlist for other configured roots on `main`.
  - file: [src/agents/skills/workspace.ts:268](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/agents/skills/workspace.ts#L268)

  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Skill source is inherited from the loading root:** `loadSingleSkillDirectory` stamps every loaded skill with `source: params.source` and matching `sourceInfo.source`, so a skill discovered via `openclaw-workspace` stays classified as workspace-origin even if its real path points elsewhere.
  - file: [src/agents/skills/local-loader.ts:77](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/agents/skills/local-loader.ts#L77)

  - file: [src/agents/skills/config.ts:62](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/agents/skills/config.ts#L62)
