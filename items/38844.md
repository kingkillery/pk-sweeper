## Summary

Current main fixes the misleading browser-control timeout wrapping, but the broader stale-ref upload/click flake is still plausible because the browser tool still exposes split `upload` and `act` calls and does not automatically force the atomic upload+click path the issue asks for.

## Evidence

- **Timeout misreport path fixed:** Dispatcher-path failures now preserve the original in-route error instead of always being rewritten as `Can't reach the OpenClaw browser control service`; local-dispatch errors go through `enhanceDispatcherPathError(...)` while only real fetch/reachability failures use the generic service wrapper.
  - file: [extensions/browser/src/browser/client-fetch.ts:310](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/extensions/browser/src/browser/client-fetch.ts#L310)
  - command: `nl -ba extensions/browser/src/browser/client-fetch.ts | sed -n '280,322p'`
- **Regression test for preserved dispatcher context:** Test asserts a dispatcher rejection like `Chrome CDP handshake timeout` keeps that message and explicitly omits `Can't reach the OpenClaw browser control service`.
  - file: [extensions/browser/src/browser/client-fetch.loopback-auth.test.ts:200](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/extensions/browser/src/browser/client-fetch.loopback-auth.test.ts#L200)

  - file: [CHANGELOG.md:3190](https://github.com/openclaw/openclaw/blob/c971c58fc7592344539ac3cf87c7c4229be9978e/CHANGELOG.md#L3190)
