## Summary

Current `main` has browser WebSocket auto-reconnect and some reconnect-state recovery, but the Control UI still hard-switches back to the login gate whenever `state.connected` is false. That means the issue's core UX request — staying in-place and transparently resuming after a short gateway restart instead of bouncing to the login screen — is not implemented well enough to conservatively close.

## Evidence

- **Browser client does auto-reconnect with backoff:** `GatewayBrowserClient` schedules reconnects after recoverable socket closes, starting at 800 ms and backing off up to 15 s.
  - file: [ui/src/ui/gateway.ts:334](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/gateway.ts#L334)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '327,368p'`
- **Control UI reconnect path reuses existing host state:** `connectGateway()` swaps in a new client and, on reconnect hello, reloads data and resumes deferred state instead of forcing a fresh login flow.
  - file: [ui/src/ui/app-gateway.ts:253](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/app-gateway.ts#L253)
  - command: `nl -ba ui/src/ui/app-gateway.ts | sed -n '253,336p'`
- **UI still gates the entire dashboard on connection state:** `renderApp()` returns the login gate whenever `!state.connected`, so any transient restart still replaces the active Control UI with the login screen during the reconnect window.
  - file: [ui/src/ui/app-render.ts:663](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/app-render.ts#L663)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '655,667p'`
- **Login gate is the full disconnected surface:** The disconnected view is the full login form, not an in-place reconnect overlay or preserved chat surface.
  - file: [ui/src/ui/views/login-gate.ts:9](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/views/login-gate.ts#L9)
  - command: `nl -ba ui/src/ui/views/login-gate.ts | sed -n '1,130p'`
- **Tests cover reconnect internals, not staying on the same UI surface:** Current tests verify reconnect/backoff and shutdown-message handling, which shows partial progress, but there is no evidence of the requested 'stay on same chat/context instead of login page' UX being implemented.
  - file: [ui/src/ui/gateway.node.test.ts:369](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/gateway.node.test.ts#L369)
  - command: `nl -ba ui/src/ui/gateway.node.test.ts | sed -n '369,466p'`

## Risks / Open Questions

- This review is inspection-based; I did not run a browser reproduction because the environment lacks `pnpm`/Node tooling.
- The existing reconnect work may reduce manual refresh/login churn, but the explicit login-gate render on disconnect means closing as `implemented_on_main` would be too aggressive.
