## Summary

Keep open. Current `main` does not implement the session binding described by PR #56420: Claude permission requests are stored without a session key, approval replies are accepted solely by `request_id`, and the MCP schema/handler do not carry `session_key`. The reported spoofing vector therefore still appears real on `main`.

## Evidence

- **Pending Claude permissions on main have no session binding:** `handleClaudePermissionRequest` stores only `toolName`, `description`, and `inputPreview` in `pendingClaudePermissions`; there is no `sessionKey` field on the stored request.
  - file: [src/mcp/channel-bridge.ts:273](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/mcp/channel-bridge.ts#L273)
  - command: `nl -ba src/mcp/channel-bridge.ts | sed -n '272,290p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Approval replies are matched only by request id:** `handleSessionMessageEvent` parses `yes/no <id>` and accepts the reply whenever `pendingClaudePermissions.has(requestId)` is true. It does not compare the inbound `sessionKey` to the originating session before sending `notifications/claude/channel/permission`.
  - file: [src/mcp/channel-bridge.ts:455](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/mcp/channel-bridge.ts#L455)
