## Summary

Keep open. Current `main` still does not implement general transient-retry behavior for OAuth refresh, so the underlying bug is not clearly fixed, but this PR is also stale against the newer `oauth-manager` refresh architecture and the review discussion identifies real regression risks in the proposed approach.

## Evidence

- **Current main still performs a single upstream OAuth refresh call:** `refreshOAuthCredential` delegates to plugin refresh/chutes refresh and then calls `getOAuthApiKey(...)` once, returning `result?.newCredentials ?? null`; there is no retry wrapper at this adapter layer.
  - file: [src/agents/auth-profiles/oauth.ts:134](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/agents/auth-profiles/oauth.ts#L134)
  - command: `nl -ba src/agents/auth-profiles/oauth.ts | sed -n '134,157p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Current locked refresh path has timeout/serialization, not transient retry:** `doRefreshOAuthTokenWithLock` serializes refreshes with file locks and wraps the refresh call in `withRefreshCallTimeout(...)`, but it invokes `adapter.refreshCredential(...)` only once inside that timeout block; no `retryAsync` or equivalent retry predicate is present in the current manager path.
  - file: [src/agents/auth-profiles/oauth-manager.ts:315](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/agents/auth-profiles/oauth-manager.ts#L315)

- **Current tests encode single-attempt fallback behavior for a known Codex compatibility path:** The regression test for the OpenAI Codex fallback path still asserts exactly one `getOAuthApiKey` call before falling back to reloaded credentials, so a naive retry patch would conflict with current expected behavior.
