## Summary

Keep open. Current main disproves the specific `max reconnect attempts (0)` configuration claim and routes Discord DM/guild inbound through the same listener stack, but this is still a live Discord-only runtime report and the reporter said it remained reproducible after follow-up. Repo inspection alone is not strong enough to conservatively close it as fixed or non-reproducible.

## Evidence

- **Supported Node range vs reported runtime mismatch:** OpenClaw currently supports Node `>=22.14.0`. The issue body reported Node `25.8.0`, while the pasted stack path showed `v24.14.0`, so the original environment details were internally inconsistent but not enough by themselves to close the bug.

  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Current Discord gateway reconnect config is not zero:** Current main sets Discord gateway reconnects to `maxAttempts: 50`, and the same setting is present in the `v2026.4.23` release tag. That contradicts the issue's implied current-config diagnosis of reconnect attempts being configured as `0`.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:378](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/discord/src/monitor/gateway-plugin.ts#L378)

That makes the reported 'guilds still work but DMs silently die after reconnect exhaustion' symptom hard to prove from code inspection alone, but not impossible enough to close.

## Risks / Open Questions

- Live Discord gateway behavior cannot be reproduced from this checkout without real bot credentials and a Discord environment, so repo-only inspection cannot safely prove the bug is gone.
- The issue's environment details are inconsistent, which weakens the original diagnosis, but it does not prove the runtime symptom was imaginary.
- Current code disproves the specific `maxAttempts=0` explanation more strongly than it disproves the reported DM-only outage, so closing would be premature.
