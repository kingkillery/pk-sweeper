## Summary

Keep open. Current `main` still treats scope-limited probe failures as unreachable in the shared `status`/`status --all` path, even though `gateway-status` now handles the same condition as reachable-but-degraded. The PR's user-facing bug is therefore not implemented on `main` yet; it likely needs rebasing onto the newer gateway-status helper rather than closing.

## Evidence

- **Shared status scan still uses strict `ok === true` reachability:** `resolveGatewayProbeSnapshot` on current `main` sets `gatewayReachable` only when `gatewayProbe?.ok === true`, so a probe that connected successfully but failed a later RPC with `missing scope: operator.read` still becomes unreachable in the main status flow.
  - file: [src/commands/status.scan.shared.ts:131](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/commands/status.scan.shared.ts#L131)
  - command: `nl -ba src/commands/status.scan.shared.ts | sed -n '121,136p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **`status --all` still renders that case as unreachable:** The status formatting code still prints `unreachable (<error>)` whenever `gatewayReachable` is false, with no degraded/scope-limited branch in this path.
  - file: [src/commands/status-all/format.ts:362](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/commands/status-all/format.ts#L362)
