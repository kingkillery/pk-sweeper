## Summary

Keep open. Current `main` still does not merge `models.providers.<id>.models[]` into `loadModelCatalog()`, so the reported catalog/capability gap remains plausible and consistent with the docs. The submitted PR also has an unresolved provider-alias dedupe correctness risk, so there is not strong evidence for a conservative close.

## Evidence

- **Current model catalog loader still omits configured providers:** `loadModelCatalog()` on current `main` reads Pi registry entries, then provider-plugin supplemental entries, and returns the sorted list. There is no merge step over `cfg.models?.providers`, so config-declared custom provider models are not added here.

  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Gateway image-support resolution depends on a catalog row:** `resolveGatewayModelSupportsImages()` loads the gateway model catalog, looks for a matching provider/model entry, and only returns `true` from catalog metadata or a few provider-specific shims. If a configured custom provider never lands in the catalog, image support falls back to `false`.
