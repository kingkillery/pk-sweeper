## Summary

Keep open. Current main partially mitigates the original owner escalation by deriving `senderIsOwner` from owner/non-owner loopback bearer tokens, but it does not implement the PR's central per-backend token-bound scope model. Session key, account id, and message channel still come from mutable request headers, and current tests explicitly assert that behavior.

## Best Possible Solution

Keep the item open or explicitly supersede it with a narrower maintainer-owned fix for #64993. The best code path is to rebase onto current main, preserve the shipped owner/non-owner mitigation, add a per-backend token registry that binds `sessionKey`, `accountId`, `messageProvider`, and `senderIsOwner`, stop emitting scope headers/env vars, invalidate scoped tool-cache entries on token revocation, and update the loopback tests so header spoofing cannot alter any scoped field.
