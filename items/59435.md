## Summary

Keep open. The issue is recent, concrete, and plausible against current main: the update service-refresh path still appears to reinstall the LaunchAgent from the current process environment rather than first preserving the existing plist's OPENCLAW_CONFIG_PATH/port for custom multi-instance services. I did not find enough source-backed evidence to close as implemented or obsolete.

## Evidence

- **Concrete recent report:** Issue #59435 was opened 2026-04-02 by a non-maintainer, has no protected labels, and includes specific macOS LaunchAgent multi-instance paths, labels, ports, expected behavior, actual behavior, and workaround. It is not old enough for stale_insufficient_info.
- **Update refresh uses current process env:** Current main's update flow refreshes the gateway service by running `gateway install --force` and passes `env: resolveServiceRefreshEnv(process.env, ...)`. I did not find evidence in this path that it reads the existing LaunchAgent plist's OPENCLAW_CONFIG_PATH before overwriting it.
  - file: [src/cli/update-cli/update-command.ts:100](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/cli/update-cli/update-command.ts#L100)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **Refresh targets env-resolved service:** The update command decides whether to refresh/restart via `resolveGatewayService().isLoaded({ env: process.env })`, then prepares a restart script and sets the refresh flag. That still makes correctness depend on the caller's current env/label for custom multi-instance setups.
  - file: [src/cli/update-cli/update-command.ts:1290](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/cli/update-cli/update-command.ts#L1290)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **LaunchAgent install overwrites plist from passed args:** The launchd installer writes a new plist from the provided `programArguments` and `environment`, then activates it. That matches the reported failure mode if update invokes install with the wrong OPENCLAW_CONFIG_PATH/port environment.
  - file: [src/daemon/launchd.ts:560](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/src/daemon/launchd.ts#L560)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **Docs support profile-based multi-gateway, not this custom-label edge:** Public docs say multiple gateways require isolated OPENCLAW_CONFIG_PATH, OPENCLAW_STATE_DIR, workspace, and port, and recommend `openclaw --profile <name> gateway install` for per-profile service names. The issue describes custom labels and one default `ai.openclaw.gateway` plist being regenerated with another instance's config, which is a narrower edge not clearly closed by the docs.
  - file: [docs/gateway/multiple-gateways.md:1](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/docs/gateway/multiple-gateways.md#L1)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
