## Summary

Keep open. Current `main` still intentionally filters and suppresses `qwen3.6-plus` on Qwen Coding Plan endpoints, and the issue’s linked fix PR is still open rather than shipped.

## Evidence

- **Current main hard-blocks Coding Plan base URLs:** `isQwen36PlusSupportedBaseUrl` returns false for Qwen Coding Plan hosts, so the bundled catalog removes `qwen3.6-plus` whenever the base URL is `coding.dashscope.aliyuncs.com` or `coding-intl.dashscope.aliyuncs.com`.
  - file: [extensions/qwen/models.ts:111](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/qwen/models.ts#L111)
  - command: `nl -ba extensions/qwen/models.ts | sed -n '108,140p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Runtime config normalization strips the model:** The Qwen plugin’s `normalizeConfig` hook deletes `qwen3.6-plus` from provider config on Coding Plan endpoints before requests are made.
  - file: [extensions/qwen/index.ts:168](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/qwen/index.ts#L168)
  - command: `nl -ba extensions/qwen/index.ts | sed -n '160,196p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Runtime suppression still returns an explicit unsupported-model error:** `suppressBuiltInModel` suppresses `qwen3.6-plus` for `qwen`/`modelstudio` providers on Coding Plan endpoints with the message that the model is not supported there.
  - file: [extensions/qwen/index.ts:177](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/qwen/index.ts#L177)
  - command: `nl -ba extensions/qwen/index.ts | sed -n '160,196p'`

  - file: [extensions/qwen/provider-catalog.test.ts:10](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/qwen/provider-catalog.test.ts#L10)
