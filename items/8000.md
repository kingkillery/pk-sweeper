## Summary

Current main intentionally routes niche messaging integrations to the external channel-plugin path. An Agora relay integration matches the documented ClawHub/npm plugin model better than OpenClaw core or a new bundled plugin.

## Close Comment

Codex review: closing this as a better fit for the external plugin path, not OpenClaw core. Reviewed against `main` at `cec34821754ad935c780805f3116f99449d46ac2`.

The repo’s current scope is explicit here: `VISION.md:54`-`59` says optional capability should usually ship as plugins, be hosted and maintained in the author’s own repository, and that the bar for adding optional plugins to core is intentionally high. The docs also already define the intended path for this exact kind of work: `docs/plugins/building-plugins.md:16`-`19` says plugins should be published to ClawHub or npm instead of being added to this repo, and `docs/plugins/sdk-channel-plugins.md:11`-`24` is the dedicated guide for building messaging channel plugins. Current main also supports external/custom channel integrations directly (`src/channels/plugins/types.adapters.ts:266`-`270`, `src/channels/plugins/read-only.test.ts:603`-`635`).

Because this request is a niche Agora relay integration tied to a specific relay/UI stack, it fits the ClawHub/npm channel-plugin route much better than a bundled OpenClaw feature. If an external Agora plugin ships and later shows strong demand for first-party bundling, that can be reconsidered with concrete adoption evidence.
