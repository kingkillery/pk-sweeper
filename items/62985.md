## Summary

Keep open. Current `main` still rejects `channels.telegram.accounts.<id>.agentId`; Telegram multi-account routing now works through top-level `bindings[]`, and I did not find a compatibility or `doctor --fix` migration for older account-level `agentId` configs. The linked discussion and open PR suggest this is an acknowledged upgrade regression, not a stale or incoherent report.

## Evidence

- **Telegram account schema is strict and does not include `agentId`:** `TelegramAccountSchemaBase` ends with `.strict()` and the declared account fields do not include `agentId`, so `channels.telegram.accounts.<id>.agentId` is still an additional-property validation error on current `main`. The runtime type `TelegramAccountConfig` also omits `agentId`.
  - file: [src/config/zod-schema.providers-core.ts:207](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/config/zod-schema.providers-core.ts#L207)
  - command: `nl -ba src/config/zod-schema.providers-core.ts | sed -n '180,380p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Current Telegram routing reads top-level bindings, not account-local `agentId`:** `resolveBindingAccount` looks only at `cfg.bindings[].match.channel/accountId` plus top-level `binding.agentId`, and `listTelegramAccountIds`/default account selection incorporate those bindings. There is no account-config `agentId` path here.
  - file: [extensions/telegram/src/account-selection.ts:40](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/extensions/telegram/src/account-selection.ts#L40)

- **Docs now direct users to `bindings[].match.accountId` for multi-account agent routing:** The public config docs say multi-account channel setups should use `bindings[].match.accountId` to route each account to a different agent, which matches the workaround in the issue comments.
  - file: [docs/gateway/config-channels.md:739](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/docs/gateway/config-channels.md#L739)

A commenter reported a working workaround on 2026.4.8 by removing per-account `agentId` and adding top-level `bindings[]`, and the reporter replied that they had been unable to find that documentation after upgrading.
