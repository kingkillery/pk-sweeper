## Summary

Keep open. Current `main` has bounded per-run retry exhaustion, auth-profile cooldown/failover, and overflow-specific fresh-session recovery, but it does not implement the requested gateway-side session circuit breaker: there is no persisted unhealthy-session state, no consecutive-failure/session-signature tracking, no config for session breaker thresholds/cooldowns, and no operator-visible pause/resume workflow for unhealthy sessions.

## Evidence

- **Per-run retry cap exists, but it only blocks the current run:** The embedded runner stops after `MAX_RUN_LOOP_ITERATIONS` and returns a `retry_limit` error with `livenessState: "blocked"`, telling the user to retry or use `/new`. This is a single-run guard, not a persisted gateway session breaker across repeated session failures.

- **Retry-limit handling does not mark sessions unhealthy or pause future sends:** `handleRetryLimitExhaustion()` emits an error payload and `error.kind = "retry_limit"`, but it does not store per-session failure counters, a breaker state, cooldown, or auto-rotation metadata.
