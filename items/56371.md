## Summary

Keep open. Current `main` still implements WebUI message deletion as a client-side hide backed by browser localStorage, and the gateway session protocol still has no per-message delete/mutate operation for chat history persistence.

## Evidence

- **Delete state is browser-local only:** `DeletedMessages` stores hidden message keys under the `openclaw:deleted:` localStorage prefix and only reads/writes that browser-local set; there is no backend call here.
  - file: [ui/src/ui/chat/deleted-messages.ts:3](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/chat/deleted-messages.ts#L3)
  - command: `nl -ba ui/src/ui/chat/deleted-messages.ts | sed -n '1,120p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Chat delete action only hides the rendered group:** The chat view loads `DeletedMessages` for the session, skips rendering when `deleted.has(item.key)`, and the delete handler only calls `deleted.delete(item.key)` plus `requestUpdate()`. No RPC or persistence beyond local state occurs.
  - file: [ui/src/ui/views/chat.ts:142](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/views/chat.ts#L142)

- **UI wording presents this as a real delete:** The rendered control is labeled `Delete message` / `Delete this message?`, which matches the reporter's expectation of an actual deletion rather than a temporary hide.
  - file: [ui/src/ui/chat/grouped-render.ts:524](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/ui/src/ui/chat/grouped-render.ts#L524)
