## Summary

Current `main` already rejects non-Telegram path-like targets in the outbound messaging-tool path before any Telegram API send, so this PR is obsolete. Its proposed low-level throw in `stripTelegramInternalPrefixes` would also change the current normalize-helper contract from `undefined`-on-invalid-input to throwing.

## Evidence

## Close Comment

Closing from a Codex review because current `main` already handles this in the outbound messaging-tool path. Telegram targets are normalized through `extensions/telegram/src/normalize.ts:9` and only accepted when they resolve to a numeric chat ID or valid Telegram handle/link via `extensions/telegram/src/targets.ts:47`; otherwise outbound resolution stops with `Unknown target "<raw>" for Telegram` before any send call (`src/infra/outbound/message-action-runner.ts:297`, `src/infra/outbound/target-resolver.ts:444`, `src/infra/outbound/target-errors.ts:17`). That makes the requested guard obsolete on `main`.

This PR’s approach would also change the current helper contract by making `normalizeTelegramChatId()` / `normalizeTelegramLookupTarget()` throw for some invalid inputs instead of returning `undefined`, which does not match the current test expectations in `extensions/telegram/src/targets.test.ts:94`. Reviewed against `main` at `eb6e1245ac881566a7f31adc7d94d9e3c721a6ac`; the current implementation path is present in commit `04066d246abc4e13a9507e1a93e12be75ae41753`.
