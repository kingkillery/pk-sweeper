## Summary

Keep open. Current `main` still leaves `openclaw message send --channel telegram` on the old message-action path that does not apply `channels.telegram.defaultAccount` when `--account` is omitted, so a named default Telegram account can still be skipped for outgoing sends.

## Evidence

- **CLI send does not pass a resolved default account into the runner:** `messageCommand` calls `runMessageAction` with `cfg`, `action`, `params`, and gateway metadata, but does not set `defaultAccountId` from the channel plugin.
  - file: [src/commands/message.ts:68](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/commands/message.ts#L68)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Message-action runner only uses explicit accountId, injected defaultAccountId, or agent bindings:** When resolving outbound account context, `runMessageAction` reads `params.accountId` or `input.defaultAccountId`, then falls back to an agent binding. There is no plugin-level `defaultAccountId` lookup on this path.
  - file: [src/infra/outbound/message-action-runner.ts:849](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/infra/outbound/message-action-runner.ts#L849)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Telegram send action still resolves token from the omitted accountId directly:** `handleTelegramAction` uses `resolveTelegramToken(cfg, { accountId })` for `sendMessage`, and forwards `accountId: accountId ?? undefined`; if the caller omitted `--account`, this path never rewrites to the configured named default account first.
  - file: [extensions/telegram/src/action-runtime.ts:357](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/telegram/src/action-runtime.ts#L357)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Telegram token resolution ignores channels.telegram.defaultAccount when accountId is omitted:** `resolveTelegramToken` normalizes an omitted accountId to the literal default account and looks up `accounts.default`; it does not consult `channels.telegram.defaultAccount` before token lookup.
  - file: [extensions/telegram/src/token.ts:102](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/telegram/src/token.ts#L102)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Telegram gating already honors defaultAccount, so the remaining gap is specifically token/action routing:** `createTelegramActionGate` resolves omitted account context via `resolveDefaultTelegramAccountId`, and the test suite explicitly checks that omitted action-gate account context uses configured `defaultAccount: "work"`. That matches the issue comment narrowing: gating/default selection was fixed separately, but the send/action token path is still inconsistent.
  - file: [extensions/telegram/src/accounts.ts:94](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/telegram/src/accounts.ts#L94)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Docs still promise outbound default-account routing when accountId is omitted:** Channel routing docs say `channels.<channel>.defaultAccount` chooses which account is used when an outbound path does not specify `accountId`, and Telegram docs repeat that `channels.telegram.defaultAccount` makes default routing explicit in multi-account setups.
  - file: [docs/channels/channel-routing.md:18](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/docs/channels/channel-routing.md#L18)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
