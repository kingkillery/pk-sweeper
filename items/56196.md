## Summary

Current `main` still contains the exact two code paths this PR changes, so the reported `/skill` no-alias bug does not appear to be implemented on `main` and should remain open.

## Evidence

- **Directive resolution still suppresses skill command loading when no model aliases exist:** `resolveReplyDirectives` only loads workspace skill commands when `rawAliases.length > 0`. If `cfg.agents.defaults.models.*.alias` is unset, `rawAliases` becomes `[]`, so `/skill` commands are never loaded upstream.
  - file: [src/auto-reply/reply/get-reply-directives.ts:250](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/get-reply-directives.ts#L250)
  - command: `nl -ba src/auto-reply/reply/get-reply-directives.ts | sed -n '236,266p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Inline action fallback still treats empty preloaded array as loaded:** `handleInlineActions` still branches on `params.skillCommands` truthiness rather than non-empty length. An empty upstream array therefore blocks the fresh `listSkillCommandsForWorkspace(...)` load and leaves skill invocation resolution with no commands.
  - file: [src/auto-reply/reply/get-reply-inline-actions.ts:234](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/auto-reply/reply/get-reply-inline-actions.ts#L234)
