## Summary

Current `main` already classifies transport failures into the timeout/failover path and has tests for cross-provider fallback, but I did not find evidence that it short-circuits repeated provider transport retries or suppresses repeated user-visible connection-error churn. The specific UX regression in #62848 is therefore still plausible and not clearly implemented on `main`.

## Risks / Open Questions

- `pnpm` is unavailable in this environment, so I could not run a live reproduction against `main`; this review is based on current code, docs, and tests.
- The reported 20-24s delay can still happen if the provider SDK spends that time in short internal transport retries before OpenClaw receives the final error.
- I did not find code or tests specifically proving suppression of repeated connection-error output on TUI/Telegram surfaces during provider transport failure.
