## Summary

Keep open. Current `main` has adjacent Docker sandbox work (`openclaw-sandbox-common` and Docker-gateway sandbox bootstrap), but the requested Sysbox/runtime architecture is still not implemented: sandboxing remains Docker-out-of-Docker via `docker.sock`, there is no `agents.defaults.sandbox.docker.runtime` config surface, and the sandbox container creation path never passes a Docker `--runtime` flag.

## Evidence

- **Sandbox Docker config rejects unknown runtime keys:** `SandboxDockerSchema` is strict and only defines fields such as `image`, `network`, `user`, `binds`, and hardening toggles. There is no `runtime` field for `agents.defaults.sandbox.docker.*`, so optional runtime selection is not implemented in current config parsing.
  - file: [src/config/zod-schema.agent-runtime.ts:102](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/zod-schema.agent-runtime.ts#L102)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Docs still describe Docker socket DooD:** The sandbox docs say the Docker backend runs locally via the Docker daemon socket at `/var/run/docker.sock` and explicitly document Docker-out-of-Docker constraints for containerized gateway deployments.
  - file: [docs/gateway/sandboxing.md:80](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/gateway/sandboxing.md#L80)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Docker setup still mounts docker.sock for sandboxing:** The Docker setup flow verifies the Docker CLI, then generates `docker-compose.sandbox.yml` with `${DOCKER_SOCKET_PATH}:/var/run/docker.sock` before enabling sandbox mode. That is the current supported Docker-gateway sandbox path, not Sysbox host-runtime integration.
  - file: [scripts/docker/setup.sh:565](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/scripts/docker/setup.sh#L565)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Sandbox container creation has no Docker runtime flag:** `buildSandboxCreateArgs()` assembles `docker create` arguments for labels, tmpfs, network, user, env, caps, limits, and binds, and `createSandboxContainer()` uses those args directly. The code never appends `--runtime`, so per-container runtime selection such as `sysbox-runc` is not wired up.
  - file: [src/agents/sandbox/docker.ts:348](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/sandbox/docker.ts#L348)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Repository-wide Sysbox search is empty:** A repository search for `sysbox` / `sysbox-runc` returned no matches in tracked code or docs on current `main`. That matches the local code evidence that Sysbox-specific support has not landed.
  - command: `rg -n "sysbox|sysbox-runc" src docs scripts Dockerfile Dockerfile.sandbox-common docker-compose.yml`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Adjacent sandbox-common work exists, but it is different scope:** Current `main` includes the `openclaw-sandbox-common:bookworm-slim` helper image path, which matches the related context about `sandbox-common` being completed separately from Sysbox/runtime architecture changes.
  - file: [scripts/sandbox-common-setup.sh:5](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/scripts/sandbox-common-setup.sh#L5)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
