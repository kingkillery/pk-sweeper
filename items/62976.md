## Summary

Current main narrows part of the report, but it does not fully resolve it. Runtime/plugin register failures are now isolated as per-plugin diagnostics, and startup can restore a last-known-good full config snapshot, but an existing third-party plugin entry with schema-invalid config is still a fatal config-validation error and Doctor still has no targeted repair/quarantine flow for that case.

## Evidence

- **Invalid plugin config is still fatal at config validation time:** Configured or enabled plugins are schema-validated, and any plugin config validation error is pushed into `issues`; if any issues exist, config loading returns `ok: false`. That still makes a bad third-party plugin entry block a valid config snapshot.
  - file: [src/config/validation.ts:1153](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/config/validation.ts#L1153)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Startup recovery is whole-config restore, not per-plugin quarantine:** Gateway startup first tries restoring the entire config from last-known-good or stripping a non-JSON prefix, then immediately re-asserts that the resulting snapshot is valid. There is no plugin-specific disable/quarantine path here.
  - file: [src/gateway/server-startup-config.ts:71](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-startup-config.ts#L71)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Doctor runs against invalid config, but its repair sequence does not include invalid plugin-entry cleanup:** Doctor preflight explicitly keeps running on invalid config using the source snapshot, but the repair sequence only wires channel repairs, bundled load-path repair, stale missing-plugin cleanup, allowlist/open-policy fixes, and a few legacy repairs. There is no repair step for present-but-schema-invalid third-party plugin entries.
  - file: [src/commands/doctor-config-preflight.ts:94](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/doctor-config-preflight.ts#L94)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Doctor stale-plugin cleanup only handles missing plugin ids:** The stale-plugin repair only removes ids from `plugins.allow` and `plugins.entries` when the plugin is no longer discoverable. It does not touch a plugin that still exists but has incompatible schema/runtime/config.
  - file: [src/commands/doctor/shared/stale-plugin-config.ts:120](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/doctor/shared/stale-plugin-config.ts#L120)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Runtime/register failures are isolated now:** Plugin load and register exceptions are caught, rolled back, and recorded as per-plugin diagnostics instead of aborting the whole plugin registry load. That means the `runtime init error` part of the issue is at least partially outdated.
  - file: [src/plugins/loader.ts:2434](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/plugins/loader.ts#L2434)
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
