## Summary

Current `main` has substantial Control UI i18n infrastructure, including `ja-JP` locale registration, English fallback, a generated Japanese bundle, and a language picker. But the request is not clearly fully implemented: important core UX strings are still hardcoded in English outside the translation map, so the issue remains a valid enhancement rather than a conservative close.

## Evidence

- **Japanese locale is wired into the Control UI registry:** `ui/src/i18n/lib/registry.ts` includes `ja-JP` in `LAZY_LOCALES`, lazy-loads `../locales/ja-JP.ts`, and maps navigator languages starting with `ja` to `ja-JP`.
  - file: [ui/src/i18n/lib/registry.ts:19](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/ui/src/i18n/lib/registry.ts#L19)

  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **English fallback exists:** `ui/src/i18n/lib/translate.ts` loads English by default and explicitly falls back to English when a key is missing in the active non-English locale.
  - file: [ui/src/i18n/lib/translate.ts:18](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/ui/src/i18n/lib/translate.ts#L18)

## Risks / Open Questions

- I did not audit every UI surface, so there may be additional untranslated strings beyond the concrete `cron.ts` examples.
- The current implementation appears to be newer than the latest listed release (`v2026.4.22`), so even the existing Japanese support may not be shipped in a stable release yet.
