## Summary

Keep open. Current `main` does not implement per-user or per-partition isolation for `memory-lancedb`: it uses one configured LanceDB path, one hard-coded `memories` table, and unfiltered vector search results. Docs recommend separate gateways for hostile multi-user boundaries, but that guidance does not fix the plugin-level isolation gap described in the issue.

## Evidence

- **Single shared LanceDB table and unfiltered search:** `memory-lancedb` hard-codes `TABLE_NAME = "memories"`, creates rows with `id/text/vector/importance/category/createdAt`, and searches with `table.vectorSearch(...).limit(limit)` without any partition or user filter.
  - file: [extensions/memory-lancedb/index.ts:54](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-lancedb/index.ts#L54)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Plugin config exposes only one dbPath:** The parsed config has a single `dbPath` plus embedding/capture settings. There is no `userId`, `partition`, workspace-aware override, or similar isolation field in the schema or UI hints.
  - file: [extensions/memory-lancedb/config.ts:157](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-lancedb/config.ts#L157)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **dbPath resolution is global, not workspace-aware:** `register()` resolves `cfg.dbPath` through `api.resolvePath`, and plugin API wiring currently maps that helper to `resolveUserPath(input)`, which expands user/home paths but is not tied to the active agent workspace or user partition.
  - file: [src/plugins/registry.ts:1427](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/plugins/registry.ts#L1427)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **OpenClaw docs support separate workspaces per agent:** Agent config docs show first-class separate workspaces such as `~/.openclaw/workspace-home` and `~/.openclaw/workspace-work`, which makes the plugin's single global DB path a real cross-workspace isolation gap.
  - file: [docs/gateway/config-agents.md:965](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/gateway/config-agents.md#L965)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Security docs recommend separate gateways, not per-user auth inside one gateway:** Security guidance says one user/trust boundary per gateway is the supported posture and that DM session isolation is only a messaging-context boundary. That explains current scope, but it does not implement the requested memory partitioning in `memory-lancedb`.
  - file: [docs/gateway/security/index.md:21](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/gateway/security/index.md#L21)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
