## Summary

Current `main` appears to have fixed the comment-thread root cause (`is_from_me` outbound rows being re-ingested), and that fix was already shipped in `v2026.4.23`. But the issue body's original request is narrower and still unimplemented: the iMessage monitor does not persist a restart cursor or pass any `since-rowid`/`start` parameter on `watch.subscribe`, so there is not enough evidence to conservatively close the issue as fully implemented.

## Risks / Open Questions

- If `imsg watch` really can replay backlog after restart, the absence of persisted `rowid`/cursor state means duplicate inbound processing may still happen even though the self-loop amplification path is now reduced.
- The issue now mixes two behaviors: the comment-thread `is_from_me` bug appears fixed, but the original restart/backlog replay request remains unverified and likely still open.
