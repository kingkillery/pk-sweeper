## Summary

Keep open. Current `main` contains a Gateway shutdown hardening for lingering WebSocket clients, but the available evidence only proves bounded shutdown/restart behavior, not that the reported runtime `CLOSE_WAIT`/`FIN_WAIT_2` accumulation on Windows/Node 24 is fixed. The issue is also recent and has a follow-up comment reporting the symptom on a newer build, so the bar for closure is not met.

## Evidence

- **Current code hardens shutdown, not steady-state connection lifecycle:** `createGatewayCloseHandler` closes tracked clients, then bounds `wss.close()` with a grace window and force-terminates lingering websocket clients. That addresses stuck shutdown/restart behavior, but it is only in the shutdown path.
  - file: [src/gateway/server-close.ts:209](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-close.ts#L209)
- **Regression coverage is shutdown-only:** The tests added for this area verify that shutdown terminates lingering websocket clients and continues when websocket close hangs without tracked clients. They do not reproduce or prove a fix for long-running runtime accumulation of `CLOSE_WAIT`/`FIN_WAIT_2` sockets.
  - file: [src/gateway/server-close.test.ts:100](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-close.test.ts#L100)
- **Per-connection runtime handling still centers on close/error/handshake timeout:** The websocket connection handler defines a `close()` helper, reacts to socket `error` and `close`, and times out pre-auth handshakes. I did not find runtime ping/pong or idle-socket reap logic here that would strongly disprove the report's claim about lingering half-closed connections over time.
  - file: [src/gateway/server/ws-connection.ts:265](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server/ws-connection.ts#L265)
- **Gateway 'keepalive' is application-level tick broadcast:** The maintenance loop's keepalive is a periodic `tick` event broadcast, not a transport-level websocket ping/heartbeat implementation.
  - file: [src/gateway/server-maintenance.ts:61](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-maintenance.ts#L61)
- **A related fix already shipped, but it is narrower than this report:** The changelog for `2026.4.5` says: `Gateway/shutdown: bound websocket-server shutdown even when no tracked clients remain, so gateway restarts stop hanging until the watchdog kills the process.` That matches restart/shutdown hangs, not necessarily runtime socket leakage.
  - file: [CHANGELOG.md:1429](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/CHANGELOG.md#L1429)
- **Issue is recent and has follow-up evidence:** Issue #56215 was opened on 2026-03-28 and updated on 2026-04-06 with a comment stating the reporter still sees `CLOSE_WAIT` behavior on a newer version. That is too recent and too specific to treat as stale or obsolete without stronger contrary proof.

## Risks / Open Questions

- Closing this now could hide a live Gateway stability bug that the current shutdown hardening does not actually fix.
- The report specifically mentions Windows + Node 24 behavior, and I did not find reproduction evidence or tests on current `main` that rule that environment out.
