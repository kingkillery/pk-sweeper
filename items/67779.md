## Summary

Keep #67779 open. Current main has general WebChat final-event/history-reload and optimistic-message safeguards, but the reported non-default agents use mixed-case session keys (`agent:DB_Dude:main`, `agent:ViewDude:main`). The UI still stores the raw `?session=` value and filters live `chat`/`session.message` events by exact string equality, while the gateway canonicalizes session keys to lowercase before registering runs and broadcasting events. That leaves the specific reported path still plausible: history loads by raw key can succeed after refresh, but live events for the canonical key are ignored.

## Best Possible Solution

Keep the issue open and fix the WebChat session-key canonicalization path. A conservative fix would normalize or replace the active UI session key with the gateway canonical key before live event filtering, or have `chat.send`/session APIs return the canonical key and update the host state. Add regression coverage for a mixed-case direct URL such as `?session=agent:DB_Dude:main` verifying that live `chat` final events and `session.message` reloads update the active WebChat without refresh.
