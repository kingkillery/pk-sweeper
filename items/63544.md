## Summary

Keep open. Current `main` still normalizes echo-cache text with CRLF folding plus `.trim()` only, so a leading NUL-prefixed echoed iMessage body would miss the text-key match the report describes. The 4-second text TTL is also still intentional current behavior, and the latest release has no touched-surface change for this path.

## Evidence

- **Echo cache still does not strip NUL prefixes:** `normalizeEchoTextKey()` only replaces CRLF and trims whitespace before building the cache key; there is no `\0` removal in the current implementation.
  - file: [extensions/imessage/src/monitor/echo-cache.ts:26](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/imessage/src/monitor/echo-cache.ts#L26)
