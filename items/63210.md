## Summary

Keep open. Current `main` still lacks the requested main-session truncation recovery: one OpenAI Responses/WebSocket path derives stop reason from content shape and then collapses non-tool calls to `"stop"`, the embedded runner only treats `"error"` as a terminal recovery case for visible-output turns, and there is no main-session config or guard for `onOutputTruncation`/tool-parameter completeness validation. The issue discussion also contains a concrete confirmation of truncated tool-call arguments in a recent build.

## Evidence

- **OpenAI Responses conversion still overwrites provider stop reason from content shape:** `buildAssistantMessageFromResponse()` computes `hasToolCalls` and sets `stopReason` to `"toolUse"` or `"stop"` solely from message content. There is no provider stop-reason input or preserved `_providerStopReason` on this path.
  - file: [src/agents/openai-ws-message-conversion.ts:596](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/openai-ws-message-conversion.ts#L596)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **WebSocket completion path still drops `length` on final event:** When a websocket response completes, the stream emits `reason` as `"toolUse"` for tool calls and `"stop"` otherwise. That logic discards any upstream `"length"` reason before the final event is pushed.
  - file: [src/agents/openai-ws-stream.ts:1175](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/openai-ws-stream.ts#L1175)

- **Embedded-runner incomplete-turn recovery still keys off `error`, not `length`:** The main-session incomplete-turn heuristics only classify zero-visible-text `toolUse` as incomplete, and for visible-output turns they return early unless `stopReason === "error"`. A `"length"` stop with visible text therefore falls through without the warning/recovery behavior requested here.
