## Summary

Keep open. Current `main` has several mitigations around session-lock teardown and stale-lock recovery, but I did not find strong evidence that the exact reported regression is fixed: there is no targeted coverage for the same-PID, no-gateway-restart, second-idle-timeout path, and the issue was cross-linked to another open report describing a similar stuck-running/session-lock symptom.

## Evidence

- **Attempt teardown is supposed to release the session lock:** The embedded runner acquires a session write lock at attempt start and, in the main `finally` block, always routes teardown through `cleanupEmbeddedAttemptResources(...)` with the held `sessionLock`.
