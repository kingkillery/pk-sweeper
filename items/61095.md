## Summary

Keep open. Current `main` still routes exec host environments through the shared host-env sanitizer without any special handling for `OPENCLAW_SERVICE_*`, and the gateway credential path still changes behavior when `OPENCLAW_SERVICE_KIND=gateway` is present. I did not find evidence on `main` that the reported env-leak regression has been fixed.

## Evidence

- **Exec host env still uses shared sanitizer:** The exec tool builds `inheritedBaseEnv` from `process.env` and, for non-sandbox host execution, passes it through `sanitizeHostExecEnvWithDiagnostics`; the resulting env is then used for gateway exec runs.
  - file: [src/agents/bash-tools.exec.ts:1536](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/bash-tools.exec.ts#L1536)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '1500,1715p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Shared sanitizer only drops policy-blocked inherited keys:** `sanitizeHostExecEnvWithDiagnostics` removes inherited keys only when `isDangerousHostInheritedEnvVarName(key)` returns true, then returns `markOpenClawExecEnv(merged)`. There is no explicit stripping of `OPENCLAW_SERVICE_*` or `OPENCLAW_SYSTEMD_UNIT` here.
  - file: [src/infra/host-env-security.ts:209](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/infra/host-env-security.ts#L209)
  - command: `nl -ba src/infra/host-env-security.ts | sed -n '209,239p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Current host env policy does not block the reported service markers:** A direct check of `src/infra/host-env-security-policy.json` showed `OPENCLAW_SERVICE_MARKER`, `OPENCLAW_SERVICE_KIND`, `OPENCLAW_SERVICE_VERSION`, and `OPENCLAW_SYSTEMD_UNIT` are all absent from the blocked key sets, so the shared sanitizer would preserve them when inherited from the gateway process.
