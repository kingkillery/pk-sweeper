## Summary

Keep open. Current `main` has configurable tool-loop detection, but the relevant guards are still opt-in and explicitly disabled by default, so the reported identical valid-tool loop behavior remains plausible unless a deployment turned `tools.loopDetection.enabled` on.

## Risks / Open Questions

- No live repro was executed because this review was constrained to read-only, no-artifact inspection; conclusion is based on current code, docs, and tests.
- A deployment that already enables `tools.loopDetection` globally or per-agent may not exhibit the reported behavior, so severity depends on runtime config not included in the issue.
- Existing guardrails are weaker than the report's requested hard-stop semantics for generic repeated calls, so partial prior implementation should not be treated as a full fix.
