## Summary

Keep open. Current `main` only partially addresses the request: `memorySearch.provider = "local"` now defaults to an auto-downloaded GGUF model, but the setup is still not zero-config because `node-llama-cpp` remains a separate optional dependency and the docs/doctor flow still require manual native-build steps.

## Evidence

- **Local provider auto-downloads the default GGUF, but only after `node-llama-cpp` loads:** `DEFAULT_LOCAL_MODEL` points at the Hugging Face GGUF, and `createLocalEmbeddingProvider()` resolves that model through `node-llama-cpp`'s `resolveModelFile()` when `provider: "local"` is used.
  - file: [src/memory-host-sdk/host/embeddings.ts:20](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/memory-host-sdk/host/embeddings.ts#L20)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Docs still say local embeddings require manual native-build setup:** The memory config reference says the default model is auto-downloaded, but immediately states: `Requires native build: pnpm approve-builds then pnpm rebuild node-llama-cpp.`
  - file: [docs/reference/memory-config.md:203](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/reference/memory-config.md#L203)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Top-level memory docs still describe `provider: "local"` as requiring `node-llama-cpp`:** The memory-search docs say: `For local embeddings with no API key, use provider: "local" (requires node-llama-cpp).`
  - file: [docs/concepts/memory-search.md:32](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/concepts/memory-search.md#L32)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Package metadata keeps `node-llama-cpp` optional instead of bundled/automatic:** `package.json` declares `node-llama-cpp` under `peerDependencies` and marks it optional in `peerDependenciesMeta`, so OpenClaw does not own this as a guaranteed bundled runtime dependency.
  - file: [package.json:1614](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/package.json#L1614)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Changelog explicitly says normal installs stopped auto-installing `node-llama-cpp`:** The changelog notes: `Windows/install: stop auto-installing node-llama-cpp during normal npm CLI installs...` That is the opposite of the issue's zero-config request.
  - file: [CHANGELOG.md:2668](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/CHANGELOG.md#L2668)

  - file: [extensions/memory-core/src/memory/provider-adapters.ts:48](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/memory-core/src/memory/provider-adapters.ts#L48)

## Risks / Open Questions

- The feature is partially implemented because the model file itself can auto-download, so the issue may look superficially resolved unless the native dependency/install path is checked too.
- I did not run an end-to-end install or reproduction because this review had to remain read-only, but the code, docs, package metadata, and changelog all align that zero-config local setup is not fully shipped.
