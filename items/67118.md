## Summary

Keep open. Current `main` still runs cron isolated `agentTurn` through `runWithModelFallback(...)` without a result classifier, then inspects `runResult.meta?.error` only after the fallback loop has already accepted that run result. I did not find code or tests on `main` proving the reported meta.error-to-fallback gap is fixed for cron isolated runs.

## Evidence

- **Cron executor does not classify returned embedded results before fallback accepts them:** `createCronPromptExecutor` calls `runWithModelFallback(...)` with `cfg`, `provider`, `model`, `runId`, `agentDir`, `fallbacksOverride`, and `run`, but no `classifyResult`. The embedded branch awaits `runEmbeddedPiAgent(...)` and returns its result directly; the fallback result is then accepted and stored.
  - file: [src/cron/isolated-agent/run-executor.ts:110](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/cron/isolated-agent/run-executor.ts#L110)
  - command: `nl -ba src/cron/isolated-agent/run-executor.ts | sed -n '105,220p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Model fallback already has a hook for non-throw terminal results:** `runWithModelFallback` explicitly supports `classifyResult?: ModelFallbackResultClassifier<T>` and passes it into `runFallbackAttempt(...)`, where classified non-throw results are converted into errors for fallback progression.
  - file: [src/agents/model-fallback.ts:725](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/agents/model-fallback.ts#L725)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '721,876p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Cron inspects `meta.error` only after the fallback run has finished:** After `executor.getState()`, the cron path feeds `runResult.meta?.error` into `resolveCronPayloadOutcome(...)` for interim/output handling. That is downstream of the model-fallback decision, so it does not advance to another candidate when a returned run result carries `meta.error`.
  - file: [src/cron/isolated-agent/run-executor.ts:348](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/cron/isolated-agent/run-executor.ts#L348)
  - command: `nl -ba src/cron/isolated-agent/run-executor.ts | sed -n '338,365p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Embedded result classification exists elsewhere but is not wired into cron isolated execution:** `classifyEmbeddedPiRunResultForModelFallback(...)` exists as the repo's mechanism for turning returned embedded terminal results into fallback classifications, but `run-executor.ts` does not reference it. The timeline also links a related external fork PR (`EronFan/openclaw-fork#11`), which is consistent with this gap still being open on upstream `main`.
  - file: [src/agents/pi-embedded-runner/result-fallback-classifier.ts:79](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/agents/pi-embedded-runner/result-fallback-classifier.ts#L79)
