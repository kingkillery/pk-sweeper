## Summary

Keep open. The issue has a plausible current-main failure mode: large active session transcripts are still read synchronously on hot reply/history paths, and the current maintenance features look like mitigations rather than a demonstrated fix for an oversized active `agent:main:main` transcript freezing gateway/channel work.

## Evidence

- **Full transcript still parsed synchronously:** `readSessionMessages()` still does `fs.readFileSync(filePath, "utf-8").split(...)` and `JSON.parse` over every line, so a very large `.jsonl` transcript can still block the event loop when this helper is used.
  - file: [src/gateway/session-utils.fs.ts:93](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/session-utils.fs.ts#L93)
  - command: `nl -ba src/gateway/session-utils.fs.ts | sed -n '90,140p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Hot reply path still falls back to full transcript scan:** Preflight compaction on live runs still falls back to `estimatePromptTokensFromSessionTranscript()`, which calls `readSessionMessages()` when persisted token counts are missing/stale. That keeps the large-transcript parse on a real agent-run path, not just an admin/debug path.
  - file: [src/auto-reply/reply/agent-runner-memory.ts:413](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/agent-runner-memory.ts#L413)
  - command: `nl -ba src/auto-reply/reply/agent-runner-memory.ts | sed -n '310,335p' && nl -ba src/auto-reply/reply/agent-runner-memory.ts | sed -n '403,418p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Maintenance does not prove active huge transcript is fixed:** Disk-budget cleanup first removes archive or unreferenced transcript files, then skips the `activeSessionKey` entirely while evicting entries. That means maintenance is not evidence that a single oversized active main transcript is automatically bounded away.
  - file: [src/config/sessions/disk-budget.ts:258](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/config/sessions/disk-budget.ts#L258)

- **Related docs describe cleanup as operator action/workaround:** Current docs expose `openclaw sessions cleanup` and `--active-key` protection, which is useful mitigation, but that is not the same as proof that update/startup hangs from a large active transcript are resolved in core.
  - file: [docs/cli/sessions.md:56](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/cli/sessions.md#L56)

That aligns with the remaining synchronous whole-file transcript parsing on main.
