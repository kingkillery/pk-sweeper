## Summary

Current `main` has an opt-in generic tool-loop detector, but it does not implement the issue's requested exec-runtime same-file reread normalization. The shipped guard is disabled by default, hashes full tool params, and only special-cases polling-style `process`/`command_status` loops rather than read-only exec commands like varying `sed`/`nl` file rereads.

## Evidence

- **Loop detection is present but disabled by default:** User-facing docs describe `tools.loopDetection` as an optional guardrail and state that it is disabled unless `enabled: true` is configured.
  - file: [docs/tools/loop-detection.md:10](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/tools/loop-detection.md#L10)

That would not collapse different `sed`/`nl` windows on the same file into one reread signature.
  - file: [src/agents/tool-loop-detection.ts:428](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tool-loop-detection.ts#L428)

  - file: [src/agents/tool-loop-detection.ts:155](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tool-loop-detection.ts#L155)

The e2e tests cover repeated identical `read` calls and alternating `read`/`list` ping-pong, which overlaps the problem space but does not prove same-file exec reread normalization.

## Risks / Open Questions

- There is existing overlap with the generic tool-loop detector, so future work should decide whether to extend that system or add a narrower exec-command normalizer to avoid duplicate guardrails.
- Because the current guard is opt-in and exact-hash based, closing this as already implemented would overstate coverage for the specific same-file reread incident described in the issue.
