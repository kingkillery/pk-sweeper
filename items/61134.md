## Summary

Keep open. Current `main` does not contain this fix: `waitForDiscordGatewayStop()` still calls `gateway.disconnect()` directly from the abort/reject paths with no catch, the lifecycle does not pass a runtime/logger into that helper, and the existing tests do not cover a synchronous throw from `disconnect()`. The PR discussion also shows the author already addressed the review feedback by separating expected Carbon throws from unexpected errors and adding coverage for both.

## Evidence

- **Abort and reject paths still call disconnect without a catch:** On current `main`, `waitForDiscordGatewayStop()` invokes `gateway?.disconnect?.()` directly inside both `finishResolve()` and `finishReject()`. A synchronous throw from `disconnect()` would escape those callback paths before the promise settles; there is no local catch/suppression in this helper.
