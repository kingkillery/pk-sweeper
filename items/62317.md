## Summary

Current `main` still appears vulnerable to the reported phantom-provider grouping. `loadModelCatalog()` accepts registry entries as-is, and `/model status` renders every distinct `entry.provider` group with `auth: missing` when no configured auth exists. I did not find a merged main-branch fix or a regression test covering inferred vendor-provider rows for aggregator IDs like `openrouter/google/...`.

## Evidence

- **Catalog loader still trusts registry provider values:** `loadModelCatalog()` iterates every `ModelRegistry` entry, keeps `entry.provider` verbatim, and pushes it into the catalog. The only later dedupe is for supplemental plugin rows by exact `provider::id`; there is no filtering step for inferred native-provider rows that duplicate an aggregator-owned model.
  - file: [src/agents/model-catalog.ts:139](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/model-catalog.ts#L139)

  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **`/model status` renders every provider bucket, including missing-auth ones:** Status output groups the picker catalog by normalized `entry.provider`, then always prints a `[provider] ... auth: ...` header for each bucket. If the catalog contains both `openrouter/google/...` and an inferred `google/...`, the latter would still render as its own `[google] ... auth: missing` group.
