## Summary

Keep open. Current `main` still defines the advanced-tab replay/daily-log bucket in terms of `groundedCount`, and the PR's one-line badge remap would make the waiting/promoted badges disagree with the staged section semantics without adding the requested regression coverage.

## Evidence

- **Current waiting/promoted badge logic uses grounded replay for the daily-log label:** `describeWaitingEntryOrigin()` returns `originDailyLog` when `groundedCount > 0`, while the same view's staged section is built from `groundedEntries = shortTermEntries.filter((entry) => entry.groundedCount > 0)` and hard-codes the same `originDailyLog` badge for that bucket. Changing only the helper to `dailyCount > 0` would split the meaning of the same label inside one screen.
