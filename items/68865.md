## Summary

Keep open. Current `main` still lacks application-level Feishu WebSocket reconnection, and the PR discussion shows an unresolved P1 in the proposed implementation rather than an already-landed or obsolete change.

## Evidence

- **Current main still uses a single fire-and-forget WS start:** `monitorWebSocket` on current `main` creates one `WSClient`, stores it, and calls `void wsClient.start({ eventDispatcher })` inside a Promise that only resolves on abort or synchronous throw. There is no outer reconnect loop, backoff, or retry path after runtime disconnects / SDK retry exhaustion.
