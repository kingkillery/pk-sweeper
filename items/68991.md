## Summary

Keep open. Current `main` still persists the runtime provider/model/context into session state, so an automatic fallback can still become the next-turn selected model across the touched paths. The PR addresses a real unfixed bug and adds coverage that does not exist on `main`.

## Evidence

- **Session persistence helper still overwrites selection with runtime values:** `persistSessionUsageUpdate` only accepts `modelUsed`/`providerUsed`/`contextTokensUsed`, and both update branches write those runtime fields back into `sessionEntry.modelProvider`, `sessionEntry.model`, and `sessionEntry.contextTokens`. There is no separate persisted-selection path on current `main`.
  - file: [src/auto-reply/reply/session-usage.ts:71](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/session-usage.ts#L71)
  - command: `nl -ba src/auto-reply/reply/session-usage.ts | sed -n '71,220p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Auto-reply path passes only runtime model/provider into persistence:** `runReplyAgent` computes `contextTokensUsed` from `providerUsed`/`modelUsed` and calls `persistRunSessionUsage` with only those runtime fields. It does not pass the originally selected provider/model for next-turn persistence.
  - file: [src/auto-reply/reply/agent-runner.ts:1333](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/agent-runner.ts#L1333)

Those paths still make fallback runtime sticky on current `main`.
