## Summary

Keep open. Current `main` still does not wire the TUI to Gateway session transcript events, so messages appended from the web UI or another channel will not show live in `openclaw tui`; they only appear after a history reload/reconnect.

## Evidence

- **TUI ignores session transcript events:** The TUI event loop handles only `chat`, `chat.side_result`, and `agent`. It does not handle `session.message`, `session.tool`, or `sessions.changed` events.
  - file: [src/tui/tui.ts:1034](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/src/tui/tui.ts#L1034)
  - command: `nl -ba src/tui/tui.ts | sed -n '1028,1115p'`
  - sha: [84fc00afdacb](https://github.com/openclaw/openclaw/commit/84fc00afdacbbb67e39fc1341242eedad09243df)
- **TUI never subscribes to session live-update feeds:** On connect, the TUI refreshes agents and loads history, but there is no call to `sessions.subscribe` or `sessions.messages.subscribe`, so it would not receive transcript-driven live updates from other clients.
  - file: [src/tui/tui.ts:1046](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/src/tui/tui.ts#L1046)
  - command: `nl -ba src/tui/tui.ts | sed -n '1028,1115p'`
  - sha: [84fc00afdacb](https://github.com/openclaw/openclaw/commit/84fc00afdacbbb67e39fc1341242eedad09243df)
- **Gateway already exposes the missing live-update API:** Gateway methods and events include `sessions.subscribe`, `sessions.messages.subscribe`, `session.message`, and `sessions.changed`, and the session handlers register those subscriptions.
  - file: [src/gateway/server-methods-list.ts:77](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/src/gateway/server-methods-list.ts#L77)
  - command: `nl -ba src/gateway/server-methods-list.ts | sed -n '70,160p'`
  - sha: [84fc00afdacb](https://github.com/openclaw/openclaw/commit/84fc00afdacbbb67e39fc1341242eedad09243df)
- **Transcript updates are delivered only to subscribed clients:** Gateway tests show appended transcript messages emit `session.message` only to clients that called `sessions.subscribe`, which matches the reported symptom when TUI is not subscribed.
  - file: [src/gateway/session-message-events.test.ts:192](https://github.com/openclaw/openclaw/blob/84fc00afdacbbb67e39fc1341242eedad09243df/src/gateway/session-message-events.test.ts#L192)
  - command: `nl -ba src/gateway/session-message-events.test.ts | sed -n '181,260p'`
  - sha: [84fc00afdacb](https://github.com/openclaw/openclaw/commit/84fc00afdacbbb67e39fc1341242eedad09243df)

## Risks / Open Questions

- I did not run a live reproduction because this review was read-only and `pnpm` is unavailable in the checkout environment, but the code-path gap is direct and sufficient.
- The symptom assumes the web UI/other channel is writing into the same session key the TUI is viewing; if session keys differ, that would be a separate usage issue.
