## Summary

The request is a useful automation pattern, but current main treats this kind of optional scheduling/gating logic as plugin or hook-pack territory rather than OpenClaw core. Core already provides the scheduler, wake endpoints, and plugin/hook surfaces needed to package heartbeat gatekeepers externally, and VISION explicitly says optional capability should usually ship as plugins and new skills should go to ClawHub first.

## Close Comment

Closing this as a Codex review because the current main codebase points to plugin/ClawHub scope rather than OpenClaw core for this feature.

The scope anchor is `VISION.md:52-70`: core stays lean, optional capability should usually ship as plugins, and new skills should go to ClawHub first. The runtime already exposes the primitives your design needs without adding a new core heartbeat config surface: cron can enqueue main-session wakes and isolated runs (`docs/automation/cron-jobs.md:34-245`), `openclaw system event --mode now` is the documented immediate wake path (`docs/cli/system.md:24-41`), and OpenClaw supports standalone/plugin-managed hooks plus installable hook packs (`docs/automation/hooks.md:9-148`). Current plugin surfaces also allow hooks, commands, services, and network handlers (`docs/plugins/architecture.md:53-86`, `docs/plugins/architecture.md:455-456`, `src/plugins/types.ts:1971-2009`).

Given that, a reusable heartbeat gatekeeper looks like a good ClawHub plugin or hook pack, not a core scheduler feature. If someone ships it externally, it can reuse the existing wake/event surfaces instead of adding more core heartbeat semantics.
