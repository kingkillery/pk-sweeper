## Summary

Current `main` does not implement the phantom-provider filtering described by PR #62323, and the PR discussion surfaced unresolved correctness risks in the proposed approach. This is not a conservative close candidate.

## Evidence

- **Current model catalog still accepts all discovered registry entries:** On current `main` (`51dd4f288f67ef4df4c7a121f078f508535df1fd`), `loadModelCatalog()` reads `registry.getAll()` and pushes each entry after only string normalization and built-in suppression. There is no logic that detects or removes entries split out from aggregator model IDs such as `openrouter` models with slashed IDs.
  - file: [src/agents/model-catalog.ts:139](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/model-catalog.ts#L139)
  - command: `nl -ba src/agents/model-catalog.ts | sed -n '1,260p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Mainline tests do not cover the reported phantom-provider case:** The current `model-catalog` tests cover cache behavior, stale built-ins, supplemental dedupe, and provider-alias lookup, but there is no test for filtering a phantom provider entry inferred from an aggregator model ID prefix. The PR adds that coverage because it is absent on `main` today.
  - file: [src/agents/model-catalog.test.ts:341](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/model-catalog.test.ts#L341)
