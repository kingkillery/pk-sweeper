## Summary

Close as `clawhub`: the request is coherent and useful, but current `main` positions this kind of workspace-file integrity monitoring as extension/plugin territory rather than OpenClaw core.

## Risks / Open Questions

- If maintainers later decide signed workspace integrity should be a default security posture rather than optional automation, this could deserve a core issue instead of a ClawHub plugin/hook pack.
- There is no single built-in 'workspace integrity' feature on `main`; the extension path would need to combine hook/plugin logic with cron or other automation.

## Close Comment

Closing as `clawhub` after a Codex review of current `main` (`cec3482`). This is a useful idea, but it fits OpenClaw's extension surfaces better than core. `VISION.md:52` says core stays lean and optional capability should usually ship as plugins. The needed integration points already exist: `docs/automation/hooks.md:34` documents `agent:bootstrap` and `gateway:startup`, `src/agents/bootstrap-hooks.ts:7` passes the workspace bootstrap files into that hook path, and `src/hooks/internal-hook-types.ts:3` gives hooks a `messages` array for user-visible alerts. For periodic verification, `docs/automation/cron-jobs.md:10` already provides Gateway-native scheduling and chat/webhook delivery. Since the request is specifically about operator-owned workspace files like `AGENTS.md` and `SOUL.md` (`docs/concepts/agent-workspace.md:62`), this is better handled as a ClawHub plugin or hook pack than as OpenClaw core.
