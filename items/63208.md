## Summary

Keep open. Current `main` still treats `memory-wiki` page groups as flat directories: compile, query, and status each use a single `fs.readdir(..., { withFileTypes: true })` over `sources/`, `entities/`, `concepts/`, `syntheses/`, and `reports/` and only accept direct child `.md` files. That matches the issue report that subdirectories under `sources/` are ignored, so the request is not implemented on `main` and remains actionable.

## Evidence

- **Compile scan is non-recursive:** `collectMarkdownFiles()` reads one directory with `fs.readdir`, filters direct child `.md` files, and maps them as `path.join(relativeDir, entry.name)`. It does not descend into nested folders before `readPageSummaries()` compiles pages.
  - file: [extensions/memory-wiki/src/compile.ts:235](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-wiki/src/compile.ts#L235)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Query path assumes flat page groups too:** `listWikiMarkdownFiles()` in the query path repeats the same pattern: one `readdir` per top-level wiki directory, then only direct child `.md` files excluding `index.md`. Nested source pages would also be invisible to `wiki_search` / `wiki_get` fallback reads.
  - file: [extensions/memory-wiki/src/query.ts:97](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-wiki/src/query.ts#L97)
