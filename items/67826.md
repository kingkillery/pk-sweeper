## Summary

Keep PR #67826 open. Current main does not contain the defensive null/non-string hardening proposed by the PR, but the PR likely needs maintainer rework before merge because current Telegram setup now treats allowFrom as numeric sender IDs only and the public setup/resolver contracts still pass string arrays.

## Best Possible Solution

Keep the PR open, but ask for a current-main rework: preserve the numeric-only allowFrom setup contract, add focused malformed-input regression coverage only for reachable or intentionally supported boundaries, and avoid widening SDK resolver/input types unless maintainers explicitly want that public contract change.
