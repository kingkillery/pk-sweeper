## Summary

Keep open. Current main still appears to write explicit custom-provider entries with `models` but no resolved auth into `models.json`, so the first reported bug is not clearly fixed. The second symptom is less conclusive as a bug: current code and docs intentionally treat `agents.defaults.models` as a session-override allowlist, while `openclaw models list --all` shows a broader catalog/discovery view.

## Evidence

- **Unresolved custom providers are still preserved during apiKey normalization:** `resolveMissingProviderApiKey()` returns the provider unchanged when it has `models` but no configured/env/profile apiKey, so explicit custom providers can remain in-memory without auth instead of being filtered out.
  - file: [src/agents/models-config.providers.secret-helpers.ts:279](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/models-config.providers.secret-helpers.ts#L279)
  - command: `nl -ba src/agents/models-config.providers.secret-helpers.ts | sed -n '255,320p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **models.json planner writes final provider map without dropping unresolved providers:** `planOpenClawModelsJsonWithDeps()` normalizes, merges, and applies compat to providers, then serializes `finalProviders` directly. There is no guard here that excludes explicit custom providers whose `models` remain unresolved.
  - file: [src/agents/models-config.plan.ts:97](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/models-config.plan.ts#L97)
