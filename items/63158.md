## Summary

Keep open. Current `main` still has the misleading QMD availability messaging in the doctor/health-check path: `noteMemorySearchHealth()` probes QMD with the agent workspace as `cwd` and, on `ENOENT`, tells operators the QMD binary could not be started and suggests reinstalling QMD. The runtime/gateway path in `extensions/memory-core` already handles missing workspaces differently, so the report is partially addressed there, but the linked issue remains valid for doctor-style availability checks.

## Evidence

- **Doctor path still blames the QMD binary:** `noteMemorySearchHealth()` calls `checkQmdBinaryAvailability()` with `cwd: resolveAgentWorkspaceDir(cfg, agentId)`. If that probe fails, it emits `QMD memory backend is configured, but the qmd binary could not be started` plus install instructions, which is the misleading behavior described in the issue when the real problem is the workspace/cwd.
