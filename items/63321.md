## Summary

Keep open. Current `main` still has the unguarded `JSON.parse` calls this PR targets, so the request is not implemented, but the PR’s current approach would also change existing error semantics in two places and needs maintainer revision rather than conservative cleanup closure.

## Evidence

- **Current main still throws on malformed noisy JSON:** `parsePossiblyNoisyJsonObject` on current `main` still calls `JSON.parse` without a local `try/catch`, so malformed or partial Tailscale output will still throw from this helper.
  - file: [src/infra/tailscale.ts:14](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/infra/tailscale.ts#L14)
  - command: `nl -ba src/infra/tailscale.ts | sed -n '1,40p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Whois path currently relies on thrown parse failures for short error TTL:** `readTailscaleWhoisIdentity` wraps the command and parse path in an outer `try/catch`; on any thrown failure it caches `null` for `errorTtlMs` (default 5s), while successful parses are cached for `cacheTtlMs` (default 60s). Swallowing parse failures inside the shared helper would change that behavior.
  - file: [src/infra/tailscale.ts:497](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/infra/tailscale.ts#L497)
  - command: `nl -ba src/infra/tailscale.ts | sed -n '482,513p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Funnel status path currently distinguishes parse/command failure from disabled funnel:** `ensureFunnel` currently parses `funnel status --json` directly inside the outer `try`, and true failures fall into the generic error handler at `src/infra/tailscale.ts:365`. The PR discussion correctly flagged that converting malformed JSON to `{}` would instead route users into the 'Funnel is not enabled' remediation path at `src/infra/tailscale.ts:330`.
  - file: [src/infra/tailscale.ts:329](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/infra/tailscale.ts#L329)
  - command: `nl -ba src/infra/tailscale.ts | sed -n '319,405p'`

- **PR discussion already identified unresolved semantic regressions:** The provided review comments on PR #63321 noted two nontrivial behavior changes: malformed `whois --json` output would bypass the short error TTL path, and malformed `funnel status --json` output would be treated as confirmed disabled funnel state. That makes this more than a straightforward already-validated cleanup close.

## Risks / Open Questions

- Closing this PR would discard a still-relevant hardening request even though current `main` has not implemented the proposed resilience change.
- Treating parse failures as empty objects in the shared helper risks longer-than-intended whois cache suppression and misleading funnel setup guidance.
