## Summary

Keep open. Current `main` still routes Feishu MP3 as a generic file instead of a voice bubble, and it still lacks both per-agent `agents.list[].tts` config and Microsoft-specific model-directive parsing for TTS voice overrides.

## Evidence

- **Feishu audio routing still excludes MP3/WAV voice-bubble handling:** `resolveFeishuOutboundMediaKind()` only maps `.opus`/`.ogg` or `audio/opus`/`audio/ogg` to `msgType: "audio"`; everything else falls through to generic file routing. `sendMediaFeishu()` uses that routing directly with no conversion step before upload.
  - file: [extensions/feishu/src/media.ts:524](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/feishu/src/media.ts#L524)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '520,635p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Feishu tests still assert MP3 uploads are sent as files:** There is an explicit test named `falls back to generic file for unsupported audio formats` that loads `song.mp3` with `audio/mpeg` and expects `file_type: "stream"` and `msg_type: "file"`.
  - file: [extensions/feishu/src/media.test.ts:239](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/feishu/src/media.test.ts#L239)
  - command: `nl -ba extensions/feishu/src/media.test.ts | sed -n '228,265p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **TTS directive parser still ignores bare `[[tts:voice]]` tokens and depends on provider plugins for key support:** `parseTtsDirectives()` skips tokens without `=` and only applies overrides when some provider's `parseDirectiveToken` handles the key.
  - file: [src/tts/directives.ts:88](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/tts/directives.ts#L88)
  - command: `nl -ba src/tts/directives.ts | sed -n '45,160p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Microsoft speech provider still has no directive-token parser:** `buildMicrosoftSpeechProvider()` defines config resolution, talk overrides, voice listing, and synthesis, but no `parseDirectiveToken`, so Microsoft-specific inline override keys are not implemented on `main`.
  - file: [extensions/microsoft/speech-provider.ts:177](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/microsoft/speech-provider.ts#L177)
  - command: `nl -ba extensions/microsoft/speech-provider.ts | sed -n '165,285p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Per-agent config schema still has no `tts` field:** `AgentConfig` includes fields such as `model`, `params`, `skills`, `heartbeat`, `subagents`, `sandbox`, `tools`, and `runtime`, but not `tts`, so the proposed `agents.list[].tts` override shape is not supported on current `main`.
  - file: [src/config/types.agents.ts:70](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/types.agents.ts#L70)

- **Docs still describe model-driven TTS override keys without Microsoft voice support:** The TTS docs list directive keys for `voice` (OpenAI), `voiceName` (Google), and `voiceId` (ElevenLabs / MiniMax / xAI); no Microsoft-specific voice override key is documented. The per-agent config docs likewise show no `agents.list[].tts` field.
