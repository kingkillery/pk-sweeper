## Summary

Keep #67252 open. Current main still has the reported behavior: `feishu_doc` write clears existing content through a single non-paginated `documentBlock.list` call, then deletes only the returned range before inserting replacement content. The supplied discussion points to a fork PR/commented fix, but that pagination change is not present in this checkout.

## Best Possible Solution

Keep the issue open and land a focused Feishu plugin fix: update `extensions/feishu/src/docx.ts` so `clearDocumentContent` paginates `client.docx.documentBlock.list` until the response has no next page, then add a regression test where `write` sees multiple list pages and deletes the full top-level range before inserting replacement content. The existing open fork PR/comment can be used as prior art, but the fix still needs to be incorporated into `openclaw/openclaw` main.
