## Summary

Keep #67670 open. Current main improves Cloudflare/CDN HTML error handling, but it does not implement browser-like TLS fingerprinting, a `tlsProfile`/Chrome transport option, or a documented `cloudscraper` workaround for the reported mainland-China proxy path. For the specific 403 HTML case, current code still classifies HTML 403 responses as `auth_html_403`; the linked #67717 PR remains open and explicitly says it only improves Cloudflare challenge detection/copy, not transport behavior.

## Best Possible Solution

Keep #67670 open as the canonical issue for the Cloudflare 403/TLS-fingerprint proxy failure. A conservative path is to merge a reviewed version of #67717 or equivalent for Cloudflare-specific 403 messaging and add docs for the reverse-proxy workaround; any built-in browser-like TLS profile should be treated as a separate core transport/API design decision with maintainer review.
