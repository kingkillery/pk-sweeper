## Summary

Keep open. Current `main` has not implemented this fix, and the PR discussion surfaces a real semantic conflict: empty `providerOverride` is still used intentionally as a sentinel for provider-less runtime models, so normalizing it away would regress existing behavior.

## Evidence

- **Current main still uses raw status-card overrides:** `buildStatusMessage` still reads `entry?.providerOverride` and `entry?.modelOverride` directly when choosing the selected model, with no `normalizeOptionalString` on those fields. That means the PR's proposed fix is not already on `main`.
  - file: [src/status/status-message.ts:484](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/status/status-message.ts#L484)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Empty provider override is intentional in current tool flow:** `session_status` still synthesizes `providerOverride: ""` when there is a runtime model but no known runtime provider, specifically to keep the status card provider-less for legacy/unknown runtime models.
  - file: [src/agents/tools/session-status-tool.ts:527](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tools/session-status-tool.ts#L527)

## Risks / Open Questions

- The original context-window bug described by the PR may still be real on current `main` for provider-qualified models that arrive with empty status-card overrides.
- Closing this PR would discard useful evidence, but merging it as-is would risk regressing the intentional provider-less legacy-runtime-model path.
