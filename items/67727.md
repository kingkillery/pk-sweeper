## Summary

Keep open. Current main does not implement `messages.inbound.botDebounceMs`: the config type/schema/generated help expose only `debounceMs` and `byChannel`, and `resolveInboundDebounceMs` still resolves `overrideMs -> byChannel -> debounceMs -> 0`. The PR is a plausible core config/API change, but its own discussion confirms the production `senderIsBot` wiring is deferred, so the branch would still be unreachable until real channel call sites pass bot-origin metadata.

## Best Possible Solution

Keep the PR open and request end-to-end implementation before merge: add the config type/schema/generated artifacts, then thread sender bot metadata through the relevant channel debounce call sites or shared helper so `botDebounceMs` is actually selected for bot-originated inbound messages. The current PR commits that regenerate generated config artifacts are useful, but the deferred runtime wiring should be part of the same merge or tracked by a maintainer-approved canonical follow-up.
