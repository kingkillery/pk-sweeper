## Summary

Current main already ships cron isolated run-session cleanup via `cron.sessionRetention`, with source, tests, docs, and release evidence. PR #67604 reworks the same `...:cron:<jobId>:run:<uuid>` surface rather than deleting base cron parent rows, and review comments identify regressions that would make the shipped cleanup less reliable for failed or log-rotated runs.

## Best Possible Solution

Close this PR and keep the shipped `cron.sessionRetention` reaper. If maintainers want additional safeguards for active background execs, child tasks, or embedded runs, the best path is a smaller follow-up PR with a reproduced current-main failure, no hard dependency on retained run-log rows, English comments, and regression coverage for failed and log-rotated runs.
