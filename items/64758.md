## Summary

Keep open. Current `main` still does not implement the non-streaming steer fallback this PR proposes, and the PR discussion includes an unresolved regression risk for heartbeat turns. That is not strong enough evidence for any auto-close path.

## Evidence

- **Current main still only steers while streaming:** `runReplyAgent` still gates steer injection on `shouldSteer && isStreaming`, so an active-but-not-streaming run will skip the steer fast-path entirely.
  - file: [src/auto-reply/reply/agent-runner.ts:972](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/agent-runner.ts#L972)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '850,1005p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Current main only aborts active runs for interrupt mode:** `resolvePreparedReplyQueueState` aborts the active run only when `queueMode === "interrupt"`; there is no equivalent abort path for `steer` or `steer-backlog` when the run is active but not streaming.
  - file: [src/auto-reply/reply/get-reply-run-queue.ts:29](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/get-reply-run-queue.ts#L29)

  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Queue policy still routes active steer runs to followup, not immediate execution:** For active runs, heartbeat is dropped, `shouldFollowup` and `queueMode === "steer"` become `enqueue-followup`, and everything else falls through to `run-now`. This matches the reported gap: steer followups remain queued unless the mid-stream injection path succeeds.
  - file: [src/auto-reply/reply/queue-policy.ts:14](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/queue-policy.ts#L14)

- **PR review already identified a behavioral regression to resolve:** The provided PR discussion includes a review comment noting that aborting before queue policy evaluation would let heartbeat turns terminate an active non-streaming run even though heartbeat policy later resolves to `drop`. That means the proposed change still needs maintainer judgment or revision, not cleanup closure.
  - sha: [cff8df1c9c3a](https://github.com/openclaw/openclaw/commit/cff8df1c9c3a3289768863b293a2a2bb2c7eb5ce)

## Risks / Open Questions

- Users in `steer` mode can still have followups blocked behind an active run that stopped streaming, which matches the bug this PR is trying to address.
- Closing this PR as implemented or obsolete would be incorrect: current `main` lacks the fallback, but merging the PR as-is may regress heartbeat handling per the existing review feedback.
