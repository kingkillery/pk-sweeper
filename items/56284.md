## Summary

Current `main` still handles Windows gateway restart as a forceful Scheduled Task stop/relaunch path, without wiring in restart deferral or restart-sentinel recovery for `openclaw gateway restart`, so #56284 is not implemented on main.

## Evidence

- **CLI restart path only performs service restart plus post-health checks:** `runDaemonRestart()` delegates to `runServiceRestart()` and its `postRestartCheck` only waits for health and stale-PID cleanup. There is no call to `deferGatewayRestartUntilIdle()` or restart-sentinel helpers in this path.
  - file: [src/cli/daemon-cli/lifecycle.ts:197](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/cli/daemon-cli/lifecycle.ts#L197)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Windows Scheduled Task restart is still forceful:** `restartScheduledTask()` sends `schtasks /End`, terminates gateway listeners/startup runtime, waits for the port to free, then runs the task again. That matches the reported abrupt-stop behavior rather than a graceful in-process restart.
  - file: [src/daemon/schtasks.ts:968](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/daemon/schtasks.ts#L968)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Current tests expect listener termination during Windows restart:** The Scheduled Task restart tests assert that `restartScheduledTask()` kills lingering gateway listeners before relaunching the task, which reinforces that current behavior is terminate-and-restart rather than wait-until-idle.
  - file: [src/daemon/schtasks.stop.test.ts:171](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/daemon/schtasks.stop.test.ts#L171)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Graceful restart deferral exists elsewhere:** `deferGatewayRestartUntilIdle()` is implemented and used by gateway config-reload handling to defer restart until active operations drain, so the issue’s proposed mechanism exists in-tree.
  - file: [src/gateway/server-reload-handlers.ts:257](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-reload-handlers.ts#L257)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Restart-sentinel session recovery exists separately:** `writeRestartSentinel()` and `consumeRestartSentinel()` are implemented for restart continuity, but the Windows CLI restart path above does not invoke them.
  - file: [src/infra/restart-sentinel.ts:80](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/infra/restart-sentinel.ts#L80)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
