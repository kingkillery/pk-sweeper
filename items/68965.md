## Summary

Current `main` still resolves and merges implicit bundled-provider catalogs before it considers `models.mode`, so the reported `models.mode: "replace"` behavior remains inconsistent with the documented contract. The linked fix PR was closed unmerged for queue hygiene, not because the bug was invalid or already fixed.

## Evidence

- **Current implementation still always loads implicit providers:** `resolveProvidersForModelsJsonWithDeps` unconditionally calls `resolveImplicitProvidersImpl(...)` and then `mergeProviders({ implicit, explicit })` before `resolveProvidersForMode` runs, so `mode: "replace"` cannot exclude implicit providers at this stage.
  - file: [src/agents/models-config.plan.ts:47](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/agents/models-config.plan.ts#L47)
  - command: `nl -ba src/agents/models-config.plan.ts | sed -n '1,260p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Docs/schema say replace should use only configured providers:** The schema allows `"merge" | "replace"`, and the help text explicitly says `"replace" uses only your configured providers`, which matches the issue report and conflicts with the current implementation.
  - file: [src/config/schema.help.ts:789](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/config/schema.help.ts#L789)
