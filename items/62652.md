## Summary

Keep open. Current `main` has explicit ways to disable Bonjour/mDNS and contains the bonjour advertiser more defensively, but I did not find strong evidence that the reported production restart/SIGKILL loop from `2026.4.5` is actually fixed on `main` well enough to close this issue conservatively.

## Evidence

- **Current gateway config can disable local mDNS entirely:** `startGatewayDiscovery` skips local discovery when `discovery.mdns.mode` is `"off"` or `OPENCLAW_DISABLE_BONJOUR` is truthy, and only starts discovery services when that gate passes.
  - file: [src/gateway/server-discovery-runtime.ts:25](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/gateway/server-discovery-runtime.ts#L25)

  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Gateway startup reads the persisted mDNS mode:** Gateway startup passes `cfgAtStart.discovery?.mdns?.mode` into discovery startup, so the `off|minimal|full` config path is live on current `main`.
  - file: [src/gateway/server-startup-early.ts:61](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/gateway/server-startup-early.ts#L61)

- The issue is recent and contains a concrete production regression report with version, environment, and timeline details, so `stale_insufficient_info` is not appropriate.
- Current `main` clearly adds mitigations/workarounds (`mdns.mode: off`, env disable, plugin disableability), but that is weaker than proving the underlying restart/SIGKILL bug is fixed.
- Without a reproduction against current `main`, closing would risk discarding a still-plausible production-only load regression.
