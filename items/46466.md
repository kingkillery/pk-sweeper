## Summary

Keep open. Current `main` still treats `launchctl bootstrap ... 125: Domain does not support specified action` as a macOS LaunchAgent session-context failure, not a resolved product change, and the docs still say macOS daemon install requires a logged-in user session. The report is too sparse to verify as user error versus a real regression in interactive macOS sessions, and at 41 days old it does not qualify for `stale_insufficient_info`.

## Evidence

- **Current launchd path still errors on unsupported GUI domain:** `resolveGuiDomain()` uses `gui/<uid>`, and `throwBootstrapGuiSessionError()` turns bootstrap 125 into the same logged-in-GUI-session / wrong-user / headless guidance shown in the issue body. That means current `main` still expects this failure mode to happen in some contexts rather than having removed it.
