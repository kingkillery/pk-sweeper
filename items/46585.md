## Summary

Keep open. Current `main` still appears to ignore `facing` whenever a macOS camera `deviceId` is provided, and Continuity Camera devices are discovered/listed as a single external camera with no front/back-specific selection path. I did not find a fix on `main` or in the latest release code.

## Evidence

- **macOS picker ignores facing when deviceId is set:** `pickCamera` returns the first device whose `uniqueID` matches `deviceId` before it evaluates the requested front/back position, so `--facing front` cannot change selection for a chosen external/Continuity camera.
  - file: [apps/macos/Sources/OpenClaw/CameraCaptureService.swift:199](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/apps/macos/Sources/OpenClaw/CameraCaptureService.swift#L199)
  - command: `nl -ba apps/macos/Sources/OpenClaw/CameraCaptureService.swift | sed -n '195,212p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Continuity/external cameras are exposed as generic discovered devices:** macOS camera discovery uses `.continuityCamera` and external camera types with discovery position `.unspecified`, and `listDevices()` reports each device's existing `position` value. That matches the report of a single iPhone camera entry with `position: "unspecified"` and provides no front/back-specific device selection path.
  - file: [apps/macos/Sources/OpenClaw/CameraCaptureService.swift:41](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/apps/macos/Sources/OpenClaw/CameraCaptureService.swift#L41)

- **No extra routing in macOS node invoke path:** `MacNodeRuntime` forwards both `facing` and `deviceId` directly into `CameraCaptureService.snap/clip`; there is no Continuity Camera-specific handling later in the stack.
