## Summary

Current `main` does not show a conservative close signal for #63723. `sessions_send` still waits on `agent.wait`, returns `status: "timeout"` when that wait times out, and only starts the agent-to-agent follow-up path after a successful reply. That matches the issue's reported failure mode more than a fix, so the item should stay open.

## Evidence

- **sessions_send returns timeout before any follow-up delivery path:** After starting the target run, `sessions_send` calls `waitForAgentRunAndReadUpdatedAssistantReply(...)`. If that result is `timeout`, it immediately returns `{ status: "timeout" }`. The A2A follow-up is only started later, after a successful `ok` result.
  - file: [src/agents/tools/sessions-send-tool.ts:358](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tools/sessions-send-tool.ts#L358)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **run wait classifies gateway wait failures as timeout:** `waitForAgentRun` wraps `agent.wait` and converts gateway timeout errors into `{ status: "timeout" }`, which is exactly the path `sessions_send` consumes.
  - file: [src/agents/run-wait.ts:118](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/run-wait.ts#L118)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **docs still define sessions_send as either fire-and-forget or wait-for-reply:** The current session-tool docs describe only two modes: `timeoutSeconds: 0` to enqueue immediately, or a timed wait that returns the reply inline. There is no documented delivery-receipt mode independent of the target's reply.
  - file: [docs/concepts/session-tool.md:68](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/concepts/session-tool.md#L68)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **A2A announce flow is documented as happening after the target replies:** The ACP docs describe the `sessions_send` follow-up path as: wait for the target session's reply, optionally exchange bounded follow-up turns, then produce and deliver an announce message. That is consistent with the timeout behavior reported in the issue when the target is slow.
  - file: [docs/tools/acp-agents.md:357](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/tools/acp-agents.md#L357)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **tests cover success cases, not delivered-but-timed-out recovery:** Current tests assert fire-and-forget, inline wait success, and ping-pong/announce success for `sessions_send`, but there is no matching test showing a separate delivery receipt or later sender-visible completion after an `agent.wait` timeout.
  - file: [src/agents/openclaw-tools.sessions.test.ts:781](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/openclaw-tools.sessions.test.ts#L781)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **issue discussion still points to the same gap and workaround:** The provided comments include a concrete case where the target receives the message, `sessions_send(timeout=30)` still times out, and the later completion does not route back to the sender; another commenter notes `message.send + message.read` as a workaround rather than a core fix.
