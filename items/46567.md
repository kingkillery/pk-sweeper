## Summary

Current main already implements durable Telegram DM pairing via the allowFrom store, and the repeated-pairing symptom is covered by a shipped Telegram fix that ignores self-authored bot message updates. The report also checks the wrong persistence file: approved senders are stored in `telegram-allowFrom.json`, while `telegram-pairing.json` only holds pending requests.

## Evidence

## Close Comment

Closing this from a Codex review because the requested behavior is already implemented on current main and is present in the shipped `v2026.4.22` release. Two concrete points from the current codebase: `docs/channels/pairing.md:39` documents that `telegram-pairing.json` is only the pending-request queue, while approved Telegram DM access is persisted in `telegram-allowFrom.json`; and `src/pairing/pairing-store.ts:674` shows `openclaw pairing approve telegram <code>` writes the approved sender into the allowFrom store after removing the pending request. The repeated re-pair symptom also matches a shipped Telegram fix noted in `CHANGELOG.md:1977`, and current runtime code now ignores self-authored Telegram DM `message` echoes in `extensions/telegram/src/bot-handlers.runtime.ts:1953`, with regression coverage in `extensions/telegram/src/bot.create-telegram-bot.test.ts:754`. If this still reproduces on `v2026.4.22` or newer with the persisted `telegram-allowFrom.json` entry present, please open a fresh issue with exact container image tag, config/account shape, and the contents of the allowFrom file after approval.
