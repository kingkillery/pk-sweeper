## Summary

Keep open. The linked discussion identified a real Feishu/webchat routing regression, but current `main` does not show that fix on the production path, and the existing tests still stub `feishu` as routable instead of exercising the real `isRoutableChannel` guard.

## Evidence

- **Current routability guard is still narrow:** `isRoutableChannel` still returns true only when `normalizeChatChannelId(channel)` or `normalizeChannelId(channel)` succeeds; there is no fallback here for arbitrary non-`webchat` channel ids.
  - file: [src/auto-reply/reply/route-reply.ts:274](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/auto-reply/reply/route-reply.ts#L274)
  - command: `nl -ba src/auto-reply/reply/route-reply.ts | sed -n '250,330p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Imported normalizeChannelId only covers bundled chat ids:** In the current tree, `route-reply.ts` imports `normalizeChannelId` from `src/channels/plugins/index.ts`, which re-exports `src/channels/registry.ts:normalizeChannelId`; that helper is just `normalizeChatChannelId(raw)`, so both sides of the `isRoutableChannel` check remain bundled-catalog based.
  - file: [src/channels/registry.ts:64](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/channels/registry.ts#L64)
