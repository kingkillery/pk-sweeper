## Summary

Keep open. Current `main` still silently swallows unexpected filesystem errors in the pairing allow-from path, so this PR's core fix has not been implemented on `main` and remains relevant.

## Evidence

- **Async write path still swallows all stat errors:** `writeAllowFromState` still does `await fs.promises.stat(filePath)` inside `catch {}` with no `ENOENT` filter, so unexpected errors like `EACCES`/`EIO` are still hidden on current `main`.
  - file: [src/pairing/pairing-store.ts:302](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/pairing/pairing-store.ts#L302)
  - command: `nl -ba src/pairing/pairing-store.ts | sed -n '297,315p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Sync stat path still converts non-ENOENT errors into missing-file results:** `readAllowFromFileSyncWithExists` catches `fs.statSync` errors and returns `{ entries: [], exists: false }` for any code other than `ENOENT`, which is the same silent-swallow behavior the PR discussion called out.
  - file: [src/pairing/allow-from-store-file.ts:230](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/pairing/allow-from-store-file.ts#L230)
  - command: `nl -ba src/pairing/allow-from-store-file.ts | sed -n '225,238p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Sync read path also swallows unexpected read errors:** The same helper also returns `{ entries: [], exists: false }` for any non-`ENOENT` `fs.readFileSync` failure, so current `main` still masks unexpected filesystem failures in the sync pairing allow-from path.
  - file: [src/pairing/allow-from-store-file.ts:252](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/pairing/allow-from-store-file.ts#L252)
