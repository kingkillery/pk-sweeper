## Summary

Current `main` does not implement the cleanup this PR is trying to add, and the PR itself still has unresolved correctness concerns from review. The item should stay open.

## Evidence

- **`cron.remove` still only deletes the job record:** On current `main`, `remove()` filters the job out of `state.store.jobs`, persists, re-arms the timer, and emits `removed`. It does not delete `cron/runs/<jobId>.jsonl` or purge any session-store entries.
  - file: [src/cron/service/ops.ts:332](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/cron/service/ops.ts#L332)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '330,347p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Current reaper explicitly keeps the base cron session:** `src/cron/session-reaper.ts` documents that the base session `...:cron:<jobId>` is kept as-is, and the implementation only prunes keys matching `isCronRunSessionKey(...)`. That means deleting a job does not remove its base cron session on current `main`.
  - file: [src/cron/session-reaper.ts:1](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/cron/session-reaper.ts#L1)
  - command: `nl -ba src/cron/session-reaper.ts | sed -n '1,150p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Session listing keeps the base cron session and hides only the run alias:** The search/listing test shows a store containing both `agent:main:cron:job-1` and `agent:main:cron:job-1:run:run-abc`, and expects only the base key to remain visible. If `cron.remove` leaves that base key behind, the ghost session remains visible.
  - file: [src/gateway/session-utils.search.test.ts:193](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/session-utils.search.test.ts#L193)
  - command: `nl -ba src/gateway/session-utils.search.test.ts | sed -n '195,220p'`
