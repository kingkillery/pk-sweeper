## Summary

Keep open. Current `main` still scans auto-discovery roots with `fs.readdirSync(..., { withFileTypes: true })` and only admits `entry.isFile()` or `entry.isDirectory()` entries, so symlinked plugin directories under `~/.openclaw/extensions` or `<workspace>/.openclaw/extensions` still fall through without discovery. The separate `plugins.load.paths` code path still uses `fs.statSync()` and would follow symlinks, which matches the reported workaround rather than disproving the bug.

## Evidence

- **Auto-discovery still skips non-directory Dirent entries:** `discoverInDirectory()` reads Dirents and, after handling plain files, immediately `continue`s when `!entry.isDirectory()`. There is no `entry.isSymbolicLink()` or follow-up `fs.statSync()` branch before scanning a directory candidate, so symlinked plugin directories are not considered.
  - file: [src/plugins/discovery.ts:845](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/plugins/discovery.ts#L845)

  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Configured load paths still follow symlinks:** `discoverFromPath()` calls `fs.statSync(resolved)` before deciding whether a configured path is a file or directory. That follows symlinks and aligns with the issue's documented workaround via `plugins.load.paths`.
