## Summary

Current main already implements the needed daily-reset behavior for conversation sessions and protects the route-metadata path from refreshing session activity. The issue's proposed root cause (`mergeSessionEntry()` refreshing `updatedAt` before freshness evaluation) does not match the current session-init flow, and shipped release notes for `v2026.4.22` describe the maintenance/activity fix.

## Close Comment

Closing from a Codex review because current `main` already covers this. The session-init path evaluates staleness from the stored entry before writing the next session state (`src/auto-reply/reply/session.ts:404`), and the inbound route-metadata path now preserves `updatedAt` instead of treating route bookkeeping as activity (`src/config/sessions/store.ts:718`, `src/config/sessions/types.ts:405`). There is also regression coverage for daily reset rollover on direct-session keys and for `updateLastRoute` not bumping `updatedAt` (`src/auto-reply/reply/session.test.ts:1247`, `src/config/sessions.test.ts:359`). This also shipped in `v2026.4.22`, whose release notes call out keeping daily reset / idle-maintenance bookkeeping from bumping session activity (`appcast.xml:52`). If you still hit a reset miss on a newer build, please open a fresh issue with the exact routed session key plus the stored `updatedAt` immediately before the inbound turn.
