## Summary

Keep open. Current `main` still does not sanitize WhatsApp auto-reply text before Markdown/table conversion, while the shared delivery sanitizer already exists and is tested to remove tool-call XML. The PR appears directionally correct, but the current logging path still records raw `replyResult.text`, so the bot's logging-consistency concern remains a real review item rather than a reason to close.

## Evidence

- **Current WhatsApp auto-reply path still delivers unsanitized text:** `deliverWebReply` normalizes WhatsApp payload text and then immediately runs `convertMarkdownTables` and `markdownToWhatsApp` on `normalizedReply.text`; there is no `sanitizeAssistantVisibleText` or equivalent in this path on current `main`. That means raw tool-call XML would still flow into outbound reply text.
