## Summary

Keep open. Current main still exposes only JSON output mode on `openclaw infer model run` and `openclaw agent`; neither surface accepts a JSON Schema contract, validates model text against a schema, or defines a repair/retry failure path. The adjacent `llm-task` plugin provides schema-validated JSON for workflow tool calls, but it is optional plugin/tool functionality rather than the requested first-class core CLI surface.

## Best Possible Solution

Keep the issue open and implement a core structured-output contract on `openclaw infer model run` first, then decide whether `openclaw agent` should share it. Add `--json-schema <path-or-inline>` parsing, validate returned JSON with the shared schema validator/AJV, return machine-readable errors in the existing JSON envelope, define bounded repair/retry semantics, and cover both local and gateway paths. The `llm-task` plugin can remain the workflow/tool-oriented alternative but should not be treated as closing this CLI feature.

## Risks / Open Questions

- Implementation should coordinate with #69030 because schema validation semantics may differ between the current agent-runtime path and any future raw inference path.
- Provider-native JSON Schema support exists for some providers, so the best fix needs a provider-neutral CLI contract with provider-specific native schema mode as an optimization, not a hard dependency.
