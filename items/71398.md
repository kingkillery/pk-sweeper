## Summary

Keep open. Current main supports QQ Bot SecretRefs in the structured SecretInput object form, but the reported `secretref:/...` string form is not resolved by the current SecretInput path and would still be treated as a plaintext secret string before being sent to the QQ API. I found no current-main fix, release note, or duplicate/superseding item that closes this exact upgrade regression.

## Best Possible Solution

Keep the issue open. Maintainers should decide whether `secretref:/...` is an old supported file-provider shorthand that needs migration/runtime compatibility, or invalid syntax that should fail validation/doctor with a clear remediation. The fix likely belongs in shared SecretInput migration/validation plus QQ Bot regression tests for `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, with docs updated to show the supported structured file SecretRef form.

## Evidence

- **Issue context:** The provided issue body reports a recent v2026.4.21 to v2026.4.23 regression with QQ Bot plugin v2026.4.20: `channels.qqbot.accounts.nuonuo.clientSecret` is set to `secretref:/QQBOT_NUONUO_CLIENT_SECRET`, the raw value appears to reach QQ OpenAPI, and hardcoding the secret works.
- **QQ Bot runtime resolution path:** QQ Bot resolves `accountConfig.clientSecret` through the platform adapter when a secret is configured, otherwise falls back to `clientSecretFile` or `QQBOT_CLIENT_SECRET`. There is no QQ Bot-specific handling for `secretref:/...`.

## Risks / Open Questions

- A compatibility fix must avoid reinterpreting legitimate plaintext secrets that happen to start with `secretref:`.
- QQ Bot docs say `clientSecret` accepts SecretRef input, but the canonical SecretRef credential surface does not list `channels.qqbot.clientSecret`, so maintainers should align docs, target registry, and runtime behavior if this path is intended to be first-class.
