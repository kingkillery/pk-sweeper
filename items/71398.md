## Summary

Keep open. Current `main` still shows a QQ Bot SecretRef contract mismatch: the QQ Bot docs/schema advertise `clientSecret` as SecretRef-capable, but runtime resolution still throws on unresolved SecretRefs and there is no QQ Bot secret-contract artifact for the channel secret resolver to load. The reporter's exact `secretref:/...` syntax is not a documented SecretRef format, so this is not strong enough for an auto-close; it needs maintainer triage as a real docs/runtime inconsistency.

## Evidence

- **QQ Bot docs advertise SecretRef support:** The QQ Bot user docs explicitly say `clientSecret` accepts SecretRef input.
  - file: [docs/channels/qqbot.md:87](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/docs/channels/qqbot.md#L87)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Canonical SecretRef docs omit QQ Bot and require structured objects:** The supported SecretRef surface lists many channel credential paths but not `channels.qqbot.clientSecret`, and the notes say `openclaw.json` SecretRefs must be structured objects like `{ "source": "env", "provider": "default", "id": "..." }`.
  - file: [docs/reference/secretref-credential-surface.md:23](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/docs/reference/secretref-credential-surface.md#L23)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **QQ Bot runtime expects pre-resolved secret input:** `resolveQQBotAccount()` treats any configured `clientSecret` secret input as something that must already be resolved by the platform adapter; it does not resolve file/env/exec SecretRefs itself on this path.
  - file: [extensions/qqbot/src/bridge/config.ts:54](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/extensions/qqbot/src/bridge/config.ts#L54)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Tests lock in unresolved SecretRef failure at runtime:** QQ Bot tests accept SecretRef objects in schema, then explicitly assert that runtime `resolveQQBotAccount()` throws `unresolved SecretRef`, while setup/status flows only mark the account configured when `allowUnresolvedSecretRef` is set.
  - file: [extensions/qqbot/src/config.test.ts:76](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/extensions/qqbot/src/config.test.ts#L76)
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **No QQ Bot secret-contract artifact exists for channel secret collection:** The channel secret loader only looks for `secret-contract-api.js` or `contract-api.js`, but `rg --files` for QQ Bot only found `extensions/qqbot/setup-entry.ts` and `extensions/qqbot/setup-plugin-api.ts`, not a QQ Bot secret-contract artifact.
  - file: [src/secrets/channel-contract-api.ts:57](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/secrets/channel-contract-api.ts#L57)
