## Summary

Keep open. Current `main` adds partial TUI mitigation for stale streaming state, but the underlying sequencing the issue describes still exists: lifecycle end can still be delayed until async flush/finalization work completes, and the new fallback only forces `idle` after a 30s silence watchdog rather than promptly after the last visible token.

## Evidence

- **TUI still relies on terminal events, with only a 30s silence fallback:** `createEventHandlers` now defines `DEFAULT_STREAMING_WATCHDOG_MS = 30_000` and arms a timer on each chat delta. If no more deltas arrive, it resets `activityStatus` to `idle` and emits a system note saying the backend may have dropped the run silently.
  - file: [src/tui/tui-event-handlers.ts:50](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/tui/tui-event-handlers.ts#L50)
  - command: `nl -ba src/tui/tui-event-handlers.ts | sed -n '1,180p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **TUI does not transition promptly on content completion:** On chat deltas, the TUI sets `activityStatus` to `streaming` and arms the watchdog. It only clears immediately on chat `final` or lifecycle `end`; otherwise the fallback waits for the watchdog window.
