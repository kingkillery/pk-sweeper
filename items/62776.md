## Summary

Keep open. Current `main` only supports remote MCP servers through a URL plus optional static headers; there is no OpenClaw-side OAuth login/callback/persistence flow for remote MCP servers like Heptabase, so the issue is not implemented on `main` and is still actionable.

## Evidence

- **User-facing MCP docs still describe static-header auth only:** The remote MCP docs list `url`, `headers`, and `connectionTimeoutMs` for SSE, and `url`, `transport`, `headers`, and `connectionTimeoutMs` for streamable HTTP. Both examples use a manually supplied `Authorization: Bearer <token>` header; there is no documented OAuth browser flow, callback, or persisted remote MCP auth state.
  - file: [docs/cli/mcp.md:446](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/docs/cli/mcp.md#L446)
  - command: `nl -ba docs/cli/mcp.md | sed -n '440,520p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Config schema has no OAuth fields for MCP servers:** `McpServerConfig` only exposes stdio fields plus remote HTTP `url`, `transport`, `headers`, and `connectionTimeoutMs`. There is no config surface for OAuth metadata, client registration, callback handling, token storage, or refresh behavior.
  - file: [src/config/types.mcp.ts:1](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/config/types.mcp.ts#L1)
  - command: `nl -ba src/config/types.mcp.ts | sed -n '1,120p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Transport layer only forwards static headers into MCP SDK transports:** For remote MCP, OpenClaw constructs `StreamableHTTPClientTransport` with `requestInit.headers` or `SSEClientTransport` with `requestInit.headers`/custom fetch headers. There is no OpenClaw logic here to detect a 401 challenge, open a browser, complete OAuth, or persist resulting credentials.
  - file: [src/agents/mcp-transport.ts:78](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/mcp-transport.ts#L78)
  - command: `nl -ba src/agents/mcp-transport.ts | sed -n '1,220p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Remote MCP startup failure path just logs and drops the server:** When a configured MCP server fails during `client.connect()` or tool listing, the bundle MCP runtime logs `failed to start server ...` and disposes the session. That matches the reported 401 behavior; there is no alternate OAuth remediation path on failure.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:192](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/pi-bundle-mcp-runtime.ts#L192)
  - command: `nl -ba src/agents/pi-bundle-mcp-runtime.ts | sed -n '130,230p'`

That strengthens the request as a real ecosystem need, but it is not evidence that `openclaw/openclaw` main has shipped support.
