## Summary

Keep #67735 open. Current main still implements `identityLinks` in the external direct-message routing path, while WebChat/Control UI continues to use an explicitly selected session key that defaults to the main session. There is a manual session picker, but no WebChat `webchat:default` identity resolution or explicit `main` versus canonical-direct mode, and the docs still describe WebChat as main-session-default behavior.

## Best Possible Solution

Keep the issue open for maintainer product/API decision. A complete fix should either add a supported WebChat identity/session mode that resolves a configured WebChat identity such as `webchat:default` through `session.identityLinks` into the canonical direct session, or explicitly document that WebChat is always main-session/default-session scoped unless manually switched. The implementation likely spans gateway chat session resolution, Control UI state/settings, tests for identity-linked WebChat/Telegram continuity, and the session/channel WebChat docs.
