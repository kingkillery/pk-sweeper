## Summary

Keep open. Current `main` still receives Feishu bot added/removed events but only logs them; it does not emit any internal or plugin hook for those events, and the existing hook catalogs do not expose an equivalent channel-membership hook surface.

## Evidence

- **Feishu bot add/remove handlers only log:** `registerEventHandlers` handles `im.chat.member.bot.added_v1` and `im.chat.member.bot.deleted_v1`, but both branches only parse the payload and write a log line. There is no hook dispatch or callback into the hook system in this code path.
  - file: [extensions/feishu/src/monitor.account.ts:276](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/extensions/feishu/src/monitor.account.ts#L276)
  - command: `nl -ba extensions/feishu/src/monitor.account.ts | sed -n '160,300p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Plugin hook catalog lacks a Feishu membership event:** The typed plugin hook list includes message and lifecycle hooks such as `message_received`, `message_sending`, and `message_sent`, but nothing for Feishu bot membership add/remove events.
  - file: [src/plugins/hook-types.ts:68](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/plugins/hook-types.ts#L68)
  - command: `nl -ba src/plugins/hook-types.ts | sed -n '64,118p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Internal hook types are limited to command/session/agent/gateway/message:** The internal hook event model only defines top-level types `command`, `session`, `agent`, `gateway`, and `message`, so the requested `channel:feishu:bot-added` shape is not part of the current internal hook contract.
  - file: [src/hooks/internal-hook-types.ts:1](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/hooks/internal-hook-types.ts#L1)
  - command: `nl -ba src/hooks/internal-hook-types.ts | sed -n '1,20p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Current inbound hook bridge only emits message_received:** Inbound routing currently bridges only normal inbound messages into plugin/internal hooks via `message_received` and `message:received`; there is no parallel bridge for Feishu bot-added/bot-removed events.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:600](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/auto-reply/reply/dispatch-from-config.ts#L600)

## Risks / Open Questions

- Implementing this likely needs a small hook-surface design decision, not just a Feishu-only log change, because the current internal hook taxonomy has no `channel` event type.
- The requested payload includes `chatName`, but the current Feishu add/remove payload parsing in `extensions/feishu/src/monitor.account.ts` only validates `chat_id` plus operator info, so that field may require an additional lookup or a revised event contract.
