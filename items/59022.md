## Summary

Keep open. The report is recent, has concrete reproduction details, and current docs explain that unauthenticated `/__openclaw__/a2ui/` access can return `Unauthorized` on non-loopback binds, but I do not have strong source/test proof that the reported macOS `canvas.a2ui.push` failure is fixed on current main.

## Evidence

- **issue_has_actionable_repro:** Issue #59022 reports OpenClaw 2026.3.31 on macOS with a specific `canvas a2ui_push` command failing as `UNAVAILABLE: A2UI_HOST_UNAVAILABLE: A2UI host not reachable`, while other canvas commands work.
- **current_docs_a2ui_surface:** Current docs say the Gateway serves A2UI at `/__openclaw__/a2ui/` on the Gateway port.
  - file: [docs/gateway/configuration-reference.md:727](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/docs/gateway/configuration-reference.md#L727)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **current_docs_auth_context:** Current docs also say non-loopback canvas routes require Gateway auth, and paired nodes should receive node-scoped capability URLs because node WebViews typically do not send auth headers. That explains the reporter's curl `Unauthorized`, but does not by itself prove the macOS node path is repaired.
  - file: [docs/gateway/configuration-reference.md:731](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/docs/gateway/configuration-reference.md#L731)
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **current_docs_troubleshooting_not_mac_proof:** Android README mentions an A2UI reachability retry/auto-refresh path, but the issue is for a macOS node, so this is not sufficient close evidence for the reported surface.
  - file: [apps/android/README.md:444](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/apps/android/README.md#L444)
