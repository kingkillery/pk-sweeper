## Summary

The remaining bug in #70577 is the same live-tool stale embedding provider state already tracked by #61098: `memory_search` continues using an old provider/error after `agents.defaults.memorySearch.provider` changes and restart/reload attempts. #71678 is adjacent memory embedding work for #71522, but its stated scope is remote indexing/diagnostics rather than the stale live tool-host/provider-cache behavior. Current main still has the same relevant surfaces: the memory-core plugin registers `memory_search`, the tool resolves a manager from captured tool config, and manager/runtime caches remain process-global/current-config keyed without a dedicated memorySearch reload hook.

## Best Possible Solution

Close #70577 as a duplicate of the open canonical stale-provider tracker #61098. Keep the implementation work and verification focused there: live `memory_search` should re-resolve current `memorySearch` config after config changes/restart boundaries, retire stale manager/provider state, and add a regression test that distinguishes the live tool path from CLI `memory status/search`.
