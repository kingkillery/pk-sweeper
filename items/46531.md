## Summary

Current main still has the setup/runtime mismatch described in #46531. Env-backed provider secrets are validated against the onboarding shell, daemon install persists only durable service env/auth-profile refs, model-provider refs remain active at startup when the provider is enabled, and gateway startup still hard-fails on unresolved active secrets.

## Evidence

- **Env refs are validated against the current session during setup:** `promptEnvSecretRefForSetup` accepts an env SecretRef when the variable is present in the current `env`/`process.env`, then stores that ref; there is no daemon-runtime check in this path.
  - file: [src/plugins/provider-auth-ref.ts:104](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/plugins/provider-auth-ref.ts#L104)
  - command: `nl -ba src/plugins/provider-auth-ref.ts | sed -n '96,150p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Daemon install only carries durable service env and auth-profile env refs:** Gateway install environment is built from state-dir/config durable env plus auth-profile env refs. It does not scan `models.providers.*.apiKey` config refs and inject those values into the service environment.
  - file: [src/commands/daemon-install-helpers.ts:52](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/commands/daemon-install-helpers.ts#L52)

- **Model-provider secrets are still active whenever the provider is enabled:** Secrets collection marks `models.providers.<id>.apiKey` active when `provider.enabled !== false`; it does not limit activation to the currently selected/default provider.

## Risks / Open Questions

- Some users may avoid the crash by putting keys in `~/.openclaw/.env` or opting into `env.shellEnv`, but that is a workaround, not evidence that the reported product gap is fixed.
- I did not find a main-branch fix that preflights provider env refs against the installed daemon environment or automatically downgrades unresolved secondary provider refs to inactive.
