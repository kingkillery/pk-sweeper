## Summary

Keep open. On current `main` (`7d471837369885c12d540d6bafa319ff41129b89`), queued followup items still capture an original provider `messageId` at enqueue time, but collect-mode draining drops that field and the queued followup runner does not pass any `currentMessageId` into `runEmbeddedPiAgent`. That means queued turns still lack the original platform message id needed for platform-native reply APIs like Feishu/Lark `im.message.reply`.

## Risks / Open Questions

- There may be narrower non-collect or plugin-specific reply paths that still work, but the default queued collect path on `main` does not preserve usable current-message identity.
- A fix likely needs an explicit contract for batch replies: whether to expose one selected `currentMessageId`, per-item message ids, or both, so tooling and reply threading stay deterministic.
