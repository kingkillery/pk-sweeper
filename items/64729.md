## Summary

Keep open. Current `main` still has no dedicated CLI error path for a stale allowlisted plugin id that does not map to any registered command, even though config validation and doctor already warn about phantom `plugins.allow` entries elsewhere. The issue body is internally inconsistent about the exact observed error text, but the underlying stale-entry UX gap is not implemented away on `main`.

## Evidence

- **Missing-command helper still has no stale-allowlist branch:** `resolveMissingPluginCommandMessage` only emits custom text for `plugins.allow` exclusion, disabled plugin entries, or known command-alias ownership. If the requested id is already present in `plugins.allow` and there is no matching plugin entry or alias owner, the function falls through to `return null`.
  - file: [src/cli/run-main.ts:83](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/cli/run-main.ts#L83)
  - command: `nl -ba src/cli/run-main.ts | sed -n '74,150p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **CLI startup only raises a custom plugin error when that helper returns a message:** After lazy plugin registration, `runCli` checks whether the primary command was registered. It throws only when `resolveMissingPluginCommandMessage(primary, config)` returns a non-null string; otherwise parsing continues with no stale-entry-specific guidance.
  - file: [src/cli/run-main.ts:277](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/cli/run-main.ts#L277)
