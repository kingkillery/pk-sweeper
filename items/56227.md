## Summary

Keep open. Current `main` still uses stop-then-start restart semantics, and the code/tests explicitly accept a failed restart leaving the gateway stopped rather than preserving the last healthy instance.

## Evidence

- **In-process restart closes the running server before attempting the next start:** The restart signal path always calls `server.close(...)` and only then continues into the restart flow; there is no parallel bring-up of a replacement listener before teardown.
  - file: [src/cli/gateway-cli/run-loop.ts:204](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/cli/gateway-cli/run-loop.ts#L204)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '204,215p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Restart startup failure leaves no serving gateway instance:** On restart-time startup failure, the code sets `server = null`, releases the lock, writes a `gateway.restart_startup_failed` stability bundle, and logs `Process will stay alive; fix the issue and restart.` That preserves the process, not the previous gateway listener.
  - file: [src/cli/gateway-cli/run-loop.ts:284](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/cli/gateway-cli/run-loop.ts#L284)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '284,296p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **CLI restart path explicitly models total-stop failure:** `formatRestartFailure` has a dedicated `stopped-free` case with the message `Gateway restart failed ... service stayed stopped and port ... stayed free`, which matches the issue's reported failure mode rather than avoiding it.
  - file: [src/cli/daemon-cli/lifecycle.ts:49](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/cli/daemon-cli/lifecycle.ts#L49)

  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Linux managed restart is a plain systemd restart:** The systemd implementation shells out to `systemctl restart <unit>` via `runSystemdServiceAction`; there is no atomic swap, health-verified cutover, or rollback to the previous listener.
  - file: [src/daemon/systemd.ts:614](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/daemon/systemd.ts#L614)

  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)

  - file: [src/daemon/schtasks.ts:987](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/daemon/schtasks.ts#L987)

## Risks / Open Questions

- An issue closer could mistake the newer diagnostics and 'process stays alive' behavior for a fix, but it does not preserve the prior working gateway instance or avoid disconnect.
- The current implementation appears intentionally tolerant of restart failure for process survivability, so closing as implemented would overstate what shipped.
