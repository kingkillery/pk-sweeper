## Summary

Current `main` still hard-rejects inbound image attachments above 5 MB in the gateway attachment parser and all main callers continue to pass `maxBytes: 5_000_000`. There is partial mitigation elsewhere (the iOS share extension recompresses before upload, and managed outgoing image attachments can be resized for dimension/pixel limits), but the requested gateway-side auto-resize behavior is not implemented and the rejection path is still covered by tests.

## Evidence

## Risks / Open Questions

- There may be product or performance reasons to prefer client-side transcoding, but current main does not satisfy the specific request to auto-resize instead of rejecting.
- A broader redesign may be needed because size limits are enforced in multiple layers and surfaces, not just one gateway file.
