## Summary

Keep open. Current main still does not implement phase-level `openclaw memory status` output, and the related duplicate PR is also open and not a clean superseding canonical. The PR is concrete but needs maintainer/product triage because the current code treats managed dreaming scheduling as deep-config driven, while the requested light/rem phase status implies broader phase semantics.

## Best Possible Solution

Keep the item open for maintainer triage. The best path is a narrowed PR that first decides the product contract: either make managed dreaming truly phase-aware and render effective light/deep/REM state, including disabled-by-limit/threshold details, or document that deep enablement controls the managed sweep and make `memory status` explain that explicitly. Split unrelated `--force`, timezone, and iMessage changes into separate work.
