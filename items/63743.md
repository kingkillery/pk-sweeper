## Summary

Current `main` does not implement general IPv6 gateway listening. The gateway still resolves non-loopback binds to IPv4-only addresses (`0.0.0.0` or an IPv4 `customBindHost`), and the only dual-stack behavior in tree is an extra `::1` loopback listener when the primary bind host is `127.0.0.1`. That means the reported LAN/public IPv6 gap is still real and should stay open.

## Evidence

- **Gateway bind resolution is still IPv4-only for LAN/custom:** `resolveGatewayBindHost()` returns `0.0.0.0` for `lan`, and `custom` only accepts/binds a valid IPv4 host before falling back to `0.0.0.0`. There is no non-loopback IPv6 bind path here.
  - file: [src/gateway/net.ts:298](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/net.ts#L298)
  - command: `nl -ba src/gateway/net.ts | sed -n '280,360p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Runtime config explicitly rejects non-IPv4 custom bind hosts:** `resolveGatewayRuntimeConfig()` enforces `gateway.bind=custom` with a `valid IPv4 customBindHost`, so operators cannot configure `::` or any other IPv6 address through the supported config surface.
  - file: [src/gateway/server-runtime-config.ts:67](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/server-runtime-config.ts#L67)
  - command: `nl -ba src/gateway/server-runtime-config.ts | sed -n '1,260p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Only loopback gets an IPv6 companion listener:** `resolveGatewayListenHosts()` adds `::1` only when the resolved bind host is `127.0.0.1`; otherwise it passes the original host through unchanged. The corresponding test expects `0.0.0.0` to stay as-is and only `127.0.0.1` to expand to `["127.0.0.1", "::1"]`.
  - file: [src/gateway/net.ts:398](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/gateway/net.ts#L398)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Current tests and docs still describe IPv4 bind targets:** Tests still assert `lan -> 0.0.0.0`, and the public gateway docs describe `lan` as `0.0.0.0` while listing `::`/`::1` only as legacy host aliases, not supported bind-mode values.
  - file: [docs/gateway/configuration-reference.md:304](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/gateway/configuration-reference.md#L304)

- **Issue discussion aligns with the remaining gap:** The issue body reports IPv6 SYNs being reset because the gateway only listens on IPv4, and the existing comment specifically asks for dual-stack listening plus an IPv6 loopback regression test. Current `main` only covers the loopback alias case, not general IPv6 listening.
