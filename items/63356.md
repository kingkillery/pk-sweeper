## Summary

Keep open. Current `main` still does not implement the PR’s UUID-preserving shared Signal allowlist normalization, and the checked-in code remains internally inconsistent with existing Signal UUID sender support.

## Evidence

- **Shared Signal formatter still coerces UUID entries as phone numbers:** `signalConfigAdapter.formatAllowFrom` maps every non-`*` entry through `normalizeE164(entry.replace(/^signal:/i, ""))`, so `uuid:...` entries are not preserved in the shared adapter on current `main`. That is the exact gap this PR addresses.
  - file: [extensions/signal/src/shared.ts:39](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/signal/src/shared.ts#L39)
  - command: `nl -ba extensions/signal/src/shared.ts | sed -n '1,220p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Shared DM-policy normalizer still lacks UUID handling:** `signalSecurityAdapter.normalizeDmEntry` also normalizes every entry as E.164 after stripping `signal:`, so shared DM-policy normalization still cannot represent `uuid:` sender ids correctly.
  - file: [extensions/signal/src/shared.ts:59](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/signal/src/shared.ts#L59)
