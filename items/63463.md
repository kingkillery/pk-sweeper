## Summary

Current `main` does not show the obvious `mlx/<id>` prefixing bug described in the issue: custom-provider resolution preserves the configured model ID, OpenAI-compatible requests send `model.id` on the wire, and the docs now explicitly classify `curl` success plus agent-runtime failure as commonly caused by backend compatibility limits rather than provider/model ref normalization. That said, I could not run a live MLX repro in this environment, so the evidence is not strong enough for a conservative auto-close.

## Evidence

- **Custom model keys are not blindly double-prefixed:** `modelKey()` returns the raw model id unchanged when it already starts with `<provider>/`, which is the main guard against `provider/provider/model` regressions.
  - file: [src/agents/model-ref-shared.ts:13](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/model-ref-shared.ts#L13)

  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Configured fallback models keep the raw configured `modelId`:** When resolving a configured custom provider fallback, OpenClaw builds the runtime model with `id: modelId` and does not prepend the provider into the wire model id.
  - file: [src/agents/pi-embedded-runner/model.ts:540](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/pi-embedded-runner/model.ts#L540)
