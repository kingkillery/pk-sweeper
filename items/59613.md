## Summary

Keep open. The report is recent, has concrete Docker mount/log evidence showing the sandbox workspace mounted read-only under `workspaceAccess: "none"`, has an independent confirmation after upgrading to 2026.4.2, and the timeline links a related still-open sandbox access request. I could not perform the required local source/history verification because every read-only shell command failed before execution with the runner sandbox error, so there is not enough high-confidence proof to close as implemented on current main.

## Evidence

- **Issue reproduction evidence:** Reporter provided Docker inspect output showing the sandbox bind `/home/node/.openclaw/sandboxes/agent-main-f331f052:/workspace:ro,z` with `RW: false`, plus tool logs where writes to `/workspace/IDENTITY.md` fail as read-only.
- **Independent confirmation:** A second user commented that they saw the same issue after upgrading to 2026.4.2, so this is not a single unclear report.
- **Related open discussion:** Timeline cross-references #59718, an open request about writable custom binds with `workspaceAccess: "none"`, which suggests the surrounding sandbox access behavior still needs maintainer judgment rather than automated closure.
- **Intended behavior docs:** Current sandbox docs describe `workspaceAccess: "none"` as tools seeing a sandbox workspace under `~/.openclaw/sandboxes`, while `"ro"` is the mode that mounts the agent workspace read-only at `/agent`. This supports the report as a plausible behavior bug if `/workspace` is read-only in `none` mode, but docs alone are not source/test proof of a fix.
  - file: [docs/gateway/sandboxing.md](https://github.com/openclaw/openclaw/blob/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d/docs/gateway/sandboxing.md)
