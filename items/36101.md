## Summary

Keep open. Current `main` does not implement per-chat-type model routing within a single agent. The only built-in model override here is `channels.modelByChannel`, which is keyed by channel/conversation identifiers, and the runtime resolver does not select models by `chatType`. OpenClaw does have a workaround via multi-agent bindings plus per-agent models, including wildcard `peer.kind` routing for `direct` vs `group`, but that requires separate agents and changes isolation/session semantics compared with the requested same-agent config.

## Evidence

- **No chat-type model config surface found:** `rg -n 'byChatType|model.*chatType|chatType.*model' src docs ui extensions packages -g '!**/dist/**'` returned no matches, so there is no existing `model.byChatType`-style config or runtime path.
  - command: `rg -n 'byChatType|model.*chatType|chatType.*model' src docs ui extensions packages -g '!**/dist/**'`
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Channel override type is keyed by channel id only:** `ChannelModelByChannelConfig` is `Record<string, Record<string, string>>`, and `channels.modelByChannel` is documented as `Map provider -> channel id -> model override`, not a chat-type map.
  - file: [src/config/types.channels.ts:27](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/config/types.channels.ts#L27)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Runtime model override resolver does not branch on DM vs group:** `resolveChannelModelOverride()` accepts `groupId`, `groupChatType`, `groupChannel`, `groupSubject`, and `parentSessionKey`, but it only builds lookup candidates from conversation ids, parent candidates, channel names, and subjects. There is no branch that chooses one model for `direct` and another for `group` as classes.
  - file: [src/channels/model-overrides.ts:33](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/channels/model-overrides.ts#L33)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Existing supported workaround is separate agents plus bindings:** Docs explicitly say per-agent defaults can override `agents.defaults.model` via `agents.list[].model` plus bindings, and binding match fields support `match.peer` with `{ kind: direct|group|channel, id }`.
  - file: [docs/concepts/models.md:33](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/docs/concepts/models.md#L33)
  - sha: [7d4718373698](https://github.com/openclaw/openclaw/commit/7d471837369885c12d540d6bafa319ff41129b89)
- **Wildcard peer-kind routing exists, but only at agent-routing level:** Tests cover wildcard peer bindings for `peer: { kind: "direct", id: "*" }` and `peer: { kind: "group", id: "*" }`, proving DM-vs-group routing is available only by sending those chats to different agents.
  - file: [src/routing/resolve-route.test.ts:930](https://github.com/openclaw/openclaw/blob/7d471837369885c12d540d6bafa319ff41129b89/src/routing/resolve-route.test.ts#L930)

## Risks / Open Questions

- Closing this as implemented would hide a real gap: current main cannot choose different models for DM vs group within one agent config.
- The current workaround requires separate agents, which changes workspace/auth/session isolation and is materially different from the requested same-agent model override.
