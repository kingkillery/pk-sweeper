## Summary

Current `main` already provides a typed, plugin-facing pre-tool authorization surface plus gateway-backed plugin approval flows across general tool execution paths, so this feature request is effectively implemented.

## Risks / Open Questions

- The implementation uses the existing plugin hook and plugin-approval contract rather than introducing a separately named `GuardrailProvider` interface/config block, so the close rationale depends on treating that existing seam as the requested pluggable provider surface.

## Close Comment

Closing as implemented on `main` after a Codex review. OpenClaw already exposes a typed pre-tool authorization seam through `before_tool_call` (`src/plugins/hook-types.ts:318`), and the runtime enforces it for tool execution in `src/agents/pi-tools.before-tool-call.ts:247`, where plugins can block a call, rewrite params, or require approval through `plugin.approval.request` / `plugin.approval.waitDecision`. The gateway-side plugin approval API is already implemented in `src/gateway/server-methods/plugin-approval.ts:26`, plugin authors are already documented to use this surface in `docs/plugins/building-plugins.md:184`, and tests cover non-exec paths including MCP and HTTP tool execution in `src/mcp/plugin-tools-serve.test.ts:92` and `src/gateway/tools-invoke-http.test.ts:441`. I also verified the same capability is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
