## Summary

Keep open. Current `main` does not expose an HTTP JSON-RPC `/rpc` gateway endpoint, and the PR discussion still contains unresolved security and runtime-parity concerns against the proposed implementation. This is neither implemented-on-main nor a safe conservative cleanup close.

## Evidence

- **Current main has no HTTP RPC route:** `createGatewayHttpServer` builds HTTP stages for hooks, models, tools invoke, session kill/history, OpenAI-compatible endpoints, and plugin HTTP handling, but there is no `http-rpc` stage or `handleHttpRpcEndpoint` on `main`. That means the PR's feature is not already shipped.
  - file: [src/gateway/server-http.ts:927](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/gateway/server-http.ts#L927)

  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Current docs do not describe a gateway `/rpc` endpoint:** The user-facing RPC docs still describe JSON-RPC only as an adapter pattern for external CLIs such as `signal-cli` and legacy `imsg`. There is no documented gateway-side HTTP JSON-RPC endpoint on current `main`.
  - file: [docs/reference/rpc.md:9](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/docs/reference/rpc.md#L9)
