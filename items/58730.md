## Summary

Keep open. The issue is a concrete, recent feature/security request and current evidence shows it is only partially covered on main: OpenClaw has exec approvals, tool policy, transcript persistence, and compaction infrastructure, but the requested OS-level exec sandboxing is still represented by an open draft PR rather than landed main behavior.

## Evidence

- **Exec approvals exist but are not the full sandbox request:** Current docs describe exec approvals as a host guardrail where policy, allowlist, and optional user approval must agree, and state that approvals stack on top of tool policy/elevated gating. This partially addresses the permission-model part but is not the requested OS-level bwrap/sandbox-exec isolation.
  - file: [docs/tools/exec-approvals.md:251](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/docs/tools/exec-approvals.md#L251)
  - sha: [44114328b4e7](https://github.com/openclaw/openclaw/commit/44114328b4e73074ac72126f1230ddbf9c1ea8b5)
- **Current permission knobs are three-state plus ask modes, not the proposed four-level skill permission model:** Docs list security modes `deny`, `allowlist`, and `full`, and ask modes `off`, `on-miss`, and `always`; that is useful existing safety surface but does not exactly implement the issue's proposed `automatic` / `askOnce` / `askAlways` / `deny` per-skill model.
  - file: [docs/tools/exec-approvals.md:390](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/docs/tools/exec-approvals.md#L390)
  - sha: [44114328b4e7](https://github.com/openclaw/openclaw/commit/44114328b4e73074ac72126f1230ddbf9c1ea8b5)
- **Session persistence and compaction are already real but do not close the whole bundled request:** Session docs describe `sessions.json`, JSONL transcripts, compaction entries, auto-compaction triggers, and pre-compaction memory flushes, so parts 3 and 4 of the issue are substantially represented. The issue also asks for exec sandboxing and broader hooks/permission design, so this is not sufficient to close the whole item as implemented.
