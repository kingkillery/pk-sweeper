## Summary

Keep open. Current main does not implement offline APNs chat reply alerts, and #67493 contains plausible but incomplete work rather than obsolete or shipped work. The PR is not wired into the chat lifecycle, its proposed helper reuse would need a chat-specific APNs payload kind, and current iOS notification routing has no chat-reply tap path. Related #63800 is open related work, not a merged/superseding implementation that makes this PR safely closeable.

## Best Possible Solution

Keep this PR open, but require it to become an end-to-end implementation before merge: add a dedicated APNs chat reply payload/send helper with a chat-specific kind, add iOS handling for that kind, wire both agent-event and direct chat.send completion paths with request device/connection metadata and canonical session-key checks, cover direct/relay/stale-registration/offline-suppression paths in tests, and reconcile whether #67493 supersedes or replaces the older #63800 branch.

## Risks / Open Questions

- Merging #67493 as-is would add isolated APNs chat code but no chat lifecycle wiring, so users would not receive offline chat reply alerts.
- The proposed use of sendApnsAlert would emit the diagnostic push.test payload kind unless the implementation adds a dedicated chat.reply payload and matching iOS handling.
- The remaining wiring needs careful canonical session-key handling and per-device/per-connection offline detection to avoid suppressing valid main-session alerts or sending alerts while the requester is still connected.
