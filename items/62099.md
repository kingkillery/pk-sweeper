## Summary

Keep open. Current main still rewrites `auth-profiles.json` during post-success auth bookkeeping, and a Windows `EPERM` in that write path can still bubble through `markAuthProfileGood` / `markAuthProfileUsed` and fail an otherwise successful gateway request. The `auth-state.json` split is only a partial mitigation, not a full fix for the reported cascade.

## Evidence

- **Post-success bookkeeping still rewrites auth-profiles.json:** `saveAuthProfileStore` always calls `saveJsonFile(authPath, payload)` before writing `auth-state.json`, so even pure routing-state updates still touch `auth-profiles.json`. That preserves the reported failure trigger if the auth file is read-only.
  - file: [src/agents/auth-profiles/store.ts:396](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/auth-profiles/store.ts#L396)
  - command: `nl -ba src/agents/auth-profiles/store.ts | sed -n '375,397p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **markAuthProfileGood can still throw on save failure:** `markAuthProfileGood` falls back to a direct `saveAuthProfileStore(store, agentDir)` when the locked update returns `null`. `updateAuthProfileStoreWithLock` returns `null` on any caught write/lock error, so a write failure can immediately retry outside the catch and escape the request path.
  - file: [src/agents/auth-profiles/profiles.ts:153](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/auth-profiles/profiles.ts#L153)

  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **markAuthProfileUsed repeats the same fatal fallback:** `markAuthProfileUsed` has the same `updated ? return : saveAuthProfileStore(...)` pattern, and the embedded runner awaits it immediately after `markAuthProfileGood` on successful runs. A readonly-auth-store failure can therefore still cascade after the model already succeeded.
  - file: [src/agents/auth-profiles/usage.ts:345](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/auth-profiles/usage.ts#L345)

  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Windows EPERM handling does not clear readonly or downgrade the error:** `renameJsonFileWithFallback` handles `EPERM`/`EEXIST` only by retrying with `copyFileSync`. There is no readonly-attribute clearing and no non-fatal handling here, so a readonly target can still fail the save.
  - file: [src/infra/json-file.ts:80](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/infra/json-file.ts#L80)
