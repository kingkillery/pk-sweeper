## Summary

Keep open. Current `main` still builds a single shared `MemoryDB` inside `memory-lancedb`, so the reported cross-agent leakage is not already fixed, and the PR discussion identifies unresolved regressions in the proposed patch.

## Evidence

- **Current main still shares one MemoryDB across all tool calls:** `register()` constructs one `MemoryDB` instance (`const db = new MemoryDB(...)`) and the tool handlers reuse that same closure-backed `db` for recall/store/forget operations. There is no per-agent DB selection on current `main`.
  - file: [extensions/memory-lancedb/index.ts:312](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-lancedb/index.ts#L312)
  - command: `nl -ba extensions/memory-lancedb/index.ts | sed -n '280,470p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Plugin tool context always resolves an agentId:** `resolveOpenClawPluginToolInputs()` always sets `context.agentId` from `resolveSessionAgentId()`, and `resolveSessionAgentId()` falls back to the configured default agent when no explicit agent is encoded. That matches the PR review warning that a truthy `agentId` check would send ordinary single-agent runs to `<dbPath>/main` instead of the legacy root path.
  - file: [src/agents/openclaw-tools.plugin-context.ts:33](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/openclaw-tools.plugin-context.ts#L33)

- **PR discussion already surfaced unresolved defects in the proposed fix:** The provided PR review context on commit `ee677bc30552ce334ed0eb2f0ffa3c9d111e48b5` flags two nontrivial issues: unsanitized `agentId` path joining and a backward-compat regression where default single-agent runs would move from the historical root DB path to `<dbPath>/main`. That makes the PR useful but not clearly ready to close as obsolete or fully resolved.
