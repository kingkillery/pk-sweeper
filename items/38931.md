## Summary

Keep open. Current `main` does not implement the requested SSRF confirmation workflow. Browser SSRF remains a strict-vs-dangerous boolean policy with hostname exceptions, while some of the request's safety goals are already partially covered by existing hard blocks for reserved hostnames and special-use/private IP ranges.

## Evidence

- **Browser SSRF config only exposes boolean opt-in plus hostname exceptions:** `BrowserSsrFPolicyConfig` contains `dangerouslyAllowPrivateNetwork`, `allowedHostnames`, and `hostnameAllowlist`; there is no `confirm` mode, timeout, blocklist, or per-request approval state.
  - file: [src/config/types.browser.ts:19](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/config/types.browser.ts#L19)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Schema matches the limited config surface:** The validated config schema for `browser.ssrfPolicy` only accepts `dangerouslyAllowPrivateNetwork`, `allowedHostnames`, and `hostnameAllowlist`.
  - file: [src/config/zod-schema.ts:375](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/config/zod-schema.ts#L375)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Docs describe only strict mode, dangerous opt-in, and explicit hostname exceptions:** Current browser docs say private-network access is off by default, can be enabled with `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork`, and otherwise uses `hostnameAllowlist`/`allowedHostnames` for exceptions.
  - file: [docs/tools/browser.md:148](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/docs/tools/browser.md#L148)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)
- **Runtime policy is still binary, not approval-based:** `isPrivateNetworkAllowedByPolicy()` returns true only for the two boolean flags, and hostname resolution enforces allowlist/private-network checks immediately by throwing `SsrFBlockedError`; there is no pause-and-ask flow.
  - file: [src/infra/net/ssrf.ts:89](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/infra/net/ssrf.ts#L89)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)

  - file: [src/infra/net/ssrf.ts:63](https://github.com/openclaw/openclaw/blob/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe/src/infra/net/ssrf.ts#L63)
  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)

  - sha: [4d7c4b329892](https://github.com/openclaw/openclaw/commit/4d7c4b3298920c0e6e2c5cc8bd7a1546e8122abe)

## Risks / Open Questions

- The issue body overstates what is missing: cloud metadata and other special-use/private destinations are already blocked by default on current `main`, so any future work should target approval UX/state rather than re-adding existing blocks.
- Implementing interactive SSRF approvals would cross a core browser security boundary and likely needs product/security-owner review for approval UX, timeout behavior, redirect handling, and audit logging.
