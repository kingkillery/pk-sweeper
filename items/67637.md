## Summary

Keep open. Current main still does not add tool-call trace data to `QaBusMessage` or the qa-bus state/client path. The existing mock-provider `/debug/requests` plumbing is useful adjacent work and several YAML flows now use `plannedToolName` for mock-backed tool-call assertions, but that is not equivalent to the requested `QaBusMessage.toolCalls` field populated from the agent run's tool invocation log. It is mock-provider scoped, separate from outbound bus messages, and the linked parity PR #67540 is closed unmerged in the provided context.

## Best Possible Solution

Keep the issue open and implement an additive qa-lab/qa-channel contract: add an optional tool-call trace field to the public qa-channel protocol, thread it through qa-bus state, cloning, HTTP client, snapshots, and suite transport helpers, and populate it from the real agent tool-call path or plugin hook events with redaction. If maintainers intentionally prefer the existing mock `/debug/requests` route as the canonical assertion surface, document that decision and close manually rather than auto-closing this request.

## Risks / Open Questions

- Adding tool-call arguments to qa-bus messages can expose sensitive tool parameters; the implementation should bound and redact trace payloads before publishing them over the bus.
- `QaBusMessage` is a public plugin SDK protocol type, so this should remain additive and optional with tests for clone/snapshot/search/client roundtrips.
