## Summary

The issue asks for workflow-specific reliability guardrails on top of automation primitives that current main already exposes. Project scope and current docs point to packaging this as a ClawHub skill/plugin or workflow layer, not additional OpenClaw core behavior.

## Close Comment

Closing this as a ClawHub/plugin-scope item after a Codex review against current `main` (`cec34821754ad935c780805f3116f99449d46ac2`). The repo already has the core primitives for this class of automation: cron can target `current` or persistent `session:<id>` sessions (`docs/automation/cron-jobs.md:82`), browser profiles keep login/storage state outside a single run and the managed browser auto-starts on demand (`docs/cli/browser.md:90`, `extensions/browser/src/browser/server-context.availability.ts:204`, `extensions/browser/src/browser/routes/tabs.ts:183`), and OpenClaw already exposes higher-level workflow layers through Task Flow, Lobster, and the optional `llm-task` plugin (`docs/automation/taskflow.md:10`, `docs/tools/lobster.md:9`, `docs/tools/llm-task.md:9`).

What remains here is a workflow-specific reliability package for market-intelligence automation: preflight checks, freshness stamping, and orchestration guardrails. That matches the project scope guidance to keep core lean, ship optional capability as plugins, and publish new skills to ClawHub first (`VISION.md:52`, `VISION.md:66`, `VISION.md:99`, `docs/tools/clawhub.md:10`). If you still want this, the best next step is a ClawHub skill/plugin proposal that defines the concrete workflow contract and guardrails to package on top of the existing cron/browser/task surfaces.
