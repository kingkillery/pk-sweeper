## Summary

Keep open. Current main still has the shared managed-response wrapper releasing guarded fetch dispatchers only on natural EOF, stream error, explicit cancel, or no-body responses. The issue’s requested abandoned-stream safety net is not present, and the existing tests do not cover the reported abandoned/GC path. The newer Anthropic transport does drain normal SSE responses, which may reduce the happy-path leak, but that is not enough evidence to close the specific buildManagedResponse lifecycle bug.

## Best Possible Solution

Keep the issue open and fix the shared provider transport lifecycle. The best code path is to make buildManagedResponse robust when the returned body is not fully drained: add deterministic cleanup on abandoned/early stream teardown where possible, consider a FinalizationRegistry or bounded watchdog as a backstop, cancel or release the inner reader safely, and add focused tests that prove release() is called exactly once for natural EOF, explicit cancel, stream error, and abandoned wrapper cases. Also verify Anthropic Vertex/provider-owned SDK streaming still reaches the guarded cleanup contract.

## Risks / Open Questions

- A FinalizationRegistry-only fix would be nondeterministic and should be paired with deterministic cleanup where the transport owns the stream lifecycle.
- An idle-timeout fallback could prematurely close very slow but valid provider streams unless it is carefully scoped and tested.
- Current direct Anthropic draining may hide the issue on the common happy path, so regression coverage should include explicit abandoned-body and provider-owned SDK stream scenarios.
