## Summary

Current `main` still leaves `openclaw mcp serve` without the stderr-routing guard used by the other MCP stdio server, so the reported stdout pollution remains plausible. The PR’s production change matches an existing pattern, but the only added test described in review uses `InMemoryTransport` and does not actually validate stdio isolation, so this is not a candidate for conservative auto-close.

## Evidence

- **Channel MCP stdio path still lacks stderr routing on main:** `serveOpenClawChannelMcp()` creates the stdio transport and starts the server, but there is no `routeLogsToStderr()` call before `createOpenClawChannelMcpServer()`/`loadConfig()` can emit warnings.
  - file: [src/mcp/channel-server.ts:65](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/mcp/channel-server.ts#L65)
  - command: `nl -ba src/mcp/channel-server.ts | sed -n '1,140p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Config load happens before bridge startup:** `createOpenClawChannelMcpServer()` resolves `opts.config ?? loadConfig()` immediately, so any config-warning console output can happen before the stdio session is serving protocol frames.
  - file: [src/mcp/channel-server.ts:26](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/mcp/channel-server.ts#L26)

  - file: [src/mcp/tools-stdio-server.ts:24](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/mcp/tools-stdio-server.ts#L24)

- **Logging subsystem confirms the missing guard matters:** `routeLogsToStderr()` flips `loggingState.forceConsoleToStderr`, and the console forwarder writes all console output to `process.stderr` when that flag is set; without it, diagnostics continue to the original stdout console path.
  - file: [src/logging/console.ts:116](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/logging/console.ts#L116)
