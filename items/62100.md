## Summary

Keep open. Current `main` does not implement OpenRouter native slash-ref resolution for inputs like `minimax/...` or `qwen/...`, and the PR discussion identified real P1 regressions in the proposed patch: it would rewrite explicit provider-qualified refs to `openrouter` and widen the allowlist so provider-scoped entries could be bypassed.

## Evidence

- **Current parser still treats first path segment as the provider:** `parseModelRef` on `main` splits at the first slash and returns `normalizeModelRef(providerRaw, model, ...)` with no OpenRouter-native slash exception, so `minimax/foo` is still parsed as provider `minimax`, not `openrouter`.
  - file: [src/agents/model-selection-normalize.ts:75](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/model-selection-normalize.ts#L75)

  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Provider inference only runs for bare model ids, not slash-qualified ones:** `resolveAllowedModelRefFromAliasIndex` only calls `inferUniqueProviderFromConfiguredModels(...)` when the raw model string does not contain `/`. Slash-qualified inputs keep `defaultProvider` and flow through normal parsing, so `minimax/...` remains provider-qualified instead of being remapped to `openrouter`.
