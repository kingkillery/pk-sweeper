## Summary

Current `main` does not implement the requested cleanup. `cron.remove` only deletes the cron job record; isolated cron runs still persist a base `agent:<agentId>:cron:<jobId>` session entry, `sessions.list` still returns that base cron session, and the cron reaper explicitly only prunes `:run:` entries later. Control UI now hides cron sessions by default, which can mask the dropdown symptom, but that is not backend cleanup.

## Evidence

- **`cron.remove` only edits the cron store:** The remove path filters the job out of `state.store.jobs`, persists, re-arms the timer, and emits `removed`. There is no session-store mutation and no run-log deletion in this path.
  - file: [src/cron/service/ops.ts:332](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/cron/service/ops.ts#L332)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '330,360p'`
- **Isolated cron runs persist both base and run session keys:** When an isolated cron run starts, it writes the shared session entry under both the base cron key and the per-run `:run:` key, so deleting the job later needs explicit cleanup if the base key should disappear.
  - file: [src/cron/isolated-agent/run-session-state.ts:22](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/cron/isolated-agent/run-session-state.ts#L22)
  - command: `nl -ba src/cron/isolated-agent/run-session-state.ts | sed -n '1,220p'`
- **Cron run-session retention keeps the base cron session:** The session reaper comment and implementation explicitly target `...:cron:<jobId>:run:<uuid>` entries and state that the base `...:cron:<jobId>` session is kept as-is.
  - file: [src/cron/session-reaper.ts:1](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/cron/session-reaper.ts#L1)
  - command: `nl -ba src/cron/session-reaper.ts | sed -n '1,170p'`
- **Gateway session listing still exposes base cron sessions:** `sessions.list` filters out only cron run-session keys via `isCronRunSessionKey(key)`, not base cron session keys, so the leftover base entry remains listable.
  - file: [src/gateway/session-utils.ts:1483](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/session-utils.ts#L1483)
  - command: `nl -ba src/gateway/session-utils.ts | sed -n '1470,1515p'`
- **UI now hides cron sessions by default, but only as a filter:** The Control UI session selector defaults `sessionsHideCron` to `true` and skips cron sessions when building dropdown options. That can hide the symptom by default, but it does not remove the stale session record from `sessions.json`.
  - file: [ui/src/ui/app.ts:342](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/ui/src/ui/app.ts#L342)
  - command: `nl -ba ui/src/ui/app.ts | sed -n '336,346p'`
- **Docs describe retention for run sessions, not immediate delete cleanup:** Current docs say `cron.sessionRetention` prunes isolated run-session entries and `runLog.*` prunes log files; they do not describe `cron.remove` deleting the base cron session record on removal.
  - file: [docs/automation/cron-jobs.md:386](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/docs/automation/cron-jobs.md#L386)
  - command: `nl -ba docs/automation/cron-jobs.md | sed -n '380,390p'`
