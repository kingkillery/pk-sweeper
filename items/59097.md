## Summary

Keep open. The report is recent, not protected/maintainer-authored, and is coherent: it describes a real interaction between the buffered dispatcher’s chunking behavior and a non-chunked SSE-style channel/plugin. The issue may ultimately belong in the external atticus-channel plugin, but I do not have enough high-confidence current-main proof to close it as implemented, cannot-reproduce, or clawhub.

## Evidence

- **Issue scope:** Issue #59097 describes `dispatchReplyWithBufferedBlockDispatcher` chunking replies around the 4000-character default and an external `atticus-channel` delivery buffer that joins chunks with blank lines, creating visible breaks. The reporter recommends changing that plugin to a non-buffered dispatcher path.
- **Current behavior docs:** Public streaming/chunking docs describe that `message_end` still uses the chunker if buffered text exceeds `maxChars`, and that channel text chunk limits clamp the chunk size. This supports the plausibility of the reported behavior for code paths using buffered block dispatch.
