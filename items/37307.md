## Summary

Current main already covers both logged failure signatures from #37307: cron runs no longer use the old default LLM idle watchdog path that produced 15s `FailoverError: LLM request timed out.` failures, and cron-delivered message-tool runs now bind the resolved chat target so `Action send requires a target.` is regression-tested away.

## Close Comment

Closing this as a Codex review because current `main` already covers the failure path from this report. The scheduled-run timeout shown in the logs matches the old cron LLM idle-timeout behavior, and OpenClaw now disables that default watchdog for cron runs unless you configure one explicitly (`src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:54`, shipped in `v2026.4.9` per `CHANGELOG.md:1012`). Current `main` also binds cron delivery context into embedded/message-tool runs (`src/cron/isolated-agent/run-executor.ts:155`), with regression coverage asserting the resolved target is forwarded instead of failing with `Action send requires a target.` (`src/cron/isolated-agent/run.message-tool-policy.test.ts:143`). If this still reproduces on a current build, please open a new issue with fresh `openclaw cron runs --id <jobId>` output, the exact cron delivery config, and current logs.
