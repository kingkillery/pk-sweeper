## Summary

Keep open. Current main has a new sanitizer for some leaked tool-call text, but I did not find evidence that WhatsApp now converts or handles the issue’s exact `[TOOL_CALL]{...}[/TOOL_CALL]` raw-text form. The reported root cause remains plausible on main.

## Evidence

- **Issue discussion points to unexecuted text-form tool calls:** The reporter’s follow-up comments say WhatsApp receives raw text like `[TOOL_CALL]{tool => "web_search", args => {...}}[/TOOL_CALL]` rather than structured `toolCall` content, so the tool never executes and no real tool result is produced.
- **WhatsApp path still delegates to shared reply flow:** `processMessage` builds the WhatsApp reply pipeline and forwards to the shared reply resolver/dispatcher; `shouldSuppressWhatsAppPayload` only hides tool lifecycle payloads from user delivery, it does not parse textual tool-call syntax into executable calls.
  - file: [extensions/whatsapp/src/auto-reply/monitor/process-message.ts:326](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/whatsapp/src/auto-reply/monitor/process-message.ts#L326)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/process-message.ts | sed -n '320,414p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)

It does not match the issue’s uppercase bracket form `[TOOL_CALL]...[/TOOL_CALL]`.
