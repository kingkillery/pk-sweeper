## Summary

Keep open. Current `main` has session listing, agent-grouped session switching, and checkpoint branching, but it does not implement the requested Control UI project trees, threaded entry-point routing, message-split triage, or project metadata model.

## Evidence

- **Control UI docs only advertise flat session management:** The Control UI docs list current session support as `sessions.list` plus per-session overrides via `sessions.patch`; there is no documented project tree, thread split, or multi-thread routing feature.
  - file: [docs/web/control-ui.md:105](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/web/control-ui.md#L105)
  - command: `nl -ba docs/web/control-ui.md | sed -n '90,190p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Sessions tab renders a flat table, not a tree:** The Sessions view is a searchable/paginated table with columns for key, label, kind, updated time, tokens, compaction, and overrides. There is no tree component or project hierarchy in this UI.
  - file: [ui/src/ui/views/sessions.ts:326](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/ui/src/ui/views/sessions.ts#L326)
  - command: `nl -ba ui/src/ui/views/sessions.ts | sed -n '246,560p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Chat session picker groups by agent only:** `resolveSessionOptionGroups()` builds `<optgroup>` buckets by `agentId` or `Other Sessions`. It does not use parent/child session structure or project metadata to render a project tree.
  - file: [ui/src/ui/chat/session-controls.ts:447](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/ui/src/ui/chat/session-controls.ts#L447)
  - command: `nl -ba ui/src/ui/chat/session-controls.ts | sed -n '447,586p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Chat send path targets exactly one session key:** Control UI chat sends always call `chat.send` with a single `sessionKey` from UI state. There is no entry-point selector that fans one input out across multiple threads or routes to a different target thread.
  - file: [ui/src/ui/controllers/chat.ts:232](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/ui/src/ui/controllers/chat.ts#L232)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '232,243p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Persisted session model has ancestry fields but no project field:** Session persistence includes `spawnedBy` and `parentSessionKey`, plus labels/display names and channel context, but no persisted `project` metadata field matching the issue's proposed UI model.
  - file: [src/config/sessions/types.ts:132](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/sessions/types.ts#L132)

## Risks / Open Questions

- Backend/session primitives such as `spawnedBy`, `parentSessionKey`, and compaction branching exist, so a future implementation could build on them even though the requested Control UI workflow is not present now.
- `gh` inspection was unavailable in this environment because `GH_TOKEN` was not set, so related GitHub-side discussion/PR context could not be checked; the repo state on `main` was sufficient for this keep-open decision.
