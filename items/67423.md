## Summary

Current main does not provide enough evidence to close this. The runtime auth resolver only lets a configured provider apiKey beat ambient profiles when the provider entry also has auth: "api-key"; otherwise it builds the auth-profile order first and only falls back to the configured key later. The status label path also checks auth-profile order before models.json, matching the issue's reported /status symptom. This should remain open for a focused fix and regression test around split OpenRouter-style provider entries and explicit per-entry auth selection.

## Best Possible Solution

Keep the issue open. The best fix is to define the intended contract for models.providers.<id>.apiKey when it names or resolves to an auth profile, then update runtime auth resolution and status-label rendering to honor the resolved provider entry before canonical-provider auth.order rotation. Add a regression test using multiple OpenRouter-compatible provider entries with distinct apiKey/profile mappings and a /status or resolveModelAuthLabel assertion.
