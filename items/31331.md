## Summary

Current main still appears to have the reported Docker-outside-of-Docker sandbox bind-source bug. The Docker sandbox path builds managed `-v` mounts from the gateway-visible workspace path directly, and I found no host mount-table remapping implementation or regression test on main. The linked fix PRs in the provided GitHub context were closed unmerged, and commenters still reported the issue as recently as 2026-04-22.

## Evidence

- **workspaceAccess rw keeps gateway-internal workspace path:** `ensureSandboxWorkspaceLayout` resolves `agentWorkspaceDir` from the configured workspace path and sets `workspaceDir = agentWorkspaceDir` when `workspaceAccess === "rw"`; this is the container-internal path in the Docker gateway install described by the issue.
  - file: [src/agents/sandbox/context.ts:38](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/agents/sandbox/context.ts#L38)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '29,90p'`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **docker create uses workspaceDir without host remap:** `createSandboxContainer` passes `workspaceDir` and `agentWorkspaceDir` straight into validation and `appendWorkspaceMountArgs`; there is no inspection of the gateway container mounts or source-path translation before creating the sandbox container.
  - file: [src/agents/sandbox/docker.ts:482](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/agents/sandbox/docker.ts#L482)
