## Summary

Keep open. Current `main` does not implement the requested channel tool restrictions, and the proposed PR still has unresolved correctness/regression concerns around owner-only enforcement and over-broad provider scoping.

## Evidence

- **Current main still leaves non-voice providers unfiltered:** `filterToolNamesByMessageProvider` only deny-lists `voice` and allow-lists `node`; every other non-empty provider falls through to `return [...toolNames]`. That means the reported gap is still present on `main` for channel providers such as Telegram/Discord/Slack/Signal/WeChat.

## Risks / Open Questions

- The underlying security gap remains on current `main`: external channel providers still pass through with the full tool list unless explicitly deny-listed.
- Merging this PR as-is still appears risky because its fallback is broad enough to affect internal/trusted providers and memory-flush behavior, and its `exec` owner-only hardening is incomplete against the actual runtime authorization path.
