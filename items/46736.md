## Summary

Current `main` does not show a stock WebChat code path that would prepend an operator skill-store policy block to the user's message. The Control UI sends only the raw message to `chat.send`, prompt-hook/operator additions are applied only to the prompt/system prompt, and chat history rendering strips injected metadata before display.

## Evidence

- **Stock WebChat send path:** The Control UI `requestChatSend` call sends only `sessionKey`, raw `message`, `deliver`, `idempotencyKey`, and optional attachments. It does not send any operator/system receipt field that could prefix policy text.
  - file: [ui/src/ui/controllers/chat.ts:232](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/ui/src/ui/controllers/chat.ts#L232)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '220,255p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Gateway stores raw message text:** `chat.send` builds `messageForAgent` from the parsed message plus an optional explicit `systemProvenanceReceipt`; it stores `Body` as that value and only adds the timestamp to `BodyForAgent`. There is no stock WebChat-side operator policy injection here.
  - file: [src/gateway/server-methods/chat.ts:2273](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/server-methods/chat.ts#L2273)

## Risks / Open Questions

- This review was read-only and code-path based; I did not run an end-to-end WebChat session against current `main`.
- A custom client, proxy, or local middleware that injects `systemProvenanceReceipt` outside the stock Control UI path could still prepend text, but that is not reproducible from the built-in WebChat code path on current `main`.

## Close Comment

Codex review against current `main` (`56fe2aab9c87596515f2280c8060dc740ea64a88`) could not reproduce a current stock WebChat path for this. The built-in Control UI only sends the raw `message` to `chat.send` (`ui/src/ui/controllers/chat.ts:232`), the gateway stores that raw message/timestamped variant rather than hook-added operator prompt text (`src/gateway/server-methods/chat.ts:2273`), and prompt-hook additions are applied only to the runtime prompt/system prompt (`src/agents/pi-embedded-runner/run/attempt.ts:1980`). Current history/display paths also strip injected metadata before rendering (`src/gateway/chat-sanitize.ts:51`, `ui/src/ui/chat/message-normalizer.ts:372`). On that basis this report no longer matches current `main`. If you can still reproduce it on a current build, please open a fresh issue with the exact current version and whether anything custom is sitting in front of the stock `chat.send` path.
