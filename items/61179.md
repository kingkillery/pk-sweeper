## Summary

Keep open. Current `main` still contains the exact probe path that can produce repeated pre-connect `1008` device-required closes on loopback, and it still logs those closes at `warn` level. I did not find strong evidence that this issue was implemented away on `main` or shipped in a later release.

## Evidence

- **`gateway status` still probes by default:** The status command is described as `Show gateway service status + probe connectivity/capability`, and the implementation passes `probe: Boolean(cmdOpts.probe)` with `--no-probe` as the opt-out. That means repeated `openclaw gateway status` calls still drive repeated probe handshakes unless the caller disables probing.
  - file: [src/cli/daemon-cli/register-service-commands.ts:53](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cli/daemon-cli/register-service-commands.ts#L53)
  - command: `nl -ba src/cli/daemon-cli/register-service-commands.ts | sed -n '52,74p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Default status probe still uses the lightweight hello-only path:** `probeGatewayStatus()` calls `probeGateway(... includeDetails: false)` for normal `gateway status`, so the default path still does a connect-capability probe rather than a richer RPC path.
  - file: [src/cli/daemon-cli/probe.ts:69](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cli/daemon-cli/probe.ts#L69)

  - file: [src/gateway/probe.ts:157](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/gateway/probe.ts#L157)
