## Summary

Keep open. Current `main` does not expose a user-configurable WhatsApp/Baileys socket passthrough such as `web.socketConfig`, even though the report's older proxy details are partly outdated because ambient env proxy support now exists.

## Evidence

- **`createWaSocket` still hardcodes the Baileys option bag:** `createWaSocket` only accepts `{ authDir?, onQr? }` and calls `makeWASocket(...)` with fixed fields (`auth`, `version`, `logger`, `browser`, `syncFullHistory`, `markOnlineOnConnect`, `agent`, `fetchAgent`). There is no config-derived spread/passthrough for `defaultQueryTimeoutMs`, `keepAliveIntervalMs`, `connectTimeoutMs`, or arbitrary socket options.
  - file: [extensions/whatsapp/src/session.ts:121](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/whatsapp/src/session.ts#L121)
  - command: `nl -ba extensions/whatsapp/src/session.ts | sed -n '121,164p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **WhatsApp config schema has no socket passthrough field:** The WhatsApp zod schema defines channel/account fields like DM policy, history, media, reactions, debounce, heartbeat, and health monitor, but no `socketConfig` or Baileys timing/proxy config field.
  - file: [src/config/zod-schema.providers-whatsapp.ts:51](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/zod-schema.providers-whatsapp.ts#L51)
  - command: `nl -ba src/config/zod-schema.providers-whatsapp.ts | sed -n '51,159p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Public WhatsApp config types also omit socket config:** The exported `WhatsAppConfig` / `WhatsAppAccountConfig` types include standard channel settings plus `authDir`, but no socket-level passthrough for Baileys connection timing or proxy options.
  - file: [src/config/types.whatsapp.ts:53](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/types.whatsapp.ts#L53)
  - command: `nl -ba src/config/types.whatsapp.ts | sed -n '53,147p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Current docs describe only env-proxy plus heartbeat/reconnect knobs:** The WhatsApp docs say transport honors host `HTTPS_PROXY`/`HTTP_PROXY` env vars and list `web.enabled`, `web.heartbeatSeconds`, and `web.reconnect.*` as the operational tuning fields. They do not document any user-facing socket passthrough.
  - file: [docs/channels/whatsapp.md:148](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/docs/channels/whatsapp.md#L148)
  - command: `nl -ba docs/channels/whatsapp.md | sed -n '148,153p;570,575p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Runtime tuning reader only consumes heartbeat/reconnect from `cfg.web`:** The reconnect helper reads `cfg.web?.heartbeatSeconds` and `cfg.web?.reconnect` only. There is no reader for a socket option object that could reach `makeWASocket`.
  - file: [extensions/whatsapp/src/reconnect.ts:23](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/whatsapp/src/reconnect.ts#L23)
