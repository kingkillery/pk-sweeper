## Summary

Keep open. Current `main` has adjacent hardening (`tools.fs.workspaceOnly`, sandbox `workspaceAccess`, sandbox bind mounts), but it does not implement the requested `tools.fileAccess` allowlist/denylist config or equivalent host-side per-path enforcement for read/write/edit/exec.

## Evidence

- **No `tools.fileAccess` config surface on main:** `FsToolsConfig` only exposes a single boolean `workspaceOnly` for filesystem tools; there is no `fileAccess`, `allowedPaths`, or `denyPaths` config in the tool types.
  - file: [src/config/types.tools.ts:296](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/types.tools.ts#L296)
  - command: `nl -ba src/config/types.tools.ts | sed -n '280,310p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Runtime schema still rejects unknown FS keys:** The Zod schema for `tools.fs` is a strict object with only `workspaceOnly`, so `tools.fileAccess` is not part of the accepted config on current `main`. A repo search for `fileAccess` returned no matches under `src/`, `docs/`, `packages/`, or `extensions/`.
  - file: [src/config/zod-schema.agent-runtime.ts:484](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/config/zod-schema.agent-runtime.ts#L484)
  - command: `nl -ba src/config/zod-schema.agent-runtime.ts | sed -n '476,492p'; rg -n "fileAccess" src docs packages extensions`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Current enforcement is workspace-only, not allow/deny path lists:** Tool setup resolves a single `workspaceOnly` flag, applies it to read/write/edit/apply_patch, and passes `exec` a workspace `cwd`; there is no per-path allowlist/denylist enforcement for exec or arbitrary host paths.
  - file: [src/agents/pi-tools.ts:426](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/pi-tools.ts#L426)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '426,556p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Host FS remains unrestricted when `workspaceOnly` is off:** The host write/edit operations explicitly allow reads and writes to arbitrary resolved absolute paths when `workspaceOnly` is false, which does not meet the requested sandbox/allowlist behavior.
  - file: [src/agents/pi-tools.read.ts:760](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/pi-tools.read.ts#L760)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '757,830p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Docs describe different mitigations than the requested feature:** Security docs currently point users to sandbox `workspaceAccess`, tool deny lists, `tools.fs.workspaceOnly`, and sandbox bind mounts for allowlisted folders. That is useful adjacent functionality, but not the requested host-side `tools.fileAccess.allowedPaths/denyPaths` feature.
  - file: [docs/gateway/security/index.md:1022](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/gateway/security/index.md#L1022)

## Risks / Open Questions

- There is meaningful overlap with existing hardening knobs (`tools.fs.workspaceOnly`, sandbox `workspaceAccess`, sandbox Docker `binds`), so maintainers may eventually want to consolidate or retitle this request instead of implementing the exact proposed schema.
- The issue specifically asks for host-side tool-layer enforcement without full sandboxing; current main does not provide that, so closing as implemented would be inaccurate.
