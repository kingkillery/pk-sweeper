## Summary

Keep open. Current `main` still builds `openclaw health` channel summaries from a reduced snapshot that omits runtime/account metadata, while `channels status` uses the full runtime-backed account snapshot path. For Telegram, that still allows `health --json` to fall back to `running: false`, `tokenSource: "none"`, and `mode: null` even when `channels status --json` reports live runtime state.

## Evidence

- **`health` still drops runtime/account fields before summary formatting:** `getHealthSnapshot()` creates `snapshot` with only `accountId`, `enabled`, `configured`, plus optional `probe`/`lastProbeAt`, then passes that directly to `plugin.status.buildChannelSummary()`. No runtime snapshot, `tokenSource`, `running`, `lastStartAt`, or `mode` is populated on this path.
  - file: [src/commands/health.ts:324](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/commands/health.ts#L324)
  - command: `nl -ba src/commands/health.ts | sed -n '300,370p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **`channels status` still uses the runtime-backed snapshot path:** `channels.status` resolves `runtimeSnapshot`, passes it into `buildChannelAccountSnapshot()`, and only then calls `buildChannelSummary()`. That path preserves runtime lifecycle fields and account-derived metadata before channel summary formatting.
  - file: [src/gateway/server-methods/channels.ts:228](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/server-methods/channels.ts#L228)
  - command: `nl -ba src/gateway/server-methods/channels.ts | sed -n '170,320p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Shared summary helpers still default missing fields to the reported false/null values:** `buildBaseChannelStatusSummary()` defaults missing lifecycle fields to `running: false`, `lastStartAt: null`, `lastStopAt: null`; `buildTokenChannelStatusSummary()` defaults missing `tokenSource` to `"none"` and missing `mode` to `null`. With the reduced `health` snapshot, the reported Telegram output in the issue is still the expected result.
  - file: [src/plugin-sdk/status-helpers.ts:105](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/plugin-sdk/status-helpers.ts#L105)
  - command: `nl -ba src/plugin-sdk/status-helpers.ts | sed -n '78,120p'; nl -ba src/plugin-sdk/status-helpers.ts | sed -n '296,350p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Telegram still exposes the missing fields only through its computed account snapshot path:** Telegram's status adapter adds credential-derived fields via `projectCredentialSnapshotFields(account)` and runtime-derived `mode`/`lastError` in `resolveAccountSnapshot()`. `buildChannelSummary()` then formats those fields. Since `health` bypasses `buildAccountSnapshot()`, it still misses the data that `channels status` includes.
  - file: [extensions/telegram/src/channel.ts:762](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/telegram/src/channel.ts#L762)

- **No current regression test covers runtime parity for `health`:** `src/commands/health.snapshot.test.ts` still validates probe/configured behavior only. It does not assert parity with runtime-backed fields like `running`, `tokenSource`, `lastStartAt`, or `mode`, so there is no evidence on `main` that this issue was fixed.
