## Summary

Keep open. Current `main` still appears to leave the reported `modelByChannel` `/status` mismatch unresolved: the request path can switch the effective model for the session, but the status rendering path does not pass that resolved context window as a runtime snapshot, and when the stored session runtime differs from the selected model it still prefers persisted/session-derived context-window data.

## Evidence

- **`modelByChannel` only changes the effective request model:** `get-reply.ts` resolves `channels.modelByChannel` and overwrites the in-flight `provider`/`model` for the reply flow when there is no session override, but this block does not update `sessionEntry.modelProvider`, `sessionEntry.model`, or `sessionEntry.contextTokens`. That matches the issue's premise: selected runtime can change while persisted session fields stay stale until a later run persists new usage.
  - file: [src/auto-reply/reply/get-reply.ts:315](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/get-reply.ts#L315)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '315,358p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **`/status` knows the resolved context window but does not pass it as a runtime snapshot:** `buildStatusText()` forwards the current `provider`/`model` and `contextTokens` into `buildStatusMessage()`, but it places the token limit under `agent.contextTokens` and does not set `runtimeContextTokens`. That matters because the stale-runtime branch in `buildStatusMessage()` only consults `runtimeContextTokens`, not `agent.contextTokens`.
  - file: [src/status/status-text.ts:297](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/status/status-text.ts#L297)
  - command: `nl -ba src/status/status-text.ts | sed -n '297,315p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **When selected and stored runtime differ, status still prefers persisted/session runtime values:** `buildStatusMessage()` detects when the active runtime model differs from the selected model, but in that branch it falls back to `persistedContextTokens` and `activeContextTokens` from the stored session entry unless `runtimeContextTokens` was explicitly provided. It does not use the selected model's `agent.contextTokens` there. For a session whose stored runtime/context still point at the previous model, this logic still yields the stale window the issue describes.
  - file: [src/status/status-message.ts:633](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/status/status-message.ts#L633)
  - command: `nl -ba src/status/status-message.ts | sed -n '592,674p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Current tests do not cover the reported `modelByChannel` stale-context case:** Main has a test that `/status` labels channel overrides, and separate tests that recompute stale context windows for fallback sessions, but there is no regression test covering a `modelByChannel`-selected model with stale persisted session runtime/context fields. That leaves this exact report unproven on `main`.
  - file: [src/auto-reply/status.test.ts:508](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/status.test.ts#L508)
