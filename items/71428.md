## Summary

Keep open. Current main implements the config-level `plugins.entries.<id>.hooks.allowConversationAccess` path, but not the requested manifest-declared `permissions.conversationAccess` capability. The manifest type/loader does not carry a permissions field, and hook enforcement still receives only `plugins.entries.<id>.hooks` as policy. This is a core plugin-manifest/security contract request, not something a plugin can satisfy externally without a new OpenClaw manifest/API path.

## Best Possible Solution

Add an explicit, additive manifest-level permission contract for conversation-access hooks, normalize it through `src/plugins/manifest.ts` and `src/plugins/manifest-registry.ts`, merge it with the existing config override in the loader/registry policy path, and document/test precedence. Keep the existing `plugins.entries.<id>.hooks.allowConversationAccess` path for backward compatibility and operator override.
