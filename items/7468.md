## Risks / Open Questions

- If maintainers later decide token-threshold routing needs first-class built-in UX rather than plugin/provider ownership, this issue may need to be reopened.
- The current hook seam exposes prompt and attachment metadata, not a guaranteed final token-count estimate after full prompt assembly, so some routing strategies may need plugin-side heuristics or an external router.

## Close Comment

Closing this after a Codex review of current `main` (`cec34821754ad935c780805f3116f99449d46ac2`). The request is useful, but it fits OpenClaw's plugin/provider extension model rather than core. Current main still documents core model selection as fixed `primary -> fallbacks -> provider auth failover` in `docs/concepts/models.md:14`, while `VISION.md:52` says optional capability should usually ship as plugins. The plugin architecture also explicitly reserves model/provider override work for `before_model_resolve` in `docs/plugins/architecture.md:96`, and that hook already runs before model resolution and can override provider/model using the incoming prompt and attachment metadata in `src/plugins/hook-before-agent-start.types.ts:7` and `src/agents/pi-embedded-runner/run/setup.ts:58`. For router/proxy style solutions, the provider SDK already exposes `resolveDynamicModel` in `docs/plugins/sdk-provider-plugins.md:264`, and external routing is already documented through LiteLLM in `docs/providers/litellm.md:130`. Given that scope and the existing extension seams, this belongs as a ClawHub/community plugin or external router/provider integration rather than an OpenClaw core feature.
