## Summary

Keep open. Current `main` does not implement the requested `sessions.patch` plugin extension seam: the wire schema has no additive plugin payload field, the gateway patch path only applies built-in fields and then fires an internal post-patch hook, and the public plugin hook catalog exposes no `sessions.patch`-style mutation hook. This looks like an in-scope plugin API gap rather than a ClawHub-only request, because `VISION.md` explicitly welcomes plugin-API extensions when a useful feature cannot yet be built as a plugin.

## Evidence

- **`sessions.patch` schema has no plugin extension payload:** `SessionsPatchParamsSchema` lists the built-in session override fields (`label`, `thinkingLevel`, `model`, `sendPolicy`, etc.) and closes the object with `additionalProperties: false`; there is no `extension` field or other plugin-owned payload slot.

## Risks / Open Questions

- There are adjacent plugin surfaces (`registerInteractiveHandler`, plugin-owned RPC methods, session-store helpers), so maintainers may still want to decide whether the right answer is a generic `sessions.patch` seam or a different plugin-owned control path.
- The issue is brand new and includes a concrete external proof case (Smarter-Claw patches), so auto-closing now would be overly aggressive for conservative maintainer cleanup.
