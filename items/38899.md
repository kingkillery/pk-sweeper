## Summary

Keep open. Current `main` has user-visible sanitization for leaked Qwen-style `<tool_call>` XML and a narrow empty-error retry, but I did not find a parser-side fix that would reliably prevent the reported llama.cpp/Qwen parse failure. The report also has a recent April 2026 corroborating comment, so there is not strong evidence the bug is obsolete or fixed on `main`.

## Evidence

- **Qwen XML handling is display sanitization, not parser recovery:** `src/shared/text/assistant-visible-text.test.ts` adds coverage for stripping Qwen-style nested `<tool_call><function=...><parameter=...>` payloads from visible assistant text. That proves UI/history cleanup, but it does not show OpenClaw now converts or prevents those malformed tool-call dumps before the model/provider parse error happens.
  - file: [src/shared/text/assistant-visible-text.test.ts:155](https://github.com/openclaw/openclaw/blob/d3dd61f2c5bd6cea175544f3a76a68ec11fa17c2/src/shared/text/assistant-visible-text.test.ts#L155)

- **Current retry is narrow:** `runEmbeddedPiAgent` only retries when the failed turn has `stopReason === "error"`, zero output tokens, empty content, and no potential side effects. That is a mitigation for silent empty failures, not a general fix for parse-input tool-call errors.
