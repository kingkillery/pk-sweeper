## Summary

Keep open. Current `main` still uses the same OpenAI-compatible interactive transport/error-classification path the report describes, and I did not find an IDN/punycode-specific fix or regression coverage that would justify closing this conservatively.

## Evidence

- **Interactive provider requests still go through the guarded fetch path:** `buildGuardedModelFetch()` still wraps OpenAI-compatible model traffic with `fetchWithSsrFGuard(...)`; there is no hostname-form special case here for Unicode/IDN or punycode hosts.
  - file: [src/agents/provider-transport-fetch.ts:153](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/provider-transport-fetch.ts#L153)
  - command: `nl -ba src/agents/provider-transport-fetch.ts | sed -n '120,220p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Hostname normalization does not canonicalize IDN forms:** `normalizeHostname()` only lowercases, trims a trailing dot, and strips IPv6 brackets. It does not convert Unicode hostnames to ASCII/punycode or normalize punycode back to a canonical form.
  - file: [src/infra/net/hostname.ts:3](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/infra/net/hostname.ts#L3)
  - command: `nl -ba src/infra/net/hostname.ts | sed -n '1,80p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **SSRF/DNS pinning still keys off the normalized hostname string:** `resolveHostnamePolicyChecks()` and `resolvePinnedHostnameWithPolicy()` continue to validate and DNS-resolve the normalized hostname string directly. I did not find IDN/punycode-specific handling in this path.
  - file: [src/infra/net/ssrf.ts:217](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/infra/net/ssrf.ts#L217)
  - command: `nl -ba src/infra/net/ssrf.ts | sed -n '210,390p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **The reported symptom still maps to current error handling:** Current embedded-runner error classification still maps HTML `403` provider responses to `auth_html_403` and formats them as an auth failure, matching the issue's `failoverReason/profileFailureReason: auth` symptom.
  - file: [src/agents/pi-embedded-helpers/errors.ts:891](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/pi-embedded-helpers/errors.ts#L891)
  - command: `nl -ba src/agents/pi-embedded-helpers/errors.ts | sed -n '880,1020p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **URL parsing canonicalizes Unicode hostnames to punycode, but that alone does not prove a fix:** A local Node check on current environment shows `new URL('https://münich.example/v1').hostname` becomes `xn--mnich-kva.example`, so OpenClaw's URL-based paths already see punycode for Unicode input. That means the report's 'Unicode and punycode both fail' claim is still plausible, not disproven.
