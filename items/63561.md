## Summary

Keep open. Current `main` has partial mitigation for Linux systemd probe failures, but the reported `Permission denied` path is still treated as a real unavailability signal rather than a false negative, so there is not strong evidence that #63561 is fully fixed on current `main` or in `v2026.4.23`.

## Evidence

- **Current status path still treats `Permission denied` as unavailable:** `execSystemctlUser` explicitly refuses any machine-scope fallback when the failing `systemctl --user` detail contains `permission denied`, which matches the issue’s reported stderr.
  - file: [src/daemon/systemd.ts:357](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/daemon/systemd.ts#L357)

  - file: [src/daemon/systemd.test.ts:172](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/daemon/systemd.test.ts#L172)
