## Summary

Keep open. Current `main` still appears to have the stale system-event leak on session reset, so the underlying bug looks real, but this PR is not safe to close as implemented and its proposed change has unresolved correctness risks on the failed-reset path.

## Evidence

- **Current reset cleanup still skips system-event queues:** `ensureSessionRuntimeCleanup` clears followup/command queues, then waits for the embedded run to end, but it does not drain system events anywhere on `main` before returning success.
  - file: [src/gateway/session-reset-service.ts:213](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/gateway/session-reset-service.ts#L213)
  - command: `nl -ba src/gateway/session-reset-service.ts | sed -n '213,244p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Fresh runs still consume queued system events from the session key:** The next reply build drains queued system events with `drainFormattedSystemEvents({ sessionKey, isNewSession, ... })`, so stale events left behind by reset can still be injected into the first prompt after `/new`.
  - file: [src/auto-reply/reply/get-reply-run.ts:517](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/auto-reply/reply/get-reply-run.ts#L517)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '500,530p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Failed reset path is real on current main:** There is an existing test proving `sessions.reset` can return `UNAVAILABLE` when the active embedded run does not stop, while the original session entry remains in place. Draining system events before that wait completes would discard notifications from a session that did not actually reset.
  - file: [src/gateway/server.sessions.gateway-server-sessions-a.test.ts:3125](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/gateway/server.sessions.gateway-server-sessions-a.test.ts#L3125)
  - command: `nl -ba src/gateway/server.sessions.gateway-server-sessions-a.test.ts | sed -n '3125,3158p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **PR patch drains too early relative to the failure point:** The proposed change inserts `drainSystemEventEntries(key)` immediately after `clearSessionQueues()`. That happens before `waitForEmbeddedPiRunEnd(...)` decides whether reset succeeded, matching the Codex review concern about data loss on failed `/new` resets.
  - file: [src/gateway/session-reset-service.ts:263](https://github.com/openclaw/openclaw/blob/df2d25fe20309bf9bf5adf1731fb740831ce870d/src/gateway/session-reset-service.ts#L263)
  - command: `GitHub PR #67049 patch and review comments in provided context`
  - sha: [df2d25fe2030](https://github.com/openclaw/openclaw/commit/df2d25fe20309bf9bf5adf1731fb740831ce870d)
- **PR also has an input-normalization mismatch:** `clearSessionQueues` skips empty/falsy keys via `normalizeOptionalString`, but `drainSystemEventEntries` calls `requireSessionKey` and throws on empty input. The Greptile review correctly identified that the raw loop added by this PR is asymmetric with the existing cleanup API.

## Risks / Open Questions

- The stale system-event leak described in the PR body still looks plausible on current `main`, so closing this PR would likely leave a real bug unaddressed.
- Merging the PR as-is risks dropping pending system notifications when `/new` or `sessions.reset` fails after cleanup starts.
- The PR does not include regression coverage for the stale-event leak or for failed-reset behavior.
