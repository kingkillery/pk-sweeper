## Summary

Keep open. Current `main` and the latest release still ship the broken `/export-session` HTML template, and the PR’s restored-placeholder approach is not yet safe because `generateHtml()` uses string replacement with JS payloads that contain `$` replacement sequences.

## Evidence

- **Broken placeholders are still on main and in the latest release:** `src/auto-reply/reply/export-html/template.html` still contains formatter-broken inline script blocks (`MARKED_JS;`, `HIGHLIGHT_JS;`, `JS;`) instead of replaceable `{{...}}` tokens. `git show a9797214338ba31c52c796adbb75afb16e0684a9:...` shows the same content in release `v2026.4.23`.
  - file: [src/auto-reply/reply/export-html/template.html:61](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/auto-reply/reply/export-html/template.html#L61)
  - command: `nl -ba src/auto-reply/reply/export-html/template.html | sed -n '1,180p'; git show a9797214338ba31c52c796adbb75afb16e0684a9:src/auto-reply/reply/export-html/template.html | nl -ba | sed -n '57,90p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Current export generator uses raw string replacement for JS payloads:** `generateHtml()` still does `.replace("{{JS}}", templateJs)`, `.replace("{{MARKED_JS}}", markedJs)`, and `.replace("{{HIGHLIGHT_JS}}", hljsJs)`. If the PR restores the `{{...}}` placeholders without changing this insertion method, replacement-text `$` sequences remain semantically active.

## Risks / Open Questions

- `/export-session` remains broken on current `main` and in release `v2026.4.23` because the checked-in template still emits raw placeholder identifiers instead of inline scripts.
- Merging this PR as-is would likely trade one regression for another: `$&` in `highlight.min.js` reinserts `{{HIGHLIGHT_JS}}`, and `$$` in `template.js` collapses to `$`, so the exported HTML would not be a byte-faithful inline bundle.
