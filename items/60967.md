## Summary

Keep open. Current `main` still wires bundle MCP timeouts only into server connection/setup, not individual tool calls, and the public config/docs surface still exposes only `connectionTimeoutMs` with no per-tool/request timeout override.

## Evidence

- **Bundle MCP runtime still omits per-call timeout:** `createSessionMcpRuntime()` constructs the MCP `Client` with empty options, uses `connectWithTimeout(...)` only for initial connect, and later calls `session.client.callTool({ ... })` without any timeout/options argument.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:176](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/pi-bundle-mcp-runtime.ts#L176)
  - command: `nl -ba src/agents/pi-bundle-mcp-runtime.ts | sed -n '160,310p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Resolved MCP transport config only carries connection timeout:** The resolved transport types and parser only define/read `connectionTimeoutMs`; there is no request/tool timeout field propagated alongside the session transport config.
  - file: [src/agents/mcp-transport-config.ts:14](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/mcp-transport-config.ts#L14)
