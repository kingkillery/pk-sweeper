## Summary

Current `main` does not implement first-class per-agent MCP env overrides for the same MCP server id. MCP credentials are still defined on top-level named `mcp.servers` entries, and the stdio launcher receives only that server config env. The adjacent #65612-style tool-policy path is now present and useful: operators can create separate named Plane MCP server entries per identity and restrict each agent to its matching tool prefix. That is a viable workaround, but it is not equivalent to the requested per-agent env injection because OpenClaw still lacks an `agents.list[].mcp.overrides` contract and does not build MCP runtime/process config from the active agent.

## Best Possible Solution

Keep this open as a focused core MCP runtime/config feature. The best implementation is a schema-backed per-agent MCP credential overlay or identity mechanism, applied before MCP runtime creation and included in the runtime cache fingerprint, with tests proving two agents can use the same MCP server id with different effective env credentials and correct tool attribution. Document the current duplicate-`mcp.servers` plus per-agent `tools.deny` workaround until that exists.

## Risks / Open Questions

- A duplicate-server workaround can solve attribution operationally, but it changes tool names, requires explicit per-agent policy maintenance, and still discovers/starts configured MCP servers before final tool filtering.
- The product decision needs care around process lifecycle, runtime cache keys, hot reload, and secret handling: per-agent env overlays probably need to affect the MCP runtime fingerprint and server process identity.
