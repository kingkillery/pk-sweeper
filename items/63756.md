## Summary

Keep open. Current `main` still has a credible native-Windows bug for globally installed `@tobilu/qmd`: OpenClaw resolves the wrapper with `packageName: "qmd"`, while the published package is scoped and exposes `bin/qmd` as a POSIX shell launcher. The docs note about WSL2 is already covered, and builtin fallback is graceful, but the wrapper-resolution path itself does not look fixed on `main`.

## Evidence

- **QMD runtime still resolves Windows wrappers as unscoped `qmd`:** Both the availability probe and the actual QMD runner still pass `packageName: "qmd"` into Windows spawn resolution, so the resolver is keyed to an unscoped package name rather than the published `@tobilu/qmd` package. See `src/memory-host-sdk/host/qmd-process.ts:44` and `extensions/memory-core/src/memory/qmd-manager.ts:1790`.
  - file: [src/memory-host-sdk/host/qmd-process.ts:44](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/memory-host-sdk/host/qmd-process.ts#L44)
  - command: `nl -ba src/memory-host-sdk/host/qmd-process.ts | sed -n '42,49p'; nl -ba extensions/memory-core/src/memory/qmd-manager.ts | sed -n '1788,1796p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Package-json fallback only searches unscoped package directories:** `src/plugin-sdk/windows-spawn.ts:171` only probes `../<packageName>` and `node_modules/<packageName>` when resolving a `.cmd` wrapper from package metadata. That does not cover a scoped global install layout like `node_modules/@tobilu/qmd`.
  - file: [src/plugin-sdk/windows-spawn.ts:171](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/plugin-sdk/windows-spawn.ts#L171)

That is not compatible with a package bin that is itself a shell launcher.
  - file: [src/plugin-sdk/windows-spawn.ts:233](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/plugin-sdk/windows-spawn.ts#L233)

The published `package/bin/qmd` file starts with `#!/bin/sh` and shells into `dist/cli/qmd.js`, so it is not a direct `.js` or `.exe` entrypoint.
