## Summary

Keep open. Current `main` still creates long-lived Docker sandbox containers whose PID 1 is `sleep`, and the sandbox create path does not enable a Docker init/reaper. That matches the report's core failure mode: orphaned `docker exec` descendants can be adopted by PID 1 without any explicit reap path, so zombie accumulation remains plausible on current `main`, especially under shell-heavy subagent workloads.

## Evidence

- **Sandbox container PID 1 is still `sleep`:** The default sandbox image ends with `CMD ["sleep", "infinity"]`, so the steady-state container entrypoint is a plain `sleep` process rather than an init/reaper.
  - file: [Dockerfile.sandbox:24](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/Dockerfile.sandbox#L24)
  - command: `nl -ba Dockerfile.sandbox | sed -n '1,80p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Create path keeps that long-lived `sleep` process and does not add `--init`:** `buildSandboxCreateArgs` assembles `docker create` flags including optional `--pids-limit`, but no `--init`/reaper flag. `createSandboxContainer` then appends `cfg.image, "sleep", "infinity"` before `docker start`, so current main still launches sandboxes with `sleep` as PID 1.
  - file: [src/agents/sandbox/docker.ts:378](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/agents/sandbox/docker.ts#L378)
  - command: `nl -ba src/agents/sandbox/docker.ts | sed -n '360,520p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Sandbox runtimes are designed to stay alive across runs:** Docs describe Docker sandbox containers as persistent runtimes that are recreated only when explicitly requested or after idle/max-age pruning, so any unreaped zombies can accumulate inside the same container over time rather than disappearing after each command.
  - file: [docs/cli/sandbox.md:16](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/docs/cli/sandbox.md#L16)
  - command: `nl -ba docs/cli/sandbox.md | sed -n '1,40p;150,165p'`
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Issue discussion narrows the surface to workload churn, not steady-state uptime:** The thread already includes a negative repro for a quiet watchdog-supervised deployment and a follow-up hypothesis from the reporter that the zombies appear after sub-agent shell-heavy activity. That discussion is consistent with the current implementation: nothing in the Docker sandbox runtime adds a PID-1 reaper for orphaned descendants from completed `docker exec` work.
  - command: `GitHub issue #68691 comments 4275091277 and 4316794529 (provided context)`

## Risks / Open Questions

- I did not run a live Docker repro in this read-only review; this conclusion is based on current `main` code, docs, and the issue discussion.
- The exact fix path is still open: it likely needs an explicit PID-1 reaper strategy (`--init` or a custom entrypoint) plus regression coverage for orphaned `docker exec` descendants under subagent-style shell workloads.
