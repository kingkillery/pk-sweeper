## Summary

Keep open. Current `main` at `c997a9f9788e0a96e8eab05e7ae94959e76cdb4a` still lacks the PR’s webchat fix: the UI only suppresses exact `NO_REPLY`, not lowercase variants or streamed prefix fragments like `NO`/`NO_`, so the reported behavior is not implemented on `main`.

## Evidence

- **Client exact-match regex is still case-sensitive on main:** `SILENT_REPLY_PATTERN` is still `^\s*NO_REPLY\s*$` without the `i` flag, so lowercase `no_reply` would not match the client-side silent-reply guard.
  - file: [ui/src/ui/controllers/chat.ts:13](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/ui/src/ui/controllers/chat.ts#L13)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '8,50p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Delta/final/aborted handlers still do not filter NO_REPLY prefix fragments:** `handleChatEvent` only checks `isSilentReplyStream(...)` for `delta`, final fallback, and aborted fallback. There is no prefix-fragment guard before assigning/persisting `state.chatStream`, so streamed `NO`/`NO_` fragments can still survive into visible messages.
  - file: [ui/src/ui/controllers/chat.ts:451](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/ui/src/ui/controllers/chat.ts#L451)
