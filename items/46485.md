## Summary

Keep open. Current `main` does not expose `api.runtime.gateway.dispatchMethod`, and the underlying need is still unresolved: plugin runtime only exposes `subagent` and `nodes`, while the sanitized `chat.history` path remains distinct from `sessions.get`.

## Evidence

- **Plugin runtime type still has no gateway surface:** `PluginRuntime` on current `main` includes `subagent`, `nodes`, and `channel`, but no `gateway` property.
  - file: [src/plugins/runtime/types.ts:78](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/plugins/runtime/types.ts#L78)
  - command: `nl -ba src/plugins/runtime/types.ts | sed -n '50,100p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Runtime factory does not inject a gateway helper:** `createPluginRuntime()` wires `subagent` and `nodes` only; there is no `_options.gateway` or unavailable `gateway` fallback on current `main`.
  - file: [src/plugins/runtime/index.ts:203](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/plugins/runtime/index.ts#L203)

## Risks / Open Questions

- The PR was based on `main` from 2026-03-14 and will need rebasing onto a newer runtime shape that now includes late-bound `nodes` support and gateway-binding options.
- Current plugin runtime wrapping in `src/plugins/registry.ts` only adds plugin-id scoping for `subagent`; if this PR lands, maintainers should check whether a new `gateway` surface also needs equivalent scoping or policy review.
