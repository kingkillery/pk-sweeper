## Summary

Keep open. Current `main` has partial idempotency for gateway `agent` requests, but it does not implement the broader retry-safe sub-agent/tool-call identity the issue asks for. Tool executions still lack a generic propagated `runId`/idempotency context, and key sub-agent paths (`sessions_send`, `sessions_spawn`) mint fresh UUIDs per invocation, so a timeout-driven retry can still produce duplicate side effects.

## Evidence

## Risks / Open Questions

- The existing gateway dedupe could make the issue look fixed in narrow `agent` RPC tests while duplicate side effects still occur when a higher-level tool invocation is retried with a new UUID.
- Without a generic propagated retry identity, plugin/skill/tool authors must solve idempotency ad hoc, so duplicate external POSTs remain plausible on timeout recovery paths.
- `sessions_spawn` itself is still not retry-idempotent, so parent-side recovery that repeats a spawn can create duplicate child work even though the child's internal `agent` start call is deduped.
