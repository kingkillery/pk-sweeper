## Summary

Current `main` does not show a complete fix for #36614. The routing layer now distinguishes isolated DM sessions via `lastRoutePolicy: "session"`, but the iMessage inbound monitor still calls `recordInboundSession(... updateLastRoute.sessionKey = decision.route.mainSessionKey)`, and the allowlist pin guard still returns `null` whenever `session.dmScope` is not `main`. That means the reported write-side contamination path remains plausible on `main`, so there is not strong evidence to close this issue.

## Evidence

The route builder still keeps a separate `mainSessionKey` (`agent:...:main`).

## Risks / Open Questions

- Write-side last-route contamination still appears possible for iMessage on `session.dmScope: per-channel-peer`, which is the core report in #36614.
- The behavior is inconsistent across channels on current `main`: some handlers honor session-scoped last-route updates while others still overwrite the main session, so closing this issue would be premature.
