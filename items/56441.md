## Summary

Keep open. Current `main` still lets host `exec` inherit provider auth env vars from `process.env`; ACP bridge spawns strip those vars explicitly, but the host-exec sanitizer does not, so the reported leak is not strongly shown as fixed on `main` or in `v2026.4.22`.

## Evidence

- **Host exec sanitizer still forwards inherited env unless blocked by policy:** `sanitizeHostExecEnvWithDiagnostics` copies every inherited env entry except keys matched by `isDangerousHostInheritedEnvVarName`; there is no provider-auth-specific stripping in this path.
  - file: [src/infra/host-env-security.ts:209](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/infra/host-env-security.ts#L209)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Exec tool still uses that sanitizer for host runs:** The host exec path builds `hostEnvResult` from `coerceEnv(process.env)` via `sanitizeHostExecEnvWithDiagnostics(...)` and then uses `hostEnvResult.env` for execution.
  - file: [src/agents/bash-tools.exec.ts:1536](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/agents/bash-tools.exec.ts#L1536)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Provider auth env vars are known centrally:** Core provider env candidates still include `OPENAI_API_KEY` and related auth env vars, and `listKnownProviderAuthEnvVarNames()` returns the combined provider-auth/setup env set.
  - file: [src/secrets/provider-env-vars.ts:11](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/secrets/provider-env-vars.ts#L11)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **ACP path has an explicit fix that host exec lacks:** `buildAcpClientStripKeys()` adds every known provider auth env var to the strip set for default ACP bridge spawns, which shows the repo has targeted stripping logic for ACP but not for host exec.
  - file: [src/acp/client-helpers.ts:190](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/acp/client-helpers.ts#L190)
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Current denylist still omits provider auth env vars:** The checked-in host env security policy blocks entries like `GITHUB_TOKEN`, `DATABASE_URL`, and AWS creds, but not `OPENAI_API_KEY`/`ANTHROPIC_API_KEY`; the latest release tag shows the same omission.
  - file: [src/infra/host-env-security-policy.json:218](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/infra/host-env-security-policy.json#L218)

  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Release-note mention is not sufficient proof of a fix:** `CHANGELOG.md` for `2026.4.10` mentions broad `host env denylisting` hardening, but the current and latest-release code still do not denylist provider auth env vars in the host exec sanitizer.
