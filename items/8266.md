## Summary

The request is still not implemented in core debug log strings on current `main`, but OpenClaw already exposes tool arguments/results through the plugin hook surface intended for external observability, so this belongs as a plugin/ClawHub integration rather than a core logging change.

## Close Comment

Closing this as `clawhub` after a Codex review. Current `main` still keeps the low-level debug line minimal in `src/agents/pi-embedded-subscribe.handlers.tools.ts:604`, so this specific core log-format change is not implemented there. But the observability data the request wants already lives on the plugin/hook surface that OpenClaw expects external integrations to use: tool start emits structured `args` in `src/agents/pi-embedded-subscribe.handlers.tools.ts:608`, and `after_tool_call` exposes `params` plus `result` in the public hook contract at `src/plugins/hook-types.ts:339`, wired from `src/agents/pi-embedded-subscribe.handlers.tools.ts:1122`. That matches the project scope in `VISION.md:54`, which says core stays lean and optional capability should usually ship as plugins, and it matches existing logging/audit precedent in the bundled `command-logger` hook at `src/hooks/bundled/command-logger/HOOK.md:16`. For richer external monitoring, the intended path is a ClawHub/plugin hook that writes its own structured log stream rather than expanding core debug logs.
