## Summary

Keep open. Current `main` still has the two code paths that match the report: subsystem loggers cache a child file logger once and never refresh it when the parent logger is rebuilt, and status output reports the configured path from config rather than the logger's live sink.

## Evidence

- **Subsystem logger keeps a stale child file logger:** `createSubsystemLogger()` stores `fileLogger` in a closure and only initializes it once with `getChildLogger(...)`; later emits reuse that cached child without any generation or rebuild check.
  - file: [src/logging/subsystem.ts:325](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/logging/subsystem.ts#L325)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '325,430p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Root logger can rebuild, but subsystem children are not invalidated:** `getLogger()` rebuilds the root logger when resolved settings change, but current `main` has no generation counter or invalidation path for already-created subsystem child loggers.
  - file: [src/logging/logger.ts:504](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/logging/logger.ts#L504)
  - command: `nl -ba src/logging/logger.ts | sed -n '504,527p'`
  - sha: [6f72b74cecf8](https://github.com/openclaw/openclaw/commit/6f72b74cecf8d1fec5c147ae932161bf0f37a68b)
- **Logger settings do honor `logging.file` when resolved:** `resolveSettings()` reads `cfg?.file ?? defaultRollingPathForToday()`, so the intended contract is that file logs follow `logging.file`; the bug is in logger lifecycle/application, not schema/docs.
  - file: [src/logging/logger.ts:374](https://github.com/openclaw/openclaw/blob/6f72b74cecf8d1fec5c147ae932161bf0f37a68b/src/logging/logger.ts#L374)

That matches the reporter's observation that status can show the configured path even if writes still land elsewhere.
