## Summary

Keep open. Current `main` still appears to preserve failed dev-preflight candidate steps as blocking failures even after a later candidate succeeds, which matches the issue’s reported `preflight lint (...)` terminal error after a successful fallback commit.

## Evidence

- **Final status still depends on any blocking step in the accumulated git update steps:** After the dev-update flow finishes rebase/build/doctor, the result is computed from `findBlockingGitFailure(steps)`, and the returned `reason` is that failing step name. This means earlier failed preflight steps can still determine the final status.
  - file: [src/infra/update-runner.ts:1249](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/infra/update-runner.ts#L1249)
  - command: `nl -ba src/infra/update-runner.ts | sed -n '1248,1268p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Blocking-failure filter only exempts install retries, not failed superseded preflight candidates:** `findBlockingGitFailure` returns the first nonzero step unless it is a superseded install failure. The exemption logic only handles `deps install` and `preflight deps install (ignore scripts)` retries; it does not exempt earlier failed `preflight lint (...)` or other failed candidate-validation steps after a later candidate succeeds.
  - file: [src/infra/update-runner.ts:499](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/infra/update-runner.ts#L499)
  - command: `nl -ba src/infra/update-runner.ts | sed -n '499,520p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Published docs promise fallback to the newest clean dev commit:** The user-facing CLI docs say dev updates run preflight lint/build in a temp worktree and, if the tip fails, walk back up to 10 commits to find the newest clean build. The current result aggregation above does not fully honor that promise when an earlier failed candidate remains in `steps`.
  - file: [docs/cli/update.md:105](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/docs/cli/update.md#L105)
  - command: `nl -ba docs/cli/update.md | sed -n '100,106p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Release-check script treats successful dev-update landing as the invariant to preserve:** The cross-OS release check explicitly says the dev-update lane must prove `openclaw update --channel dev` landed on the expected git checkout, and warns that masking updater regressions would create a false green. That aligns with keeping this report open until the updater no longer returns an error after a successful fallback commit.
  - file: [scripts/openclaw-cross-os-release-checks.ts:1520](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/scripts/openclaw-cross-os-release-checks.ts#L1520)

- **Issue discussion matches the current code path:** The provided issue log shows multiple `preflight lint (...)` failures, then a later candidate (`6ab359f5`) passes preflight lint and the update completes rebase/build/doctor, yet the final result remains `ERROR` with reason `preflight lint (338c7b8d)`. That behavior is consistent with the current `findBlockingGitFailure(steps)` aggregation.
