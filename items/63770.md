## Summary

Keep open. Current `main` still gates deletion on `job.schedule.kind === "at"`, so `deleteAfterRun: true` remains ineffective for `every` and `cron` jobs. The public cron schema and CLI still accept `deleteAfterRun` as a generic job field, so the report is still actionable rather than stale or incoherent.

## Evidence

- **Shared result path still hardcodes one-shot-only deletion:** `applyJobResult` sets `shouldDelete` only when `job.schedule.kind === "at"` and `job.deleteAfterRun === true` on success. That is the exact behavior reported in the issue, and it is present on current `main` at `86f8c826e20a01d1bba67c0ba6e73182fbf0455f`.
  - file: [src/cron/service/timer.ts:463](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/cron/service/timer.ts#L463)

Because recurring jobs never satisfy the `kind === "at"` guard above, successful `every`/`cron` jobs are not deleted after their first run.
