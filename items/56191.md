## Summary

Current `main` already persists session transcripts and reuses them across gateway restarts. The issue request matches existing session storage behavior rather than a missing feature.

## Evidence

## Risks / Open Questions

- A user can still appear to 'lose' history if they restart against a different profile or `OPENCLAW_STATE_DIR`; the migration guide explicitly calls that out as a configuration pitfall rather than a missing feature.
- This review is code-and-docs based; I did not mutate the checkout or run a live restart flow.

## Close Comment

Closing from a Codex review because current `main` already implements this. OpenClaw persists session transcripts on disk (`docs/concepts/session.md:74`, `src/gateway/server-methods/chat.ts:1486`) and restores history by reading those transcript files back through `chat.history` (`src/gateway/server-methods/chat.ts:1891`). The session store also reuses an existing `sessionId` when a session key already exists instead of starting fresh (`src/gateway/server-methods/agent.ts:621`). The migration guide further documents that copying the state directory preserves `Sessions -- conversation history and agent state` (`docs/install/migrating.md:13`). If you are still seeing history disappear after a restart, that looks more like a specific bug or profile/state-dir mismatch than a missing feature; please open a new issue with the exact restart path, whether `--profile` or `OPENCLAW_STATE_DIR` is involved, and any relevant logs.
