## Summary

Keep #27996 open. Current main fixes the normal `openclaw cron edit` / `cron.update` path, but the remaining user-reported direct `jobs.json` edit path is still plausible on current main because startup/load preserves still-future `nextRunAtMs` values instead of detecting schedule/timezone drift.

## Evidence

- **CLI/API schedule edits are handled:** `update()` applies the patch, detects `patch.schedule`, and recomputes `job.state.nextRunAtMs` with `computeJobNextRunAtMs()` before persisting. This covers the normal `openclaw cron edit --cron/--tz` path.
  - file: [src/cron/service/ops.ts:347](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/cron/service/ops.ts#L347)
  - command: `nl -ba src/cron/service/ops.ts`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Startup recompute preserves future stale values:** `start()` loads with `skipRecompute: true` and then calls `recomputeNextRuns()`. `recomputeNextRuns()` only recomputes when `nextRunAtMs` is missing or past due, explicitly preserving a still-future value. That leaves the direct-file-edit repro unresolved when old runtime state is still in the future.
  - file: [src/cron/service/jobs.ts:482](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/cron/service/jobs.ts#L482)
  - command: `nl -ba src/cron/service/jobs.ts`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Store split keeps runtime state separate from job definitions:** `loadCronStore()` merges `jobs-state.json` state by job id and ignores inline state when the sidecar exists, so editing only `jobs.json` schedule fields does not necessarily update the persisted `nextRunAtMs`.
  - file: [src/cron/store.ts:174](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/cron/store.ts#L174)
  - command: `nl -ba src/cron/store.ts`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Docs make direct jobs.json edits a real supported-adjacent workflow:** The cron docs state job definitions live in `~/.openclaw/cron/jobs.json` and runtime execution state lives in `~/.openclaw/cron/jobs-state.json`; this aligns with the commenter’s direct-edit reproduction and means the remaining issue is not just obsolete inline-state behavior.
  - file: [docs/automation/cron-jobs.md:34](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/docs/automation/cron-jobs.md#L34)
  - command: `nl -ba docs/automation/cron-jobs.md | sed -n '1,80p'`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Issue discussion has concrete remaining reproduction:** The March 18 and later comments describe an agent directly updating `jobs.json` while preserving the old `nextRunAtMs`, then a stale run firing because startup used the valid future timestamp. That maps to the current load/recompute behavior above.
