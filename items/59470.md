## Summary

Keep open. The report is recent, author is not a maintainer, has no protected labels, and the current main evidence still matches the reported failure mode: plugin gateway methods can do work and return a value, while the gateway RPC path only completes the CLI request when a handler explicitly calls `respond`. I do not see strong enough evidence that main has implemented a return-value-to-response adapter for plugin gateway methods.

## Evidence

- **plugin registration stores raw gateway handler:** Current main records plugin gateway handlers directly in `registry.gatewayHandlers[trimmed] = handler` after conflict/scope handling; I did not find an adapter here that converts a returned plugin value into a gateway response.
  - file: [src/plugins/registry.ts:3115](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/src/plugins/registry.ts#L3115)
  - command: `GitHub web inspection of current main at 44114328b4e73074ac72126f1230ddbf9c1ea8b5; local read-only shell was unavailable due sandbox bwrap loopback failure`
  - sha: [44114328b4e7](https://github.com/openclaw/openclaw/commit/44114328b4e73074ac72126f1230ddbf9c1ea8b5)
- **gateway dispatch ignores handler return value:** `handleGatewayRequest` selects `opts.extraHandlers?.[req.method] ?? coreGatewayHandlers[req.method]`, constructs `invokeHandler`, and awaits it inside `withPluginRuntimeGatewayRequestScope`; the shown code does not consume the handler's returned value or synthesize a response if no `respond` call occurred.
  - file: [src/gateway/server-methods.ts:881](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/src/gateway/server-methods.ts#L881)
  - command: `GitHub web inspection of current main at 44114328b4e73074ac72126f1230ddbf9c1ea8b5`
  - sha: [44114328b4e7](https://github.com/openclaw/openclaw/commit/44114328b4e73074ac72126f1230ddbf9c1ea8b5)
- **websocket request completion depends on respond:** After handshake, the websocket message handler defines `respond` to send `{ type: "res", id: req.id, ok, payload, error }` and passes it to `handleGatewayRequest` with `extraHandlers`. If a plugin handler returns normally without calling `respond`, this path does not obviously emit the response frame the CLI waits for.
  - file: [src/gateway/server/ws-connection/message-handler.ts:1530](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/src/gateway/server/ws-connection/message-handler.ts#L1530)
  - command: `GitHub web inspection of current main at 44114328b4e73074ac72126f1230ddbf9c1ea8b5`
  - sha: [44114328b4e7](https://github.com/openclaw/openclaw/commit/44114328b4e73074ac72126f1230ddbf9c1ea8b5)
- **public SDK docs support return-style examples:** The public Plugin SDK reference describes `registerGatewayMethod(method, handler)` as exposing JSON-RPC endpoints and shows an async handler returning `{ result: "success" }`, which aligns with the issue's expectation that returning normally should complete `openclaw gateway call`.
  - file: [docs/plugins/plugin-sdk.md](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/docs/plugins/plugin-sdk.md)
