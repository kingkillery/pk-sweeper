## Summary

Keep open. Current `main` still does not implement node-registered agent tools: node handshake/state only carries `caps`/`commands`/`permissions`, the node registry only tracks those fields, and runtime tool catalogs only admit `core`/`plugin`/`channel` sources. The documented extension path remains plugin-registered tools, so the feature request is still unmet rather than stale or already shipped.

## Evidence

- **Node protocol has no tool-registration field:** The WebSocket `connect` schema accepts `caps`, `commands`, and `permissions`, but no `tools` payload for nodes. Pairing metadata likewise only records `caps` and `commands`.
  - file: [src/gateway/protocol/schema/frames.ts:37](https://github.com/openclaw/openclaw/blob/c0a7b6a510758645c5267734b984f4885a605b67/src/gateway/protocol/schema/frames.ts#L37)
  - command: `nl -ba src/gateway/protocol/schema/frames.ts | sed -n '20,45p'; nl -ba src/gateway/protocol/schema/nodes.ts | sed -n '1,35p'`
  - sha: [c0a7b6a51075](https://github.com/openclaw/openclaw/commit/c0a7b6a510758645c5267734b984f4885a605b67)
- **Gateway node registry only stores commands/caps:** `NodeRegistry.register()` reads `connect.caps`, `connect.commands`, `permissions`, and `pathEnv`; there is no registration, storage, or teardown path for node-defined LLM tools.
  - file: [src/gateway/node-registry.ts:45](https://github.com/openclaw/openclaw/blob/c0a7b6a510758645c5267734b984f4885a605b67/src/gateway/node-registry.ts#L45)
  - command: `nl -ba src/gateway/node-registry.ts | sed -n '35,80p'`
  - sha: [c0a7b6a51075](https://github.com/openclaw/openclaw/commit/c0a7b6a510758645c5267734b984f4885a605b67)
- **Tool catalogs do not admit node-owned tools:** `ToolCatalogEntrySchema` only allows `source: "core" | "plugin"`, and effective tool inventory only allows `core | plugin | channel`. The protocol docs match that: `tools.catalog` returns `core` or `plugin`, and `tools.effective` covers `core`, `plugin`, and `channel` tools.
  - file: [src/gateway/protocol/schema/agents-models-skills.ts:369](https://github.com/openclaw/openclaw/blob/c0a7b6a510758645c5267734b984f4885a605b67/src/gateway/protocol/schema/agents-models-skills.ts#L369)
  - command: `nl -ba src/gateway/protocol/schema/agents-models-skills.ts | sed -n '369,440p'; nl -ba docs/gateway/protocol.md | sed -n '428,440p'`
  - sha: [c0a7b6a51075](https://github.com/openclaw/openclaw/commit/c0a7b6a510758645c5267734b984f4885a605b67)
- **Current architecture still routes node capability through generic node commands and plugins:** The built-in `nodes` tool is still a single generic tool with fixed actions plus `invoke`, and the CLI docs still describe node access as `nodes invoke` / `exec host=node`. Separately, the repo vision and plugin docs still say optional capability should usually ship as plugins and document `api.registerTool(...)` as the agent-tool extension path.
  - file: [src/agents/tools/nodes-tool.ts:19](https://github.com/openclaw/openclaw/blob/c0a7b6a510758645c5267734b984f4885a605b67/src/agents/tools/nodes-tool.ts#L19)
