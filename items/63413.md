## Summary

Current `main` still accepts `--limit <n>` for `ltm search` and passes `Number.parseInt(opts.limit, 10)` straight through to the LanceDB search path without any `NaN`/positive-integer guard. Because `db.search()` then forwards that value directly to `.limit(limit)`, the PR still addresses a plausible unfixed bug rather than something already implemented or safely obsolete.

## Evidence

- **Current CLI path still lacks validation:** The `ltm search` command on current `main` parses `opts.limit` with `Number.parseInt(opts.limit, 10)` and immediately passes it into `db.search(...)`; there is no fallback or warning for invalid input.
  - file: [extensions/memory-lancedb/index.ts:554](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/memory-lancedb/index.ts#L554)
  - command: `nl -ba extensions/memory-lancedb/index.ts | sed -n '540,560p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Database layer forwards limit unsafely:** `MemoryLanceDb.search()` accepts `limit` as a number argument and forwards it directly to `this.table!.vectorSearch(vector).limit(limit).toArray()` without sanitization, so `NaN` would propagate beyond the CLI parser.
