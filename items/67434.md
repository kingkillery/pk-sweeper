## Summary

Keep open. Current main has addressed part of the report: the docs now explain strict-agentic auto-exposure, `tools.experimental.planTool=false` overrides the auto-enable path, and `update_plan` now returns compact empty content. But I did not find strong evidence that the specific 290-call `update_plan`-only burst is fixed: the planning-only retry guard still requires visible assistant planning text, tests only cover `update_plan` paired with assistant prose, and general tool-loop detection is disabled by default.

## Best Possible Solution

Keep #67434 open as the focused tracker for excessive consecutive `update_plan` bursts. The likely fix should add a runtime guard that treats pure `update_plan` tool-only activity as non-progress even when there is no visible assistant text, caps consecutive plan-tool-only turns or calls before retry/blocking, and adds regression coverage for many consecutive `update_plan` calls with no non-plan tool activity. Keep the current docs and `planTool: false` mitigation because they address the operator confusion described in the follow-up comment.
