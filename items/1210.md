## Summary

Current main still routes inbound Discord/media attachments back into inline base64 `image` blocks for agent session prompts, then applies a later pruning mitigation for already-persisted image history. The requested transcript-level media-reference storage is not implemented on `main`.

## Evidence

- **Discord ingress stores files, not a new session transcript image schema:** Discord inbound attachments are downloaded with `saveMediaBuffer(...)` and pushed forward as saved file paths plus MIME metadata; `buildDiscordMediaPayload()` then exposes those as `MediaPath`/`MediaPaths` for the reply pipeline.
  - file: [extensions/discord/src/monitor/message-utils.ts:395](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/discord/src/monitor/message-utils.ts#L395)
  - command: `nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '368,405p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Inbound prompt text carries media references:** Inbound media is rendered into `[media attached: ...]` note lines from `MediaPath`/`MediaPaths`; this path does not add the issue's proposed transcript object shape such as `mediaRef`/`mediaPath` on session messages.
  - file: [src/auto-reply/media-note.ts:103](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/auto-reply/media-note.ts#L103)
