## Summary

Keep open. Current `main` still has the raw `pending ?? {}` / `paired ?? {}` loads in device pairing, so the PR is not implemented on `main`, and the underlying corruption path remains plausible. The review comments also raise valid follow-up concerns: no regression test was added, and the helper as proposed would still pass through other non-record JSON values.

## Evidence

- **Current main still lacks the guard in device pairing:** `loadState()` in device pairing still assigns parsed JSON directly with only nullish fallback: `pendingById: pending ?? {}` and `pairedByDeviceId: paired ?? {}`. That means array-shaped JSON would still survive load on current `main`.

  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)

  - file: [src/infra/json-files.ts:29](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/infra/json-files.ts#L29)
