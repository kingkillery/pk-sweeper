## Summary

Keep open. Current `main` still rotates/retries assistant-stage timeouts without checking whether visible reply text was already surfaced, and the existing Telegram duplicate guards are aimed at channel-side preview/edit ambiguity rather than provider-timeout replay across attempts.

## Evidence

- **Assistant timeout still counts as retry/failover on main:** `shouldRotateAssistant()` returns true whenever `timedOut && !timedOutDuringCompaction`, with no guard for prior visible assistant text or prior channel delivery.
  - file: [src/agents/pi-embedded-runner/run/failover-policy.ts:88](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/pi-embedded-runner/run/failover-policy.ts#L88)
  - command: `nl -ba src/agents/pi-embedded-runner/run/failover-policy.ts | sed -n '84,93p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Run loop immediately retries after assistant-timeout failover:** The main run loop resolves assistant failover, and when the outcome is `retry` it `continue`s into another attempt. This path also does not inspect `attempt.assistantTexts` before retrying.
  - file: [src/agents/pi-embedded-runner/run.ts:1685](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/pi-embedded-runner/run.ts#L1685)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '1685,1754p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Current duplicate suppression is intra-attempt, not cross-attempt retry suppression:** `message_end` dedupe only skips a second send when the same attempt already delivered via `text_end` or the messaging tool. It does not record or suppress content replayed by a later provider retry attempt.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:793](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/pi-embedded-subscribe.handlers.messages.ts#L793)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.messages.ts | sed -n '793,820p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Telegram tests cover transport-side duplicate avoidance, not provider-timeout replay:** There is explicit regression coverage for swallowing a post-connect Telegram preview-edit timeout to avoid duplicate messages, which is a different failure mode from the reported provider-timeout + retry replay.
