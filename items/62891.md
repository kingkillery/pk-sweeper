## Summary

Keep open. Current `main` adds a partial mitigation by delaying cron/heartbeat activation until post-attach sidecars finish, but it still does not wait for channel transports such as Discord to become connected. `startChannels()` only hands channel lifecycles off to background tasks, and readiness explicitly tolerates disconnected channels during startup grace, so the reporter's follow-up failure mode remains plausible on current `main`.

## Evidence

- **Reporter follow-up narrowed the bug to channel readiness, not just slow startup:** The April 8, 2026 follow-up says cron jobs fire before Discord WS is connected, with `ready` logged minutes before Discord becomes usable, and explicitly suggests a channel-readiness gate for cron startup.
  - command: `issue context from #62891 comments dated 2026-04-08T03:44:33Z and 2026-04-08T05:44:59Z`
- **Cron now starts after post-attach sidecars, but not after channel connection:** `server.impl.ts` awaits `startGatewayPostAttachRuntime(...)` and only then calls `activateGatewayScheduledServices(...)`. That is a mitigation for earlier startup races, but the activation point is tied to sidecars finishing, not to channels reporting connected/ready.
  - file: [src/gateway/server.impl.ts:815](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server.impl.ts#L815)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '660,890p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Current code documents the gate as sidecars/chat.history only:** `activateGatewayScheduledServices` is documented as running after sidecars are started and `chat.history` is available. There is no channel-connected prerequisite in this path.
  - file: [src/gateway/server-runtime-services.ts:97](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-runtime-services.ts#L97)
  - command: `nl -ba src/gateway/server-runtime-services.ts | sed -n '1,240p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Sidecar startup does not wait for channel lifecycle completion:** `startGatewaySidecars()` awaits `params.startChannels()`, but `startChannels()` only calls `startChannel(...)` for each plugin. In `startChannelInternal`, the actual provider lifecycle is launched as a background `task` and stored in `store.tasks`; the startup function resolves once that task is handed off, not when the provider reaches connected state.
  - file: [src/gateway/server-channels.ts:418](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-channels.ts#L418)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '300,520p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Startup readiness still tolerates disconnected channels for a grace window:** Channel health uses a 120000ms startup connect grace, and readiness tests assert that a disconnected Discord channel is still considered ready during early startup. That reinforces that current startup sequencing is not a strict channel-connected gate before other work begins.
  - file: [src/gateway/channel-health-policy.ts:56](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/channel-health-policy.ts#L56)
