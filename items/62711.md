## Summary

Keep open. Current `main` still uses the same Signal SSE client and autostart behavior that shipped in `v2026.4.5`, and I did not find code, tests, or history evidence that the reported `signal-cli 0.14.x` `TypeError: terminated` receive regression was fixed or disproven. The report may involve an external-daemon misconfiguration (`httpUrl` with `autoStart: true`), but that footgun still exists on `main`, so there is not strong evidence for a conservative close.

## Evidence

- **Issue report points at external-daemon + autostart combination:** The issue body configures `channels.signal.httpUrl` to an already-running daemon and also sets `autoStart: true`, with logs showing `Signal SSE stream error: TypeError: terminated` followed by `signal daemon exited (source=process code=null signal=SIGKILL)`. That makes the report plausible and not obviously obsolete.
- **Current main still uses the same raw SSE fetch path:** `streamSignalEvents` still connects to `/api/v1/events` with a plain `fetch(..., { headers: { Accept: "text/event-stream" }, signal })` and then reads the response body stream directly. I compared the local file to `v2026.4.5` via GitHub raw; the relevant block is unchanged.

## Risks / Open Questions

- The report may be caused by unsupported or conflicting config (`httpUrl` plus `autoStart: true`), but current `main` does not guard against that combination, so closing would be premature.
- I did not run a live Signal reproduction in this read-only review, so I cannot rule out an upstream `signal-cli`/Java interaction; that uncertainty favors keeping the issue open.
