## Summary

Keep open. Current `main` still compares `boundNode` and `requestedNode` as raw strings before canonical node resolution, so a session bound to a node ID can still reject a name-based request with `exec node not allowed`, matching the reported failure mode.

## Evidence

- **Raw mismatch check still precedes node resolution:** `executeNodeHostCommand` throws when `boundNode !== requestedNode` at raw-string level, then only afterwards resolves the node query to a canonical node ID. That means an ID-vs-display-name mismatch can still fail before canonicalization.
  - file: [src/agents/bash-tools.exec-host-node.ts:68](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/agents/bash-tools.exec-host-node.ts#L68)
  - command: `nl -ba src/agents/bash-tools.exec-host-node.ts | sed -n '56,90p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Node resolution accepts display names as aliases:** Node matching explicitly resolves a query by exact node ID, remote IP, normalized display name, or long ID prefix. So the same node can legitimately be referred to by display name in one place and node ID in another.
  - file: [src/shared/node-match.ts:74](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/shared/node-match.ts#L74)
  - command: `nl -ba src/shared/node-match.ts | sed -n '60,120p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Session exec defaults preserve the raw node string:** Reply exec overrides carry `sessionEntry.execNode` through unchanged, and the current-session prompt hint renders that raw value back to the model as `node=...`. A stale display-name value can therefore persist across turns in a long-lived session.
  - file: [src/auto-reply/reply/get-reply-exec-overrides.ts:24](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/auto-reply/reply/get-reply-exec-overrides.ts#L24)
  - command: `nl -ba src/auto-reply/reply/get-reply-exec-overrides.ts | sed -n '1,80p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Product contract still allows name-based node bindings:** The exec config type and docs still describe `tools.exec.node` / `exec.node` as accepting a node `id/name`, so name-based bindings are still part of the supported surface on current `main`.
  - file: [src/config/types.tools.ts:249](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/config/types.tools.ts#L249)
  - command: `nl -ba src/config/types.tools.ts | sed -n '240,254p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Issue timeline already points to an unmerged fix direction:** The provided GitHub timeline cross-references PR #66985 titled `fix(agents): resolve requestedNode to canonical ID before boundNode comparison`, which directly matches the bug. In this checkout, that change is not present on `main`.
