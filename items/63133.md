## Summary

Keep open. Current `main` still skips memory collection on the fast `status --json` path without preserving why, and the status formatter still renders `memory: null` as `enabled (...) · unavailable`. The PR remains relevant because its head adds an explicit `memoryCollection` state and regression coverage rather than being already implemented on `main`.

## Evidence

- **Fast `status --json` still skips memory probes without recording that skip on `main`:** `scanStatusJsonFast` returns `null` for `resolveMemory` whenever `--all` is not set, but the current implementation returns the raw scan result with no companion field to distinguish `skipped` from a real unavailable probe.
  - file: [src/commands/status.scan.fast-json.ts:57](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/status.scan.fast-json.ts#L57)

- **Current formatter still reports all null memory as unavailable:** `buildStatusMemoryValue` treats any enabled plugin with `memory === null` as `enabled (plugin …) · unavailable`; there is no branch for an intentional fast-path skip.
