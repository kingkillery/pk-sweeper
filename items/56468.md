## Summary

Current `main` does not contain the PR's core behavior. `src/config/io.ts` still dedups suspicious config reads only via the persisted `lastObservedSuspiciousSignature`, with no in-process fallback when health-state writes fail, and `src/infra/clawhub.ts` still lacks the wildcard `*`/`x`/`X` comparator handling from the PR. This PR remains substantive, and no allowed close reason is supported by the evidence.

## Evidence

- **Current config observe path has no in-process dedup:** On current `main`, both `observeConfigSnapshot` and `observeConfigSnapshotSync` return early only when `entry.lastObservedSuspiciousSignature === suspiciousSignature`, then immediately write a new `.clobbered.*` snapshot. There is no process-level `Set` or equivalent secondary guard in these paths.
  - file: [src/config/io.ts:633](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/io.ts#L633)

  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Health-state persistence is best-effort only:** `writeConfigHealthState` and `writeConfigHealthStateSync` swallow write errors. If that write fails, the next read reloads `{}` from disk and the current code has no other dedup state to prevent another clobbered backup.
  - file: [src/config/io.ts:372](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/io.ts#L372)
