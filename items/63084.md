## Summary

Keep open. Current `main` still does not emit `gateway:shutdown` or `gateway:pre-restart` internal hooks, and the PR discussion identified a real shutdown-hang risk because `triggerInternalHook()` is synchronous and unbounded in the shutdown path.

## Evidence

- **Current shutdown path has no internal hook dispatch:** `createGatewayCloseHandler()` normalizes `reason`/`restartExpectedMs`, stops services, and broadcasts the websocket `shutdown` event, but there is no import or call to `createInternalHookEvent()` / `triggerInternalHook()` anywhere in the close path on current `main`. That means the PR request is not already implemented.
  - file: [src/gateway/server-close.ts:105](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-close.ts#L105)
  - command: `nl -ba src/gateway/server-close.ts | sed -n '105,170p;209,260p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **User-facing internal-hook docs only expose gateway startup:** The automation hook docs list `gateway:startup` in the supported event table and do not list any shutdown or pre-restart gateway internal hook. This matches the code gap in `server-close.ts`.
  - file: [docs/automation/hooks.md:46](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/docs/automation/hooks.md#L46)

- **Existing startup hook dispatch is intentionally non-blocking:** Gateway startup defers internal-hook dispatch with `setTimeout(...); void triggerInternalHook(hookEvent)`, so current lifecycle-hook precedent avoids blocking startup on hook completion. The proposed shutdown change would add a different, synchronous behavior unless revised.
  - file: [src/gateway/server-startup-post-attach.ts:268](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/gateway/server-startup-post-attach.ts#L268)

- **Internal hook execution is awaited sequentially with no timeout:** `triggerInternalHook()` awaits each handler in order and has no timeout or cancellation guard. That makes the PR review warning credible: awaiting shutdown hooks directly in `server-close.ts` could block restart/exit indefinitely if a hook hangs.

That is useful context from the surrounding surfaces, but it is not the same internal-hook automation event this PR proposes.
