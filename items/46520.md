## Summary

Keep open. Current `main` still fails exact allowlist matching for custom URL schemes like `tauri://` and `electron://`, so the reported bug is still reproducible from the origin-check implementation itself.

## Evidence

- **Allowlist compares against `URL.origin`:** `parseOrigin()` normalizes `new URL(trimmed).origin`, and `checkBrowserOrigin()` only accepts an allowlist hit on `parsedOrigin.origin`. For opaque custom-scheme URLs, that value is not the original `tauri://...` string, so an exact configured allowlist entry cannot match.
  - file: [src/gateway/origin-check.ts:22](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/origin-check.ts#L22)
  - command: `nl -ba src/gateway/origin-check.ts | sed -n '1,120p'`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Read-only reproduction shows the mismatch on current runtime:** A direct Node reproduction against the same logic returns `parsed.origin: "null"` for `tauri://localhost`, and the resulting allowlist check with `['tauri://localhost']` returns `{ ok: false, reason: 'origin not allowed' }`. The same URL parsing behavior also occurs for `electron://app`.
  - command: `node -e "const parseOrigin=(originRaw)=>{const trimmed=(originRaw??'').trim(); if(!trimmed||trimmed==='null') return null; try { const url=new URL(trimmed); return {origin:(url.origin||'').trim().toLowerCase(),host:(url.host||'').trim().toLowerCase(),hostname:(url.hostname||'').trim().toLowerCase()}; } catch { return null; }}; const parsed=parseOrigin('tauri://localhost'); const allowlist=new Set(['tauri://localhost']); console.log(JSON.stringify({parsed,allowlistHas:allowlist.has(parsed.origin),result:(allowlist.has('*')||allowlist.has(parsed.origin))?{ok:true}:{ok:false,reason:'origin not allowed'}}));"`
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Gateway handshake still uses this helper for reported failure path:** The WebSocket connect handler still calls `checkBrowserOrigin()` with `gateway.controlUi.allowedOrigins`, and on failure emits the same `origin-mismatch` / `origin not allowed` path described in the issue.
  - file: [src/gateway/server/ws-connection/message-handler.ts:476](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/src/gateway/server/ws-connection/message-handler.ts#L476)
