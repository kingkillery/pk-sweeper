## Summary

Keep open. Current `main` still routes QMD probe failures through generic 'qmd binary could not be started' messaging, and the probe path does not pre-validate the configured workspace `cwd`, so the missing-workspace case this PR targets is not implemented on `main` yet.

## Evidence

- **Doctor still misclassifies workspace-cwd failures as qmd binary failures:** `noteMemorySearchHealth` always reports a failed QMD probe as a binary startup problem and suggests reinstalling/configuring `qmd`, even though it passes the agent workspace directory as `cwd` to the probe.
  - file: [src/commands/doctor-memory-search.ts:337](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/commands/doctor-memory-search.ts#L337)
  - command: `nl -ba src/commands/doctor-memory-search.ts | sed -n '300,360p'`
  - sha: [245451b6a9c5](https://github.com/openclaw/openclaw/commit/245451b6a9c529464e177afa65013adca0d63eb5)
- **Probe path still lacks workspace cwd validation:** `checkQmdBinaryAvailability` resolves the spawn invocation and immediately calls `spawn(..., { cwd: params.cwd ?? process.cwd() })`; there is no preflight check for missing or invalid workspace directories on current `main`.
  - file: [src/memory-host-sdk/host/qmd-process.ts:36](https://github.com/openclaw/openclaw/blob/245451b6a9c529464e177afa65013adca0d63eb5/src/memory-host-sdk/host/qmd-process.ts#L36)
