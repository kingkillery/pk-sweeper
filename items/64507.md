## Summary

Keep open. Current `main` has partial session-route fixes for seeded delivery context, but it does not implement the requested per-agent `delivery.defaultTarget` / `delivery.defaultChannel` fallback. The shared message tool still only auto-fills a target from session/run context, and still errors when that context is absent.

## Evidence

- **No per-agent delivery config exists:** `AgentConfig` defines per-agent fields like model, skills, heartbeat, subagents, sandbox, params, tools, and runtime, but no `delivery`, `defaultTarget`, or `defaultChannel` block.
  - file: [src/config/types.agents.ts:70](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/config/types.agents.ts#L70)
  - command: `nl -ba src/config/types.agents.ts | sed -n '60,120p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Message tool still depends on tool context for implicit target:** `normalizeMessageActionInput` only infers a missing target from `toolContext.currentChannelId`. If that is missing, it throws `Action ${action} requires a target.` There is no fallback to agent config.
  - file: [src/infra/outbound/message-action-normalization.ts:36](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/infra/outbound/message-action-normalization.ts#L36)

That helps when a session already has a resolved route, but it does not add the requested per-agent default destination.

## Risks / Open Questions

- Recent delivery-context work narrows the bug surface for spawned/subagent sessions, so the issue should be framed as missing agent-level fallback config rather than a blanket failure across every cross-session path.
- `pnpm` and authenticated `gh` access were unavailable in this environment, so this review is based on current source, checked-in tests, and local history rather than fresh command/test execution against GitHub.
