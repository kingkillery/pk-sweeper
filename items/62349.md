## Summary

Keep open. Current `main` still caches subsystem child loggers indefinitely, so the reported midnight roll bug remains plausible, and the proposed PR patch has unresolved review feedback showing the invalidation logic is incomplete.

## Evidence

- **Current main still permanently caches subsystem file loggers:** `createSubsystemLogger()` stores a `fileLogger` in closure state and only initializes it once before reusing it for all later file writes; there is no refresh path when the root logger's resolved file path changes on a new day.
  - file: [src/logging/subsystem.ts:327](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/logging/subsystem.ts#L327)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '325,354p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Raw logging path has the same stale-cache behavior:** `raw()` also reuses the same cached `fileLogger` once created, so subsystem-only traffic can continue writing through an old child logger after a date roll.
  - file: [src/logging/subsystem.ts:414](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/logging/subsystem.ts#L414)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '414,420p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Root logger only rebuilds when `getLogger()` is called with changed settings:** The active log file path is derived from `defaultRollingPathForToday()`, and the root logger is rebuilt only inside `getLogger()` when `settingsChanged()` detects a new file path. Once a subsystem child logger is cached, later subsystem writes do not call back into `getLogger()` on current `main`.
  - file: [src/logging/logger.ts:127](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/logging/logger.ts#L127)

  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **Main code confirms no generation-based invalidation exists yet:** `loggingState` on current `main` has no `loggerGeneration` field or equivalent mechanism, so the PR's proposed fix is not already implemented elsewhere on `main`.
  - file: [src/logging/state.ts:1](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/logging/state.ts#L1)
  - command: `nl -ba src/logging/state.ts | sed -n '1,19p'`
  - sha: [51dd4f288f67](https://github.com/openclaw/openclaw/commit/51dd4f288f67ef4df4c7a121f078f508535df1fd)
- **PR has unresolved production-significant review feedback:** The review on PR commit `f8c3267f8a1c3ee14205b9f9fe11df351535557d` points out that `getFileLogger()` compares generations before calling `getChildLogger()`, while `loggerGeneration` only increments inside `getLogger()`. That means subsystem-only traffic after a date roll can still reuse the stale child logger unless some other path rebuilds the root first.
