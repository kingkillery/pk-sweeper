## Summary

Current `main` still does not support per-agent TTS overrides. The proposed PR addresses a real gap: agent config does not expose `tts`, validation does not accept `agents.list[].tts`, and the runtime TTS paths still resolve from global `messages.tts` only.

## Evidence

- **Agent config surface still lacks per-agent TTS:** `AgentConfig` on current `main` has fields for identity, groupChat, sandbox, tools, and runtime, but no `tts` property, so `agents.list[].tts` is not part of the typed config surface.
  - file: [src/config/types.agents.ts:70](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/config/types.agents.ts#L70)
  - command: `nl -ba src/config/types.agents.ts | sed -n '55,130p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Validation schema still rejects agent-level TTS entries:** `AgentEntrySchema` enumerates allowed per-agent keys and does not include `tts`, which matches the PR description that current `main` rejects `agents.list[].tts` during config validation.
  - file: [src/config/zod-schema.agent-runtime.ts:812](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/config/zod-schema.agent-runtime.ts#L812)

- **Core TTS resolution is still global-only:** `resolveConfiguredTtsMode` and `shouldAttemptTtsPayload` read only `cfg.messages?.tts` and take no `agentId`, so the runtime still resolves TTS behavior from the global config block only.
  - file: [src/tts/tts-config.ts:9](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/tts/tts-config.ts#L9)

  - file: [src/auto-reply/reply/dispatch-from-config.ts:1122](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/auto-reply/reply/dispatch-from-config.ts#L1122)
