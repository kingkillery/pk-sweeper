## Summary

Keep open. Current `main` still only intercepts paste events when the clipboard exposes `image/*` items, so a paste that arrives as text/plain `data:image/...;base64,...` will still fall through to the textarea and can match the reported bad UX. The send/draft restore concern is already centralized in the send pipeline, but that does not resolve the composer paste bug, and I did not find targeted regression coverage for this paste path.

## Evidence

- **Paste handler only handles clipboard image items:** `handlePaste` scans `e.clipboardData.items`, keeps only entries whose MIME type starts with `image/`, and returns early when none are found. In that early-return case it does not call `preventDefault()`, so text payloads are left to the textarea.
