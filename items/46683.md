## Summary

Keep open. Current `main` renders Slack interactive replies and enqueues a `Slack interaction` system event, but the click/select is still routed to a peer/channel session key only, not the originating Slack thread session. The handler carries `threadTs` in the payload, yet does not use it for `sessionKey` resolution or `deliveryContext`, and the tests explicitly assert that non-threaded routing behavior. The same logic is present in release `v2026.4.22`, so this is not implemented on main or in the latest release.

## Evidence

- **Block-action enqueue path ignores thread id for routing:** `enqueueSlackBlockActionEvent` builds an event payload that includes `threadTs`, then resolves the destination with `resolveSlackSystemEventSessionKey({ channelId, channelType, senderId })` and enqueues only that `sessionKey`; it does not pass any thread id or `deliveryContext` when queueing the system event.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:630](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/slack/src/monitor/events/interactions.block-actions.ts#L630)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Slack system-event session resolution is peer-scoped, not thread-scoped:** `resolveSlackSystemEventSessionKey` accepts only `channelId`, `channelType`, and `senderId`, then routes through `resolveAgentRoute(... peer ...)` or falls back to `resolveSessionKey(...)`. There is no thread/session suffix handling here, unlike normal inbound Slack thread routing.
  - file: [extensions/slack/src/monitor/context.ts:178](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/slack/src/monitor/context.ts#L178)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Tests codify the missing thread-session routing:** The interaction test asserts the payload contains `threadTs: "100.100"`, but also asserts `resolveSessionKey` was called with only `{ channelId: "C1", channelType: "channel", senderId: "U123" }`. That matches the current implementation and shows thread context is not used to pick the target session.
  - file: [extensions/slack/src/monitor/events/interactions.test.ts:338](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/slack/src/monitor/events/interactions.test.ts#L338)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Normal inbound Slack thread turns do use thread session keys:** Slack inbound routing separately resolves `canonicalThreadId` and builds `:thread:<threadTs>` session keys for true thread replies, so the block-action path is inconsistent with the regular message path the issue references.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:106](https://github.com/openclaw/openclaw/blob/56fe2aab9c87596515f2280c8060dc740ea64a88/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L106)
  - sha: [56fe2aab9c87](https://github.com/openclaw/openclaw/commit/56fe2aab9c87596515f2280c8060dc740ea64a88)
- **Latest release still contains the same routing behavior:** `git show v2026.4.22:extensions/slack/src/monitor/events/interactions.block-actions.ts | nl -ba | sed -n '628,640p'` shows the release tag still enqueues the interaction event using only the peer-scoped `sessionKey`, and `git show v2026.4.22:extensions/slack/src/monitor/events/interactions.test.ts | nl -ba | sed -n '330,342p'` shows the same test expectation.
  - command: `git show v2026.4.22:extensions/slack/src/monitor/events/interactions.block-actions.ts | nl -ba | sed -n '628,640p'; git show v2026.4.22:extensions/slack/src/monitor/events/interactions.test.ts | nl -ba | sed -n '330,342p'`
