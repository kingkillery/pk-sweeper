## Summary

Keep open. Current `main` still blocks the first manual ACP turn behind the global ACP dispatch gate, so `sessions_spawn(runtime="acp")` remains coupled to `acp.dispatch.enabled` despite docs describing that flag as only pausing dispatch while keeping ACP controls available.

## Evidence

- **ACP session turns still enforce dispatch policy:** When an existing ACP session receives a turn, the ACP path in `agentCommandInternal` loads `resolveAcpDispatchPolicyError(cfg)` and throws before `acpManager.runTurn(...)`, so any ACP session run is blocked when `acp.dispatch.enabled=false`.
  - file: [src/agents/agent-command.ts:458](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/agent-command.ts#L458)
  - command: `nl -ba src/agents/agent-command.ts | sed -n '448,478p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Manual ACP spawn only checks `acp.enabled` up front:** `spawnAcpDirect` rejects `acp.enabled=false` but does not reject `acp.dispatch.enabled=false` before creating the ACP session, which matches the issue report that the failure happens on the immediate first turn rather than at spawn validation.
  - file: [src/agents/acp-spawn.ts:1024](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/acp-spawn.ts#L1024)
  - command: `nl -ba src/agents/acp-spawn.ts | sed -n '1024,1062p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **ACP spawn immediately sends the first task through gateway `agent`:** After creating the ACP session, `spawnAcpDirect` calls gateway method `agent` with the new ACP `sessionKey` and the requested task, which routes into the ACP turn path above and hits the dispatch gate.
  - file: [src/agents/acp-spawn.ts:1264](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/acp-spawn.ts#L1264)
  - command: `nl -ba src/agents/acp-spawn.ts | sed -n '1258,1298p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Docs still describe dispatch=false as preserving ACP controls:** The ACP setup docs say `acp.dispatch.enabled=false` should 'pause ACP dispatch while keeping /acp controls', which is broader than the current implementation because manual ACP session turns still fail through the shared dispatch gate.
  - file: [docs/tools/acp-agents-setup.md:46](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/tools/acp-agents-setup.md#L46)
  - command: `nl -ba docs/tools/acp-agents-setup.md | sed -n '42,48p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)

## Risks / Open Questions

- The current behavior contradicts the published ACP docs and likely surprises users who disable auto-dispatch but still expect explicit ACP spawns to work.
- Because ACP spawns submit the initial task immediately, users cannot currently use `sessions_spawn(runtime="acp")` as an on-demand/manual escape hatch while auto-dispatch is disabled.
