## Summary

Keep #71416 open. Current main still automatically falls back from the Gateway CLI path to embedded execution after any gateway request failure, with the fallback diagnostic emitted via stderr. The existing tests and docs codify that behavior rather than adding an opt-in flag, nonzero exit, stdout banner, or transport field that distinguishes CLI embedded fallback from a successful Gateway run.

## Best Possible Solution

Keep the issue open for a focused CLI behavior change. The best fix is to make fallback explicit and script-detectable: either fail closed unless an `--allow-embedded-fallback` flag is passed, or add a stable JSON field such as `meta.transport`/`meta.fallback` plus a clear non-JSON warning path, then update `src/commands/agent-via-gateway.ts`, CLI option registration, tests, and docs together.
