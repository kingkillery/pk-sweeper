## Summary

Keep #67709 open. Current main contains relevant startup mitigations, but the exact reported symptom remains plausible and not proven fixed: `models.list` still awaits the gateway model catalog directly, and `chat.history` can also await that same catalog to compute the thinking default, so both RPCs can still share one slow first catalog load and report nearly identical long durations. The related title-search items concern transcript visibility/session behavior rather than this startup latency report, so they are not canonical duplicates.

## Best Possible Solution

Keep the issue open and investigate the first `loadGatewayModelCatalog()` path used by Control UI startup. Best maintainer path: add bounded timing around `models.list`, `chat.history`, `ensureOpenClawModelsJson`, and provider catalog discovery; avoid making `chat.history` wait on model catalog for a thinking default; add a timeout or stale cached fallback for `models.list`; then verify on Windows or a throttled local profile with `OPENCLAW_DEBUG_INGRESS_TIMING=1`.
