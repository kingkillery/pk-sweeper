## Summary

Keep open. Current `main` does not support the PR's stated reproduction as written: the selected subscription id is chosen from the same in-memory `subs` array that is later searched, so deleting a subscription after listing would not make `subs.find(...)` return `undefined`. But the patch is still a reasonable defensive hardening because wizard session answers are accepted as unvalidated `unknown` values in the generic session path, so there is not strong enough evidence to close this PR as `cannot_reproduce` or `incoherent`.

## Evidence

- **Reported edge case is not shown by current auth flow:** `entraIdAuthMethod.run` builds the select options directly from `subs`, then immediately searches that same local array by the returned id. No second subscription fetch happens between selection and lookup, so an external deletion after listing would not change this in-memory lookup.
