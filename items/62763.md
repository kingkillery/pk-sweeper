## Summary

Keep open. Current `main` still contains the doctor migration that invents `channels.signal.accounts.default` from top-level multi-account Signal settings, and the Signal runtime still treats that inserted `default` entry as a real account candidate rather than shared defaults. The reporter's related note about #62764 does not change this item's status; this Signal corruption path still appears unfixed on `c997a9f9788e0a96e8eab05e7ae94959e76cdb4a`.

## Evidence

- **Doctor still inserts a phantom `accounts.default` block:** `seedMissingDefaultAccountsFromSingleAccountBase()` iterates any channel with named `accounts`, deletes movable top-level keys, and always writes them into `accounts.default`, with the exact change message quoted in the report.
  - file: [src/commands/doctor/shared/legacy-config-core-normalizers.ts:120](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/commands/doctor/shared/legacy-config-core-normalizers.ts#L120)
  - command: `nl -ba src/commands/doctor/shared/legacy-config-core-normalizers.ts | sed -n '120,180p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **The generic migration still targets Signal-style shared defaults:** The common movable-key set still includes `account`, `cliPath`, `httpUrl`, `httpHost`, `httpPort`, `dmPolicy`, `allowFrom`, `groupPolicy`, and `groupAllowFrom`. When named accounts exist and a channel does not provide `namedAccountPromotionKeys`, all of those keys are moved.
  - file: [src/channels/plugins/setup-promotion-helpers.ts:11](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/channels/plugins/setup-promotion-helpers.ts#L11)
  - command: `nl -ba src/channels/plugins/setup-promotion-helpers.ts | sed -n '1,45p;120,145p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Signal still treats every account-map key as a real account id:** `createAccountListHelpers()` uses raw `Object.keys(accounts)` as configured account ids, and `resolveMergedAccountConfig()` only merges top-level channel config with the requested account entry. After doctor removes the top-level shared keys, named Signal accounts stop inheriting them while a new `default` account id appears.
  - file: [src/channels/plugins/account-helpers.ts:39](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/channels/plugins/account-helpers.ts#L39)
  - command: `nl -ba src/channels/plugins/account-helpers.ts | sed -n '14,70p;120,175p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Signal runtime can treat the synthetic `default` entry as configured and start it:** `resolveSignalAccount()` marks an account configured when `cliPath`, `httpUrl`, `httpHost`, or `httpPort` is present even if `account` is missing, and gateway startup starts every id returned by `plugin.config.listAccountIds(cfg)`. `monitorSignalProvider()` then spawns a daemon using that account record, with `account` potentially undefined.
  - file: [extensions/signal/src/accounts.ts:47](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/signal/src/accounts.ts#L47)

- **Current docs still describe Signal multi-account as shared-base config:** The public channel config docs still say base channel settings apply to all accounts unless overridden, and the Signal docs point multi-account users at that shared pattern. That is inconsistent with doctor stripping shared Signal keys off the channel root into a synthetic `accounts.default` entry.
  - file: [docs/gateway/config-channels.md:736](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/docs/gateway/config-channels.md#L736)
