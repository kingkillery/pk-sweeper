## Summary

Keep open. Current `main` contains mitigations for repeated `LiveSessionModelSwitchError` retry loops, but it still intentionally normalizes that error into an `overloaded` failover reason, and I did not find strong evidence that the reported local-vLLM timeout/fallback cascade is fully resolved on `main`.

## Evidence

- **Live-session switch is still classified as overloaded:** `runWithModelFallback` catches `LiveSessionModelSwitchError` and wraps it as `new FailoverError(..., { reason: "overloaded" })`, then records the failed candidate and continues fallback.
  - file: [src/agents/model-fallback.ts:921](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/model-fallback.ts#L921)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '880,960p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Docs describe that normalization as current behavior:** The model-failover docs explicitly say `LiveSessionModelSwitchError` is normalized into the failover path so a stale persisted model does not create an outer retry loop.
  - file: [docs/concepts/model-failover.md:258](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/concepts/model-failover.md#L258)
