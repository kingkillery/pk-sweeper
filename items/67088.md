## Summary

Keep open. Current `main` still short-circuits browser launch on macOS when `SSH_CLIENT`/`SSH_TTY`/`SSH_CONNECTION` are present and no `DISPLAY`/`WAYLAND_DISPLAY` is set, so the reported `openclaw dashboard` behavior remains plausible and is not implemented on `main`.

## Evidence

- **Current browser-open logic still blocks darwin SSH sessions:** `resolveBrowserOpenCommand()` returns `reason: "ssh-no-display"` for any SSH session without display variables on every non-Windows platform. That condition runs before the later macOS `open` handling, so darwin still gets classified as headless.
  - file: [src/infra/browser-open.ts:42](https://github.com/openclaw/openclaw/blob/7875092f4ddc6f2dbe218bea8506c9ac1da87c39/src/infra/browser-open.ts#L42)
  - command: `nl -ba src/infra/browser-open.ts | sed -n '35,80p'`
  - sha: [7875092f4ddc](https://github.com/openclaw/openclaw/commit/7875092f4ddc6f2dbe218bea8506c9ac1da87c39)
- **Dashboard command still turns browser-open failure into the reported SSH hint:** `dashboardCommand()` calls `detectBrowserOpenSupport()`, only opens the browser when `ok`, and otherwise prints `formatControlUiSshHint(...)`. That matches the issue's reported `No GUI detected. Open from your computer:` output.
