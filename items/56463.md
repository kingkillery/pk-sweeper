## Summary

Current main already contains a shipped ACPX queue-owner/session-recovery fix that matches this report’s failure mode, and the report was filed against the older 2026.3.23-2 build.

## Evidence

- **Changelog entry for shipped fix:** Release 2026.4.1 explicitly says ACPX/runtime repaired `queue owner unavailable` session recovery by replacing dead named sessions and retrying without stale resume ids.
  - file: [CHANGELOG.md:1666](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/CHANGELOG.md#L1666)
  - command: `nl -ba CHANGELOG.md | sed -n '1635,1668p'`
- **Fresh-session repair on persistent ensure:** Current ACP manager prepares a fresh persistent session before ensure when prior metadata lacks a stable session id, preventing reuse of a dead named session handle.
  - file: [src/acp/control-plane/manager.core.ts:1423](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/acp/control-plane/manager.core.ts#L1423)
  - command: `nl -ba src/acp/control-plane/manager.core.ts | sed -n '1418,1482p'`
- **Retry without stale resume id:** If resuming a persisted ACP session id fails, current main strips the stale ids and retries `ensureSession()` without `resumeSessionId`.
  - file: [src/acp/control-plane/manager.core.ts:1448](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/acp/control-plane/manager.core.ts#L1448)

## Close Comment

Closing from a Codex review because this looks fixed on current `main` and was already shipped after the version in the report. Your repro was on `2026.3.23-2`, while `CHANGELOG.md:1666-1667` shows the ACPX `queue owner unavailable` recovery fix shipped in `v2026.4.1`. Current code now prepares a fresh persistent ACP session before reuse when only stale metadata exists and retries session init without a stale persisted resume id (`src/acp/control-plane/manager.core.ts:1423-1477`), with regression coverage in `src/acp/control-plane/manager.test.ts:1194-1209` and `src/acp/control-plane/manager.test.ts:1541-1610`. The old `queueOwnerTtlSeconds` setting from the report is also now explicitly treated as a legacy ignored compatibility field (`extensions/acpx/openclaw.plugin.json:129-131`). If this still reproduces on `v2026.4.1` or newer, please open a fresh issue with updated gateway logs and exact `acpx` / `cursor-agent` versions.
