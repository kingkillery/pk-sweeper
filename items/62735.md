## Summary

Keep open. Current `main` still ties Ollama runtime ownership to the canonical provider id `ollama`, so an arbitrary custom provider entry with `api: "ollama"` is not shown as implemented end-to-end. The OpenAI-compatible `num_ctx` path is also still gated by `isOllamaCompatProvider()` heuristics rather than treating `injectNumCtxForOpenAICompat: true` as an explicit override.

## Evidence

- **Runtime stream registration keys off the configured provider id:** `registerProviderStreamForModel()` resolves a provider stream only through `resolveProviderStreamFn({ provider: params.model.provider, ... })`. If the model uses a custom provider id, the runtime must first map that id to the Ollama plugin or no stream is registered for `api: "ollama"`.

  - command: `nl -ba src/agents/provider-stream.ts | sed -n '1,80p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Provider runtime ownership still exact-matches provider refs:** `resolveProviderRuntimePlugin()` scopes plugin loading to `providerRefs: [params.provider]` and then `matchesProviderId(...)`. That means the plugin first has to be owned by the exact provider ref being used.
  - file: [src/plugins/provider-hook-runtime.ts:149](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/plugins/provider-hook-runtime.ts#L149)

There is no manifest alias or `activation.onProviders` entry here for arbitrary custom ids, so `ollama2`/proxy-specific ids are not owned by the plugin through the normal provider-owner path.
  - file: [extensions/ollama/openclaw.plugin.json:4](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/ollama/openclaw.plugin.json#L4)
