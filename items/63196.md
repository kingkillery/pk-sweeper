## Summary

Keep open. Current `main` includes the earlier cron-maintenance fix for stale active rows, but I did not find any code path that re-validates an already-terminal `lost` cron task against cron run history or lets `tasks maintenance --apply` clear that state before normal retention pruning. The reported disagreement between `cron runs` showing `finished/ok` and `tasks audit` still surfacing `lost` therefore still looks plausible on `main`.

## Evidence

- **Maintenance only reconciles active tasks:** `shouldMarkLost()` returns false for any non-active task, and the maintenance sweep only has three actions: mark active tasks lost, stamp `cleanupAfter` on terminal tasks, or prune expired terminal tasks. There is no terminal-state repair path for a fresh `lost` row.
