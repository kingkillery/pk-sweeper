## Summary

Current main already avoids the repeated plugin-registration path this PR targets, so the proposed `registerTool()` dedupe guard is obsolete on main even though that exact code was never merged.

## Evidence

- **Gateway runtime schema no longer loads full plugin runtime:** `loadGatewayRuntimeConfigSchema()` reads manifest metadata via `loadPluginManifestRegistry(...)` and builds schema from that, instead of activating the full plugin registry. This removes the repeated `registerFull(...)` path that previously re-ran Feishu registration on schema reads.
  - file: [src/config/runtime-schema.ts:11](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/runtime-schema.ts#L11)
  - command: `nl -ba src/config/runtime-schema.ts | sed -n '1,40p'`
- **Regression guard explicitly covers Feishu re-registration:** Current test documents the original bug: repeated `config.schema` / `config.get` calls used to activate a fresh full plugin registry and re-run `registerFull` for channel plugins including Feishu. The regression test now asserts repeated schema loads do not replace the active registry or bump its version.
  - file: [src/config/runtime-schema.test.ts:227](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/runtime-schema.test.ts#L227)
  - command: `nl -ba src/config/runtime-schema.test.ts | sed -n '224,242p'`
- **Config hot reload does not reload plugin registrations:** The hot-reload handler updates hooks, cron, health monitor, Gmail watcher, and channel restarts, but it does not call plugin loader or plugin registration APIs. That contradicts the PR’s stated current-main root cause that config hot reload re-runs `registerFull(...)`.
  - file: [src/gateway/server-reload-handlers.ts:101](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-reload-handlers.ts#L101)
  - command: `nl -ba src/gateway/server-reload-handlers.ts | sed -n '100,195p'`
- **Plugin reload path clears prior plugin state before a fresh registry load:** When plugins are loaded, current main clears previously registered plugin state, then creates a new plugin registry with `createPluginRegistry(...)`. This means reloads replace registration state rather than accumulate entries into an existing registry.
  - file: [src/plugins/loader.ts:1890](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/plugins/loader.ts#L1890)
  - command: `nl -ba src/plugins/loader.ts | sed -n '1850,1915p'; nl -ba src/plugins/loader.ts | sed -n '1990,2010p'`
- **Active registry is swapped, not appended to:** `setActivePluginRegistry(...)` assigns `state.activeRegistry = registry` and advances the version. Current main replaces the active registry on reload instead of mutating an older active registry in place.
  - file: [src/plugins/runtime.ts:76](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/plugins/runtime.ts#L76)
  - command: `nl -ba src/plugins/runtime.ts | sed -n '76,90p'`
- **Exact PR patch is not present on main:** `registerTool()` still unconditionally pushes into `registry.tools`, so the proposed dedupe guard itself was not merged. The reason this PR is obsolete is that main now avoids the duplicate-registration path at a higher layer.
  - file: [src/plugins/registry.ts:334](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/plugins/registry.ts#L334)
  - command: `nl -ba src/plugins/registry.ts | sed -n '334,356p'`
