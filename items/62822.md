## Summary

Keep open. Current `main` still audits Discord channels serially and recomputes bot/guild/member permission context on every channel fetch, so the PR addresses a real unmerged performance gap rather than a stale or already-shipped change.

## Risks / Open Questions

- `channels status --probe` still does serialized Discord permission audits, so larger channel sets can still accumulate latency against the CLI's default 10s timeout.
- For channels in the same guild, current `main` still repeats the same `@me`, guild, and guild-member REST lookups per channel instead of reusing them, so the slow path scales poorly.
- Closing this PR would discard a still-relevant optimization and its regression tests without evidence that equivalent behavior already shipped another way.
