## Summary

Current `main` is only partially aligned with the issue. `hooks.token` is now intentionally documented/validated as unsupported, and Matrix/MiniMax SecretRef handling appears covered in code/tests, but the local `infer web search` path still bypasses command SecretRef resolution and the `voice-call` plugin schema still models the reported auth/TTS keys as plaintext-only strings. The issue still tracks real remaining gaps, so it should stay open.

## Evidence

- **Local `infer web search` still skips command SecretRef resolution:** `runWebSearchCommand` reads `loadConfig()` and passes that raw config directly into `runWebSearch(...)`; unlike other command paths, it does not call `resolveCommandConfigWithSecrets`/`resolveCommandSecretRefsViaGateway`, which matches the Brave local-CLI repro in the issue discussion.
  - file: [src/cli/capability-cli.ts:1114](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/cli/capability-cli.ts#L1114)

- **SecretRef resolution support exists for web-search commands, just not on that CLI path:** The command-secret gateway tests explicitly cover local fallback resolution for plugin web-search SecretRefs when the gateway is unavailable, showing the platform has a supported resolution path that `web.search` is currently bypassing.
