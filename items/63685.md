## Summary

Keep open. Current main documents and tests Gemma 4 support only for the full Google API ids like `gemma-4-26b-a4b-it`, but it still does not normalize the reporter's failing shorthand `gemma-4-26b`. The request path is built directly from the configured model id, so the reported 404 remains plausible on current main. The issue timeline also links an open PR (#68996), which is additional evidence that this is not clearly resolved on `main`.

## Evidence

- **Docs use the full Gemma 4 id:** The Google provider docs say Gemma 4 models are supported and give `gemma-4-26b-a4b-it` as the example id, matching the helpful issue comment that corrected the model name.
  - file: [docs/providers/google.md:149](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/providers/google.md#L149)

- **Tests cover only the full API id:** The current Google provider tests verify forward-compat handling for `gemma-4-26b-a4b-it`; there is no corresponding coverage for the shorthand `gemma-4-26b` from the bug report.
  - file: [extensions/google/provider-models.test.ts:355](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/google/provider-models.test.ts#L355)

- **Model normalizer does not repair the reporter's shorthand:** `normalizeGoogleModelId` only rewrites a few Gemini aliases and otherwise returns the incoming id unchanged, so `gemma-4-26b` would stay `gemma-4-26b`.
  - file: [extensions/google/model-id.ts:3](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/google/model-id.ts#L3)

  - file: [extensions/google/transport-stream.ts:183](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/google/transport-stream.ts#L183)
