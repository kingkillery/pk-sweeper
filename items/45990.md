## Summary

Keep open. Current `main` still emits `llm_input` before `activeSession.prompt(...)` and emits `llm_output` once after the attempt completes, so the PR is not already implemented. The review thread also surfaces real unresolved API/regression risks in the proposed approach: moving hooks to per-round stream events would change `llm_output.usage` and `assistantTexts` semantics and can skip `llm_input` entirely for rounds that fail before the first `message_start`.

Those concerns are consistent with current main behavior and were not resolved in the provided discussion.

## Risks / Open Questions

- Per-round `llm_input`/`llm_output` observability for embedded agent loops is still not available on current `main`.
- Merging the PR as described would likely break existing hook consumers that treat `usage` and `assistantTexts` as run-level data.
- Moving `llm_input` emission to stream start would reduce failure observability for auth/rate-limit/network errors that happen before the first streamed assistant event.
