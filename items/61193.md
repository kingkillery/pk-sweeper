## Summary

Keep open. Current `main` still skips any allowlist entry whose pattern lacks a path separator, so a bare glob like `python3 *` would not match. The latest shipped release `v2026.4.23` still has the same logic.

## Evidence

- **Current matcher still ignores bare patterns:** `matchAllowlist` trims each pattern, computes `hasPath`, and immediately `continue`s when the pattern has no `/`, `\\`, or `~`. That means `python3 *` never reaches glob matching against the command.
  - file: [src/infra/exec-command-resolution.ts:351](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/infra/exec-command-resolution.ts#L351)
  - command: `nl -ba src/infra/exec-command-resolution.ts | sed -n '318,360p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Tests only cover path-based allowlist patterns:** The current node-host allowlist tests exercise exact and wildcard path patterns such as `/usr/bin/python3` and `/opt/**/python*`, but there is no coverage for a bare command pattern like `python3 *`.
  - file: [src/infra/exec-approvals-config.test.ts:61](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/infra/exec-approvals-config.test.ts#L61)

- **Docs still describe path-based semantics:** The public docs/examples still frame allowlist entries as resolved binary paths and show only path examples, which matches the current implementation rather than the promised bare-command behavior from the issue thread.
  - file: [docs/platforms/macos.md:105](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/docs/platforms/macos.md#L105)
