## Summary

Keep open. The discussion clarified an important distinction: `messages.queue.collect` is followup batching only after a run is active, while idle rapid-message batching should come from `messages.inbound.debounceMs`. Current main has Feishu inbound debounce plumbing, but the reporter's Feishu DM repro remains plausible: the issue comments ruled out missing `chat_id`, sender id, and `message_type`, while current code still has Feishu-specific key/text/control bypasses and no direct regression test for the exact idle DM burst case.

## Best Possible Solution

Keep the issue open and add a focused Feishu regression test around `createFeishuMessageReceiveHandler` that sends three same-sender direct text events inside `messages.inbound.debounceMs` while the agent is idle and asserts one merged dispatch. If that passes, add diagnostic/docs clarification for `messages.queue.collect` versus `messages.inbound`; if it fails, fix the Feishu debounce key/text/config path before the per-chat sequential dispatch queue.
