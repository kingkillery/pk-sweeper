## Summary

Keep open. Current `main` still hard-fails config validation when an `auth.profiles.*` entry lacks `provider`, and the auth-profile resolution path still expects `profile.provider` instead of inferring it from the profile id. I found no shipped fix or test coverage for the reported orphan-profile case.

## Evidence

- **Config schema still requires auth profile provider:** `OpenClawSchema` defines `auth.profiles` entries with `provider: z.string()` as a required field, so a leftover profile like `qwen-portal:default` without an explicit provider remains schema-invalid.
  - file: [src/config/zod-schema.ts:431](https://github.com/openclaw/openclaw/blob/a7c1376b207e337cd7c3058b3a163fac1235fd7d/src/config/zod-schema.ts#L431)

  - sha: [a7c1376b207e](https://github.com/openclaw/openclaw/commit/a7c1376b207e337cd7c3058b3a163fac1235fd7d)
- **Validation path still treats schema failure as fatal:** `validateConfigObjectRaw()` directly runs `OpenClawSchema.safeParse(raw)` and returns schema issues immediately on failure, which matches the reported CLI short-circuit behavior instead of downgrading orphan auth profiles to warnings.
  - file: [src/config/validation.ts:599](https://github.com/openclaw/openclaw/blob/a7c1376b207e337cd7c3058b3a163fac1235fd7d/src/config/validation.ts#L599)
