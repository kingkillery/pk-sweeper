## Summary

Current main already fixes the central data-loss bug this PR targeted. `syncMemoryWikiBridgeSources` now checks active memory capability registration before pruning bridge-imported pages, so the empty-artifact result caused by an unavailable/unloaded memory capability no longer turns into a destructive prune with an empty `activeKeys` set.

## Best Possible Solution

Close this PR as implemented on current main. Keep the mainline capability-registration guard as the product behavior; any desire for the PR's narrower source-file-existence pruning heuristic or extra regression coverage should be handled as a separate follow-up, not by keeping this stale PR open.

## Evidence

- **Current main skips bridge prune when memory capability is absent:** `syncMemoryWikiBridgeSources` imports `getMemoryCapabilityRegistration()`, reads the current capability, and sets `removedCount` to `0` instead of calling `pruneImportedSourceEntries` when memory-core is not loaded, explicitly to avoid removing bridge-imported entries in CLI/unloaded-capability contexts.
