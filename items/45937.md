## Summary

Keep open. Current `main` does not implement the requested MiniMax/CN-provider handling, and this PR is not a conservative cleanup close because the proposed branch introduces false-positive failover classification and loses the concrete failover reason for the new text-response path.

## Evidence

- **Current main still lacks the proposed Chinese rate-limit patterns:** `ERROR_PATTERNS.rateLimit` on current `main` contains only the existing generic/English tokens; there is no MiniMax/CN-specific Chinese matcher such as `速率限制`, `请求量较高`, or code `2062`. That means the reported gap is not already implemented on `main`.

## Risks / Open Questions

- The underlying MiniMax/CN-provider failover gap still appears plausible on current `main`, because Chinese rate-limit text is not matched and non-error assistant text is still ignored by `isFailoverAssistantError()`.
- A safe fix likely needs provider-scoped matching plus targeted tests, and if text-only assistant responses are supported, the matched failover reason must be propagated instead of synthesized later as `unknown`.
- The PR review feedback about `"请稍后重试"` being overly broad is consistent with current main's conservative classifier/test design, so auto-closing this PR would discard a plausible bug report without a shipped fix.
