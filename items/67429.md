## Summary

Keep PR #67429 open. Current main still does not implement the requested null-clear behavior for cron payload model, fallbacks, or thinking overrides, and I found no superseding main implementation in the affected cron service/schema/normalization paths. The PR is a focused, plausible bug fix with useful author follow-up, but it needs revision for the unresolved review concern: preserving null in the shared payload normalizer can regress cron.add by letting null reach the non-null create schema.

## Best Possible Solution

Keep the PR open and ask for a small revision: make null-clear a patch-only contract in CronPayloadPatch types/schema, preserve null through normalizeCronJobPatch without changing normalizeCronJobCreate behavior, delete model/fallbacks/thinking in mergeCronPayload when patch values are null, coerce null to undefined only when building a replacement payload, and add regression tests for both cron.update null-clear and cron.add optional-null non-regression.
