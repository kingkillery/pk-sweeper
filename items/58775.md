## Summary

Keep open. The issue is recent, author is not protected, labels are not protected, and the discussion plus linked PR indicate a plausible provider regression that still needs maintainer resolution rather than cleanup closure. The linked PR #60860 is still open and explicitly claims to fix/close this issue by adding separate Google Vertex ADC/global endpoint handling, so the conservative decision is not to close #58775 from this review.

## Evidence

- **Current docs still define google-vertex as a distinct ADC-backed provider:** The current main documentation lists `google-vertex` separately from `google-gemini-cli` and says Vertex uses gcloud ADC, matching the reporter's expected transport/auth split.
  - file: [docs/concepts/model-providers.md:367](https://github.com/openclaw/openclaw/blob/44114328b4e73074ac72126f1230ddbf9c1ea8b5/docs/concepts/model-providers.md#L367)
  - sha: [44114328b4e7](https://github.com/openclaw/openclaw/commit/44114328b4e73074ac72126f1230ddbf9c1ea8b5)
- **Issue report is concrete and recent:** #58775 gives version 2026.3.28, affected provider/model refs, root-cause hypothesis, expected behavior, and auth/URL impact. It is less than 60 days old, so stale-insufficient-info does not apply.
- **Related PR remains open:** PR #60860 is cross-referenced from the issue and its summary says it registers a separate `google-vertex` provider using ADC and Vertex endpoint routing, with `Closes #58775`; because the PR is open, this is evidence of active unresolved work rather than evidence that main already fixed it.
- **Follow-up user comment supports configuration sensitivity, not invalidity:** A commenter reported `GOOGLE_APPLICATION_CREDENTIALS`, `GOOGLE_CLOUD_PROJECT`, and `GOOGLE_CLOUD_LOCATION=global` made `google-vertex/gemini-3.1-pro-preview` work for them, which is useful troubleshooting evidence but does not disprove the reported regression for the 2026.3.28 provider normalization path.
