## Summary

Keep #59165 open. It is a recent, coherent RFC for a core plugin-SDK/runtime extension, not a stale support report or a request that clearly belongs only in ClawHub. Current main documents many plugin registration surfaces, but not the proposed credential-provider capability, and the issue discussion includes a concrete AWS Secrets Manager plugin implementation offer that depends on the core interface landing first.

## Evidence

- **Issue is recent and actionable:** #59165 was opened 2026-04-01 and updated 2026-04-19, so it is well under the 60-day stale threshold. It describes a concrete CredentialProvider abstraction, CredentialRef shape, lifecycle operations, migration phases, and links related credential-management issues and a full RFC gist.
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **Discussion supports keeping the core interface design open:** The 2026-04-19 comment by wirjo says the plugin interface design is solid and offers to implement an AWS Secrets Manager provider inside the Amazon plugin once the core plugin interface lands. That is evidence of downstream plugin interest, not a reason to close as already handled.
  - sha: [2e0ae56b1a58](https://github.com/openclaw/openclaw/commit/2e0ae56b1a580103f71b9b7f79b3d638e3c3698d)
- **Vision allows plugin API extension discussions:** VISION.md says core should stay lean, optional capability should usually ship as plugins, and if a useful feature cannot be built as a plugin yet the project welcomes PRs and design discussions that extend the plugin API instead of adding one-off core behavior.
