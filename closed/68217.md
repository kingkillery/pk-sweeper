## Summary

Current `main` does not appear to mis-rewrite mixed-case or `vbscript:` absolute-scheme links in the localized docs rewriter. The proposed change hardens an early scheme guard, but the existing control flow already returns any non-`/` target unchanged, so the reported behavior does not reproduce on `main`.

## Evidence

- **Non-route targets already return unchanged:** `localizeURL` trims the value, skips anchors/protocol-relative URLs, then returns the original string unchanged whenever `pathPart` does not start with `/`. For inputs like `Javascript:alert(1)` or `vbscript:msgbox(1)`, `pathPart` is the full scheme string, so current `main` returns `raw` before any route localization.
  - file: [scripts/docs-i18n/localized_links.go:293](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/scripts/docs-i18n/localized_links.go#L293)
  - command: `nl -ba scripts/docs-i18n/localized_links.go | sed -n '200,340p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Case-sensitive scheme helper is not user-visible here:** `hasURLScheme` still checks lowercase prefixes only, but that helper is followed immediately by the `/`-prefix gate above. In this call site, failing the scheme check does not cause mixed-case absolute schemes to be localized; they still fall through to the unchanged return path.
