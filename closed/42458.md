## Summary

Closing because the reported self-deadlock path no longer matches current main: manual `cron.run` still queues on `CommandLane.Cron`, but isolated cron agent work is normalized onto the nested lane before the embedded agent run, and that behavior is covered by tests and already present in the latest release tag I could inspect.

## Evidence

- **Manual runs still queue on cron lane:** `enqueueRun()` still fire-and-forgets manual runs onto `CommandLane.Cron`, so the only way the reported deadlock could still happen is if inner isolated execution also tried to occupy that same lane.

- **Cron isolated jobs enter the isolated runner with lane="cron":** Gateway cron wiring still passes `lane: "cron"` into `runCronIsolatedAgentTurn()`, which matches the PR's premise up to this point.
  - file: [src/gateway/server-cron.ts:333](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/gateway/server-cron.ts#L333)
