## Summary

Keep open. Current `main` has a newer media-server rewrite that hardens active-content handling, but it does not implement the PR's requested response headers on `/media/:id`, so this is not `implemented_on_main`. The PR is also not a clean obsolete close because review feedback raised real concerns about `Cache-Control: no-store` and the current server design has changed materially since the PR was opened.

## Evidence

- **Current main still only sets `nosniff` on media responses:** `createMediaRequestHandler()` sets `X-Content-Type-Options: nosniff` before request validation, but there is no `X-Frame-Options`, `Referrer-Policy`, `Content-Security-Policy`, or `Cache-Control` header on the `/media/:id` handler.
  - file: [src/media/server.ts:137](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/media/server.ts#L137)
  - command: `nl -ba src/media/server.ts | sed -n '126,190p'`
  - sha: [84dc9f12f1b9](https://github.com/openclaw/openclaw/commit/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5)
- **Main now mitigates active-content execution differently:** Instead of adding CSP/XFO headers, current main detects active MIME types (`text/html`, `image/svg+xml`, XML, JS) and forces them to download as `application/octet-stream`, with `Content-Disposition: attachment`. That is a newer defense not present in the PR branch.
  - file: [src/media/server.ts:83](https://github.com/openclaw/openclaw/blob/84dc9f12f1b9e36f7bfc9bea438bce3852ed9dd5/src/media/server.ts#L83)

There are still no assertions for `X-Frame-Options`, `Referrer-Policy`, `Content-Security-Policy`, or `Cache-Control`.
